# Screen Spec(SCR-NNN) 작성 규칙

> D2 문서 규격 (org-design-v2.md §6). A12(Planning Reviewer)가 G1 체크리스트로 기계 검증하는 기준이므로,
> 이 규칙의 위반은 곧 반려다.

## 1. 파일 규칙

- 경로: `docs/plan/ui/SCR-NNN-<slug>.md` (예: `SCR-001-login.md`) — NNN은 3자리 일련번호 (기존 최대 번호 +1). 재사용·재배정 금지. slug는 화면명의 kebab-case
- 1 화면 = 1 파일. 탭/모달이 독립 URL 또는 독립 Flow 노드를 가지면 별도 SCR로 분리
- frontmatter 필수 필드:

```yaml
---
id: SCR-012
title: 회원 관리 — 목록
flow: FLOW-003#node-member-list   # docs/plan/ux/flows/의 노드 ID (1:1 매핑 근거)
status: draft                      # draft | in_review | approved
date: 2026-07-14
producer: A11
---
```

## 2. 본문 필수 섹션 (순서 고정)

1. **개요** — 화면의 목적 1~3문장, 대상 사용자 역할
2. **화면 구성** — 영역(헤더/필터바/테이블/사이드패널 등) 목록과 각 영역의 역할. 시각 값(px/색) 금지 — 그것은 A14의 Visual Spec 영역
3. **기능 정의** — 화면이 다루는 CRUD 기능(목록/검색/필터/등록/수정/삭제)별 하위 섹션. **각 기능마다 아래 4개 상태를 전부 정의**:

   | 상태 | 정의할 내용 |
   |---|---|
   | 빈 상태 (Empty) | 조건(데이터 0건/검색 결과 없음 구분), 표시 문구, 후속 행동(CTA) |
   | 에러 (Error) | 에러 유형별(네트워크/서버/유효성) 문구·재시도 수단 |
   | 로딩 (Loading) | 로딩 표현 방식 요구(스켈레톤/스피너/프로그레스 — 선택은 디자인 영역), 중복 요청 방지 규칙 |
   | 권한없음 (Unauthorized) | 진입 차단 vs 요소 비활성 구분, 표시 문구, 이동처 |

   해당 없는 기능·상태는 **"해당 없음 — 사유"** 로 명시적으로 기재 (침묵 = 누락 = 반려)
4. **컴포넌트 인벤토리** — 표 형식 필수:

   | 컴포넌트 | 분류 | 근거 |
   |---|---|---|
   | Table | 기존 | contracts/Table.contract.json@1.4.0 |
   | Select | **확장(EXTEND)** | reports/reuse/2026-07-14-select.json — 유사도 78%, multiple prop 추가 필요 |
   | StatusBadge | **신규(CREATE)** | reports/reuse/2026-07-14-statusbadge.json — 최고 유사도 41% |

   - `기존`: contract 경로 + 버전 명시
   - `확장`/`신규`: **A75 Reuse Guard 판정 리포트 경로 + 판정 + 유사도 필수 첨부**. 미첨부는 G1 blocker
   - 유사도 ≥ 85% 항목을 신규로 표기 금지 (자동 차단 대상)
5. **비즈니스 규칙** — 권한 매트릭스(역할 × 기능 O/X)와 유효성 규칙(필드 | 형식 | 범위 | 필수 | 에러 문구)
6. **데이터 요구** — 표시/입력 필드 목록, 정렬·페이징 기본값, 검색·필터 대상 필드

## 3. 모호 문구 금지 목록 (G1 체크리스트: 0건)

아래 단어·표현이 본문에 있으면 반려된다. 반드시 측정 가능·판정 가능한 문장으로 치환한다.

| 금지 문구 | 치환 예 |
|---|---|
| 적절히, 적당히 | "페이지당 20건, 최대 100건까지 선택 가능" |
| 직관적으로 | "필터 적용 즉시(별도 버튼 없이) 목록 갱신" |
| 자연스럽게 | "저장 성공 시 목록 화면으로 이동하고 상단에 토스트 표시" |
| 깔끔하게, 심플하게, 예쁘게 | (시각 표현은 SCR에서 금지 — 기능 요구로 재기술) |
| 유연하게 | "컬럼 표시/숨김을 사용자가 설정 가능, 설정은 계정별 저장" |
| 빠르게 | "검색 입력 후 300ms 디바운스로 요청" |
| 쉽게, 사용자 친화적으로 | 구체 행동 시나리오로 기술 |
| 필요 시, 가능하면, 상황에 따라 | 조건을 명시: "N건 초과 시", "권한 X인 경우" |
| 일반적인, 표준적인 | 참조 대상 명시: "docs/tds/patterns/list-page.md 패턴 준수" |
| 등, 등등, 기타 | 전수 나열 또는 범위 정의 |
| 알아서, 임의로 | 판정 주체와 규칙 명시 |

자가 검사(Grep 패턴 예):

```
적절히|적당히|직관적|자연스럽|깔끔|심플하게|예쁘게|유연하게|알아서|등등
```

## 4. 흔한 반려 사유 (A12 실측 기준)

- 검색과 필터를 한 문단에 뭉뚱그림 → 기능별 섹션 분리 + 각각 4개 상태 정의
- "권한 없는 사용자는 볼 수 없음" → 진입 차단인지 요소 숨김인지, 어떤 화면으로 보내는지 미정의
- 컴포넌트 인벤토리에 "공통 버튼 사용" → contract 경로·버전 미명시
- 신규 컴포넌트에 Reuse Guard 리포트 경로 대신 "검색해봤는데 없음" 산문 → 판정 리포트 필수
