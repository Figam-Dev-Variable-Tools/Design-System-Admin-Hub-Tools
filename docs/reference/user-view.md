# 고객 화면 축 (User View) — 명세 규약

이 문서는 **명세를 쓰는 방법**을 정한다. 화면 하나를 기술하지 않는다.
`quality-bar.md` 가 '품질 축을 화면에 대는 법' 을 정하듯, 이 문서는 **'어드민의 값이 고객 화면에서 무엇이 되는가' 를 적는 법**을 정한다.

기준일: 2026-07-22 · 대상: `specs/**` 의 모든 FS 문서

---

## 1. 왜 이 축이 필요한가

이 어드민은 **혼자 서 있지 않다.** 앞단에 고객이 보는 홈페이지가 붙고, 문의·주문·프로그램 신청이 거기서 들어온다.
그런데 지금까지의 FS 는 **어드민 쪽만** 기술했다. 그래서 다음 세 질문에 명세가 답하지 못한다:

1. 이 값을 끄면 **고객 화면에서 무엇이 바뀌는가**
2. 어드민이 `sent` 라 부르는 것을 **고객은 뭐라고 읽는가**
3. 이 값 때문에 고객 화면에서 **아예 사라지는 것**이 있는가

이 셋은 운영자가 매일 하는 판단이다. 명세가 답하지 않으면 그 판단은 **화면 밖에서** 이뤄지고, 앱은 자기가 무엇을 바꿨는지 모른다.

### 1.1 이 축이 '관찰' 이 아니라 '계약' 인 이유 — 앞단이 아직 없다

**고객 홈페이지는 이 리포에 존재하지 않는다.**

| 사실 | 근거 |
|---|---|
| 워크스페이스의 앱은 `apps/admin` **하나뿐**이다 | `apps/` 디렉터리 실측(2026-07-22) — `admin` 외 없음 |
| 홈페이지는 **바깥에 따로 있는 사이트**이고, **연동 UI 조차 어드민이 갖지 않는다** — 연동은 백엔드가 홈페이지 템플릿별로 뚫는다 | 코드 실측: `apps/admin/src/pages/settings/**` 에 연동 화면이 없고 `nav-config.ts` 의 시스템 설정 잎 6개(`site`·`api-keys`·`oauth`·`payment`·`plan`·`notifications`)에도 없다 |
| 백엔드도 아직 없다 — 결제 하나만 봐도 **프론트가 절대 못 하는 일**(결제창 세션 서명·승인·금액 재검증·웹훅 수신)이 서버 몫으로 남아 있다 | `apps/admin/src/shared/commerce/payment-settings.ts:290-304` (`TODO(backend)`) |

> **연동 화면이 없어도 연동 계약은 필요하다.** 어드민이 그 UI 를 갖지 않기로 한 것이지, 홈페이지와 어드민이
> 주고받을 것이 없다는 뜻이 아니다. 그 세 엔드포인트 계약은 [`site-connect-contract.md`](site-connect-contract.md) 가 갖는다.
> 그 문서는 이 문서와 **같은 층**이다 — 화면 하나가 아니라 여러 화면을 가로지르는 규약이고, `specs/_axes/` 에 놓인다.
> **방향만 반대다**: 이 문서는 '어드민의 값 → 고객 화면', 그 문서는 **'고객 홈페이지의 사건 → 어드민'** 을 정한다.
> 그 문서의 모든 행은 §2 의 `[계약]` 이다 — **대응하는 코드가 리포에 하나도 없다.**

그러므로 이 축에 적는 문장의 **대부분은 관찰이 아니다.** 없는 화면을 본 것처럼 쓰는 것이 이 리포에서 가장 무거운 금지이므로,
**모든 행은 아래 세 표기 중 하나를 반드시 단다.**

## 2. 세 표기 — `[확인]` · `[계약]` · `[미확정]`

| 표기 | 뜻 | 필수 부가정보 | 쓰면 안 되는 때 |
|---|---|---|---|
| `[확인]` | **이 리포의 코드가 실제로 그렇게 한다.** 어드민 코드가 만들어 내는 사실이거나, 코드가 고객용으로 만든 문자열이 실재한다 | `파일:줄` 또는 심볼명 | 고객 화면의 렌더 결과를 주장할 때. 우리는 그 화면을 갖고 있지 않다 |
| `[계약]` | **우리가 지금 정한다.** 앞단이 생길 때 이대로 만들어야 한다 | 누가·무엇을 근거로 정했는지 한 줄 | 코드에 이미 답이 있는데 게을러서 안 찾았을 때 |
| `[미확정]` | 답이 **아직 정해지지 않았다.** 다른 트랙의 결정을 기다린다 | 무엇을 기다리는가 + 어디서 결론이 나는가 | 추측으로 메울 수 있을 때. 메우지 말고 이 표기를 써라 |

**규칙 UV-1.** 근거 없는 `[확인]` 은 결함이다. 리뷰는 `[확인]` 행의 `파일:줄` 을 먼저 본다.
**규칙 UV-2.** `[계약]` 은 '바람' 이 아니다 — 앞단 구현자가 그대로 따를 수 있을 만큼 구체적이어야 한다('적절히 안내한다' 는 계약이 아니다).
**규칙 UV-3.** `[미확정]` 은 **개수를 센다.** 각 FS §8 의 마지막 소절이 그 목록을 갖고, 0건이면 0건이라고 적는다.

## 3. 어디에 적는가 — **FS 안의 표준 절 §8** 이다

새 문서 종류를 만들지 않는다. `README.md` §1 이 문서 종류를 **FS·BE·NFR 셋으로** 못박고 있고,
이 축은 '무엇이 보이고 무엇이 일어나는가' 이므로 성격상 **FS 의 것**이다. BE(엔드포인트 계약)도 NFR(품질 판정)도 아니다.

| 판단 | 내용 |
|---|---|
| **채택** | FS 문서 끝에 **§8 고객 화면 (User View)** 을 붙인다. §1–§7 의 번호·내용은 건드리지 않는다 |
| 기각 — 화면마다 `UV-nnn` 새 문서 | 문서 종류가 넷이 되고, 한 화면의 사실이 두 파일로 갈라진다. 갈라진 명세는 반드시 한쪽만 갱신된다(README §4.5 의 044 가 그 전례다) |
| 기각 — NFR 의 축으로 편입 | NFR 은 `quality-bar.md` 의 축을 대고 **pass/gap 을 판정**하는 문서다. User View 는 판정이 아니라 **서술**이다 |
| 기각 — BE 에 편입 | 고객 화면의 문구·존재 여부는 엔드포인트 계약이 아니다. 다만 **고객 행동이 어드민에 도착하는 지점**은 BE 가 받아야 하므로, §8.7 이 BE 로 이관 항목을 남긴다 |

이 문서(규약)와 `quality-bar.md`(품질 축)는 **같은 층**이다 — 둘 다 `docs/reference/` 에 놓이고, 화면 문서가 이것을 참조한다.

## 4. §8 의 표준 구성

FS 의 §8 은 아래 아홉 소절을 **이 순서로** 갖는다. 해당 없음이면 'N/A + 사유' 를 적는다(빈칸 금지 — FS §4 의 규율과 같다).

| 소절 | 이름 | 무엇을 적는가 |
|---|---|---|
| 8.0 | 전제 | 이 단계에 고객 접점이 **있는가 없는가**. 없으면 여기서 그 사실을 못박고 나머지는 전부 `[계약]` 이 된다 |
| 8.1 | 고객이 보는 자리 | 어떤 화면·어떤 경로에서 보는가. **볼 수 있기는 한가** |
| 8.2 | 값 대응표 | **어드민 값 → 고객 화면 결과.** 값 하나가 한 행 |
| 8.3 | 어휘 대조표 | 어드민 표기 / 고객 표기. 다르면 다르다고 적는다 |
| 8.4 | 사라지는 것 | 값에 따라 고객 화면에서 없어지는 요소. **숨김·비활성·없음을 구분한다** |
| 8.5 | 누출 금지 목록 | 고객에게 **절대 새면 안 되는** 어드민 내부 값 |
| 8.6 | '모름'(null)의 고객 화면 | `null` 이 고객에게 무엇으로 보이는가 |
| 8.7 | 고객 행동 → 어드민 도착 지점 | 고객이 할 수 있는 일과, 그것이 어드민의 **무엇으로** 들어오는가 |
| 8.8 | 알림 | 상태가 바뀔 때 고객에게 **무엇이 가는가** |
| 8.9 | 이 절의 `[미확정]` | 개수와 목록. 기다리는 결정과 그 결정이 날 자리 |

### 4.1 §8.2 값 대응표 — 필수 열과 판정 규칙

| 열 | 규칙 |
|---|---|
| 어드민 값 | **enum 은 값마다 한 행**이다. '나머지는 동일' 로 접지 않는다 — 접은 자리가 정확히 나중에 갈리는 자리다 |
| 어드민 표기 | 지금 화면에 뜨는 한국어 라벨. 근거는 라벨 상수 |
| 고객 화면 결과 | 문구·버튼·존재 여부. **'보이지 않는다' 도 결과다** |
| 표기 | `[확인]`/`[계약]`/`[미확정]` |
| 근거 | `파일:줄` 또는 계약 근거 한 줄 |

**규칙 UV-4 (전수).** 그 화면이 저장하는 상태 enum 은 **한 값도 빠지지 않는다.** 빠진 값이 있으면 §8 은 미완이다.
**규칙 UV-5 (부울도 값이다).** 토글 하나는 두 행(`true`/`false`)이다.

### 4.2 §8.3 어휘 대조 — '같아야 하는데 다른 것' 은 결함이다

| 상황 | 어떻게 적는가 |
|---|---|
| 어드민과 고객이 **다르게 부른다** (의도적) | 두 표기를 적고 **왜 달라야 하는지** 한 줄. 예: 어드민 `발송` ↔ 고객 '견적서가 도착했습니다' — 주체가 반대다 |
| 어드민 안에서 **같은 것을 다르게 부른다** | ⚠ 로 표시하고 §8.9 `[미확정]` 또는 §7 미결로 이관한다. **고객 표기를 정하기 전에 어드민 표기부터 하나여야 한다** |
| 어드민과 고객이 **같은 말을 쓴다** | 그것도 적는다. '같다' 는 판정이지 공백이 아니다 |

### 4.3 §8.4 '사라지는 것' — 세 낱말을 구분한다

| 낱말 | 뜻 | 고객이 알 수 있는가 |
|---|---|---|
| **숨김**(hidden) | 요소가 DOM 에 없다. 그러나 **그 기능은 존재한다** — 조건이 바뀌면 돌아온다 | 모른다 |
| **비활성**(disabled) | 요소는 보이는데 누를 수 없다. **왜 못 누르는지 함께 말해야 한다** | 안다 |
| **없음**(absent) | 이 사이트에는 그 기능 자체가 **없다** | 모르고, 알 필요도 없다 |

**규칙 UV-6.** 셋 중 무엇인지 적지 않은 '사라진다' 는 서술은 §8 을 통과하지 못한다.
**규칙 UV-7 (선례).** 이 구분은 코드에 이미 있다 — 문의 메뉴는 `open`·`archive`·`hidden` 세 상태를 갖고,
'남은 것이 없을 때만' 사라진다(`apps/admin/src/shared/commerce/inquiry-backlog.ts:112-123` `inquiryMenuState`). 고객 화면도 같은 결로 적는다.

### 4.4 §8.5 누출 금지 — 이것은 보안 요구다

어드민에는 **고객에게 보이면 안 되는 값**이 있다. 명세가 그것을 열거하지 않으면, 앞단 구현자는 API 응답에 실린 필드를 전부 그린다.

**규칙 UV-8.** §8.5 는 **필드 이름으로** 적는다('내부 정보' 같은 뭉뚱그린 말 금지).
**규칙 UV-9.** 누출 금지 항목마다 **왜** 를 적는다. 이유 없는 금지는 다음 사람이 지운다.

### 4.5 §8.6 '모름'(null) — 이 리포의 규율을 고객 화면까지 끌고 간다

`null` 은 '없음' 이 아니라 **'모른다'** 다. 이 규율은 이미 코드의 것이다:

| 선례 | 무엇이 null 인가 | null 을 어떻게 읽는가 |
|---|---|---|
| `shared/commerce/inquiry-backlog.ts:50-53` (`readInquiryBacklog`) | 조회기 미배선 | **fail-open** — 메뉴를 남긴다(`:120-121` '모르면 남긴다 … 지우는 쪽으로 실패하면 복구가 없다') |
| `shared/domain/quote-issue.ts:106-107` (`issuer`) | 발행기 미배선 | 거절 **사유 문자열**로 드러낸다(`QUOTE_ISSUE_UNWIRED`) — 조용히 아무 일도 하지 않는 경로를 만들지 않는다 |
| `shared/commerce/payment-settings.ts:134` (`PgCredentialHealth`) | 자격증명을 확인한 적 없음 | 불리언으로 접지 않는다 — `'unverified'` 를 별도 상태로 둔다 |

**규칙 UV-10.** 고객 화면의 '모름' 은 **0 이나 빈 목록으로 그리지 않는다.** '아직 확인되지 않았습니다' 인지, 요소를 **그리지 않는 것**인지 둘 중 하나를 고르고 그 이유를 적는다.

### 4.6 §8.8 알림 — 지금 있는 것과 없는 것

| 사실 | 근거 | 표기 |
|---|---|---|
| 알림 종류 8종은 **전부 운영자용**이다 — 각 종류가 어드민 사이드바 잎 하나를 가리키고, 그 잎의 권한(`page:{to}`)으로 걸러진다 | `shared/notifications/catalog.ts:44-101` · `:13-15` ('알림이 권한 우회로가 되는 경로가 애초에 없다') | `[확인]` |
| **고객에게 가는 알림은 이 카탈로그에 한 건도 없다** | 같은 표의 `leafPath` 8개가 전부 `/products/inquiries`·`/orders`·`/users/members` 등 어드민 경로다 | `[확인]` |
| 마케팅의 SMS·이메일은 **세그먼트 대상 캠페인**이고 상태 전이에 걸린 트리거가 아니다 | `pages/marketing/sms/types.ts:8-28` (`SmsCampaign` — `segmentIds`·`scheduledAt`·`isAd`) | `[확인]` |

**따라서 상태 전이에 붙는 고객 알림은 전부 `[계약]` 이다.** §8.8 은 최소한 이 넷을 적는다:
**무엇이 계기인가 · 어느 채널인가 · 문구의 뼈대 · 고객이 끌 수 있는가.**

## 5. 이미 코드가 답한 대응 — 새로 발명하지 않는다

아래는 **`[확인]`** 이다. 어드민 값이 고객 화면을 바꾼다는 사실이 이미 코드에 문장으로 적혀 있다.
새 §8 을 쓸 때 이 표를 먼저 보고, 겹치면 **여기를 인용한다**(같은 사실을 두 번 다르게 적지 않는다).

| # | 어드민 값 | 고객 화면 결과 | 근거 |
|---|---|---|---|
| UV-P1 | `usePg = false` (또는 자격증명 미완/거절) | 상품 CTA `구매하기` → **`문의하기`**, 프로그램 CTA `후원하기` → **`문의하기`** | `shared/commerce/payment-settings.ts:88-93,221-235` (`checkoutCta`) |
| UV-P2 | 같은 값 | **금액 자리가 `가격문의` 로 바뀐다** — 저장된 판매가가 있어도 그렇다 | `shared/commerce/price-display.ts:34,71-88` (`resolvePriceDisplay` — `kind: 'inquiry'`) |
| UV-P3 | 같은 값 | 문의 버튼 **아래에 안내 문구**가 붙는다. 기본값: '현재 온라인 결제를 준비 중입니다. 문의를 남겨 주시면 담당자가 확인 후 연락드립니다.' | `shared/commerce/payment-settings.ts:271-273` (`DEFAULT_INQUIRY_GUIDE`) · 운영자가 고친다(`PaymentSettings.inquiryGuide` — `:50-51`) |
| UV-P4 | 안내 문구가 **빈 값** | 고객은 **왜 살 수 없는지 모른 채 '문의하기' 만 본다** — 그래서 검증이 빈 값을 막는다 | FS-078-EL-011 이 인용하는 `settings/payment/validation.ts:76` |
| UV-P5 | `usePg = true` + 자격증명 완비 | 고객의 문의는 **더 이상 들어오지 않는다.** 어드민 문의 메뉴는 남은 건수가 0 이 될 때까지 `· 읽기 전용` 꼬리표를 달고 남았다가 스스로 사라진다 | `shared/commerce/inquiry-backlog.ts:112-135` (`inquiryMenuState` · `INQUIRY_ARCHIVE_SUFFIX`) |
| UV-P6 | `mode = 'test'` | 결제창은 **열리지만 돈이 움직이지 않는다** — 고객에게는 정상 결제로 보인다 | `payment-settings.ts:237-245` |

## 6. 어느 화면이 §8 을 가져야 하는가

**모든 FS 가 §8 을 갖는다.** 백로그를 따로 두지 않는다 — 백로그는 '언제 채울지' 를 문서 안에서 관리하게 만들고,
그것이 곧 상태 기록이다. 고객 접점이 없는 화면은 §8.0 에 `N/A — 고객 접점 없음` 과 사유 한 줄을 적고 끝낸다.

**고객 접점이 없다고 판정할 수 있는 부류** (그래도 §8.0 한 줄은 쓴다):

| 부류 | 사유 |
|---|---|
| 로그 4종 · 통계 6종 | 운영 기록·집계다. 고객 화면에 대응물이 없다 |
| 권한 · 운영자 관리 · API Key · OAuth · 플랜 | 어드민 내부의 설정이다. 값이 고객 화면을 바꾸지 않는다(단 **OAuth 는 예외다** — 소셜 로그인 버튼이 고객 화면에 뜬다) |
| AI 에이전트 2종 | 운영자 전용 조회 도구다 |

**반대로 §8 이 가장 무거운 부류**: 결제 설정(그 값 하나가 다른 화면들의 고객 뷰를 지배한다 — 위 UV-P1–P7),
상품·프로그램(고객이 돈을 쓰는 자리), 상품 문의·프로그램 문의(PG 를 끈 동안 고객의 유일한 접점 — `INQUIRY_PATH`),
주문·배송·클레임(상태값이 거의 전부 고객에게 그대로 보인다), 영업 사슬 4화면(문의→견적→계약→프로젝트가 하나의 흐름이다).

작성 순서는 이 문서가 정하지 않는다 — [`../FSD/plan.md`](../FSD/plan.md) 의 묶음 순서가 정본이다.

## 7. 이 규약을 유지하는 법

- FS 를 **새로 쓰면** §8 을 함께 쓴다. 고객 접점이 없으면 'N/A + 사유' 한 줄이면 된다.
- **앞단이 생기면** `[계약]` 을 하나씩 `[확인]` 으로 바꾼다 — 바꿀 때 근거(`파일:줄`)를 반드시 함께 넣는다.
- **어드민 라벨을 고치면** §8.3 의 그 행을 함께 고친다. 고치지 않으면 어휘 대조표가 조용히 거짓이 된다.
- `[미확정]` 이 정해지면 §8.9 에서 **지운다.** 결론은 그 행이 서술하는 자리(§8.2·§8.4 …)에 `[계약]` 또는 `[확인]` 으로
  들어가야 하고, '한때 미확정이었다' 는 사실은 문서가 아니라 git 이 갖는다
  ([`../FSD/id-convention.md`](../FSD/id-convention.md) — 문서는 지금의 코드만 기술한다).
