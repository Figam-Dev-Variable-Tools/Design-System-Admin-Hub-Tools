# SMS 발송

> **발송 3화면(SMS · 이메일 · 뉴스레터)은 같은 뼈대를 쓴다** — 수신자 세그먼트 · 발신자 레지스트리 ·
> 발송 상태(초안/예약/발송중/발송완료/취소) · 예약 · 발송 결과 통계 · 상태별 허용 액션이 한 모델
> (`apps/admin/src/pages/marketing/_shared/messaging.ts`)에서 온다.
> **그 뼈대의 전부를 이 문서가 갖고**, [이메일 발송](../email/index.md) · [뉴스레터](../newsletters/index.md)
> 는 자기 차이(제목·오픈/클릭율·회차)만 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | SMS 발송 |
| 화면 ID | `SCR-MARKETING-SMS` |
| 메뉴 경로 | 비즈니스 > 마케팅 관리 > SMS 발송 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 문자 발송 캠페인을 한 목록에서 보고, 유형(SMS/LMS/MMS)·대상 수·예약/발송 상태·성공률을 확인하며 등록·수정·삭제한다.

**업무 배경** — 문자는 보내고 나면 되돌릴 수 없고 건당 과금된다. 그래서 이 목록은 '무엇을 보냈나'와 '무엇이 아직 안 나갔나'를 상태로 가르고, **이미 나간 캠페인은 편집으로 열지 않는다**(§4.3).

**화면 설명** — 툴바(검색 · 발송상태 필터 · 등록 버튼) 아래 표가 있고, 열은 발송명 · 유형 · 대상수 · 예약/발송상태 · 성공률이다.

**주요 사용자** — 최상위 관리자 · 마케팅 담당 운영자.

**사용 시나리오**

1. 상태 필터를 `예약` 으로 좁혀 아직 나가지 않은 발송만 본다.
2. 유형 배지로 이 캠페인이 SMS 인지 LMS 인지 확인한다(단가가 다르다).
3. 발송완료 행의 성공률을 확인한다.
4. `SMS 발송 등록` 으로 새 캠페인을 만든다.

**선행 조건** — 로그인. 이 화면의 읽기 권한. **`SMS 발송` 모듈이 이 계정에 있어야 한다**(§9). 발송하려면 사전등록(검증)된 발신번호가 있어야 한다.

**후행 처리** — 등록·수정한 캠페인이 이 목록에 반영된다. **이 화면에서 문자가 나가지는 않는다** — 저장은 예약일 뿐이다.

**관련 화면** — [SMS 발송 등록·수정](form.md)(`SCR-MARKETING-SMS-FORM`) · 같은 뼈대의 [이메일 발송](../email/index.md) · [뉴스레터](../newsletters/index.md) · 본문을 채우는 [발송 템플릿 관리](../templates/index.md). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `비즈니스 > 마케팅 관리` 의 네 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:224`. **⚠ 플랜으로 잠긴다 — 그리고 이메일과 다른 키다.** `marketing.sms` 가 이 경로만 지배하고, 뉴스레터·이메일은 `marketing.email` 이 지배한다 — `apps/admin/src/shared/entitlements/module-resources.ts:75-80`. **`marketing.sms` 는 어떤 티어에도 번들되지 않아** 꺼지면 잠금이 아니라 **완전 숨김**이다(§9).

**관련 기능** — 시스템 설정의 사이트 기본 설정이 '메일·SMS 전용 사이트 이름'을 정하고, 그 이름이 발송 본문 앞에 붙는다([SMS 발송 등록·수정](form.md) §4). 발송 템플릿 관리가 본문의 원천이다.

**관련 API** — 목적 다섯(목록 · 단건 · 등록 · 수정 · 삭제). **실제 전송은 하지 않는다** — 발송 호출 자리는 각 어댑터의 `TODO(backend)` 주석뿐이다 — `apps/admin/src/pages/marketing/sms/data-source.ts:7`. 백엔드가 없다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

## 4. 기능 명세

### 4.1 목록 조회 · 검색 · 필터

| 항목 | 내용 |
|---|---|
| 기능 목적 | 발송 캠페인을 예약일시 최신순으로 보여 주고 조건으로 좁힌다 |
| 실행 조건 | 화면 진입 · 조건 변경 |
| 사용자 동작 | 검색어 입력 · 발송상태 선택 |
| 시스템 처리 | 두 조건 모두 **주소 쿼리가 소유한다**. 정렬은 예약일시 내림차순이며 **예약이 없는 초안은 뒤로** 간다 — `apps/admin/src/pages/marketing/sms/types.ts:66-74` |
| 데이터 처리 | 없음(읽기) |
| Validation | 주소의 필터 값은 허용 목록으로 좁힌다 |
| API 호출 여부 | Y — 목록 조회 |
| 성공 처리 | 표가 그려진다. **토스트를 띄우지 않는다** |
| 실패 처리 | 공용 껍데기의 조회 실패 인라인 배너 |
| 예외 처리 | 조건이 바뀌면 선택을 해제한다 |
| 화면 변경 사항 | 목록영역 · 오류영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | 확인하지 못했다 |

### 4.2 유형(SMS/LMS/MMS)은 자동으로 정해진다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 운영자가 고르지 않아도 **실제로 나갈 등급**을 보여 준다 |
| 실행 조건 | 행마다 |
| 사용자 동작 | 없음(읽기) |
| 시스템 처리 | 저장 시점의 판정값을 그대로 그린다. 판정 규칙은 **이미지가 있으면 MMS**, 없으면 90byte 이하이고 제목이 없을 때만 SMS, 그 밖에는 LMS 다 — `apps/admin/src/pages/marketing/_shared/messaging.ts:324-327` |
| 데이터 처리 | 없음 |
| Validation | 해당 없음 |
| API 호출 여부 | N |
| 성공 처리 | 유형 배지가 뜬다 |
| 실패 처리 | 실패 경로가 없다 |
| 예외 처리 | **제목을 한 글자라도 적으면 90byte 이하여도 LMS 로 승격된다** — 이 조건이 빠지면 화면에서는 SMS(단가가 싸다)로 보이고 실제로는 LMS 로 과금된다 — `_shared/messaging.ts:317-322`. ⚠ 다만 **이 목록의 저장값은 제목을 세지 않고 판정된다** — 목록이 쓰는 판정기는 본문과 이미지만 받는다 — `apps/admin/src/pages/marketing/sms/types.ts:39-41` |
| 화면 변경 사항 | 목록영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

### 4.3 발송 상태가 편집 진입을 가른다

| 항목 | 내용 |
|---|---|
| 기능 목적 | **이미 나간 발송의 기록을 덮어쓰지 않는다** |
| 실행 조건 | 행 클릭 또는 행의 수정 |
| 사용자 동작 | 행 클릭 |
| 시스템 처리 | 상태별 허용 액션을 공용 모델이 판정한다: **초안·예약만 편집**, 예약만 취소, 발송중만 삭제 불가 — `apps/admin/src/pages/marketing/_shared/messaging.ts:499-505`. 행 클릭과 연필이 한 콜백으로 모이므로 여기서 한 번 막으면 둘 다 막힌다 — `SmsListPage.tsx:203-206` |
| 데이터 처리 | 없음 |
| Validation | 해당 없음 |
| API 호출 여부 | N |
| 성공 처리 | 편집 가능한 행이면 수정 화면이 열린다 |
| 실패 처리 | 편집할 수 없는 행은 **비활성으로 표시되고 사유가 붙는다**: `발송이 끝나 수정할 수 없어요` — `SmsListPage.tsx:214` |
| 예외 처리 | **화면은 UX 이고 어댑터가 정본이다** — 주소창 직접 입력이 이 게이팅을 지나치므로 어댑터도 같은 판정으로 거절한다 — `SmsListPage.tsx:203-208`. 발송완료를 수정으로 열면 상태가 조용히 초안으로 강등돼 발송 이력이 뒤집히던 문제가 그 근거다 |
| 화면 변경 사항 | 화면 전체(이동) 또는 변화 없음 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**권한과는 다른 축이다** — 수정 권한이 있어도 발송완료 캠페인은 열리지 않는다.

## 6. Validation

**해당 없음 — 이 목록 화면에는 입력 검증이 없다.** 사용자가 치는 값은 검색어 하나이고 형식 제한이 없다. 주소에서 읽는 필터 값만 허용 목록으로 좁힌다. 발송 폼의 검증 규칙은 [SMS 발송 등록·수정](form.md) 이 갖는다.

**권한이 없을 때 화면이 어떻게 되는가**

| 없는 것 | 결과 | 근거 |
|---|---|---|
| 읽기(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다 | `apps/admin/src/shared/errors/ErrorScreens.tsx:73` |

**발송 채널이 두 키로 갈린 이유** — 이메일만 있는 계약이 흔해 SMS 와 한 키로 묶지 않는다 — `apps/admin/src/shared/entitlements/module-resources.ts:74-79`.

**청구 상태(미납·정지)** — 전 기능이 조회 전용으로 내려앉는다(공통 규칙).
