# 발송 템플릿 등록·수정

> **`/marketing/templates/new?kind={종류}` 와 `/marketing/templates/:id/edit` 는 같은 화면이다** —
> 한 라우트가 종류에 따라 편집기를 고른다.
>
> **템플릿 4종이 한 모델을 공유한다** — 머리(제목 · 취소 · 저장 · 발행)와 저장 배선은 넷이 같고,
> **본문 편집기만 통째로 다르다.** 그래서 이 문서는 **공통 껍데기를 한 번 쓰고 종류별 차이를 표로**
> 다룬다. 종류마다 문서를 복제하지 않는다. 모델의 설명은 [발송 템플릿 관리](index.md) 가 갖는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 발송 템플릿 등록·수정 |
| 화면 ID | `SCR-MARKETING-TEMPLATES-FORM` |
| 메뉴 경로 | 비즈니스 > 마케팅 관리 > 발송 템플릿 관리 > 등록·수정 (사이드바에 자리가 없다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 템플릿의 이름·발신 프로필·본문을 종류에 맞는 편집기로 만들거나 고치고, **초안으로 저장하거나 발행한다.**

**업무 배경** — 네 종류는 규제도 모양도 다르다. 문자는 바이트로 등급이 갈리고, 알림톡은 카카오 심사를 받으며, 브랜드 메시지는 발송 시간대가 제한되고, 이메일은 블록으로 조립한다. 한 편집기로 넷을 덮으면 어느 종류에서도 맞지 않는 화면이 된다.

**화면 설명** — 위에 `목록으로` · (수정이면) 종류 눈썹 · **편집 가능한 큰 제목** · 저장 액션이 있고, 아래에 종류별 본문 편집기가 온다.

**주요 사용자** — 최상위 관리자 · 마케팅 담당 운영자(등록·수정 권한 보유자).

**사용 시나리오**

1. 목록의 `새 템플릿` → 종류 고르기 팝업 → 그 종류의 편집기로 들어온다.
2. 큰 제목 칸에 템플릿명을 적는다.
3. 본문을 만든다 — 이메일이면 블록을 쌓고, 문자면 한 칸에 쓰고, 카카오면 유형을 고른 뒤 채운다.
4. `초안 저장` 으로 두거나 `발행` 으로 켠 채 내보낸다.
5. 발행본을 수정 중이면 액션이 `저장` 하나로 바뀐다.

**선행 조건** — 로그인. 등록은 `create`, 수정은 `update` 권한. 상세에서 수정으로 들어오려면 상태가 `사용중` 이 아니어야 한다([템플릿 상세](detail.md)).

**후행 처리** — 저장한 템플릿이 [발송 템플릿 관리](index.md) 목록에 반영되고, **발행되어 켜진 것만** 발송 화면의 '템플릿 불러오기'에 나타난다.

**관련 화면** — [발송 템플릿 관리](index.md) · [템플릿 상세](detail.md) · 종류 고르기 팝업 [`POP-MARKETING-TEMPLATES-NEW-KIND`](pop-new-kind.md) · **이 화면이 여는 팝업** [`POP-MARKETING-TEMPLATES-FORM-BLOCK-PICKER`](pop-block-picker.md). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. 권한 리소스는 잎 `/marketing/templates` 가 덮는다.

**관련 기능** — 치환변수의 정본은 공통 층의 카탈로그이며, **카탈로그에 없는 토큰은 저장을 막는다**(§6). 발신 프로필·카카오 채널은 이 폴더의 저장소가 갖는다.

**관련 API** — 목적 셋(단건 조회 · 등록 · 수정). 백엔드가 없다 — 경계는 `apps/admin/src/pages/marketing/message-templates/data-source.ts:39-40` 의 `TODO(backend)` 다.

### 3.3 이메일 빌더의 탭 둘 — 각각 다른 축이다

| 탭 묶음 | 접근성 이름 | 항목 | 무엇을 바꾸는가 | 왜 탭인가 |
|---|---|---|---|---|
| 편집 모드 | `편집 모드` — `apps/admin/src/pages/marketing/message-templates/email/EmailToolbar.tsx:162` | `편집` · `미리보기` — `EmailToolbar.tsx:164-165` | **보는 방식이 통째로 바뀐다** | 전환이므로 탭이다. 옆의 데스크톱/모바일은 같은 화면의 폭만 바꾸므로 세그먼트다 — `EmailToolbar.tsx:6-8` |
| 속성 패널 | `속성` — `apps/admin/src/pages/marketing/message-templates/email/EmailBuilder.tsx:370` | `스타일` · `속성` · `점검` — `EmailBuilder.tsx:61-68` | 오른쪽 패널의 내용 | 같은 자리에서 다른 자료를 본다 |

**두 탭 다 문서가 되지 않는다** — [인벤토리](../../inventory.md) 의 '문서가 되지 않는 탭' 표가 둘을 **같은 편집기의 모드·패널 전환**으로 분류한다. 이 문서의 §3·§8 이 그 자리를 갖는다.

**`점검` 이 패널의 탭인 이유** — 점검은 '저장을 누른 뒤에 듣는 판정'이 아니라 편집하는 내내 곁에 있어야 하는 것이다. 저장 옆에 두면 다 만든 뒤에야 열어 보게 되고, 그때는 고칠 것이 가장 많이 쌓여 있다 — `EmailBuilder.tsx:64-66`.

## 4. 기능 명세

### 4.1 종류에 따라 편집기를 고른다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 한 라우트가 네 편집기를 덮는다 |
| 실행 조건 | 화면 진입 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | **등록이면** 주소 쿼리 `?kind=` 로, **수정이면** 저장된 본문의 종류로 편집기를 고른다 — `apps/admin/src/pages/marketing/message-templates/MessageTemplateEditorPage.tsx:64,84` |
| 데이터 처리 | 없음(읽기) |
| Validation | 쿼리의 종류 값이 넷 중 하나가 아니면 **문자로 되돌린다** — 주소는 사용자가 손으로 고칠 수 있다 — `MessageTemplateEditorPage.tsx:31-37` |
| API 호출 여부 | Y — 수정일 때만 단건 조회 |
| 성공 처리 | 그 종류의 편집기가 그려진다 |
| 실패 처리 | **종류를 알기 전이라 편집기를 고를 수 없다** — 편집기 안의 배너가 아니라 여기서 끝낸다: `메시지 템플릿을 불러오지 못했어요.` + `목록으로` — `MessageTemplateEditorPage.tsx:66-80` |
| 예외 처리 | **도착 전에는 아무 편집기도 고르지 않는다** — 문자 편집기를 먼저 그렸다가 이메일이면 통째로 갈아치우게 되고, 그 사이 잠깐 뜬 폼에 입력한 값이 사라진다 — `MessageTemplateEditorPage.tsx:80-82`. 라우트를 종류별로 가르지 않은 이유는 **수정 경로도 갈라야 하고**(`:id` 만 봐서는 종류를 모른다) 목록의 링크가 항목마다 다른 경로를 계산해야 하기 때문이다 — `MessageTemplateEditorPage.tsx:5-7` |
| 화면 변경 사항 | 화면 전체 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | 확인하지 못했다 |

**수정 진입 시 두 번 조회하지 않는다** — 이 라우트와 편집기가 **같은 캐시 키**를 보므로 요청은 한 번이다 — `MessageTemplateEditorPage.tsx:9-10`.

### 4.2 저장 액션이 상태에 따라 갈린다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 초안과 발행본을 다른 행위로 다룬다 |
| 실행 조건 | 항상 |
| 사용자 동작 | `초안 저장` · `발행` · `저장` 중 하나 |
| 시스템 처리 | 초안이면 두 버튼(`초안 저장` · `발행`), 발행본을 수정 중이면 한 버튼(`저장`)이다 — 예: `apps/admin/src/pages/marketing/message-templates/TextTemplateEditor.tsx:267-295` |

## 6. Validation

검증의 정본은 스키마와 순수 함수들이며, **상한값은 도메인 파일이 갖는다** — 안내 문구와 검증이 같은 상수를 봐야 문구만 고치고 검증이 남는 사고가 없다 — `apps/admin/src/pages/marketing/message-templates/validation.ts:1-8`.

| 항목 | 내용 |
|---|---|
| 사고의 모양 | `#{member.name}` 을 `#{member.nmae}` 로 잘못 적으면 **어디에서도 티가 나지 않는다** — 편집기는 정상 토큰처럼 그리고, 검증은 통과하고, 발송도 성공한다. **수신자 화면에 그 글자가 그대로 찍힌 뒤에야** 알게 된다 — `message-templates/validation.ts:98-104` |
| 왜 경고가 아닌가 | 경고는 넘길 수 있고, 넘길 수 있는 경고는 결국 넘긴다. 이 규칙을 어기는 정당한 이유가 없다 — 카탈로그에 없는 토큰은 발송 시점에도 치환되지 않으므로 통과시켜 봐야 반드시 사고가 된다 — `message-templates/validation.ts:109-112` |
| 무엇을 하지 않는가 | **카탈로그를 모르면(조회기 미배선) 아무것도 막지 않는다** — 빈 목록으로 뭉개면 멀쩡한 본문의 토큰이 전부 오타로 신고되어 화면이 통째로 저장 불가가 된다 — `message-templates/validation.ts:114-116` |
| 왜 두 문법을 함께 보는가 | 고객센터 답변 템플릿의 `{{…}}` 는 이 화면의 발송 경로가 모른다. `#{…}` 검사는 중괄호 하나짜리만 보므로 그 문법은 그대로 빠져나가 수신자에게 간다 — **오타 토큰과 정확히 같은 사고**라 같은 자리에서 막는다 — `message-templates/validation.ts:122-125` |

삭제·다운로드·업로드·승인·반려는 **이 화면에 존재하지 않는 동작**이다. **알림톡 심사의 승인·반려는 카카오가 판정하며** 이 화면은 그 상태를 읽기만 한다.

**권한이 없을 때 화면이 어떻게 되는가** — 쓰기 권한이 없으면 저장이 §7 의 권한 배너로 막힌다. 플랜 축은 걸리지 않는다 — `apps/admin/src/shared/entitlements/module-resources.ts:75-79`.

**상태 축이 하나 더 있다** — 상세에서 `사용중` 템플릿의 `수정` 은 노출되지 않는다([템플릿 상세](detail.md) §4.1).
