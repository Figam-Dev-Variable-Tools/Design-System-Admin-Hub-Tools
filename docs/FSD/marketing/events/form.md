# 이벤트 등록·수정

> **`/marketing/events/new` 와 `/marketing/events/:id/edit` 는 같은 화면이다** — 같은 컴포넌트가
> `:id` 유무로 등록과 수정을 겸한다. 그래서 문서도 하나다.
>
> **이벤트와 프로모션의 폼은 같은 뼈대를 쓴다**(공용 폼 껍데기 + 기간·상태·대상·참조 연동·설명).
> **그 뼈대의 전부를 이 문서가 갖고**, [프로모션 등록·수정](../promotions/form.md) 은 차이만 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 이벤트 등록·수정 |
| 화면 ID | `SCR-MARKETING-EVENTS-FORM` |
| 메뉴 경로 | 비즈니스 > 마케팅 관리 > 이벤트 > 등록·수정 (사이드바에 자리가 없다 — 목록에서 들어온다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 이벤트의 이름·기간·상태·대상·혜택·연동 배너·설명을 입력해 만들거나 고친다.

**업무 배경** — 이벤트는 기간과 혜택이 함께 정해져야 성립한다. 혜택을 '쿠폰 발급'으로 골라 두고 어떤 쿠폰인지 비워 두면 운영에서 쓸 수 없는 이벤트가 저장된다. 이 화면은 그런 반쪽 상태를 저장 전에 막는다(§6).

**화면 설명** — 한 장의 카드에 이벤트명 → 기간 → (상태 · 대상) → (혜택 유형 · 혜택 상세) → 배너 연동 → 설명 순으로 놓인다. 아래에 취소·저장 버튼이 있다.

**주요 사용자** — 최상위 관리자 · 마케팅 담당 운영자(등록·수정 권한 보유자).

**사용 시나리오**

1. 목록의 `이벤트 등록` 으로 들어와 이름과 기간을 채운다.
2. 혜택 유형을 `쿠폰 발급` 으로 바꾸면 **혜택 상세 칸이 나타난다** — 쿠폰명을 적는다.
3. 배너 연동을 켜고 목록에서 배너를 고른다. 아래 `연동한 배너 상세 보기` 로 확인한다.
4. 저장하면 목록으로 돌아가고 공통 성공 토스트가 뜬다.
5. 다 채우지 않은 채 다른 메뉴를 누르면 이탈 가드가 뜬다.

**선행 조건** — 로그인. 등록은 `create`, 수정은 `update` 권한. 배너를 연동하려면 콘텐츠 관리 배너에 등록된 배너가 있어야 한다.

**후행 처리** — 저장한 값이 [이벤트 목록](index.md) 에 반영된다. **연동한 배너 쪽은 바뀌지 않는다** — 이벤트가 배너를 참조할 뿐이다. 저장 시점의 배너명이 표시용 사본으로 함께 저장된다.

**관련 화면** — [이벤트 목록](index.md)(`SCR-MARKETING-EVENTS`) · 연동 대상 `SCR-CONTENT-BANNERS-FORM` · 같은 뼈대의 [프로모션 등록·수정](../promotions/form.md). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. 권한 리소스는 잎 `/marketing/events` 가 덮는다.

**관련 기능** — 콘텐츠 관리 배너가 선택 목록의 정본이다. **카탈로그를 모르면(조회기 미배선) 빈 선택 목록을 그리지 않는다**(§4.4).

**관련 API** — 목적 셋: 단건 조회(수정일 때) · 등록 · 수정. 공통 폼 컨트롤러가 배선한다. **백엔드가 없다** — 경계는 `apps/admin/src/pages/marketing/events/data-source.ts` 의 `TODO(backend)` 다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)
