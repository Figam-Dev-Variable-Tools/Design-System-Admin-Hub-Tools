# 관리자 로그

> **로그 4화면은 같은 껍데기를 공유한다.** 좌측 필터 레일 · 상단 툴바 · 요약 줄 · 표 · 페이지네이션 ·
> 상세 다이얼로그 · 내보내기가 한 벌(`apps/admin/src/pages/logs/components/LogListShell.tsx`)이고,
> 각 화면이 주는 것은 **축의 어휘 · 컬럼 · 강조 규칙 · 상세의 모양 · 보존기간** 다섯뿐이다.
> **그 껍데기의 전부를 이 문서가 갖는다.** [회원 활동 로그](../member-activity/index.md) ·
> [API 로그](../api/index.md) · [오류 로그](../errors/index.md) 세 문서는 이 문서를 가리키고
> **자기 화면의 차이만** 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 관리자 로그 |
| 화면 ID | `SCR-LOGS-ADMIN` |
| 메뉴 경로 | 분석 · 운영 > 로그 관리 > 관리자 로그 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — "그 회원 등급 누가 바꿨어요?" 에 답한다. 운영자가 시스템에 한 일을 누가·무엇을·어떻게·됐는가의 네 축으로 남기고, 기간과 조건으로 좁혀 찾게 한다.

**업무 배경** — 어드민은 여러 사람이 같은 데이터를 고친다. 값이 바뀐 뒤에 "언제부터 이랬지"를 물으면, 답할 수 있는 곳이 이 화면 하나다. 이 화면이 없으면 변경의 책임 소재를 확인할 방법이 없고, 사고가 나도 무엇이 원인이었는지 되짚을 수 없다.

**화면 설명** — 왼쪽에 필터 레일(결과 · 액션 · 기간 · 보존기간 안내), 오른쪽에 검색·페이지 크기·내보내기 툴바와 요약 줄, 그 아래 표와 페이지네이션이 있다. 표의 행을 누르면 그 요청의 원문이 다이얼로그로 열리고, 행위자 계정이나 대상 이름을 누르면 그 대상의 화면으로 이동한다.

**주요 사용자** — 최상위 관리자 · 운영자(내보내기 권한이 있는 역할) · 감사 담당.

**사용 시나리오**

1. 회원 등급이 잘못 바뀌었다는 신고를 받는다 → 기간을 최근 7일로 좁히고 액션 `권한 변경`·검색어 `회원 계정`으로 찾아 행을 열어 요청 원문을 확인한다.
2. 요약 줄의 `이 기간의 권한 변경 3건` 을 보고 액션 축을 눌러 그 3건만 본다.
3. 감사 자료 제출을 위해 조건을 걸어 둔 채 내보내기를 눌러 CSV 를 받는다.
4. 좁혀 놓은 조건 그대로 주소를 복사해 동료에게 보낸다 — 조회 조건이 전부 주소에 실려 있어 같은 화면이 열린다.

**선행 조건** — 로그인. 이 화면의 읽기 권한. 조회 자체에는 앞선 화면에서 만들어져야 할 데이터가 없다 — 로그는 이 앱이 만들지 않는다.

**후행 처리** — **이 화면은 아무것도 바꾸지 않는다.** 읽고, 파일로 받고, 다른 화면으로 이동할 뿐이다.

**관련 화면** — 같은 껍데기를 쓰는 세 화면: `SCR-LOGS-MEMBER-ACTIVITY` · `SCR-LOGS-API` · `SCR-LOGS-ERRORS`. 행에서 이동하는 곳: `SCR-USERS-ADMINS-DETAIL` · `SCR-USERS-MEMBERS-DETAIL` · `SCR-CONTENT-NOTICES-DETAIL` · `SCR-USERS-ROLES`. 원문 다이얼로그는 로그 4화면이 공유하는 팝업 `POP-COMMON-LOG-PAYLOAD` 다 — [로그 원문 보기](../../_common/pop-log-payload.md). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `분석 · 운영 > 로그 관리` 의 첫 항목. 조건부 노출이 아니다 — `apps/admin/src/shared/layout/nav-config.ts:276-277`. 이 경로에는 플랜 모듈 매핑이 없어 플랜 축으로 잠기지 않는다(매핑이 없으면 열린다) — `apps/admin/src/shared/entitlements/module-resources.ts:19-22`.

**관련 기능** — 로그인 이력(`SCR-USERS-LOGIN-HISTORY`)이 이 화면과 같은 배치·같은 불변성 규칙의 원형이다. 권한 관리(`SCR-USERS-ROLES`)가 이 화면의 읽기·내보내기 권한을 정한다.

**관련 API** — 목적 둘. ① 조건에 맞는 한 페이지 조회(축별 건수와 총 건수를 함께 받는다) ② 같은 조건에 걸린 **전체**를 내보내기용으로 조회. **쓰기 목적의 호출이 하나도 없다** — 감사 기록은 이 앱이 만들지도 고치지도 지우지도 않는다. 백엔드는 아직 없고, 경계는 `apps/admin/src/pages/logs/admin/data-source.ts` 의 `TODO(backend)` 두 줄이다(`:57` · `:65`).

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**셸은 이 화면의 것이 아니다** — 사이드바·헤더·본문으로 건너뛰기는 인증 후 모든 화면이 공유하는 껍데기가 소유하고, 화면 제목(`<h1>`)도 헤더가 갖는다.

## 4. 기능 명세

### 4.1 목록 조회

| 항목 | 내용 |
|---|---|
| 기능 목적 | 지금 주소가 말하는 조건(기간 · 축 2개 · 검색어 · 정렬 · 페이지 · 페이지 크기)에 맞는 한 페이지와, 좌측 배지에 쓸 축별 건수, 총 건수를 가져온다 |
| 실행 조건 | 화면 진입 또는 위 조건 중 하나라도 바뀌었을 때. **기간이 유효하지 않으면 조회를 시작하지 않는다** — `LogListShell.tsx:153` |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 조건 전체가 캐시 키가 된다 — 늦게 도착한 이전 검색어의 응답이 최신 화면을 덮지 못한다 — `apps/admin/src/pages/logs/queries.ts:40-46`. 필터·기간·검색어는 **AND** 로 걸린다 — `query-engine.ts:60-75` |
| 데이터 처리 | 없음(읽기 전용). 축별 건수는 **기간 안에서만** 세며 다른 축·검색어와 무관하다 — `query-engine.ts:78-83` |
| Validation | 기간 직접 지정만 검증한다(§6). 나머지 조건은 주소에서 읽을 때 허용 목록으로 좁힌다 |
| API 호출 여부 | Y — 조건에 맞는 한 페이지 + 축별 건수 + 총 건수 |
| 성공 처리 | 요약 줄 `전체 {N}건 중 {a}–{b}`, 표, 페이지네이션, 좌측 배지가 채워진다. 경고할 사건이 있으면 §4.10 의 한 줄이 붙는다. **토스트를 띄우지 않는다** |
| 실패 처리 | 표·요약·페이지네이션 자리를 danger 배너가 대체한다: `관리자 로그를 불러오지 못했어요.` + `다시 시도` — `LogListShell.tsx:280-292`. **토스트가 아니다** — 통지가 사라지면 빈 화면만 남아 '기록이 없다'와 구분되지 않는다 — `LogListShell.tsx:279` |
| 예외 처리 | 다른 조건으로 좁혀 총 페이지가 줄면 현재 페이지가 범위를 벗어난다 — 마지막 페이지로 되돌린다 — `LogListShell.tsx:165-169`. 데이터가 이미 있는 재조회는 표를 비우지 않는다(스켈레톤은 **첫 조회에만**) — `LogListShell.tsx:258` |
| 화면 변경 사항 | 필터영역(배지) · 요약영역 · 목록영역 · 페이지네이션 · 오류영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | 확인하지 못했다 — 조회 행위를 다시 로그로 남기는 배선을 코드에서 찾지 못했다 |

**재현 손잡이(백엔드 연동 전)** — 주소에 `?fail=list` · `?fail=all` · `?fail=logs-admin:list` 를 붙이면 조회 실패를 재현할 수 있다 — `apps/admin/src/pages/logs/adapter.ts:17-19`.

### 4.9 CSV 내보내기

| 항목 | 내용 |

| 후속 동작 | 목록은 그대로다 |
| 로그 기록 여부 | 확인하지 못했다 — 액션 축에 `내보내기` 가 있으나, 이 화면의 내보내기가 자기 로그를 남기는 배선은 코드에 없다 |

**권한이 없을 때 화면이 어떻게 되는가**

| 없는 것 | 결과 | 근거 |

| 읽기(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다. 사이드바·헤더는 남아 다른 메뉴로 걸어 나갈 수 있다 | `apps/admin/src/shared/errors/ErrorScreens.tsx:73` |
| 내보내기(`export`) | 내보내기 버튼이 **사라진다**(비활성이 아니다). 대체 안내 문구는 없다 | `LogToolbar.tsx:54-55,112` |

**읽기 권한 판정은 이 화면의 일이 아니다** — 셸이 모든 라우트를 한 번에 덮는다. 화면이 스스로 게이팅하는 것은 내보내기 버튼 하나뿐이다 — `LogListShell.tsx:114-116`.

**청구 상태(미납·정지)의 영향** — 쓰기가 없으므로 읽기 전용으로 내려앉을 것이 없다. 조회와 내보내기는 청구 상태와 무관하게 열린다 — `apps/admin/src/shared/entitlements/plan.ts:396-405`.
