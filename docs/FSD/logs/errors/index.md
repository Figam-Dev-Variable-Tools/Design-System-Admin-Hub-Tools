# 오류 로그

> **껍데기는 [관리자 로그](../admin/index.md) 와 같다.** 좌측 필터 레일 · 툴바 · 요약 줄 · 표 ·
> 페이지네이션 · 원문 다이얼로그 · 내보내기 · 조회 조건의 주소 왕복 · 기간 검증 · 빈 상태 3분기 ·
> 조회 실패 인라인 배너 · 권한 게이팅이 전부 한 벌(`LogListShell`)에서 나온다.
> **이 문서는 그 껍데기를 다시 적지 않고 이 화면의 차이만 적는다.**

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 오류 로그 |
| 화면 ID | `SCR-LOGS-ERRORS` |
| 메뉴 경로 | 분석 · 운영 > 로그 관리 > 오류 로그 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — "무엇이 깨졌나?" 에 답한다. 앞의 세 화면이 '누가 무엇을 했나' 라면 이 화면은 시스템이 **스스로 실패한 사건**을 남긴다.

**업무 배경** — 고객이 신고하기 전에 우리가 먼저 알아야 하는 사건이 있다. 정산 배치가 조용히 실패하거나 결제 게이트웨이가 간헐적으로 타임아웃되면, 그 사실이 어디에도 표시되지 않으면 며칠 뒤 매출 차이로 발견된다.

**화면 설명** — 배치는 다른 로그 화면과 같다. 다른 것은 **행위자가 없다**는 점이다 — 아무도 하지 않았는데 일어난 일이다. 축은 `심각도`·`발생 위치` 이고, 표에 `발생 횟수` 라는 숫자 컬럼이 있다.

**주요 사용자** — 최상위 관리자 · 장애 대응 담당 · 개발자에게 전달할 정보를 모으는 운영자.

**사용 시나리오**

1. 아침에 이 화면을 열어 요약 줄의 `이 기간의 치명 오류 2건` 을 확인하고 그 2건을 연다.
2. 발생 위치 축 `정산 배치` 로 좁혀 지난 30일간 같은 코드가 몇 번 났는지 본다.
3. 한 건을 열어 `추적 ID` 를 복사해 개발자에게 전달한다.
4. `발생 횟수` 를 내림차순으로 정렬해 가장 시끄러운 오류부터 처리한다.

**선행 조건** — 로그인. 이 화면의 읽기 권한.

**후행 처리** — 아무것도 바꾸지 않는다. **'해결됨' 토글이 없다** — 이 화면은 이슈 트래커가 아니라 감사 로그다. 일어난 일을 나중에 덧칠할 수 있다면 그것은 이미 불변 기록이 아니다 — `apps/admin/src/pages/logs/errors/types.ts:9-14`.

**관련 화면** — 같은 껍데기: `SCR-LOGS-ADMIN` · `SCR-LOGS-MEMBER-ACTIVITY` · `SCR-LOGS-API`. **행에서 이동하는 화면이 없다** — 행위자도 대상 레코드도 없다. 원문 다이얼로그: `POP-COMMON-LOG-PAYLOAD` — [로그 원문 보기](../../_common/pop-log-payload.md).

**관련 메뉴** — 사이드바 `분석 · 운영 > 로그 관리` 의 네 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:280`. 플랜 모듈 매핑이 없어 잠기지 않는다.

**관련 기능** — 이 어드민 화면이 렌더 중에 던진 예외는 **여기에 오지 않는다.** 지금은 브라우저 콘솔에만 남고 원격 수집기가 붙어 있지 않다 — `apps/admin/src/shared/errors/ErrorBoundary.tsx:91-92`. 즉 이 화면이 담는 것은 서버 쪽 사건이다.

**관련 API** — 다른 로그 화면과 같은 두 목적. 쓰기 목적의 호출이 없다. 경계는 `apps/admin/src/pages/logs/errors/data-source.ts` 의 `TODO(backend)` 다.

**참고 문서** — [관리자 로그](../admin/index.md)(껍데기 정본) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/logs/errors/ErrorLogPage.tsx` · `errors/types.ts` · `errors/data-source.ts`

## 5. 이벤트 정의

[관리자 로그](../admin/index.md) 의 이벤트 표와 **같다.** 다만 이 화면에는 **행 안의 링크가 없어** 링크 클릭 이벤트 세 줄이 성립하지 않는다. 행 클릭·첫 칸 버튼 활성화는 언제나 원문 다이얼로그로만 간다.

**이 화면에도 확인 다이얼로그가 없다.** 상태를 바꾸는 동작이 아예 없기 때문이다.

## 6. Validation

[관리자 로그](../admin/index.md) 와 **같다.** 이 화면만의 추가 검증은 없다. 다른 것은 축 이름(`severity`·`source`)과 허용 값 목록뿐이다.

## 8. 화면 상태(State)

[관리자 로그](../admin/index.md) 의 상태 표와 **같다.** 문구의 엔티티 이름만 `오류 로그` 로 바뀐다. 이 화면만의 상태는 없다.

**0건이 '고장'이 아니라 '사실'인 화면** — 다른 화면과 달리 여기서는 빈 결과가 정상 상태다. 화면은 그것을 특별히 축하하지 않고 같은 빈 상태 카드로 그린다.

## 10. 화면 이동(Screen Flow)

| 현재 화면 | 이벤트 | 이동 화면 | 팝업 여부 | 이동 조건 | 전달 파라미터 | 복귀 화면 |
|---|---|---|---|---|---|---|
| `SCR-LOGS-ERRORS` | 행 클릭 · 첫 칸 버튼 활성화 | `POP-COMMON-LOG-PAYLOAD` | Y | 언제나 | 그 행의 필드 표 · 원문 · 원문 제목 `스택 · 컨텍스트` | 같은 목록(조건·페이지 유지) |
| `SCR-LOGS-ERRORS` | 내보내기 성공 | 이동 없음(파일 다운로드) | N | 권한 + 유효한 기간 | — | — |
| `SCR-LOGS-ERRORS` | 401 관측 | `SCR-LOGIN` | N | 세션 만료 | 원래 주소 · 만료 표식 | 재인증 후 같은 조건의 이 화면 |
| `SCR-LOGS-ERRORS` | 읽기 권한 없음 | 권한 없음 화면 | N | `read` 없음 | — | 대시보드로 나갈 수 있다 |

**이 화면에서 다른 화면으로 나가는 길이 없다** — 오류를 고치는 곳은 이 앱 밖(이슈 트래커)이며, 그 연결은 코드에 배선되어 있지 않다. 화면이 주는 것은 복사해 갈 수 있는 `추적 ID` 뿐이다.

**팝업은 공유 팝업 하나뿐이고 화면 고유 확인 다이얼로그는 0건이다.**

**결과 통지 토스트 2건(내보내기 성공·실패)은 로그 4화면이 공유한다.**
