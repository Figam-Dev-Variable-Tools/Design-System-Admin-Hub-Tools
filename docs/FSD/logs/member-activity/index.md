# 회원 활동 로그

> **껍데기는 [관리자 로그](../admin/index.md) 와 같다.** 좌측 필터 레일 · 툴바 · 요약 줄 · 표 ·
> 페이지네이션 · 원문 다이얼로그 · 내보내기 · 조회 조건의 주소 왕복 · 기간 검증 · 빈 상태 3분기 ·
> 조회 실패 인라인 배너 · 권한 게이팅이 전부 한 벌(`LogListShell`)에서 나온다.
> **이 문서는 그 껍데기를 다시 적지 않고 이 화면의 차이만 적는다.**

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 회원 활동 로그 |
| 화면 ID | `SCR-LOGS-MEMBER-ACTIVITY` |
| 메뉴 경로 | 분석 · 운영 > 로그 관리 > 회원 활동 로그 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — "이 회원이 정말 그 주문을 했나요?" · "탈퇴 전에 무슨 일이 있었나요?" 에 답한다. **회원이** 서비스에서 한 일을 남긴다.

**업무 배경** — 고객 분쟁과 이상 거래 확인은 "그 계정이 실제로 무엇을 했는가"를 시각 순으로 되짚어야 끝난다. 회원 상세 화면은 지금의 상태를 보여줄 뿐 과거의 행위를 보여주지 않는다.

**화면 설명** — 관리자 로그와 같은 배치다. 다른 것은 좌측 축이 `결과`·`활동` 이고, 표에 회원·내용·접속(IP/기기) 컬럼이 있으며, 계정을 누르면 회원 상세로 간다는 점이다.

**주요 사용자** — 최상위 관리자 · 상담원 · 감사 담당.

**사용 시나리오**

1. 결제 이의 신고를 받고 회원 계정으로 검색해 그 시각 전후의 활동을 시각 순으로 읽는다.
2. 요약 줄의 `이 기간의 탈퇴 4건` 을 보고 활동 축 `탈퇴` 를 눌러 그 4건만 본다.
3. 한 건을 열어 주문 요청 원문을 확인한다 — 배송지·결제수단은 가려진 채로 보인다.

**선행 조건** — 로그인. 이 화면의 읽기 권한.

**후행 처리** — 아무것도 바꾸지 않는다.

**관련 화면** — 같은 껍데기: `SCR-LOGS-ADMIN` · `SCR-LOGS-API` · `SCR-LOGS-ERRORS`. 행에서 이동하는 곳: `SCR-USERS-MEMBERS-DETAIL`. 원문 다이얼로그: `POP-COMMON-LOG-PAYLOAD` — [로그 원문 보기](../../_common/pop-log-payload.md).

**관련 메뉴** — 사이드바 `분석 · 운영 > 로그 관리` 의 두 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:278`. 플랜 모듈 매핑이 없어 잠기지 않는다.

**관련 기능** — 회원 관리(`SCR-USERS-MEMBERS`)가 이 로그가 가리키는 계정의 현재 상태를 갖는다. 탈퇴한 회원은 가리킬 레코드가 없어 링크가 되지 않는다.

**관련 API** — 관리자 로그와 같은 두 목적(한 페이지 조회 · 내보내기용 전체 조회)이고 대상만 다르다. 쓰기 목적의 호출이 없다. 경계는 `apps/admin/src/pages/logs/member-activity/data-source.ts` 의 `TODO(backend)` 다.

**참고 문서** — [관리자 로그](../admin/index.md)(껍데기 정본) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/logs/member-activity/MemberActivityPage.tsx` · `member-activity/types.ts` · `member-activity/data-source.ts` (공유 껍데기는 [관리자 로그](../admin/index.md) 의 구현 파일 표)

## 4. 기능 명세

기능의 목록과 절차는 [관리자 로그](../admin/index.md) 와 같다: 목록 조회 · 축 필터 · 기간 선택 · 검색 · 정렬 · 페이지 이동 · 상세 열기 · CSV 내보내기 · 요약 경고. **성공·실패·예외의 처리 방식도 같다.** 아래는 이 화면에서만 달라지는 값이다.

| 사용자 동작 | 없음(읽기) |
| 시스템 처리 | 아래 여섯 컬럼으로 그린다. 실패 행만 danger 톤이 된다 |

| Validation | 해당 없음 |

| 실패 처리 | 시각을 해석할 수 없으면 받은 문자열을 그대로 그린다 |
| 예외 처리 | **탈퇴 회원은 계정이 링크가 아니다** — 가리킬 레코드가 없다 — `member-activity/types.ts:43` |

## 5. 이벤트 정의

| 이벤트 | 발생 조건 | 처리 내용 | 대상 컴포넌트 | 결과 화면 |
|---|---|---|---|---|
| 계정 링크 클릭 | 등록 회원의 계정 클릭 | 회원 상세로 이동 | 회원 셀 | `SCR-USERS-MEMBERS-DETAIL` |
| 탈퇴 회원 계정 클릭 | 탈퇴 회원의 계정 클릭 | **아무 일도 일어나지 않는다**(링크가 아니다). 행 클릭으로 해석되어 원문이 열린다 | 회원 셀 | `POP-COMMON-LOG-PAYLOAD` |

**이 화면에도 확인 다이얼로그가 없다.**

## 6. Validation

[관리자 로그](../admin/index.md) 와 **같다** — 기간 직접 지정 검증(형식 · 미래 금지 · 시작≤종료 · 최대 90일)과 주소 값의 허용 목록 좁히기가 4화면 공유 규칙이다. 이 화면만의 추가 검증은 없다. 다른 것은 축 이름(`activity`)과 그 축이 허용하는 값 목록뿐이다.

**이 화면 고유의 위험** — 표시되는 계정·이름은 이미 가려진 형태로 내려온다는 전제이며(`member-activity/types.ts:40-43` 의 주석), **가리는 주체가 서버인지 화면인지는 코드에서 확인하지 못했다** — 화면 쪽에 계정·이름을 가리는 함수가 없다. 원문(payload)의 마스킹만 화면이 확실히 수행한다.

## 8. 화면 상태(State)

[관리자 로그](../admin/index.md) 의 상태 표와 **같다.** 문구의 엔티티 이름만 `회원 활동 로그` 로 바뀌고, 좌측 배지의 `—`(모름)과 `0`(그 기간에 없음)을 다르게 표기하는 규칙도 같다. 이 화면만의 상태는 없다.

**권한이 없을 때 화면이 어떻게 되는가** — [관리자 로그](../admin/index.md) 와 같다: 읽기가 없으면 본문이 권한 없음 화면으로 대체되고, 내보내기가 없으면 버튼이 사라진다. 플랜 축은 걸리지 않는다.

**⚠ 이 화면은 개인정보 밀도가 가장 높다.** 그런데 권한 축은 다른 로그 화면과 **동일한 두 개(`read`·`export`)뿐**이며, 개인정보 열람에 대한 별도 권한이나 열람 사유 입력 같은 추가 통제가 코드에 없다. 보존기간을 1년으로 짧게 잡은 것이 코드에서 확인되는 유일한 완화 조치다.
