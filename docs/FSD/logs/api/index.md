# API 로그

> **껍데기는 [관리자 로그](../admin/index.md) 와 같다.** 좌측 필터 레일 · 툴바 · 요약 줄 · 표 ·
> 페이지네이션 · 원문 다이얼로그 · 내보내기 · 조회 조건의 주소 왕복 · 기간 검증 · 빈 상태 3분기 ·
> 조회 실패 인라인 배너 · 권한 게이팅이 전부 한 벌(`LogListShell`)에서 나온다.
> **이 문서는 그 껍데기를 다시 적지 않고 이 화면의 차이만 적는다.**

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | API 로그 |
| 화면 ID | `SCR-LOGS-API` |
| 메뉴 경로 | 분석 · 운영 > 로그 관리 > API 로그 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — "연동이 왜 안 되죠?" · "누가 우리 API 를 두드리고 있죠?" · "어디가 느리죠?" 에 답한다. 시스템이 주고받은 HTTP 요청을 남긴다.

**업무 배경** — 외부 연동은 실패해도 조용하다. 파트너가 "안 된다"고 말할 때 우리 쪽 응답이 무엇이었는지 확인할 곳이 없으면 논의가 진행되지 않는다. 응답 시간도 이 화면 말고는 볼 곳이 없다.

**화면 설명** — 배치는 다른 로그 화면과 같다. 다른 것은 **행위자가 사람이 아니라 키(client)** 라는 점이다. 그래서 축이 `상태`(2xx/4xx/5xx)·`메서드` 이고, 검색이 경로·클라이언트·요청 ID 이며, **응답 시간이라는 숫자 컬럼이 처음 등장한다.**

**주요 사용자** — 최상위 관리자 · 연동 담당 운영자 · 장애 대응 담당.

**사용 시나리오**

1. 파트너 연동 실패 신고를 받고 상태 축 `요청 오류 (4xx)` 와 경로로 좁혀 어떤 요청이 거절됐는지 본다.
2. 요약 줄의 `이 기간의 서버 오류(5xx) 7건` 을 보고 그 7건만 열어 요청·응답을 확인한다.
3. `응답시간(ms)` 헤더를 눌러 내림차순으로 정렬해 가장 느린 호출을 찾는다.
4. 한 건을 열어 `요청 ID` 를 복사해 서버 로그와 대조한다.

**선행 조건** — 로그인. 이 화면의 읽기 권한.

**후행 처리** — 아무것도 바꾸지 않는다.

**관련 화면** — 같은 껍데기: `SCR-LOGS-ADMIN` · `SCR-LOGS-MEMBER-ACTIVITY` · `SCR-LOGS-ERRORS`. **행에서 이동하는 화면이 없다** — 행위자가 사람이 아니라 가리킬 계정 상세가 없다. 원문 다이얼로그: `POP-COMMON-LOG-PAYLOAD` — [로그 원문 보기](../../_common/pop-log-payload.md).

**관련 메뉴** — 사이드바 `분석 · 운영 > 로그 관리` 의 세 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:279`. 플랜 모듈 매핑이 없어 잠기지 않는다.

**관련 기능** — API Key 설정(`SCR-SETTINGS-API-KEYS`)이 이 로그에 찍히는 키를 발급·폐기하는 화면이다. 이 화면은 **키의 뒤 4자만** 보여 주는데, 그 목적이 바로 "어느 키였는지 대조해 폐기하는 것" 이다 — `apps/admin/src/pages/logs/api/types.ts:52-56`.

**관련 API** — 다른 로그 화면과 같은 두 목적. 쓰기 목적의 호출이 없다. 경계는 `apps/admin/src/pages/logs/api/data-source.ts` 의 `TODO(backend)` 다.

**참고 문서** — [관리자 로그](../admin/index.md)(껍데기 정본) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/logs/api/ApiLogPage.tsx` · `api/types.ts` · `api/data-source.ts`
