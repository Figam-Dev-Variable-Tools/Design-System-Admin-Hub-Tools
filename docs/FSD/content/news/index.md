# 뉴스·보도자료

> 이 화면은 콘텐츠 관리에서 **유일하게 공용 목록 껍데기(`CrudListShell`)를 쓰는 목록**이다.
> 그래서 [공지사항 목록](../notices/index.md)과 겉모습은 닮았지만 **쓰기 권한 게이팅 · 사유별 빈 상태 ·
> 재조회 표시**가 다르다(§3·§8·§9). 아래 각 절은 그 차이를 밝혀 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 뉴스·보도자료 |
| 화면 ID | `SCR-CONTENT-NEWS` |
| 메뉴 경로 | 콘텐츠 관리 > 뉴스·보도자료 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 회사가 밖으로 내보내는 소식(언론보도·수상·공시)을 찾고, 고정 여부와 발행 상태를 확인하고, 지운다.

**업무 배경** — 공지사항은 **서비스 이용자에게 알리는 글**이고 뉴스는 **회사가 밖으로 내보내는 소식**이다. 독자·톤·수명이 다르고 홈페이지에서 서는 자리도 다르다 — 한 게시판에 섞으면 방문자가 점검 안내와 수상 소식을 같은 목록에서 보게 된다 — `apps/admin/src/pages/content/news/types.ts:4-6`.

**화면 설명** — 왼쪽에 분류 필터(건수 배지 포함), 오른쪽에 검색 + 상태 필터 + 등록 버튼 + 표가 온다. **페이지네이션이 없다** — 필터를 적용한 전체를 한 번에 그린다.

**주요 사용자** — 최상위 관리자 · 콘텐츠를 다루는 운영자 · 홍보 담당자.

**사용 시나리오**

1. 홍보 담당자가 '언론보도'를 걸어 지난 보도자료를 훑는다.
2. 다음 주에 나갈 보도자료의 상태가 '예약'으로 보이는지 확인한다 — 상태는 저장값이 아니라 **지금 시각으로 계산된 값**이다.
3. 잘못 올린 글을 체크해 한 번에 지운다.

**선행 조건** — 로그인 · 콘텐츠 관리 플랜 모듈(`cms.pages`) — `apps/admin/src/shared/entitlements/module-resources.ts:74`.

**후행 처리** — 목록의 값은 홈페이지의 뉴스 게시판에 쓰인다. 홈페이지는 이 리포 밖이라 반영 경로를 코드에서 확인하지 못했다.

**관련 화면** — [뉴스 등록·수정](form.md)(`SCR-CONTENT-NEWS-FORM`) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md). **상세 화면이 없다** — 행을 누르면 수정 폼으로 간다.

**관련 메뉴** — 콘텐츠 관리 가지의 첫 잎 — `apps/admin/src/shared/layout/nav-config.ts:144`.

**관련 기능** — 없음. **첨부 기능이 없다** — 예전에는 미디어 라이브러리의 자산을 참조했으나 그 화면이 IA 에서 빠지면서 필드째 걷혔다 — `apps/admin/src/pages/content/news/types.ts:12-14`.

**관련 API** — 목록 조회 1건 · 단건 삭제 1건. **백엔드가 없다** — `apps/admin/src/pages/content/news/data-source.ts:130`. 분류도 아직 고정 목록이며 조회로 바뀔 자리가 표시되어 있다 — `apps/admin/src/pages/content/news/data-source.ts:16-18`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/content/news/NewsPage.tsx` (규칙 `types.ts` · 데이터 `data-source.ts` · 발행 판정 `apps/admin/src/shared/domain/publish-schedule.ts`)
