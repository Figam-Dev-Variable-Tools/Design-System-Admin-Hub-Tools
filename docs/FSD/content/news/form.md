# 뉴스 등록 · 수정

> 이 화면은 콘텐츠 관리에서 **유일하게 공용 폼 껍데기(`FormPageShell` + `useCrudForm`)를 쓰는 폼**이다.
> 그래서 [공지 등록·수정](../notices/form.md)과 겉모습은 닮았지만 **권한 문 · 저장 충돌 다이얼로그 ·
> 422 인라인 매핑 · 오류 참조 코드 · 동기 중복 제출 잠금 · 404 와 오류의 구분**을 모두 갖는다.
> 아래 각 절은 그 차이를 밝혀 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 뉴스 등록 · 수정 |
| 화면 ID | `SCR-CONTENT-NEWS-FORM` |
| 메뉴 경로 | 콘텐츠 관리 > 뉴스·보도자료 > 뉴스 등록 (또는 행 클릭·연필) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 보도자료 한 건의 제목·분류·상태·공개 일시·상단 고정·본문을 정한다.

**업무 배경** — 보도자료는 "언제 공개되는가"가 계약처럼 정해지는 일이 잦다(엠바고). 그래서 공개 시각을 미래로 지정할 수 있어야 하고, 그 지정이 실제로 지켜지는지 목록에서 확인할 수 있어야 한다.

**화면 설명** — 카드 하나에 제목 · (분류 · 상태 · 공개 일시) 한 줄 · 상단 고정 토글 · 본문이 놓인다. 맨 아래 취소·제출.

**주요 사용자** — 최상위 관리자 · 콘텐츠를 다루는 운영자 · 홍보 담당자.

**사용 시나리오**

1. 다음 주 공개할 보도자료를 미리 쓰고 상태를 '발행', 공개 일시를 미래로 지정한다 → 목록에서 '예약'으로 보인다.
2. 즉시 내보낼 소식을 쓰고 공개 일시를 비운 채 '발행'으로 저장한다 → 즉시 공개된다.
3. 초안 상태로 공개 일시를 적는다 → 저장이 막히고 그 이유가 공개 일시 칸에 뜬다.

**선행 조건** — 로그인 · 콘텐츠 관리 플랜 모듈 · **등록이면 등록 권한, 수정이면 수정 권한**(없으면 화면이 열리지 않는다 — §9).

**후행 처리** — 등록·수정 모두 성공하면 목록으로 간다(상세가 없다). 히스토리를 덮어쓴다 — `apps/admin/src/shared/crud/useCrudForm.ts:250`.

**관련 화면** — [뉴스·보도자료 목록](index.md)(`SCR-CONTENT-NEWS`) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자리가 없다. 권한 리소스는 잎 `/content/news` 가 덮는다.

**관련 기능** — 목록의 상태 열이 이 폼의 상태와 공개 일시로 계산된다([목록 문서](index.md) §4.4).

**관련 API** — 등록 1건 · 수정 1건 · (수정 진입 시) 상세 조회 1건. **백엔드가 없다** — `apps/admin/src/pages/content/news/data-source.ts:130`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/content/news/NewsFormPage.tsx` (검증 `validation.ts` · 규칙 `types.ts` · 예약 판정 `apps/admin/src/shared/domain/publish-schedule.ts`)

---

**⚠ 지금 코드에서 확인되는 사실** — 이 폼에는 **첨부 표면이 없다.** 예전에는 미디어 라이브러리에서 자산을 골라 붙였으나, 그 화면이 IA 에서 빠지면서 목록을 읽을 수 없게 됐고 표면만 남기면 화면이 영원히 '라이브러리를 불러올 수 없습니다'를 말하게 되므로 표면과 모델을 함께 걷었다 — `apps/admin/src/pages/content/news/NewsFormPage.tsx:5-8`.

### 4.4 저장 충돌

| 항목 | 내용 |
|---|---|
| 기능 목적 | 내가 폼을 여는 사이 다른 관리자가 같은 글을 먼저 바꿨을 때 **덮어쓰지 않는다** |
| 실행 조건 | 저장이 충돌(409/412)로 거절됐을 때 — `apps/admin/src/shared/crud/useCrudForm.ts:191-206` |
| 사용자 동작 | 다이얼로그에서 '최신 내용 불러오기' 또는 '이어서 편집' |
| 시스템 처리 | 성공 토스트도, 목록 이동도 일어나지 않는다(유령 저장 금지). **입력은 그대로 살아 있다** |
| 데이터 처리 | '최신 내용 불러오기'는 서버 최신본으로 폼을 덮는다. '이어서 편집'은 아무것도 바꾸지 않는다 |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 불러오기를 고르면 상세를 다시 조회한다 |
| 성공 처리 | 다이얼로그가 닫히고 고른 쪽에 따라 폼이 바뀌거나 그대로 남는다. **토스트를 띄우지 않는다** |
| 실패 처리 | 재조회가 실패하면 폼의 조회 실패 표시가 대신 뜬다 |
| 예외 처리 | 서버가 사유 문장을 주면 그것이 본문 앞에 온다. 없으면 기본 문장을 쓴다 — `apps/admin/src/shared/crud/useCrudForm.ts:197`. 픽스처가 주는 사유는 `다른 사용자가 먼저 삭제한 항목이에요.` — `apps/admin/src/shared/crud/crud.ts:257` |
| 화면 변경 사항 | 다이얼로그 · 입력영역 |
| 후속 동작 | 폼 값 교체 또는 계속 편집 |
| 로그 기록 여부 | 확인하지 못했다 |

**확인 다이얼로그** — `DLG-COMMON-FORM-CONFLICT`([공통 문서](../../_common/index.md) §4.5).

### 4.5 저장하지 않은 채 떠나기

공지 폼의 §4.4 와 동작이 같다(`DLG-COMMON-LEAVE-FORM`). 문구만 다르다: `뉴스에 저장하지 않은 변경 사항이 있어요. 이 화면을 벗어나면 입력한 내용이 사라져요.` — `apps/admin/src/pages/content/news/NewsFormPage.tsx:30-31`.
