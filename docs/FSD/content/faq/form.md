# FAQ 등록 · 수정

> 이 화면은 [공지 등록·수정](../notices/form.md)(`SCR-CONTENT-NOTICES-FORM`)과 **같은 골격**이다 —
> 한 화면이 등록과 수정을 겸하고, 검증의 정본은 zod 스키마이며, 저장 실패는 카드 맨 위 배너,
> 미저장 이탈은 공통 가드가 막는다. **공통 폼 컨트롤러를 쓰지 않는 것도 같다** — 그래서 저장 충돌
> 다이얼로그·422 인라인 매핑·권한 문·오류 참조 코드가 없다. 아래 각 절은 **다른 것만** 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | FAQ 등록 · 수정 |
| 화면 ID | `SCR-CONTENT-FAQ-FORM` |
| 메뉴 경로 | 콘텐츠 관리 > FAQ > FAQ 등록 (또는 상세 > 수정) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — FAQ 한 건의 질문·카테고리·정렬 순서·노출 여부·답변을 정한다.

**업무 배경** — FAQ 는 카테고리 안에서 순서가 곧 노출 우선순위다. 등록할 때 순서를 매번 계산하게 하면 실수가 잦으므로, 새 등록에서는 **현재 최대 + 1** 을 미리 채워 준다.

**화면 설명** — 카드 하나에 질문 · (카테고리 · 정렬 순서) 한 줄 · 노출 체크박스 · 답변이 놓인다. 맨 아래 취소·제출.

**주요 사용자** — 최상위 관리자 · 콘텐츠를 다루는 운영자.

**사용 시나리오**

1. 운영자가 '결제' 카테고리에 새 FAQ 를 쓴다 → 정렬 순서가 자동으로 채워져 있어 그대로 등록한다.
2. 답변의 안내 문구가 바뀌어 수정한다 → 저장하면 그 FAQ 의 상세로 돌아간다.
3. 아직 공개하고 싶지 않아 '사용자 화면에 노출' 체크를 끄고 등록한다.

**선행 조건** — 로그인 · 콘텐츠 관리 플랜 모듈. 카테고리가 **한 개 이상 있어야 등록할 수 있다**(카테고리는 필수인데 선택지가 서버에서 온다 — 없으면 고를 것이 없다). 카테고리는 [FAQ 카테고리 관리](pop-manage-categories.md) 팝업이 만든다.

**후행 처리** — 등록 성공은 목록으로, 수정 성공은 **그 FAQ 의 상세**로 간다. 둘 다 히스토리를 덮어쓴다 — `apps/admin/src/pages/content/faq/FaqFormPage.tsx:173,186`.

**관련 화면** — [FAQ 목록](index.md) · [FAQ 상세](detail.md) · [FAQ 카테고리 관리](pop-manage-categories.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자리가 없다. 권한 리소스는 잎 `/content/faq` 가 덮는다.

**관련 기능** — 목록의 노출 토글과 정렬 드래그가 같은 두 값을 바꾼다. 즉 **정렬 순서와 노출 여부에는 편집 경로가 둘**이다.

**관련 API** — 등록 1건 · 수정 1건 · 상세 조회 1건 · 카테고리 목록 조회 1건 · 다음 정렬 순서 조회 1건. **백엔드가 없다** — `apps/admin/src/pages/content/faq/data-source.ts:292`(등록) · `:299`(수정) · `:164`(상세) · `:141`(카테고리) · `:285`(다음 순서).

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/content/faq/FaqFormPage.tsx` (검증 `validation.ts` · 상수 `types.ts`)

---

**⚠ 지금 코드에서 확인되는 사실** — 저장 요청은 픽스처를 바꾸지 않는다(`apps/admin/src/pages/content/faq/data-source.ts:292-305`). 공통 폼 껍데기를 쓰지 않아 권한 문·충돌 다이얼로그·동기 중복 제출 잠금이 없다(공지 폼과 같은 이유·같은 범위).

### 4.3 정렬 순서 자동 채움 (이 화면 고유)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 새 FAQ 의 정렬 순서를 운영자가 계산하지 않게 한다 |
| 실행 조건 | **등록 진입일 때만** — 수정에서는 부르지 않는다 — `apps/admin/src/pages/content/faq/FaqFormPage.tsx:136,139` |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 현재 최대 정렬 순서 + 1 을 조회해 칸에 넣는다. **한 번만 넣는다** — 그 뒤 사용자가 고친 값을 덮지 않는다 |
| 데이터 처리 | 없음 — 읽기만 한다 |
| Validation | 채워진 값도 §6 의 정렬 순서 규칙을 그대로 지나야 한다 |
| API 호출 여부 | Y — 다음 정렬 순서 조회 1건 |

| 실패 처리 | ⚠ **아무것도 알리지 않는다.** 조회가 실패하면 칸이 빈 채로 남고, 제출하면 §6 의 `정렬 순서를 입력하세요.` 가 걸린다 |
| 예외 처리 | 응답이 늦게 도착해도 사용자가 이미 손댄 값을 덮지 않는다 |
| 화면 변경 사항 | 정렬 순서 칸 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

## 5. 이벤트 정의

공지 폼의 이벤트 표를 따른다. **다른 것 둘**:

| 이벤트 | 발생 조건 | 처리 내용 | 대상 컴포넌트 | 결과 화면 |
|---|---|---|---|---|
| 다음 정렬 순서 도착 | 등록 진입에서 조회가 끝남 | 정렬 순서 칸을 **한 번만** 채운다 | 정렬 순서 | 값이 채워진 폼 |
| 노출 체크 토글 | 체크박스 클릭 | 값 토글(폼이 dirty 가 된다) | 노출 체크박스 | 변화 없음 |

공지 폼의 '상태 변경 → 게시일 잠금' 이벤트는 이 화면에 **없다**(상태·게시일 축이 없다).

| 세션 만료 | 자기 문구가 없다 | 앱이 로그인 화면으로 보낸다 | 재로그인(입력은 사라진다) |
| 데이터 충돌 | ⚠ 충돌 처리가 없다 — 409/412 도 일반 저장 실패 문장으로 뭉개진다 | 배너 하나 | 목록에서 최신 상태 확인 |
| 중복 데이터 | 해당 없음 — 질문·정렬 순서 중복을 막는 규칙이 없다. **같은 정렬 순서를 두 FAQ 가 가질 수 있다** | — | — |
| 파일 업로드 실패 | 해당 없음 — 파일 입력이 없다 | — | — |

## 8. 화면 상태(State)

공지 폼의 §8 과 같다. 다른 것만 적는다.

| 상태 | 화면 표시 | 버튼 활성화 여부 | Loading 여부 | 사용자 안내 메시지 |
|---|---|---|---|---|
| 데이터 없음(등록 진입) | 질문·답변이 비어 있고 카테고리는 고르지 않은 상태, 노출은 **켜짐**, 정렬 순서는 자동으로 채워진다 | 취소 ○ · 등록 ○ | N | 정렬 순서 힌트 `작을수록 위에 노출돼요.` |
| 카테고리 목록이 아직 안 옴 | select 에 `카테고리를 선택하세요` 한 줄만 있다 | 취소 ○ · 제출 ○ | N | 없음 |
| 조회 실패(수정) | 폼이 사라지고 배너 하나 | 목록으로 ○ | N | `FAQ 를 불러오지 못했어요. ` |
| 읽기 전용(권한) | ⚠ 나타나지 않는다 — 권한 판정이 없다(§9) | — | N | — |
