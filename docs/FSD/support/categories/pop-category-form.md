# 문의 유형 추가·수정 (팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 문의 유형 추가 / 문의 유형 수정 |
| 화면 ID | `POP-SUPPORT-CATEGORIES-CATEGORY-FORM` |
| 메뉴 경로 | 고객센터 > 문의 유형 > (추가 · 수정) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 문의 유형 한 건의 **이름**과 **사용여부**를 받아 만들거나 고친다.

**업무 배경** — 사용여부는 삭제를 대신하는 축이다. 참조된 유형은 지울 수 없으므로, 더는 쓰지 않는 유형은 **꺼서 신규 문의·템플릿의 선택지에서만 숨긴다**(기존 참조는 그대로 유지된다) — `apps/admin/src/pages/support/_shared/domain.ts:25-26`.

**화면 설명** — 입력 두 칸(이름 · 사용여부 토글)과 버튼 둘(취소 · 추가/저장)이 전부다. 등록과 수정을 한 팝업이 겸한다.

**주요 사용자** — 최상위 관리자 · 고객센터 운영자.

**사용 시나리오**

1. 목록의 `유형 추가` 로 열어 이름을 넣고 저장한다 → 사용여부는 기본이 켜짐이다.
2. 더는 쓰지 않는 유형을 연필로 열어 **사용여부만 끄고** 저장한다.
3. 이름을 고쳐 저장한다 → 목록의 그 행과 티켓·템플릿의 유형 표기에 반영된다.

**선행 조건** — 부모 화면(`SCR-SUPPORT-CATEGORIES`)이 열려 있고, 그 화면의 추가·수정 컨트롤을 눌렀을 때.

**후행 처리** — 저장하면 팝업이 닫히고 부모 목록이 갱신되며 부모가 성공 토스트를 띄운다.

**관련 화면** — [문의 유형 목록](index.md) `SCR-SUPPORT-CATEGORIES`(부모) · [1:1 문의 목록](../tickets/index.md)(유형 필터가 이 값을 읽는다) · [문의 답변](../replies/index.md)(유형 태그가 이 값을 읽는다) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — 껍데기(포커스 트랩·Esc·딤·포커스 복귀)는 공통 모달을 재사용한다. 쓰기 배선은 공통 CRUD 프레임워크의 저수준 훅이다 — `apps/admin/src/pages/support/categories/components/CategoryFormModal.tsx:3-4`.

**관련 API** — 유형 등록 1건 또는 갱신 1건. **백엔드가 없다** — 어댑터가 픽스처 위에 배선되어 있다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/support/categories/components/CategoryFormModal.tsx` (검증 `../validation.ts` · 도메인 상수 `../../_shared/domain.ts` · 저장소 `../../_shared/store.ts`)

## 4. 기능 명세

### 4.1 유형 등록

| 항목 | 내용 |
|---|---|
| 기능 목적 | 새 문의 유형을 만든다 |
| 실행 조건 | 부모가 `유형 추가` 로 팝업을 열었을 때 |
| 사용자 동작 | 이름을 넣고 `추가` 클릭 |
| 시스템 처리 | 이름의 앞뒤 공백을 다듬어 등록 1건을 보낸다 |
| 데이터 처리 | 유형 1건이 생긴다. **사용여부 기본값은 켜짐이다** |
| Validation | §6 전부 |
| API 호출 여부 | Y — 등록 1건 |
| 성공 처리 | 팝업이 닫히고 **부모가** 토스트를 띄운다: `'{유형명}' 유형을 추가했어요.` — `apps/admin/src/pages/support/categories/CategoriesPage.tsx:251` |
| 실패 처리 | 팝업이 닫히지 않고 안쪽 danger 배너로 말한다. 입력은 그대로 남는다 |

| 후속 동작 | 티켓 목록의 유형 필터와 템플릿의 유형 태그 셀렉트에 나타난다 |

**이 팝업 자체에는 권한 축이 없다** — 부모 기능의 권한을 물려받는 것이 규약이지만, **부모도 쓰기 권한으로 컨트롤을 가리지 않는다**(§ [문의 유형 목록](index.md) §9). 그래서 읽기 권한만으로 이 팝업이 열리고 저장까지 시도된다.

**권한이 없을 때 화면이 어떻게 되는가** — 읽기 권한이 없으면 부모 화면 자체가 권한 화면이라 이 팝업에 닿을 수 없다. 쓰기 권한만 없으면 팝업이 열리되 저장이 서버에서 거절된다.
