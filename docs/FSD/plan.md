## 1. 시작 조건 — 지금 쓰면 안 되는 이유

운영자 확정: *"기능 명세, 비기능 명세는 **모든 작업이 끝나면** 작성한다."*

| 진행 중인 작업 | 명세의 어디를 낡게 만드는가 |
|---|---|
| **사용자 문구 전면 정리** | §3 Label·Placeholder · §6 오류 메시지 · §7 사용자 메시지 · §8 사용자 안내 메시지 — **네 절이 전부 문구를 인용한다** |
| 지도 항목 제거 · 주소 검색 확장 | `SCR-SETTINGS-API-KEYS` 의 연동 카탈로그 · `POP-COMPANY-DIRECTIONS-ADDRESS-SEARCH` |

**순서는 워딩 → 명세다.** 반대로 하면 첫 묶음을 다 쓴 뒤 전부 다시 고쳐야 한다.

**지금 해도 되는 것**(개별 화면 내용에 의존하지 않는다): 인벤토리 · 템플릿 · README · ID 규약 · 이 계획 · 도구 판정.
**이미 끝났다** — 이 문서가 그 마지막 항목이다.

## 3. 순서와 그 근거

```
B0 → B1 → B2 → B3 → B4 → B5 → B6 → B7 → B8 → B9 → B10 → B11 → B12 → B13 → B14 → B15 → B16
```

| 자리 | 왜 |
|---|---|
| **B0 이 맨 앞** | 공통 팝업·다이얼로그·토스트를 먼저 못박지 않으면, 각 화면 문서가 같은 것을 저마다 다르게 적는다. 그 중복이 곧 옛 명세를 무겁게 만든 것과 같은 모양이다 |
| B1–B4 는 **사이드바 순서** | 읽는 사람이 메뉴를 훑는 순서와 같다. 진도를 눈으로 확인할 수 있다 |
| B5–B10 이 가운데 | 난도가 높은 묶음을 **가장 힘이 남아 있을 때** 배치한다. 특히 B8(영업 사슬)은 다섯 화면을 한 사람이 이어서 써야 한다 |
| B11–B14 가 뒤 | 형태가 단순하거나 껍데기를 공유해 빠르다 |
| B15 는 뒤 | 다른 화면에 영향을 주는 설정이라 **앞 묶음이 다 있어야** 영향 범위를 적을 수 있다 |
| **B16 이 맨 마지막** | 연동 카탈로그가 편집 중이다. 지금 쓰면 목록·탭·자격증명 칸이 곧 어긋난다 |

**한 묶음의 완료 판정** — 그 묶음의 FSD 가 전부 있고, 각 문서의 자기 점검 항목이 전부 체크되며,
[`inventory.md`](inventory.md) 의 팝업·확인 다이얼로그·토스트 건수와 문서에 적힌 수가 **일치**한다.

## 4. 옛 `specs/` 는 어떻게 되는가

`specs/**` 218개 파일은 **이미 삭제됐다**(운영자 지시, 소실 범위를 확인한 뒤 확정).
되살리지 않는다. 참조가 필요하면 `git show HEAD:specs/<경로>` 로 읽을 수 있으나
**재작성의 기준은 문서가 아니라 코드다** — 옛 문장을 옮겨 적으면 걷어낸 누적 기록이 그대로 돌아온다.

**살려 둔 것 3건** — 화면 명세가 아니라 **축 규약**이라 성격이 다르다. `docs/reference/` 로 옮겼다.

| [`../reference/quality-bar.md`](../reference/quality-bar.md) | 9차원 102요구사항. 비기능 명세서의 판정 기준이다 |

## 5. `tools/test-coverage` — 새 양식을 읽지 못한다

`pnpm coverage:check` 는 5축 중 **축 4** 를 FS 문서에서 뽑는다. 그 입력이 사라졌고, 새 양식은 모양이 다르다.

### 5.1 지금 상태 (실측)

| 사실 | 값 |
|---|---|
| 직전 기준선 | 축 4 커버 **137칸** |
| `specs/` 삭제 직후 첫 실행 | 커버 **0칸** → `커버리지 후퇴 137칸` **BLOCKER** |
| 그 실행이 리포트를 덮어썼다 | `reports/test-coverage/all.json` 의 축 4 `covered: 0` |
| 다음 실행 | 기준선을 그 리포트에서 읽으므로 **0 → 후퇴 없음 → exit 0** |

**즉 BLOCKER 는 이미 사라졌다** — 재실행으로 확인했다(`축 4 fs-exception-axes 0/0 PASS`, blocker 0건, major 2건).
**그러나 래칫 바닥이 137 → 0 으로 세탁됐다.** 도구 자신의 자기 감사가 예고한 경로다:

> `축 4 래칫의 한계: 기준선은 reports/test-coverage/ 의 직전 리포트 파일에서 읽는다.
> 리포트가 삭제되면 기준선이 0으로 떨어져 후퇴가 은폐된다.`
> — `tools/test-coverage/src/index.ts:89`

⚠ **`reports/test-coverage/all.json` 의 변경분을 커밋하면 137칸의 보호가 영구히 사라진다.**
커밋 전에 판단이 필요하다.

### 5.2 도구가 요구하는 것 vs 새 양식

| 도구가 요구하는 것 | 근거 | 새 양식은 |
|---|---|---|
| 원천 디렉터리 `specs/` | `src/thresholds.ts:115` `SPECS_DIR = 'specs'` | **`docs/FSD/`** — 경로가 다르다 |
| 파일명 `FS-\d{3}….md` | `src/lib/specs.ts:118` | **`index.md`·`detail.md`·`form.md`·`pop-*.md`·`tab-*.md`** — 하나도 안 맞는다 |
| `## 4. 예외 명세` 절 | `src/lib/specs.ts:125` | **없다.** 예외는 **§7 예외 처리** 다 |
| 요소 × **7축** 격자 (빈 상태·로딩·실패·유효성·권한없음·경합·대량) | `src/thresholds.ts:181-189` | §7 은 **상황 12종 × 4열**(사용자 메시지·시스템 처리·복구 방법)이다. **축이 다르고 차원도 다르다** |
| 첫 열이 `FS-\d{3}-EL-\d{3}` | `src/lib/specs.ts:149` | §3 은 `No` 열(1,2,3…)이고 요소 좌표가 없다 |
| 테스트 이름에 요소 토큰 | `src/axes/fs-exceptions.ts:8-10` | e2e 4파일에 **`FS-001`~`FS-004` 토큰 107개**가 박혀 있다. 가리킬 문서가 사라졌다 |

**결론: 축 4 는 새 양식에서 지금 코드 그대로는 작동할 수 없다.** 규칙 몇 줄이 아니라 **모델이 다르다.**

### 5.3 무엇을 고쳐야 하는가 — 세 갈래 (판정 필요)

| 안 | 무엇을 하나 | 비용 | 무엇을 잃나 |
|---|---|---|---|
| **A. 새 양식으로 재조준** | `SPECS_DIR` → `docs/FSD` · 파일 필터 → `(index\|detail\|form\|pop-.*\|tab-.*\|[a-z-]+)\.md` · 절 탐색 `## 4. 예외 명세` → `## 7. 예외 처리` · `EXCEPTION_AXES` 7축 → **§7 의 상황 12종** · 대조 키 `FS-nnn-EL-nnn` → **화면 ID(`SCR-…`) + 상황명** · e2e 4파일의 테스트 이름 107개를 새 ID 로 | 도구 3파일(`thresholds.ts` · `lib/specs.ts` · `axes/fs-exceptions.ts`) + e2e 4파일 | 없음. 격자는 (화면 × 12상황)이 되어 오히려 커진다 |
| B. 축 4 폐지 | 축 1·2·3·5 만 남긴다 | 작다 | **레지스트리 `blockCondition` 이 'FS의 예외 7축' 을 명시**하므로 ADR 판정이 필요하다. 명세와 테스트를 잇는 유일한 축이 사라진다 |
| C. 방치 | 아무것도 안 한다 | 0 | 축 4가 영구히 `0/0 PASS` 로 남는다. **초록불이 아무것도 보증하지 않는다** — 도구가 스스로 경계한 '공허 통과' 다 |

**권고: A.** 새 양식의 §7 은 오히려 도구가 읽기 **쉬운** 표다 — 상황 12개가 고정 행이라
축 별칭 표를 그대로 갈아 끼우면 되고, 빈 행을 강제하는 규율(`해당 없음 — 사유`)이 이미 템플릿에 있다.

**A 를 고를 때의 마이그레이션 순서** (한 번에 바꾸면 래칫이 다시 세탁된다):

1. 도구를 **두 형식 모두** 읽도록 고친다(옛 토큰 + 새 ID 병행 인식).
2. 묶음 B1 의 문서를 쓰면서 그 화면의 e2e 테스트 이름을 새 ID 로 바꾼다 — **한 묶음씩**.
3. 커버 칸 수가 0 에서 단조 증가하는 것을 확인한 뒤 옛 토큰 인식을 뺀다.

**이 절의 작업은 이 단계의 몫이 아니다.** 무엇을 고쳐야 하는지만 여기 적었다.
