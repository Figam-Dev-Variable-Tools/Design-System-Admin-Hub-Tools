# 주문 취소 (팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 주문 취소 |
| 화면 ID | `POP-ORDERS-DETAIL-CANCEL` |
| 메뉴 경로 | 주문 관리 > 주문 > 주문 상세 > `주문 취소` |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 출고 전 주문을 취소하면서 **왜 취소하는지**를 반드시 기록한다.

**업무 배경** — 취소는 되돌릴 수 없고, 나중에 '왜 취소됐나' 를 묻는 사람은 반드시 나타난다(고객·정산·CS). 사유 없는 취소는 그 질문에 답할 수 없는 기록이다 — `apps/admin/src/pages/orders/validation.ts:11-12`. 그래서 이 조작만은 확인 다이얼로그(제목·본문·버튼 둘)로 처리하지 않고 **입력을 받는 팝업**이 된다.

**화면 설명** — 안내 한 문단 + (실패했다면) 배너 + 취소 사유 입력 3줄 + 버튼 둘(`닫기` · `주문 취소`)이 전부다.

**주요 사용자** — 최상위 관리자 · 운영자(주문 수정 권한 보유자).

**사용 시나리오**

1. 고객이 전화로 취소를 요청한다 → 상세에서 `주문 취소` → 사유에 '고객 요청 — 다른 색상으로 다시 주문 예정' 을 적고 `주문 취소` → 취소 배지가 붙고 차감돼 있던 재고가 되돌아온다.
2. 사유를 비운 채 `주문 취소` 를 누른다 → 요청이 나가지 않고 입력 아래에 `취소 사유를 입력하세요.` 가 뜬다.

**선행 조건** — 주문 상세가 열려 있고, 그 주문이 **취소 가능**해야 한다(아직 취소되지 않았고 배송이 시작되지 않았다). 취소가 막힌 주문에는 이 팝업을 여는 버튼 자체가 없다.

**후행 처리** — 취소 시각·사유가 주문에 저장되고 처리 이력에 `주문 취소` 한 줄이 붙는다. 이미 차감된 재고가 있으면 **기록된 출고 이동을 뒤집어** 복원된다. 그 뒤 이 주문은 상태를 바꿀 수 없다.

**관련 화면** — `SCR-ORDERS-DETAIL`(부모) · `SCR-ORDERS`(취소 필터 항목에 잡힌다) · `SCR-ORDERS-CLAIMS`(배송이 시작된 뒤의 되돌림은 이쪽이다).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — 클레임의 '취소' 유형은 **주문의 출고 여부를 읽어** 자기 가드를 만든다. 클레임 화면은 재고를 되돌리지 않는다 — 복원의 주체는 이 팝업이 만드는 주문 취소다 — `apps/admin/src/pages/orders/claims/types.ts:336-346`.

**관련 API** — 주문 갱신 1건(취소 사실 + 재고 복원). 자체 엔드포인트를 갖지 않고 부모의 저장 통로를 쓴다. **백엔드가 없다** — `apps/admin/src/pages/orders/data-source.ts:61-64`.

**참고 문서** — [주문 상세](detail.md) · [공통 확인 다이얼로그 · 토스트](../_common/index.md) · [ID 규약](../id-convention.md)

**구현 파일** — `apps/admin/src/pages/orders/OrderDetailPage.tsx:572-625` (부모 화면 안의 인라인 모달) · 검증 `apps/admin/src/pages/orders/validation.ts` · 취소 규칙 `apps/admin/src/shared/domain/order.ts:319-394`

| 순서 | 언제 막히는가 | 거부 사유 | 출처 |
