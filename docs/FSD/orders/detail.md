# 주문 상세 · 처리

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 주문 상세 |
| 화면 ID | `SCR-ORDERS-DETAIL` |
| 메뉴 경로 | 주문 관리 > 주문 > (행 클릭) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 한 주문의 전부(주문자·수령인·품목·금액·이력)를 보고, 그 주문을 다음 단계로 옮기거나 입금을 확인하거나 취소한다.

**업무 배경** — 목록의 일괄 처리는 '같은 일을 여러 건에' 하는 도구다. 그런데 실제 CS 는 한 건씩 온다 — 주소가 이상해 보류한 주문, 가상계좌 입금이 방금 들어온 주문, 고객이 전화로 취소를 요청한 주문. 그 셋은 각각 다른 조작이고 되돌릴 수 없으며, 왜 그렇게 했는지가 나중에 반드시 질문된다. 이 화면은 그 조작들과 그 근거(처리 이력·메모)를 한자리에 둔다.

**화면 설명** — 카드 여섯 장이 세로로 쌓인다: 요약(상태 배지 + 액션) · 주문자/수령인 · 주문 품목 · 결제 정보 · 처리 이력 · 처리 메모. 등록 폼이 없는 **처리 워크플로 화면**이라 '저장' 버튼은 메모 하나에만 있다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. 가상계좌 주문의 입금이 확인됐다 → `입금 확인` → 확인 다이얼로그 → 이력에 '입금 확인' 이 붙고, 차감 시점 설정이 '입금 확인 시' 면 재고가 이 순간 빠진다.
2. 배송준비중 주문을 다음 단계로 민다 → `배송대기 처리` → 확인 → 상태 배지와 이력이 함께 바뀐다.
3. 고객이 취소를 요청한다 → `주문 취소` → 사유를 적는 팝업 → 취소 사실이 얹히고 차감돼 있던 재고가 자동 복원된다.
4. 배송이 이미 시작된 주문에서 취소를 찾는다 → 버튼이 없고 그 자리에 `배송이 시작된 주문은 취소할 수 없어요. 교환/반품으로 접수해 주세요.` 가 있다.

**선행 조건** — 로그인 · `/orders` 리소스 조회 권한 · 목록에서 고른(또는 주소로 지목한) 주문번호가 실재해야 한다.

**후행 처리** — 상태·입금·취소는 주문 원장과 처리 이력을 함께 바꾸고, 조건이 맞으면 SKU 재고가 증감한다. 클레임 상세(`SCR-ORDERS-CLAIMS-DETAIL`)의 '취소 가능' 판정은 이 주문의 상태를 읽는다.

**관련 화면** — `SCR-ORDERS`(목록 · '목록으로') · `POP-ORDERS-DETAIL-CANCEL`(주문 취소 팝업) · `SCR-ORDERS-SHIPMENTS` · `SCR-ORDERS-CLAIMS-DETAIL`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. `주문 관리 > 주문` 아래의 상세 화면이다.

**관련 기능** — 재고 차감 시점 설정(`stockDeductAt`, 기본값 **입금 확인 시**)이 '입금 확인' 이 무엇을 하는지 바꾼다. 배송 처리 화면의 발송처리가 이 주문의 `출고` 수량을 올린다.

**관련 API** — 상세 조회 1건 · 주문 갱신 1건(상태·입금·취소·메모가 모두 같은 통로를 쓴다). **백엔드가 없다** — `apps/admin/src/pages/orders/data-source.ts:61-64`. 서버가 맡을 일: 주문 갱신과 SKU 재고 증감을 **한 트랜잭션**으로 처리하고, 재고가 모자라면 422 로 거절한다. 멱등키는 `stockAppliedAt`·`stockRestoredAt` 이다.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [주문 비기능 명세서](../../NFRS/orders.md)

**구현 파일** — `apps/admin/src/pages/orders/OrderDetailPage.tsx` (상태 기계·금액 `apps/admin/src/shared/domain/order.ts` · 검증 `validation.ts` · 설정 `_shared/store.ts`)

| 경우 | 제목 | 본문 | 확인 버튼 |
|---|---|---|---|
| 입금 확인 | `입금 확인` | `입금을 확인 처리해요. 재고 차감 시점이 '입금 확인 시'이면 이 시점에 재고가 빠지며, 되돌릴 수 없어요.` — `apps/admin/src/pages/orders/OrderDetailPage.tsx:570` | `입금 확인` |

### 4.6 이 화면의 전이·취소 가드

**전이 가드** — 목록의 §4.4 와 **같은 술어**다. 이 화면은 그 결과를 두 가지로 쓴다: 갈 수 있는 단계는 버튼이 되고, 하나도 없으면 그 이유가 문장으로 남는다 — `OrderDetailPage.tsx:314-318`.

| 상황 | 화면이 보이는 것 |
|---|---|
| 갈 수 있는 단계가 여럿 | 그 개수만큼 버튼. 첫 버튼이 기본 동작(primary), 나머지는 건너뛰는 예외 경로(secondary) |
| 미입금 상태 | 첫 버튼도 secondary 가 되고 `입금 확인` 이 primary 를 가져간다 |
| 갈 곳이 없음 | 버튼 대신 사유 한 줄 — 취소면 `취소된 주문은 상태를 바꿀 수 없어요.`, 구매확정이면 `구매확정된 주문은 더 이상 진행할 단계가 없어요.` |

**취소 가드** — `shared/domain/order.ts:320-329`.

| 순서 | 언제 막히는가 | 거부 사유 |
|---|---|---|
| ① | 이미 취소됨 | `이미 취소된 주문이에요.` |
| ② | 배송중·배송완료·구매확정(창고를 떠났다) | `배송이 시작된 주문은 취소할 수 없어요. 교환/반품으로 접수해 주세요.` |

**취소와 반품을 가르는 선은 '배송중' 이다** — 그 이후에 되돌아오는 것은 취소가 아니라 반품이고, 처리 화면도 창구도 다르다(`SCR-ORDERS-CLAIMS`).
