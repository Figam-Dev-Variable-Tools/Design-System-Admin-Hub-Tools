# 배송 처리

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 배송 처리 |
| 화면 ID | `SCR-ORDERS-SHIPMENTS` |
| 메뉴 경로 | 주문 관리 > 배송 처리 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 오늘 무엇을 내보낼 것인지 보고, 준비중으로 옮기고, 송장을 붙이고, 발송처리한다.

**업무 배경** — 주문 목록이 '무엇이 어디에 고여 있는가' 를 보는 곳이라면 여기는 **출고 작업대**다. 국내 운영에는 '운송장은 출력됐지만 아직 인수인계 전' 인 구간이 반드시 있고, 그 구간을 이름으로 부르지 못하면 운영자는 나가지도 않은 물건을 '배송중' 이라고 고객에게 말하게 된다. 그래서 송장 입력(→ 배송대기)과 발송처리(→ 배송중)는 **다른 조작**이다 — `apps/admin/src/shared/domain/shipment.ts:20-22`.

**화면 설명** — 좌측은 작업 상태 필터 네 갈래(발송대기 · 배송대기 · 배송중 · 배송완료)와 전체. 우측은 검색 · 선택 바(일괄 버튼 3개) · 표다. **표의 행은 배송 건이 아니라 주문**이다 — 배송 건을 행으로 삼으면 송장이 아직 없는 주문이 목록에서 사라진다(정작 할 일이 가장 많은 주문이).

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. '발송대기 8' 을 누른다 → 전부 선택 → `송장 입력 (8)` → 팝업에서 주문마다 택배사·송장번호를 채우고 등록 → 8건이 배송대기로 넘어간다.
2. 배송대기 건을 골라 `발송처리` → 전 품목이 나간 주문만 배송중이 되고, 일부만 나간 주문은 발송대기에 남는다(부분발송 배지가 붙는다).
3. 택배사가 송장번호로 사고를 문의한다 → 검색창에 송장번호를 넣어 그 주문을 찾는다.

**선행 조건** — 로그인 · `/orders/shipments` 리소스 조회 권한. 송장을 붙이려면 **배송 정책 화면에 택배사가 1곳 이상 등록**되어 있어야 한다(`SCR-PRODUCTS-SHIPPING`).

**후행 처리** — 송장 등록은 배송 건(송장 1장)을 만들고, 전 품목이 덮이면 주문을 배송대기로 민다. 발송처리는 배송 건을 배송중으로 옮기고 주문 품목의 **출고 수량**을 올린다. 그 결과가 주문 목록·주문 상세의 '부분배송' 배지와 '출고' 칸에 그대로 나타난다.

**관련 화면** — `SCR-ORDERS`(같은 주문의 상태 축) · `SCR-ORDERS-DETAIL`(행 클릭 목적지) · `POP-ORDERS-SHIPMENTS-INVOICE-BULK`(송장 일괄 입력) · `SCR-PRODUCTS-SHIPPING`(택배사 원장). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `주문 관리 > 배송 처리`. 조건부 노출이 아니다.

**관련 기능** — 택배사 목록의 정본은 배송 정책 화면이고, 이 화면은 **조회기로만** 그것을 읽는다(두 화면이 서로를 참조하지 않는다) — `shared/domain/shipment.ts:57-91`. 반대로 택배사 삭제를 막는 '사용 건수' 는 이 화면의 배송 원장이 답한다.

**관련 API** — 조회 2건(주문 목록 · 배송 건 목록) · 배송 건 생성 N건 · 배송 건 갱신 N건 · 주문 갱신 N건. **백엔드가 없다** — `apps/admin/src/pages/orders/shipments/data-source.ts` 의 `TODO(backend)` 와 `apps/admin/src/pages/orders/data-source.ts:61-64` 가 경계다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [배송 처리 비기능 명세서](../../../NFRS/orders/shipments.md)

**구현 파일** — `apps/admin/src/pages/orders/shipments/ShipmentListPage.tsx` (표 `components/ShipmentTable.tsx` · 행 모델·필터 `types.ts` · 배송 도메인 `apps/admin/src/shared/domain/shipment.ts` · 어댑터 `data-source.ts`)

| 배송완료 | 배송 건이 전부 `배송완료` | 끝났다 |
| 배송중 | 배송 건 중 하나라도 창고를 떠났다 | 나가는 중 |
| 배송대기 | 위 어디에도 해당하지 않는다(송장은 다 붙었고 아무것도 안 나갔다) | 인수인계 전 |

**부분발송이 '배송중' 이 아닌 이유** — 이 목록의 건수는 곧 오늘 처리할 양이어야 하고, 반쯤 끝난 일을 완료 쪽에 놓으면 그 숫자가 거짓말이 된다 — `types.ts:82-84`.

**송장번호를 검색 대상에 넣은 이유** — 고객이 부르는 것은 주문번호이지만 택배사가 부르는 것은 송장번호다. 배송 사고 문의는 대개 후자로 들어온다 — `types.ts:165-168`.

### 4.5 배송 축의 전이 규칙

**배송 건 하나의 전이** — `shared/domain/shipment.ts:435-473`.

| 순서 | 언제 막히는가 | 거부 사유 |
|---|---|---|
| ① | 목적 상태가 지금과 같거나 뒤에 있다 | `배송 상태는 되돌릴 수 없어요.` |
| ② | 배송중으로 가려는데 송장번호가 유효하지 않다 | `송장번호가 없어 발송처리할 수 없어요.` |

배송 건의 상태 순서는 `배송대기 → 배송중 → 배송완료` 이며 **주문 상태와 같은 낱말**을 쓴다 — 같은 사건에 두 어휘가 생기면 운영자는 둘이 같은 것인지 확인할 방법이 없다 — `shipment.ts:166-186`.

**주문을 배송 축의 사유로 밀 때** — `shipment.ts:414-431`. **주문 축을 먼저 묻는다.**

| 순서 | 판정 | 거부 사유 |
|---|---|---|
| ① | 주문 전이 가드 4종([주문 목록](../index.md) §4.4) | 그 사유 문장 그대로 |
| ② | 배송대기로 밀 때 — 송장이 전 품목을 덮었는가 | `송장이 붙지 않은 품목이 있어 배송대기로 넘길 수 없어요 ({덮인 수량}/{전체 수량}).` |
| ② | 배송중으로 밀 때 — 실제로 전 품목이 나갔는가 | `전 품목이 발송되어야 배송중으로 넘어가요 ({나간 수량}/{전체 수량}).` |

**커버리지는 SKU 단위로 센다** — 송장은 주문이 아니라 **품목**에 붙는다. 그래서 '이 주문이 다 나갔는가' 는 배송 건 개수로 답할 수 없고 수량을 맞춰 봐야 한다. 한 품목에 과다 입력된 수량이 다른 품목까지 '발송됨' 으로 만들지 않도록 배정은 품목 수량을 넘지 않는다 — `shipment.ts:299-328`.

**부분 발송의 경계**

| 무엇 | 어디까지가 이 화면의 일인가 |
|---|---|
| 송장을 여러 장 나눠 붙이기 | **지금 회차에 없다** — 잔량 전부가 한 송장에 실린다. 잔량이 남는 것은 앞서 붙인 송장이 일부만 덮었을 때다 |
| 부분 발송된 주문의 상태 | 주문은 **상태를 하나만 갖는다** — 부분배송을 상태로 만들지 않는다. 배지와 출고 수량이 그 사실을 전한다 |
| 부분 발송 주문의 작업 상태 | `발송대기` 로 남는다(완료 쪽으로 옮기지 않는다) |
| 일괄 처리와 부분 발송 | 일괄 발송처리는 **행 단위**로 돈다. 한 행 안에서 발송 가능한 배송 건이 여럿이면 그 전부를 함께 내보낸다 |

**송장 등록도 '수정' 축이다** — 새 배송 건을 만드는 일이지만 화면의 컨트롤은 전부 `canUpdate` 에 묶여 있다. 배송 처리는 **주문의 수정**이라는 판단이다 — `ShipmentListPage.tsx:132-133`.

**권한이 없을 때 화면이 어떻게 되는가** — 조회 권한이 없으면 본문이 `접근 권한이 없어요` 로 바뀐다. 수정 권한이 없으면 선택 체크박스와 세 일괄 버튼이 **사라진다**(비활성이 아니다).
