# 주문

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 주문 |
| 화면 ID | `SCR-ORDERS` |
| 메뉴 경로 | 주문 관리 > 주문 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 지금 무엇이 어느 단계에 고여 있는지 건수로 보고, 여러 건을 골라 다음 단계로 한 번에 옮긴다.

**업무 배경** — 주문은 고객의 결제가 만든다. 관리자가 만들 수 없고 지울 수도 없다(거래 기록이다). 그래서 이 화면에는 등록 버튼도 삭제 버튼도 없고, 남는 일은 둘뿐이다 — **어디에 얼마나 쌓였는지 확인하는 일**과 **여러 건을 한꺼번에 진행시키는 일**. 이 둘이 없으면 운영자는 주문 하나하나를 열어 상태를 바꿔야 한다.

**화면 설명** — 좌측은 상태 필터(전체 · 상태 7종 · 취소)이며 항목마다 건수 배지가 붙는다. 우측은 검색창 · 선택 바(일괄 처리 버튼 4개) · 주문 표다. 행을 누르면 주문 상세로 간다.

**주요 사용자** — 최상위 관리자 · 운영자(주문 수정 권한 보유자) · 상담원(조회만).

**사용 시나리오**

1. 아침에 이 화면을 연다 → 좌측 '배송준비중 12' 를 보고 그 필터를 누른다 → 전체 선택 후 '배송대기 처리 (12)' 를 눌러 한 번에 옮긴다.
2. 고객이 전화로 주문번호를 부른다 → 검색창에 붙여 넣는다 → 한 행이 남고, 주문번호 링크로 상세에 들어간다.
3. 결제(PG)를 아직 연동하지 않은 사이트에서 이 화면을 연다 → 표 대신 '결제를 사용하지 않아 주문이 들어오지 않는다' 는 안내와 결제 설정·상품 문의로 가는 링크가 뜬다.

**선행 조건** — 로그인 · 이 화면(리소스 `/orders`)의 조회 권한. 일괄 처리는 수정 권한이 있어야 컨트롤이 존재한다.

**후행 처리** — 상태 전이는 주문의 처리 이력에 한 줄을 남기고, 재고 차감 조건이 맞으면 SKU 재고가 함께 움직인다(`apps/admin/src/pages/orders/data-source.ts:44-59`). 통계(`SCR-STATS-ORDERS`)가 같은 상태 어휘로 집계한다.

**관련 화면** — `SCR-ORDERS-DETAIL`(행 클릭) · `SCR-ORDERS-SHIPMENTS`(같은 주문을 배송 축에서 처리한다) · `SCR-ORDERS-CLAIMS`(취소·교환·반품) · `SCR-SETTINGS-PAYMENT`(빈 상태의 링크) · `SCR-PRODUCTS-INQUIRIES`(결제를 쓰지 않는 동안 고객 요청이 쌓이는 곳). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 사이드바 `주문 관리 > 주문`. 조건부 노출이 아니다 — 결제를 쓰지 않아도 메뉴는 남는다(과거 주문이 있을 수 있다).

**관련 기능** — 결제 설정(`SCR-SETTINGS-PAYMENT`)의 사용 여부가 이 목록의 **빈 상태 문구**를 바꾼다. 배송 처리 화면의 '배송준비중 처리' 는 이 화면의 같은 이름 일괄 처리와 **같은 술어**를 읽는다.

**관련 API** — 목록 조회 1건 · 주문 갱신 N건(선택 건수만큼). **백엔드가 없다** — `apps/admin/src/pages/orders/data-source.ts:61-64` 의 `TODO(backend)` 가 경계다: `GET /api/orders` · `GET/PUT /api/orders/:id`. `POST` 는 열지 않는다(주문을 만드는 것은 고객의 결제다).

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [주문 비기능 명세서](../../NFRS/orders.md)

**구현 파일** — `apps/admin/src/pages/orders/OrderListPage.tsx` (표 `components/OrderTable.tsx` · 표시 규칙 `types.ts` · 상태 기계 `apps/admin/src/shared/domain/order.ts` · 어댑터 `data-source.ts` · 결제 판정 `_shared/store.ts`)

**상태 필터의 특이 규칙** — 취소된 주문은 **상태 필터에서 빠진다**. '배송준비중' 을 고른 운영자가 보려는 것은 지금 준비할 주문이고, 취소된 건이 섞이면 그 건수가 할 일의 양이 아니게 된다. 취소는 자기 항목에서만 보이고, '전체' 는 말 그대로 전부다 — `types.ts:61-68`.

**등록·삭제 액션은 이 화면에 존재하지 않는다** — 권한이 있어도 버튼이 없다. 주문은 고객의 결제가 만들고, 거래 기록이라 지우지 않는다.

**권한이 없을 때 화면이 어떻게 되는가** — 조회 권한이 없으면 본문이 `접근 권한이 없어요` 화면으로 바뀌고 사이드바는 남는다. 수정 권한이 없으면 **선택 체크박스와 일괄 처리 바가 사라지고**(비활성이 아니다), 좌측 안내가 그 이유를 밝힌다. 판정 순서는 인증 → 플랜 → 권한이다.
