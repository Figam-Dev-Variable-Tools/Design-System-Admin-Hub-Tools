# 취소/교환/반품 (클레임)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 취소/교환/반품 |
| 화면 ID | `SCR-ORDERS-CLAIMS` |
| 메뉴 경로 | 주문 관리 > 취소/교환/반품 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 고객이 접수한 취소·교환·반품 요청을 한 창구에서 보고, 처리할 건을 골라 상세로 들어간다.

**업무 배경** — 셋은 **같은 사건의 세 시점**이다: 출고 전에 멈추면 취소, 나갔다 돌아오면 반품, 돌아오고 다시 나가면 교환. 예전에는 유형이 '교환·반품' 둘뿐이라 취소 건이 갈 곳이 없었고, 운영자는 취소를 '반품' 으로 접수해 놓고 메모에 '실은 취소' 라고 적었다 — `apps/admin/src/pages/orders/claims/types.ts:4-8`.

**화면 설명** — 툴바(검색 + 유형 셀렉트 + 상태 셀렉트) 위에 읽기 전용 표가 선다. **선택 체크박스도 삭제도 일괄 처리도 없다** — 클레임은 고객이 만들고 관리자는 처리만 하며, 접수 사실은 감사 대상이라 지우지 않는다. 행을 누르면 클레임 처리 화면으로 간다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원.

**사용 시나리오**

1. 상태를 '검수중' 으로 걸러 오늘 판정할 건만 본다 → 행을 눌러 상세에서 완료 또는 반려한다.
2. '환불' 열이 `환불 접수` 인 건만 눈으로 훑는다 → 정산일에 맞춰 환불 완료 처리를 돈다.
3. 고객이 주문번호를 부른다 → 검색창에 넣어 그 주문의 클레임을 찾는다.

**선행 조건** — 로그인 · `/orders/claims` 리소스 조회 권한.

**후행 처리** — 이 화면 자체는 아무것도 바꾸지 않는다. 상세에서의 처리가 재고와 적립금 원장을 움직인다.

**관련 화면** — `SCR-ORDERS-CLAIMS-DETAIL`(행 클릭) · `SCR-ORDERS-DETAIL`(클레임이 가리키는 원 주문) · `SCR-PRODUCTS-SHIPPING`(반품배송비 기본값의 정본). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 화면 — 이 화면으로 오는 옛 경로** — `/products/returns` 와 `/products/returns/*` 는 이 화면으로 되돌려 보내진다(`apps/admin/src/App.tsx:335-336`). 클레임이 가리키는 것은 상품이 아니라 **주문**이라 주문 관리 아래로 옮겼다.

**관련 메뉴** — 사이드바 `주문 관리 > 취소/교환/반품`. 조건부 노출이 아니다.

**관련 기능** — 취소 유형의 처리 가능 여부는 **원 주문의 출고 여부**를 읽는다. 취소 클레임은 **재고를 움직이지 않는다** — 복원의 주체는 주문 취소다(`POP-ORDERS-DETAIL-CANCEL`).

**관련 API** — 클레임 목록 조회 1건. **백엔드가 없다** — `apps/admin/src/pages/orders/claims/data-source.ts:467-471` 의 `TODO(backend)`: `GET /api/claims` · `GET/PUT /api/claims/:id`. `POST` 는 열지 않는다(클레임을 만드는 것은 고객의 접수다).

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [클레임 비기능 명세서](../../../NFRS/orders/claims.md)

**구현 파일** — `apps/admin/src/pages/orders/claims/ClaimsListPage.tsx` (도메인·표시 규칙 `types.ts` · 환불 축 `refund.ts` · 어댑터 `data-source.ts`)

### 4.3 이 목록이 보여 주는 두 축

**클레임 처리 축과 환불 축은 나란한 별개다** — `refund.ts:4-13`.

| 축 | 값 | 무엇을 뜻하는가 |
|---|---|---|
| 처리 상태 | 접수 → 수거중 → 검수중 → 완료 · (흐름 밖) 반려 · 철회 | 물건과 심사가 어디까지 왔는가 |
| 환불 | 환불 없음 → 환불 접수 → 환불 완료 · (교환) 해당 없음 | 돈이 어디까지 갔는가 |

물건을 받아 검수까지 끝났어도(처리 완료) 정산일에 맞춰 며칠 뒤 송금하는 일이 흔하고, 반대로 불량이 확실해 돈부터 보내는 경우도 있다. 한 축에 눌러 담으면 그 사이 며칠을 표현할 자리가 없어진다.

**유형별 흐름이 다르다** — `types.ts:127-130`.

| 유형 | 흐름 | 왜 |
|---|---|---|
| 취소 | 접수 → 완료 | 물건이 아직 창고에 있어 수거·검수가 없다 |
| 교환 | 접수 → 수거중 → 검수중 → 완료 | — |
| 반품 | 접수 → 수거중 → 검수중 → 완료 | — |

반려·철회는 **흐름 밖 종료**라 이 배열에 없고, 화면이 배지로 따로 알린다.
