# 클레임 처리 (취소/교환/반품 상세)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 클레임 처리 |
| 화면 ID | `SCR-ORDERS-CLAIMS-DETAIL` |
| 메뉴 경로 | 주문 관리 > 취소/교환/반품 > (행 클릭) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 클레임 한 건의 요청 내용을 보고, **처리 축**(접수→수거→검수→완료 / 반려 / 철회)과 **환불 축**(접수 → 완료)을 각각 진행한다.

**업무 배경** — 클레임 완료와 환불 완료는 다른 사건이고 끝나는 시점도 다르다. 한 카드에 섞으면 '완료' 버튼이 무엇을 완료하는지 화면에서 읽히지 않는다. 그래서 카드를 나누고, 각 축의 막힌 동작은 **왜 못 누르는지**를 버튼 옆에 그대로 쓴다 — `apps/admin/src/pages/orders/claims/ClaimDetailPage.tsx:6-11`.

**화면 설명** — 카드가 최대 넷이다: 요청 정보(스텝퍼 + 상태 선택 + 메모 + `처리 저장`) · 교환 재발송·재고(교환일 때만) · 환불 처리(취소·반품일 때만) · 재고 이동 이력.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. 반품 건이 입고돼 검수를 마쳤다 → 상태를 `완료` 로 바꾸고 `처리 저장` → **재고 반영 확인**이 뜬다 → 확인하면 회수분이 입고되고 이력에 남는다.
2. 그 뒤 환불 카드에서 반품배송비를 확인하고 `환불 완료 처리` → **환불 완료 확인**이 뜬다 → 확인하면 적립금이 원장으로 돌아간다.
3. 교환 건이면 먼저 재발송할 옵션을 고른다 → 재고가 부족한 옵션은 선택지에서 잠겨 있다.
4. 배송이 시작된 주문의 취소 건을 연다 → 경고 배너가 뜨고 상태를 완료로 옮길 수 없다.

**선행 조건** — 로그인 · `/orders/claims` 리소스 조회 권한. 처리하려면 수정 권한이 있어야 컨트롤이 존재한다.

**후행 처리** — 완료 처리는 재고를 움직이고(취소는 제외), 환불 완료는 **적립금 원장에 지급 1건을 덧붙인다**. 둘 다 되돌릴 수 없다.

**관련 화면** — `SCR-ORDERS-CLAIMS`(목록) · `SCR-ORDERS-DETAIL`(주문번호 링크) · `SCR-PRODUCTS`(옵션·재고의 정본) · `SCR-PRODUCTS-SHIPPING`(반품배송비 기본값) · `SCR-USERS-MEMBERS-DETAIL`(복원된 적립금이 보이는 곳). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다.

**관련 기능** — 취소 유형의 가드는 **원 주문의 출고 여부**를 읽는다. 취소 클레임은 재고를 움직이지 않는다 — 복원의 주체는 주문 취소다.

**관련 API** — 상세 조회 1건 · 클레임 갱신 1건(상태·환불·메모·교환 옵션이 한 통로) · 교환일 때 상품 옵션·재고 조회 1건. **백엔드가 없다** — `apps/admin/src/pages/orders/claims/data-source.ts:467-471,484-485`. 서버는 클레임 갱신 + SKU 재고 증감 + 원장 append 를 **한 트랜잭션**으로 처리하고, 재고가 모자라면 422 로 거절해야 한다. 멱등키는 `stockAppliedAt` · `refund.completedAt` 이다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [클레임 비기능 명세서](../../../NFRS/orders/claims.md)

**구현 파일** — `apps/admin/src/pages/orders/claims/ClaimDetailPage.tsx` (도메인·가드 `types.ts` · 환불 축 `refund.ts` · 환불 카드 `components/RefundSection.tsx` · 교환 옵션 `components/ExchangeOptionField.tsx` · 재고 이력 `components/StockMovementTable.tsx` · 어댑터 `data-source.ts`)

### 4.6 다중 위반의 배치(이 화면의 실패 규칙)

**문제** — 서버·어댑터는 한 응답에 **여러 위반**을 실을 수 있다(응답의 필드별 문구는 map 이다). 그런데 이 화면에서 인라인으로 되돌릴 수 있는 자리는 **교환 옵션 필드 하나뿐**이고, 그 필드는 교환 클레임에만 렌더된다. 예전에는 화면이 첫 위반만 읽고 무조건 인라인으로 보내, 취소·반품에서 전이·환불·재고 위반이 나면 **배너도 토스트도 인라인도 없이 버튼만 조용히 다시 활성화됐다** — `types.ts:426-445,480-511`.

**규칙** — `types.ts:494-511` · `ClaimDetailPage.tsx:321-337`.

| 단계 | 판정 | 결과 |
|---|---|---|
| ① | 위반마다 '인라인 자리가 지금 실제로 떠 있는가' 를 묻는다 | 교환 옵션 필드가 렌더 중이고 확인 다이얼로그에 가려져 있지 않으면 그 필드로, 아니면 배너로 |
| ② | 인라인으로 간 문구들 · 배너로 간 문구들을 각각 한 문장으로 잇는다 | **하나도 버리지 않는다** |
| ③ | 필드를 하나도 지목하지 않은 거절이면 | 서버의 대표 문구를 배너로 보낸다(침묵하지 않는다) |
| ④ | 포커스를 옮긴다 | 인라인이 있으면 그 셀렉트로, 없으면 배너로 — `ClaimDetailPage.tsx:282-287` |

**확인 다이얼로그가 떠 있으면 포커스를 옮기지 않는다** — 그 실패는 다이얼로그 안의 danger 배너가 즉시 읽어 준다. 모달 밖으로 포커스를 끌어내면 포커스 트랩이 깨져 Esc·Tab 이 어디로 가는지 알 수 없게 된다 — `ClaimDetailPage.tsx:278-280`.

**포커스는 저장이 끝난 뒤에 옮긴다** — 실패 시점에는 아직 저장 중이라 셀렉트가 비활성이고 배너도 마운트되지 않았다. 비활성 요소의 포커스 요청은 브라우저가 조용히 무시한다 — `ClaimDetailPage.tsx:272-276`.

**400 과 422 를 한 갈래로 묶는다** — 원인은 다르지만(400=형식 위반, 422=규칙 위반) **화면이 하는 일은 같다**: 위반을 그 입력으로 되돌리고, 되돌릴 자리가 없는 것은 배너로 모은다 — `types.ts:447-465`.

| 반품배송비 | 원 단위 정수만(소수점·쉼표 불가) | 입력 중 | `반품배송비는 0 이상의 원 단위 숫자로 입력하세요.` — `refund.ts:159` | 필드 인라인 오류 · 환불 완료 버튼 비활성 |
| 반품배송비 | 0 이상의 안전한 정수 | 저장 시(서버 축) | 같은 문구 — `data-source.ts:329` | 400 으로 거절 → 배너 |
| 반품배송비 | **환불 완료 후에는 바꿀 수 없다** | 저장 시 | `환불이 완료되어 차감 내역을 바꿀 수 없어요.` — `apps/admin/src/pages/orders/claims/data-source.ts:361` | 422 → 배너. 입력도 잠겨 있다 |
| 처리 상태 전이 | §4.5 처리 축 가드 | 선택 시(사유 표기) · 저장 시 | 그 사유 문장 그대로 | 갈 수 없는 상태는 선택지에 없다. 그럼에도 걸리면 422 → 배너 |
| 환불 상태 전이 | §4.5 환불 축 가드 | 버튼 렌더 · 저장 시 | 그 사유 문장 그대로 | 버튼 비활성 + 사유 문장. 저장에서 걸리면 422 → 배너 |
| 재고 유효성 | §4.5 재고 표 | 상태를 완료로 고른 순간 · 저장 시 | 그 사유 문장 그대로 | 교환 옵션 관련은 **인라인**, 주문 옵션을 못 찾은 것은 카드 배너 |
| 상품 옵션 카탈로그 | 읽을 수 있어야 재고를 반영한다 | 저장 시 | `연결된 상품의 옵션을 확인할 수 없어 재고를 반영할 수 없어요.` — `data-source.ts:378-379` | 422 → 배너 |

**규칙의 정본은 도메인 술어다** — 화면의 버튼과 저장소가 **같은 함수**를 부른다. 입력 형식만 zod 가 아닌 순수 함수(`parseFeeInput`)로 판정하는데, 폼이 문자열을 드는 이유는 지우는 도중에 0 이 튀어나오지 않게 하기 위해서다 — `refund.ts:166-173`.

**화면 검증은 보증이 아니다** — 화면의 입력 상한·선택지·숫자 파서는 브라우저 입력만 막는다. 프로그램적으로 채워진 값은 그대로 나가고, 그것을 거르는 것은 서버의 일이다(지금은 픽스처가 그 자리에 있다) — `data-source.ts:306-317`.

| 파일 업로드 실패 | 해당 없음 — 파일 입력이 없다 | — | — |

**재고를 못 움직였으면 멱등 키를 찍지 않는다** — 적용기가 배선되지 않았으면 반영 시각을 찍지 않고 그대로 둔다. 여기서 찍으면 재고는 그대로인데 원장만 '반영 완료' 라고 말하는 상태가 남는다 — `data-source.ts:391-392`.

**클레임의 '반려' 는 권한 액션이 아니다** — 권한 모델의 액션은 다섯이다(`read`·`create`·`update`·`remove`·`export`). 반려·철회는 **상태 전이**이므로 `update` 권한 하나로 열린다.

**권한이 없을 때 화면이 어떻게 되는가** — 조회 권한이 없으면 본문이 `접근 권한이 없어요` 로 바뀐다. 수정 권한이 없으면 상태 선택·메모·교환 옵션·차감 입력이 잠기고 `처리 저장`·`환불 접수`·`환불 완료 처리` 버튼이 **사라진다**. `목록으로` 는 남는다.
