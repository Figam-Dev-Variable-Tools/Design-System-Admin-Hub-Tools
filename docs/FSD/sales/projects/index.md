# 프로젝트 목록

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 프로젝트 |
| 화면 ID | `SCR-SALES-PROJECTS` |
| 메뉴 경로 | 영업 관리 > 프로젝트 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 진행 중인 영업 기회(프로젝트)를 단계별로 훑고, 예상매출과 진척률을 한눈에 비교한다.

**업무 배경** — 프로젝트는 사슬의 마지막 칸이다. 계약이 체결되면 그 계약이 프로젝트가 되고, 여기서 마일스톤·진척을 따라간다. 문서(플로우 차트)에는 계약 → 프로젝트 화살표가 있었지만 **프로젝트에는 계약을 가리키는 필드가 한 칸도 없었다** — 체결된 계약은 그 다음이 없는 종점이었고, 프로젝트는 어디서 왔는지 앱이 답하지 못했다 — `apps/admin/src/pages/sales/projects/types.ts:45-52`.

**화면 설명** — 검색 + 단계 필터 툴바와 6열 표다. 선택 체크박스와 삭제가 있지만 **등록 버튼이 없다**. 행을 누르면 프로젝트 수정 폼이 열린다(프로젝트에는 읽기 전용 상세가 없다).

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자.

**사용 시나리오**

1. 단계를 '제안'으로 걸러 놓고 주간 회의에 그 주소를 그대로 붙여 넣는다 — 조회 조건이 주소에 실린다.
2. 예상 마감이 임박한 순서로 놓인 표(종료일 오름차순)에서 위쪽 몇 건을 먼저 연다.
3. 진척 막대와 예상매출을 나란히 보고 어디에 사람을 더 붙일지 정한다.

**선행 조건** — 로그인 · 읽기 권한 · `sales.pipeline` 모듈. **행이 생기려면 앞 칸(체결된 계약)이 있어야 한다.**

**후행 처리** — 이 목록에서 일어나는 변경은 삭제뿐이다. **프로젝트를 지우면 그 계약의 `프로젝트 만들기` 버튼이 다시 살아난다** — `apps/admin/src/pages/sales/projects/data-source.ts:133-134`.

**관련 화면** — [프로젝트 수정](form.md) `SCR-SALES-PROJECTS-FORM` · [계약 수정](../contracts/form.md) `SCR-SALES-CONTRACTS-FORM`(프로젝트가 태어나는 곳) · [거래처 상세](../accounts/detail.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `영업 관리 > 프로젝트`.

**관련 기능** — 계약에서 만들어진 프로젝트는 **단계 `수주`**·확률 100%로 시작한다. 계약이 체결됐다는 것은 파이프라인의 앞 칸(리드·상담·제안·협상)이 이미 끝났다는 뜻이고, 여기서 '리드'로 시작하면 방금 도장을 찍은 거래가 확률 10%로 집계되어 가중예상매출이 거짓이 된다 — `apps/admin/src/pages/sales/projects/types.ts:196-201`.

**관련 API** — 프로젝트 목록 조회 1건 · 삭제 N건. **백엔드가 없다** — `TODO(backend): GET/POST /api/sales/projects · GET/PUT/DELETE /api/sales/projects/:id` — `apps/admin/src/pages/sales/projects/data-source.ts:170`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/sales/projects/ProjectListPage.tsx` (규칙 `types.ts` · 저장소 `data-source.ts`)

## 4. 기능 명세

### 4.1 프로젝트 목록 조회

| 항목 | 내용 |
|---|---|
| 기능 목적 | 진행 중인 기회를 훑는다 |
| 실행 조건 | 진입 · 조회 조건 변경 · 삭제 성공 후 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 목록을 받아 **예상 마감(종료)일 오름차순**(임박이 위, 같은 날짜는 id 안정 정렬)으로 정렬한다 — `apps/admin/src/pages/sales/projects/types.ts:142-148` |
| 데이터 처리 | 읽기만 한다 |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 프로젝트 전체 목록 1건 |
| 성공 처리 | 표가 그려진다. 토스트 없음. 보조기술에는 `프로젝트 {N}건을 찾았어요.` |
| 실패 처리 | 표 자리에 danger 배너 + `다시 시도` |
| 예외 처리 | 재조회 중에는 이전 행을 유지한다 |
| 화면 변경 사항 | 목록영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**다른 목록과 정렬 방향이 반대다** — 견적·계약·문의·청구는 전부 최근이 위(내림차순)인데 프로젝트만 **임박이 위**(오름차순)다. 이 화면에서 운영자가 답할 질문이 '무엇이 곧 끝나는가'이기 때문이다.

### 4.3 프로젝트 삭제 (단건 · 일괄)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 잘못 만든 프로젝트를 지운다 |
| 실행 조건 | 삭제 권한이 있어 컨트롤이 보일 때 |
| 사용자 동작 | 행 휴지통 → `DLG-COMMON-DELETE-ONE` · 선택 후 일괄 삭제 → `DLG-COMMON-DELETE-BULK` |
| 시스템 처리 | 확인 시 삭제 요청을 보낸다 |
| 데이터 처리 | 그 프로젝트가 사라진다. **그 계약의 `프로젝트 만들기` 버튼이 다시 살아난다** — 계약 폼이 프로젝트 존재 여부를 렌더 시점에 다시 조회하기 때문이다. 수동 등록을 막은 뒤에도 잘못 만든 프로젝트에서 빠져나올 길이 남는다 |
| Validation | 선택 0건이면 확인해도 요청을 만들지 않는다 |
| API 호출 여부 | Y — 대상 건수만큼 |
| 성공 처리 | 목록이 갱신되고 공통 토스트(`MSG-COMMON-02`·`MSG-COMMON-03`)가 뜬다 |
| 실패 처리 | 다이얼로그가 닫히지 않고 배너로 이유를 말한다: `이미 삭제된 프로젝트예요.` — `apps/admin/src/pages/sales/projects/data-source.ts:122` |
| 예외 처리 | 취소·Esc·딤 클릭은 진행 중 요청을 중단하고 실패로 치지 않는다 |
| 화면 변경 사항 | 다이얼로그 · 목록영역 · 선택 바 |
| 후속 동작 | 계약 폼의 배너가 '체결이 끝난 계약…' 또는 버튼으로 되돌아간다 |
| 로그 기록 여부 | 확인하지 못했다 |

**이 화면에는 자기 토스트가 없다** — [인벤토리](../../inventory.md) §9.3 에 `pages/sales/projects/ProjectListPage.tsx` 가 없다.
