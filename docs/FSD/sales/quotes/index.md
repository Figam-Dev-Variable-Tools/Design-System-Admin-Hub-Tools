# 견적 목록

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 견적 |
| 화면 ID | `SCR-SALES-QUOTES` |
| 메뉴 경로 | 영업 관리 > 견적 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 발행된 견적을 한 표에 모아 상태·합계금액·유효기간을 훑고, 앞 칸(원본 문의)과 다음 칸(계약)으로 건너간다.

**업무 배경** — 견적은 사슬의 두 번째 칸이다. 이 목록에서 운영자가 답해야 하는 질문은 둘이다 — '지금 어느 견적이 승인을 기다리는가'와 '이 견적은 계약까지 갔는가'. 두 번째 질문에 답할 수 없다는 것이 사슬이 따로 노는 느낌의 한 조각이었고, 그래서 마지막 열이 계약 링크다 — `apps/admin/src/pages/sales/quotes/QuoteListPage.tsx:160-163`.

**화면 설명** — 검색 + 상태 필터 툴바와 7열 표다. 선택 체크박스와 삭제가 있는 일반 CRUD 목록이지만 **등록 버튼이 없다**. 행을 누르면 수정 폼이 열린다.

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자.

**사용 시나리오**

1. 상태를 '발송'으로 걸러 회신을 기다리는 견적만 본다 → 고객이 승인하면 그 행을 열어 상태를 옮긴다.
2. 고객이 문의번호를 대며 견적을 묻는다 → 검색창에 **문의번호**를 넣는다(견적번호·거래처명뿐 아니라 원본 문의번호로도 찾는다 — `apps/admin/src/pages/sales/quotes/types.ts:383-384`).
3. '계약' 열의 `계약 열기` 로 그 견적이 낳은 계약을 바로 연다.

**선행 조건** — 로그인 · 읽기 권한 · `sales.pipeline` 모듈. **그리고 앞 칸이 있어야 행이 생긴다** — 견적은 문의에서만 발행된다.

**후행 처리** — 이 목록에서 일어나는 변경은 삭제뿐이다. 상태 전이·계약 생성·청구 생성은 전부 견적 상세의 일이다.

**관련 화면** — [견적 상세](detail.md) `SCR-SALES-QUOTES-DETAIL` · [견적 수정](form.md) `SCR-SALES-QUOTES-FORM` · [문의 상세](../inquiries/detail.md) `SCR-SALES-INQUIRIES-DETAIL`(견적이 태어나는 곳) · [계약 수정](../contracts/form.md) `SCR-SALES-CONTRACTS-FORM` · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `영업 관리 > 견적`.

**관련 기능** — 원본 문의는 세 창구 중 하나다: 영업 문의(`/sales/inquiries`) · 상품 문의(`/products/inquiries`) · 프로그램 문의(`/programs/inquiries`) — `apps/admin/src/pages/sales/quotes/types.ts:58-62`. 어느 창구에서 왔든 이 목록에 같이 쌓인다.

**관련 API** — 견적 목록 조회 1건 · 삭제 N건. **백엔드가 없다.** 서버 계약은 `TODO(backend): GET/POST /api/sales/quotes · GET/PUT/DELETE /api/sales/quotes/:id` — `apps/admin/src/pages/sales/quotes/data-source.ts:201-202`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/sales/quotes/QuoteListPage.tsx` (규칙 `types.ts` · 저장소 `data-source.ts` · 거래처 셀 `../_shared/AccountLink.tsx`)

### 4.3 견적 삭제 (단건 · 일괄)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 잘못 발행된 견적을 지운다 |
| 실행 조건 | 삭제 권한이 있어 컨트롤이 보일 때 |
| 사용자 동작 | 행 휴지통 → `DLG-COMMON-DELETE-ONE` · 선택 후 일괄 삭제 → `DLG-COMMON-DELETE-BULK` |
| 시스템 처리 | 확인 시 삭제 요청을 보낸다. 문구·동작 규약은 [공통 문서](../../_common/index.md) 가 갖는다 |
| 데이터 처리 | 그 견적이 목록에서 사라진다. **이미 만들어진 계약·청구는 함께 지워지지 않는다** — 계약의 `quoteId` 는 남고, 그 계약을 열면 원 견적 링크가 죽은 목적지를 가리킨다 |
| Validation | 일괄 삭제에서 선택 0건이면 확인해도 아무 요청도 만들지 않는다 |
| API 호출 여부 | Y — 대상 건수만큼의 삭제 |
| 성공 처리 | 목록이 갱신되고 공통 토스트가 뜬다: 단건 `MSG-COMMON-02` · 일괄 `MSG-COMMON-03` |
| 실패 처리 | 다이얼로그가 닫히지 않고 배너로 이유를 말한다. 저장소가 없는 id 에 주는 사유: `이미 삭제된 견적이에요.` — `apps/admin/src/pages/sales/quotes/data-source.ts:119` |
| 예외 처리 | 취소·Esc·딤 클릭은 진행 중 요청을 중단하고 실패로 치지 않는다 |
| 화면 변경 사항 | 다이얼로그 · 목록영역 · 선택 바 |
| 후속 동작 | 목록 재조회 |
| 로그 기록 여부 | 확인하지 못했다 |

**⚠ 견적을 지워도 앞뒤 칸이 정리되지 않는다** — 원본 문의의 `quoteId` 는 그대로 남아 문의 상세가 계속 '이미 발행됨'이라고 말하고, `발행된 견적 보기` 가 없는 견적을 가리킨다. 이 되돌리기를 정리하는 코드가 없다는 사실을 그대로 적는다.

**`create` 권한은 이 화면에서 아무것도 열지 않는다** — 등록 CTA 가 아예 없고 `/sales/quotes/new` 주소도 차단 화면으로 간다([견적 수정](form.md) §4.1). 권한이 있어도 만들 수 없다는 사실이 이 화면의 규약이다.

**권한이 없을 때 화면이 어떻게 되는가** — 읽기가 없으면 본문이 통째로 권한 화면으로 바뀐다. 쓰기가 없으면 컨트롤이 사라진다(비활성이 아니다).
