# 견적 상세

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 견적 상세 |
| 화면 ID | `SCR-SALES-QUOTES-DETAIL` |
| 메뉴 경로 | 영업 관리 > 견적 > (행 선택) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 견적서를 읽고 인쇄하며, 상태를 다음 칸으로 옮기고, **계약과 청구를 만든다.** 사슬의 세 화살표(견적→계약 · 견적→청구 · 문의↔견적 역링크)가 전부 이 한 화면에서 갈라진다.

**업무 배경** — 예전에는 견적에 읽기 전용 표면이 없어 문의 화면의 '견적 보기' 링크가 실제로는 수정 폼을 열었다. 또 상태를 바꾸는 컨트롤이 하나도 없어서, 다음 칸(계약)이 '승인'에서만 열리는데 발송한 견적을 승인으로 넘길 방법이 화면에 없었다 — 계약 버튼이 영원히 잠겨 있었다 — `apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:1-12,167-174`.

**화면 설명** — 머리에 견적번호·상태 배지·액션 4개(인쇄 · 계약 만들기 · 청구 만들기 · 견적 수정)가 있고, 그 아래에 상황별 경고 배너, '상태 관리' 카드, 그리고 좌우 2단(견적서 미리보기 / 연결·요약)이 온다.

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자.

**사용 시나리오**

1. 발송한 견적을 고객이 승인했다 → '상태 관리'에서 `승인` 을 누른다 → 그 순간 머리에 `계약 만들기` 가 나타난다.
2. `계약 만들기` 를 누른다 → 계약이 생기고 **견적 상태가 '수주(계약 진행)'로 바뀌며** 계약 수정 폼으로 이동한다.
3. 수주된 견적에서 `청구 만들기` 를 누른다 → 폼 없이 청구가 생기고 청구 상세로 이동한다.
4. 계약은 있는데 상태가 '승인'에 머물러 있는 견적을 연다 → 경고 배너와 `견적 상태 맞추기` 버튼이 뜬다.

**선행 조건** — 로그인 · 읽기 권한 · `sales.pipeline` 모듈. 상태 관리·계약·청구·수정은 **수정 권한**이 있어야 컨트롤이 존재한다.

**후행 처리** — 상태 전이는 견적 목록의 배지에 반영된다. 계약 생성은 `SCR-SALES-CONTRACTS` 에 행을 하나 만들고 **이 견적의 상태를 되돌려 쓴다**. 청구 생성은 `SCR-SALES-BILLING` 에 행을 만든다.

**관련 화면** — [견적 목록](index.md) · [견적 수정](form.md) · [계약 수정](../contracts/form.md) · [청구 상세](../billing/detail.md) · [거래처 상세](../accounts/detail.md) · 원본 문의 3창구(`SCR-SALES-INQUIRIES-DETAIL` · `SCR-PRODUCTS-INQUIRIES-DETAIL` · `SCR-PROGRAMS-INQUIRIES-DETAIL`) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다.

**관련 기능** — 견적서에 인쇄되는 **공급자(자사) 정보**는 회사 정보 화면(`SCR-COMPANY-PROFILE`)의 값을 공통 층의 조회기가 받아 온다. 예전에는 견적 코드에 다른 회사가 하드코딩되어 있어 회사 정보를 고쳐도 견적서가 바뀌지 않았다 — `apps/admin/src/pages/sales/quotes/types.ts:134-150`.

### 1.1 이 화면이 정하는 어휘 — `ordered` 와 '수주(계약 진행)'

코드의 상태값은 `ordered` 이고 사람이 읽는 라벨은 `수주(계약 진행)` 다. 라벨은 한 곳에서만 정한다 — `apps/admin/src/pages/sales/quotes/types.ts:188` 의 `QUOTE_ORDERED_LABEL`.

**왜 `ordered` 를 '계약'으로 개명하지 않았는가.** 이 상태는 계약의 결과이면서 동시에 **청구의 근거**다. 청구 화면은 `수주로 전환된 견적만 청구할 수 있어요.` 라고 말하는데(`apps/admin/src/pages/sales/billing/types.ts:148`), 청구는 계약을 요구하지 않는다 — 계약 없이 수주만 된 견적에서도 청구가 나간다. 상태 이름을 '계약'으로 바꾸면 그 문장이 거짓이 된다. 그래서 코드 어휘는 `ordered` 로 두고 사람이 읽는 자리에만 두 이름을 함께 붙였다 — 운영자의 말('계약 진행')과 코드의 말('수주')이 같은 줄에서 만나게 한 것이다.

## 4. 기능 명세

### 4.1 견적 상세 조회

| 항목 | 내용 |
|---|---|
| 기능 목적 | 견적서와 그 앞뒤 연결을 한 화면에서 읽는다 |
| 실행 조건 | 주소에 견적 id 가 있을 때 |
| 사용자 동작 | 목록·문의 화면의 링크 · 주소 직접 입력 |
| 시스템 처리 | 견적을 조회하고, 같은 페이지의 계약·청구 저장소를 **동기로** 조회해 연결 여부를 정한다 — `apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:184-189` |
| 데이터 처리 | 읽기만 한다. 합계금액은 품목에서 계산한다 |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 견적 상세 1건. 계약·청구 조회는 같은 페이지 안이라 요청이 아니다 |
| 성공 처리 | 머리·경고·상태 관리·2단 카드가 그려진다. 토스트 없음 |
| 실패 처리 | **없는 것과 고장난 것을 가른다**: `견적을 찾을 수 없어요. 이미 삭제되었을 수 있어요.` 에는 `목록으로` 만, `견적을 불러오지 못했어요.` 에는 `다시 시도` + `목록으로` 를 준다 — `apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:251-271` |
| 예외 처리 | 왜 가르는가 — 삭제된 견적에 '다시 시도'는 영원히 실패한다 |
| 화면 변경 사항 | 화면 전체 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**왜 계약·청구를 동기로 보는가** — 버튼을 **그리는 순간** 답이 있어야 한다. 비동기로 나중에 알면 '만들기' 버튼이 잠깐 살아 있다가 죽고, 그 사이에 누른 사람은 두 번째 계약을 만든다 — `apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:177-183`.

| 시스템 처리 | 그 견적을 상태만 바꿔 갱신한다(목록 인라인 갱신 훅을 쓴다) |
| 데이터 처리 | 견적의 `status` 가 바뀐다. 되돌리는 전이는 없다 |
| Validation | §6 의 전이표 |
| API 호출 여부 | Y — 견적 갱신 1건 |
| 성공 처리 | 배지와 목록이 갱신된다. 공통 토스트 `MSG-COMMON-04`(success)에 이 화면이 준 문구가 실린다: `'{견적번호}'을(를) '{상태}' 상태로 바꿨어요.` (조사는 런타임에 고른다) — `apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:405` |
| 실패 처리 | 공통 토스트 `MSG-COMMON-05`(error) `변경하지 못했어요. 잠시 후 다시 시도해 주세요.` — `apps/admin/src/shared/crud/useCrudRowUpdate.ts:53`. **실패 토스트는 저절로 사라지지 않는다** |
| 예외 처리 | 요청이 중단되면(화면 이탈) 어떤 토스트도 띄우지 않는다 — `apps/admin/src/shared/crud/useCrudRowUpdate.ts:52`. 처리 중에는 세 버튼이 함께 잠긴다 |
| 화면 변경 사항 | Header 배지 · 상태 관리 카드 · 머리 액션(승인이 되면 `계약 만들기` 가 나타난다) |
| 후속 동작 | 목록의 상태 열이 바뀐다 |
| 로그 기록 여부 | 확인하지 못했다 |

**전이표 (§6 이 규칙의 정본이다)** — `draft→sent` · `sent→accepted|rejected` · `accepted` 는 **손으로 갈 곳이 없다**(다음 칸은 상태가 아니라 계약이다) · `rejected`·`expired`·`ordered` 는 종점. `ordered` 는 버튼으로 고를 수 없다 — 계약이 만든다.

### 4.3 계약 만들기 (사슬의 두 번째 화살표)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 승인된 견적으로 계약 초안을 만든다. **계약이 생기는 유일한 문이다** |
| 실행 조건 | 수정 권한 + `contractDraftBlock(상태, 기존계약id) === null` — 즉 ① 이 견적으로 만든 계약이 아직 없고 ② 상태가 `accepted` 또는 `ordered` 일 때 — `apps/admin/src/pages/sales/contracts/types.ts:213-217` |
| 사용자 동작 | `계약 만들기` 클릭 |
| 시스템 처리 | **한 번의 누름이 두 가지 일을 한다**: ① 계약 생성 ② 견적 상태를 `ordered` 로 되돌려 쓰기. 저장소가 그 둘의 결과를 **따로** 돌려준다 — `apps/admin/src/pages/sales/contracts/data-source.ts:168-180` |
| 데이터 처리 | 계약이 견적에서 승계하는 값: 계약명 `{견적번호} 기준 계약` · 거래처(id·이름) · 계약금액=견적 **합계**(공급가액+세액) · `부가세 포함`은 **과세 견적일 때만 참**(면세·영세율 합계에는 세액이 없어 '포함'이라고 적으면 거짓이 된다) · 계약유형=공급계약 · 기간=오늘부터 1년 · 상태=초안 · 서명=미발송 · `quoteId`·`quoteNo` — `apps/admin/src/pages/sales/contracts/types.ts:238-267` |
| Validation | 해당 없음 — 물어볼 것이 없다 |
| API 호출 여부 | Y — 계약 생성 1건(지금은 같은 페이지 저장소). 그 안에서 견적 되돌려 쓰기가 이어진다 |
| 성공 처리 | 계약·견적 조회 캐시를 무효화하고 **계약 수정 폼(`/sales/contracts/{id}/edit`)으로 이동한다.** 토스트는 없다 — 화면이 통째로 바뀌는 것이 곧 결과다 |
| 실패 처리 | 되돌려 쓰기가 실패하면 **이동하지 않고 이 화면에 머문다.** 계약 화면으로 밀어 버리면 운영자는 견적이 안 바뀌었다는 사실을 다음에 목록을 볼 때에야 알게 된다 — `apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:232-234`. 그 자리에 §4.4 의 danger 배너가 뜬다 |
| 예외 처리 | **중복 방어가 두 겹이다**: ① 렌더 시점 조회로 버튼의 존재를 정한다 ② 저장소가 **견적 id 를 멱등키**로 한 번 더 막는다 — 이미 계약이 있으면 새로 만들지 않고 그 계약을 돌려준다(`created: false`) — `apps/admin/src/pages/sales/contracts/data-source.ts:157-161`. 버튼이 잠깐 살아 있다가 죽는 사이에 두 번 누른 사람이 두 번째 계약을 만들지 못하게 한다 |
| 화면 변경 사항 | 안내영역 · 연결·요약(연결된 계약) · 또는 화면 전체(이동) |
| 후속 동작 | 계약 목록에 행이 생기고, 견적 목록의 '계약' 열이 `계약 열기` 로 바뀐다. 그 계약이 체결되면 프로젝트로 이어진다 |
| 로그 기록 여부 | 확인하지 못했다 |

**왜 `accepted` 가 문을 여는가** — 예전에는 운영자가 '수주 전환'을 먼저 누르고 그 다음에 '계약 초안 만들기'를 또 눌렀다. 한 사건('이 거래는 성사됐다')을 두 번 누르게 한 셈이라, 한 번을 빠뜨리면 두 화면이 다른 사실을 말했다. 이제 `ordered` 는 계약이 만들어진 **결과**이고, 계약의 입구는 그 앞 칸인 '승인'이다 — `apps/admin/src/pages/sales/contracts/types.ts:203-207`.

**`ordered` 도 통과시키는 이유** — 계약이 없는데 상태만 `ordered` 인 견적이 존재할 수 있다(되돌려 쓰기만 성공하고 계약이 지워졌거나, 이 규칙 이전의 데이터). 그때 문을 닫으면 그 견적은 영원히 계약을 가질 수 없다 — 수동 등록이 막혀 있어 우회로가 없기 때문이다. **막다른 골목을 만들지 않는다** — `apps/admin/src/pages/sales/contracts/types.ts:209-211`.

### 4.4 ⚠ 반쪽 저장 — 계약은 생겼는데 견적이 안 바뀐 상태

| 항목 | 내용 |

| 실패 처리 | 배너가 새 사유로 갱신된다. 사유는 저장소가 준 문장 그대로: `견적을 찾을 수 없어요. 이미 삭제되었을 수 있어요.` — `apps/admin/src/pages/sales/quotes/data-source.ts:156` 또는 전이 거절 문구(§6) |

## 6. Validation

**이 화면에는 입력 칸이 없다.** 검증 대신 **전이 규칙**이 그 자리를 갖는다. 판정은 `types.ts` 의 순수 함수 하나가 하고, 버튼의 비활성 조건과 저장의 거절 조건이 그 하나를 함께 읽는다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 상태 전이 — 같은 상태 | 지금 상태로는 다시 못 간다 | 버튼 렌더 · 클릭 | `이미 '{상태}' 상태예요.` — `apps/admin/src/pages/sales/quotes/types.ts:264` | 버튼 비활성 + 사유를 툴팁·aria-label 에 |
| 상태 전이 — `accepted` 에서 | 손으로 갈 칸이 없다 | 〃 | `'승인' 다음 칸은 상태가 아니라 계약이에요. '계약 만들기'를 누르면 계약이 생기고 견적이 '수주(계약 진행)'로 바뀌어요.` — `apps/admin/src/pages/sales/quotes/types.ts:267` | 〃 |
| 상태 전이 — 종점 상태 | `rejected`·`expired`·`ordered` 에서는 아무 데도 못 간다 | 〃 | `'{상태}' 상태의 견적은 상태를 되돌리거나 더 바꿀 수 없어요 — 계약·청구가 이 상태를 근거로 삼아요.` — `apps/admin/src/pages/sales/quotes/types.ts:271` | 〃 |
| 상태 전이 — 허용되지 않은 칸 | 표에 없는 목적지 | 〃 | `'{현재}'에서 '{목적지}'{조사} 바꿀 수 없어요. 지금 바꿀 수 있는 것: {허용 목록}.` — `apps/admin/src/pages/sales/quotes/types.ts:292` | 〃 |
| 계약 생성 — 이미 있음 | 그 견적의 계약이 이미 있다 | 버튼 렌더 · 생성 경로 | `이미 계약이 만들어진 견적이에요.` — `apps/admin/src/pages/sales/contracts/types.ts:193` | 버튼을 그리지 않는다 |
| 계약 생성 — 승인 전 | 상태가 `accepted`·`ordered` 가 아니다 | 〃 | `고객이 승인한 견적만 계약으로 넘길 수 있어요. 견적 상세의 상태 관리에서 ‘승인’으로 바꾸세요.` — `apps/admin/src/pages/sales/contracts/types.ts:191-192` | 〃 |
| 청구 생성 — 수주 전 | 상태가 `ordered` 가 아니다 | 〃 | `수주로 전환된 견적만 청구할 수 있어요.` — `apps/admin/src/pages/sales/billing/types.ts:148` | 〃 |
| 청구 생성 — 이미 있음 | 그 견적의 청구가 이미 있다 | 〃 | `이미 청구가 생성된 견적이에요.` — `apps/admin/src/pages/sales/billing/types.ts:149` | 〃 |
| 되돌려 쓰기 — 이미 수주 | `ordered` 면 성공으로 본다 | 되돌려 쓰기 시점 | 없음(멱등) | 사유를 만들지 않는다 |
| 되돌려 쓰기 — 승인 아님 | `accepted` 가 아닌 상태에서 시도 | 〃 | `승인된 견적만 '{수주 라벨}'로 넘길 수 있어요.` — `apps/admin/src/pages/sales/quotes/types.ts:298` | 반쪽 저장 배너에 그대로 실린다 |

**되돌리는 전이를 만들지 않는 이유** — `accepted → sent` 를 열면 곧바로 답할 수 없는 질문이 생긴다: 그 견적으로 이미 계약이 만들어졌다면 그 계약은 무엇이 되는가? 청구가 걸려 있다면 그 청구는? 백엔드가 없어 되돌릴 트랜잭션도 없다. 답할 수 없으면 막는다. 견적서 **내용**을 고치는 길은 그대로 살아 있다('견적 수정') — 막은 것은 상태를 거꾸로 돌리는 일뿐이다 — `apps/admin/src/pages/sales/quotes/types.ts:226-230`.

**'다운로드'는 인쇄·PDF 저장을 말한다** — 권한 축이 걸려 있지 않다. 화면을 볼 수 있으면 인쇄할 수 있다.

**계약·청구 생성은 `update` 권한이 연다** — 이 화면의 리소스는 `sales-quotes` 이고, 계약·청구 생성 버튼도 그 리소스의 `update` 로 게이팅된다(`apps/admin/src/pages/sales/quotes/QuoteDetailPage.tsx:158,314,319`). 계약 리소스의 `create` 권한을 따로 보지 않는다.

**승인·반려 액션은 권한 모델에 없다** — 액션은 다섯이다(`read`·`create`·`update`·`remove`·`export`). 견적의 '승인'·'반려'는 상태 전이이지 권한 축이 아니다.

**권한이 없을 때 화면이 어떻게 되는가** — 읽기가 없으면 본문이 통째로 권한 화면이 된다. 수정이 없으면 **쓰기 컨트롤이 사라지고**(비활성이 아니다) 견적서·연결·인쇄만 남는다.
