# 거래처 상세

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 거래처 상세 |
| 화면 ID | `SCR-SALES-ACCOUNTS-DETAIL` |
| 메뉴 경로 | 영업 관리 > 거래처 > (행 선택) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 거래처 한 곳의 사업자정보·거래조건·담당자를 **읽고**, 그 거래처의 **계약·견적·프로젝트 이력**을 한 화면에서 본다.

**업무 배경** — 두 가지 결함을 한 화면이 함께 푼다 — ① 거래처에는 읽기 전용 상세가 없어서 담당자·여신한도를 **보기만** 하려는 조회 전용 역할도 수정 폼을 열어야 했다(고칠 수 없는 사람에게 편집 화면을 여는 것은 그 자체로 오조작의 초대장이다) ② **거래처 → 그 거래처의 계약/견적/프로젝트 역방향 조회가 앱 전체에 없었다** — 계약도 견적도 거래처를 자유 입력 문자열로 들고 있었으니 애초에 물을 수 없는 질문이었다 — `apps/admin/src/pages/sales/accounts/AccountDetailPage.tsx:3-9`.

**화면 설명** — 머리에 상호·배지 3종·`거래처 수정`, 그 아래 2단(기본 정보 / 거래 조건·신용), 담당자 카드, 그리고 **역방향 조회 카드 3장**(계약 · 견적 · 프로젝트)이 세로로 이어진다.

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자 · 조회 전용 역할.

**사용 시나리오**

1. 견적 목록에서 거래처 이름을 눌러 이 화면에 온다 → 그 거래처의 다른 견적·계약이 몇 건인지 한 번에 본다.
2. 여신한도·결제조건을 확인해 새 계약 조건을 정한다.
3. 담당자 연락처를 확인한다 — 대표담당에는 배지가 붙는다.

**선행 조건** — 로그인 · 읽기 권한 · `sales.pipeline` 모듈.

**후행 처리** — 이 화면은 아무것도 바꾸지 않는다. 수정은 `거래처 수정` 버튼으로 폼에 가서 한다.

**관련 화면** — [거래처 목록](index.md) · [거래처 등록·수정](form.md) · [계약 수정](../contracts/form.md) · [견적 상세](../quotes/detail.md) · [프로젝트 수정](../projects/form.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다.

**관련 기능** — 역방향 조회는 `accountId` 로만 거른다. **미등록('')으로는 걸러 주지 않는다** — 그러면 서로 무관한 미등록 레코드가 한 거래처의 이력인 양 한 표에 모인다. 미등록은 '연결이 없다'는 뜻이지 '같은 거래처'라는 뜻이 아니다 — `apps/admin/src/pages/sales/_shared/account-reference.ts:72-84`.

**관련 API** — 거래처 상세 조회 1건 + **계약·견적·프로젝트 목록 조회 3건**. **백엔드가 없다** — `TODO(backend): GET /api/sales/{contracts,quotes,projects}?accountId=:id — 지금은 목록을 받아 화면에서 거른다(픽스처). 서버가 붙으면 질의만 바뀌고 아래는 그대로다` — `apps/admin/src/pages/sales/accounts/AccountDetailPage.tsx:181-182`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/sales/accounts/AccountDetailPage.tsx` (역방향 카드 `components/RelatedRecordsCard.tsx` · 필터 규칙 `../_shared/account-reference.ts`)

## 4. 기능 명세

### 4.1 거래처 상세 조회

| 항목 | 내용 |
|---|---|
| 기능 목적 | 거래처의 사실을 읽는다 |
| 실행 조건 | 주소에 거래처 id 가 있을 때 |
| 사용자 동작 | 목록의 행 클릭 · 다른 화면의 거래처 링크 · 주소 직접 입력 |
| 시스템 처리 | 상세 1건과 **계약·견적·프로젝트 목록 3건**을 각각 조회한다 |
| 데이터 처리 | 읽기만 한다 |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 4건(상세 1 + 목록 3) |
| 성공 처리 | 머리 + 2단 + 담당자 + 카드 3장이 그려진다. 토스트 없음 |
| 실패 처리 | **없는 것과 고장난 것을 가른다**: `거래처를 찾을 수 없어요. 이미 삭제되었을 수 있어요.` 에는 `목록으로` 만, `거래처를 불러오지 못했어요.` 에는 `다시 시도` + `목록으로` — `apps/admin/src/pages/sales/accounts/AccountDetailPage.tsx:282-305` |
| 예외 처리 | **역방향 카드 3장은 상세와 따로 실패한다** — 계약 조회가 실패해도 견적·프로젝트 카드는 그대로 보이고, 실패한 카드만 배너 + `다시 시도` 를 그린다 |
| 화면 변경 사항 | 화면 전체 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**한 요청으로 합치지 않은 이유** — 모듈마다 어댑터가 따로다. 한 요청으로 합치려면 서버가 필요하고, 지금은 픽스처라 세 목록을 받아 `accountId` 로 거른다. 필터는 순수 함수 하나라 세 구획이 같은 규칙을 쓴다 — `apps/admin/src/pages/sales/accounts/AccountDetailPage.tsx:11-14`.

### 4.2 역방향 조회 (계약 · 견적 · 프로젝트)

| 항목 | 내용 |
|---|---|
| 기능 목적 | '이 거래처와 무엇을 하고 있는가'에 한 화면에서 답한다 |
| 실행 조건 | 세 목록 조회가 각각 끝났을 때 |
| 사용자 동작 | 없음(자동). 각 행의 링크로 그 문서를 연다 |
| 시스템 처리 | 목록을 `accountId` 로 거른다. **미등록('')이면 빈 배열을 돌려준다** — `apps/admin/src/pages/sales/_shared/account-reference.ts:78-84` |
| 데이터 처리 | 읽기만 한다 |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 세 목록 각각 1건 |
| 성공 처리 | 카드마다 표가 그려진다. 표 캡션은 `이 거래처의 {엔티티} 목록 — 각 행의 링크로 해당 화면으로 이동해요.` — `apps/admin/src/pages/sales/accounts/components/RelatedRecordsCard.tsx:77` |
| 실패 처리 | 그 카드만 배너 + `다시 시도` |
| 예외 처리 | **세 상태를 뭉개지 않는다**: 못 불러왔다(에러 + 다시 시도) / 아직 못 읽었다(스켈레톤 2행) / 정말 없다(안내 문구). 셋을 '없습니다' 하나로 합치면 조회가 실패한 거래처가 '거래 이력 없는 거래처'로 보인다 — `apps/admin/src/pages/sales/accounts/components/RelatedRecordsCard.tsx:8-10` |
| 화면 변경 사항 | 그 카드 |
| 후속 동작 | 링크로 각 문서를 연다 |
| 로그 기록 여부 | N |

**연결되지 않은 문서는 여기 나타나지 않는다** — 문의에서 발행된 견적은 거래처가 미등록('')으로 시작한다. 그 견적을 마스터에 연결하지 않으면 이 카드에서 영원히 빠진다. 견적 폼이 그 대가를 warning 으로 미리 말한다 — `apps/admin/src/pages/sales/_shared/AccountSelectField.tsx:214-217`.

## 6. Validation

**이 화면에는 입력 칸이 하나도 없다** — 읽기 전용이다. 거래처의 검증 규칙(사업자번호 체크섬·담당자 이메일 형식 등)은 [거래처 등록·수정](form.md) 이 갖는다.

**역방향 카드에는 별도 권한 판정이 없다** — 이 화면의 읽기 권한만 통과하면 계약·견적·프로젝트 이력이 전부 보인다. 각 도메인의 `read` 권한을 따로 보지 않는다.

**권한이 없을 때 화면이 어떻게 되는가** — 읽기가 없으면 본문이 통째로 권한 화면이 된다. 수정 권한만 없으면 `거래처 수정` 버튼이 사라지고 나머지는 그대로 보인다 — 그것이 이 화면을 만든 이유다(보기만 하려는 사람이 편집 화면을 열지 않게 한다).
