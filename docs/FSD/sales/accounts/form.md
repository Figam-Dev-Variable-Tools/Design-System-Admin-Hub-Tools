# 거래처 등록·수정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 거래처 등록 / 거래처 수정 |
| 화면 ID | `SCR-SALES-ACCOUNTS-FORM` |
| 메뉴 경로 | 영업 관리 > 거래처 > (등록 · 수정) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 거래처 마스터 한 건을 만들거나 고친다. 사업자정보 · 거래 조건 · 담당자(복수) · 거래 상태를 한 폼에서 받는다.

**업무 배경** — 영업 사슬의 네 화면이 이 마스터를 가리킨다. 여기서 만든 행이 견적·계약·프로젝트 폼의 거래처 셀렉트에 나타나고, 그 연결이 있어야 [거래처 상세](detail.md)의 역방향 조회가 성립한다. **영업 관리에서 손으로 만드는 유일한 문서다** — 나머지 네 칸은 앞 칸에서만 생긴다.

**화면 설명** — `/new` 와 `/:id/edit` 가 **같은 컴포넌트**다(`:id` 유무로 등록/수정을 겸한다). 좌우 2단이다: 왼쪽에 입력 카드 4개(사업자 정보 · 거래 조건 · 담당자 · 비고), 오른쪽에 사업자 요약 미리보기.

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자.

**사용 시나리오**

1. 새 거래처와 첫 계약을 앞두고 사업자등록증을 보며 상호·번호·대표자를 넣는다 → **번호가 국세청 체크섬을 통과해야 저장된다.**
2. 담당자를 둘 넣고 한 명을 대표담당으로 지정한다.
3. 견적 폼의 '미등록 거래처' 경고에서 `거래처 등록` 링크를 눌러 이 화면으로 온다.

**선행 조건** — 로그인 · `sales.pipeline` 모듈 · 등록은 `create`, 수정은 `update` 권한.

**후행 처리** — 저장하면 거래처 목록으로 돌아간다. 새 거래처가 네 폼의 셀렉트에 나타난다.

**관련 화면** — [거래처 목록](index.md) · [거래처 상세](detail.md) · 이 마스터를 참조하는 폼 3종([견적](../quotes/form.md) · [계약](../contracts/form.md) · [프로젝트](../projects/form.md)) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다.

**관련 기능** — 견적 폼에서 거래처를 마스터에서 고르면 **사업자등록번호와 대표자가 함께 승계된다**. 그 값의 원본이 이 폼이다.

**관련 API** — 거래처 상세 조회 1건(수정 시) · 등록/갱신 1건. **백엔드가 없다** — `TODO(backend)` 로 어댑터 본문만 교체한다 — `apps/admin/src/pages/sales/accounts/data-source.ts:3-4`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/sales/accounts/AccountFormPage.tsx` (검증 `validation.ts` · 담당자 편집기 `components/AccountContactsField.tsx` · 미리보기 `components/AccountBusinessPreview.tsx` · 사업자번호 규칙 `../_shared/business.ts`)

## 4. 기능 명세

### 4.1 거래처 등록

| 항목 | 내용 |
|---|---|
| 기능 목적 | 마스터에 새 거래처를 만든다 |
| 실행 조건 | `/sales/accounts/new` 진입 · `create` 권한 |
| 사용자 동작 | 값을 채우고 `등록` 클릭 |
| 시스템 처리 | 공통 폼 컨트롤러가 검증 → 제출을 맡는다 |
| 데이터 처리 | 거래처 1건이 생긴다 |
| Validation | §6 전부 |
| API 호출 여부 | Y — 등록 1건. 재시도는 같은 멱등키를 재사용해 서버가 최초 응답을 재생한다 — `apps/admin/src/shared/crud/useCrudForm.ts:122-150` |
| 성공 처리 | 목록으로 이동하고 공통 토스트 `MSG-COMMON-01`(success) `거래처을(를) 등록했어요.`(조사는 이름의 받침이 고른다) — `apps/admin/src/shared/crud/useCrudForm.ts:249,256` |
| 실패 처리 | 폼 배너 `저장하지 못했어요. 잠시 후 다시 시도해 주세요.` + `오류 코드 {참조}`. 이동하지 않고 입력을 유지한다 |
| 예외 처리 | ⚠ **같은 사업자등록번호를 두 번 등록하는 것을 막는 규칙이 없다** — 체크섬만 보고 중복은 보지 않는다 |
| 화면 변경 사항 | 안내영역 · 화면 전체(성공 시 이동) |
| 후속 동작 | 견적·계약·프로젝트 폼의 셀렉트에 나타난다 |
| 로그 기록 여부 | 확인하지 못했다 |

### 4.2 거래처 수정

| 항목 | 내용 |
|---|---|
| 기능 목적 | 기존 거래처의 값을 고친다 |
| 실행 조건 | `/sales/accounts/{id}/edit` 진입 · `update` 권한 |
| 사용자 동작 | 값을 고치고 `저장` 클릭 |
| 시스템 처리 | 상세를 조회해 폼을 채운 뒤 저장 시 한 벌을 통째로 갱신한다 |
| 데이터 처리 | 그 거래처가 갱신된다. **이름을 고치면 거래처 상세·셀렉트의 이름은 바뀌지만, 이미 저장된 견적·계약의 라벨은 그대로다** — 견적서는 발행 시점의 상호를 남겨야 하는 문서이기 때문이다. 조회·링크의 정본은 언제나 `accountId` 이고, 상세 화면은 마스터를 다시 읽어 이름을 되맞춘다 — `apps/admin/src/pages/sales/_shared/account-reference.ts:10-13,59-70` |
| Validation | §6 전부 |
| API 호출 여부 | Y — 조회 1건 + 갱신 1건 |
| 성공 처리 | 목록으로 이동하고 공통 토스트 `MSG-COMMON-01`(success) `거래처을(를) 저장했어요.` |
| 실패 처리 | 폼 배너 + 참조 코드. 입력을 유지한다 |
| 예외 처리 | 서버 필드 거절(422)은 그 입력의 인라인 오류로 꽂히고 포커스가 옮겨간다 |
| 화면 변경 사항 | 안내영역 · 화면 전체(성공 시 이동) |
| 후속 동작 | 목록·상세·셀렉트에 새 값이 반영된다 |
| 로그 기록 여부 | 확인하지 못했다 |
