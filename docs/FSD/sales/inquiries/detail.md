# 영업 문의 처리(상세)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 문의 처리 |
| 화면 ID | `SCR-SALES-INQUIRIES-DETAIL` |
| 메뉴 경로 | 영업 관리 > 문의 > (행 선택) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 문의 한 건을 읽고, 담당자를 배정하고, 처리 상태를 옮기고, 고객답변·내부메모를 이력에 덧붙인다. **그리고 여기가 견적이 태어나는 유일한 문이다.**

**업무 배경** — 영업 사슬의 두 번째 칸(견적)에는 등록 버튼이 없다. 견적을 만들 수 있는 곳은 이 화면의 '처리 상태 → 견적 발행' 하나뿐이다. 빈 폼으로 세운 견적은 원본 문의가 없어 '어느 요청에 대한 견적인가'를 앱이 영영 답할 수 없기 때문이다 — `apps/admin/src/pages/sales/quotes/QuoteFormPage.tsx:6-9`.

**화면 설명** — 좌우 2단이다. 왼쪽 카드에 문의 정보·담당 배정·처리 상태·답변 작성칸이, 오른쪽 카드에 처리 이력 타임라인이 있다. 저장 버튼 하나가 세 가지(담당·상태·작성 글)를 **한 번에** 커밋한다.

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자.

**사용 시나리오**

1. 접수된 견적요청을 연다 → 담당자를 넣고 상태를 '처리중'으로 옮기며 내부메모를 남긴다 → '처리 저장'.
2. 금액을 잡을 준비가 되면 상태를 '견적 발행'으로 바꾼다 → 저장하는 순간 이 문의를 승계한 견적이 자동 생성되고 타임라인에 그 사실이 남는다 → '발행된 견적 보기'로 넘어가 품목·단가를 채운다.
3. 이미 견적이 나간 문의를 다시 연다 → '이미 이 문의로 견적이 발행되었어요.' 안내가 뜨고, 상태를 다시 '견적 발행'으로 골라 저장해도 **견적은 새로 생기지 않는다**.

**선행 조건** — 로그인 · 읽기 권한 · `sales.pipeline` 모듈. 저장하려면 수정 권한이 필요하다.

**후행 처리** — 담당·상태·타임라인이 저장되고 목록에 반영된다. 상태가 '견적 발행'이면 **견적 저장소에 견적이 1건 생기고**, 그 견적은 이 문의를 `sources` 로 들고 있다(양방향 링크). 그 견적이 `SCR-SALES-QUOTES` 목록에 나타난다.

**관련 화면** — [문의 목록](index.md) `SCR-SALES-INQUIRIES` · [견적 상세](../quotes/detail.md) `SCR-SALES-QUOTES-DETAIL` · [견적 등록·수정](../quotes/form.md) `SCR-SALES-QUOTES-FORM` · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다(목록의 하위 화면이다).

**관련 기능** — 견적 자동 발행은 **문의 저장의 부수효과**로 일어난다. 화면이 '문의 저장'과 '견적 생성'을 두 번 호출하면 하나만 성공하는 창이 생기고, 그 창에서 새로고침하면 상태는 '견적 발행'인데 견적이 없는 문의가 남는다 — `apps/admin/src/pages/sales/inquiries/data-source.ts:6-8`.

**관련 API** — 문의 상세 조회 1건 · 문의 갱신 1건. **백엔드가 없다.** 서버가 맡을 계약은 `TODO(backend): GET /api/sales/inquiries · GET/PUT /api/sales/inquiries/:id … '견적 발행' 전이는 견적 생성을 동반한다 — 서버는 문의 갱신 + 견적 생성 + 양방향 링크를 한 트랜잭션으로 처리하고, 이미 발행된 문의의 재발행은 기존 견적을 돌려준다(멱등)` — `apps/admin/src/pages/sales/inquiries/data-source.ts:160-162`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/sales/inquiries/InquiryDetailPage.tsx` (타임라인 `components/InquiryTimeline.tsx` · 규칙 `types.ts` · 발행 배선 `data-source.ts`)

| 사용자 동작 | 처리 상태 셀렉트에서 `견적 발행` 선택 → `처리 저장` |
| 시스템 처리 | 문의 갱신과 **한 덩이로** 견적을 만든다. 승계 값은 한 곳에서만 정한다(`buildQuoteFromSources`): 거래처 라벨=문의의 회사명 · 담당자=문의 고객명 · 원본 문의 참조(번호·창구·문의내용). 견적번호는 저장소가 `Q-YYYYMMDD-NNN` 으로 채번하고, 상태는 `작성중`, 유효기간은 견적일 +30일이다 — `apps/admin/src/pages/sales/quotes/types.ts:456-487` |
| 데이터 처리 | 문의에 `quoteId` 가 박히고 타임라인에 `견적 자동 생성 — {견적번호}` 가 덧붙는다 — `apps/admin/src/pages/sales/inquiries/data-source.ts:155`. 견적 쪽에는 이 문의가 `sources` 로 남는다(양방향) |
| Validation | 해당 없음 — 고를 것은 상태 하나다 |
| API 호출 여부 | Y — 문의 갱신 1건(견적 생성이 그 안에서 함께 일어난다) |
| 성공 처리 | 토스트 `문의 처리 내용을 저장하고 견적을 발행했어요.` — `apps/admin/src/pages/sales/inquiries/InquiryDetailPage.tsx:195`. 재조회 후 '이미 발행됨' 안내와 `발행된 견적 보기` 버튼이 생긴다 |
| 실패 처리 | §4.2 와 같은 배너. **문의도 견적도 저장되지 않는다** — 둘이 한 덩이라 반쪽 상태가 생기지 않는다 |
| 예외 처리 | **거래처는 미등록('')으로 시작한다.** 문의는 고객이 적은 회사명 문자열만 갖고 있어 거래처 마스터의 어느 행인지 모른다. 이름을 맞춰 id 를 추측하면 동명이인 거래처에 남의 견적이 붙는다 — `apps/admin/src/pages/sales/quotes/types.ts:451-454`. 그 대가(거래처 상세의 견적 이력에서 빠진다)는 견적 폼이 경고로 드러낸다. **영업 문의는 품목을 만들지 않는다** — 무엇을 파는지 모르기 때문에 품목 없는 견적으로 태어나고 운영자가 채운다 — `apps/admin/src/pages/sales/inquiries/data-source.ts:136` |
| 화면 변경 사항 | 안내영역 · 이력영역 |
| 후속 동작 | 견적 목록·거래처 역방향 조회에 그 견적이 나타난다. 이후 견적 상세에서 계약·청구로 이어진다 |
| 로그 기록 여부 | 확인하지 못했다 |

**중복 발행이 두 겹으로 막힌다** — ① 어댑터가 문의의 `quoteId` 를 보고 이미 발행됐으면 아무것도 만들지 않는다(`apps/admin/src/pages/sales/inquiries/data-source.ts:126`) ② 견적 저장소가 원본 문의 id 로 한 번 더 교차 확인한다(`apps/admin/src/pages/sales/quotes/data-source.ts:181-184`). 그래서 상태를 오가도, 저장을 연타해도 견적은 하나다.

**등록·삭제 액션이 없다** — 문의는 고객 채널이 만들고 이 화면에는 삭제가 없다.

**권한이 없을 때 화면이 어떻게 되는가** — 읽기 권한이 없으면 본문이 `접근 권한이 없어요` 화면으로 바뀐다. **수정 권한만 없을 때 이 화면은 형제 화면들과 다르게 동작한다** — 견적 상세·청구 상세는 쓰기 컨트롤을 아예 그리지 않는데(`canUpdate` 판정), 이 화면은 입력칸과 저장 버튼을 그대로 그린다. 막는 것은 서버뿐이다.
