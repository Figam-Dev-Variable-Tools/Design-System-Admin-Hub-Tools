# 계약 목록

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 계약 |
| 화면 ID | `SCR-SALES-CONTRACTS` |
| 메뉴 경로 | 영업 관리 > 계약 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 맺은 계약을 한 표에 모아 기간·금액·상태를 훑고, 갱신이 임박한 건을 집어낸다. 원 견적으로 되짚는 역링크도 여기 있다.

**업무 배경** — 계약은 사슬의 세 번째 칸이다. 문서(플로우 차트)에는 견적 → 계약 화살표가 그려져 있었지만 **코드의 계약에는 견적을 가리키는 필드가 한 칸도 없었다.** 그래서 '이 계약이 얼마짜리 견적에서 나왔는가'를 앱이 답하지 못했고, 수주 전환된 견적은 그 다음이 없는 종점이었다 — `apps/admin/src/pages/sales/contracts/types.ts:45-51`. 그 화살표를 데이터에 넣은 결과가 이 목록의 '원 견적' 열이다.

**화면 설명** — 검색 + 상태 필터 툴바와 7열 표다. 선택 체크박스와 삭제가 있지만 **등록 버튼이 없다**. 행을 누르면 계약 수정 폼이 열린다(계약에는 읽기 전용 상세가 없다).

**주요 사용자** — 최상위 관리자 · 영업 담당 운영자.

**사용 시나리오**

1. 상태를 '진행중'으로 걸러 살아 있는 계약만 본다 → 상태 열의 `갱신임박` 배지가 붙은 건을 먼저 연다.
2. 고객이 계약 금액을 묻는다 → 계약명·거래처로 검색해 행을 찾고 금액 열을 읽는다.
3. '원 견적' 열의 견적번호를 눌러 그 계약의 근거가 된 견적서를 연다.

**선행 조건** — 로그인 · 읽기 권한 · `sales.pipeline` 모듈. **행이 생기려면 앞 칸(승인된 견적)이 있어야 한다.**

**후행 처리** — 이 목록에서 일어나는 변경은 삭제뿐이다. 상태·서명·기간 편집과 프로젝트 생성은 계약 수정 폼의 일이다.

**관련 화면** — [계약 수정](form.md) `SCR-SALES-CONTRACTS-FORM` · [견적 상세](../quotes/detail.md) `SCR-SALES-QUOTES-DETAIL`(계약이 태어나는 곳) · [프로젝트 수정](../projects/form.md) `SCR-SALES-PROJECTS-FORM` · [거래처 상세](../accounts/detail.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `영업 관리 > 계약`.

**관련 기능** — 계약이 **진행중 + 서명완료**가 되면 그 계약에서 프로젝트를 만들 수 있다. 그 판정(`isConcludedContract`)은 계약 도메인이 갖고, 버튼은 계약 수정 폼에 있다 — `apps/admin/src/pages/sales/contracts/types.ts:220-236`.

**관련 API** — 계약 목록 조회 1건 · 삭제 N건. **백엔드가 없다** — `TODO(backend): GET/POST /api/sales/contracts · GET/PUT/DELETE /api/sales/contracts/:id` — `apps/admin/src/pages/sales/contracts/data-source.ts:182`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/sales/contracts/ContractListPage.tsx` (규칙 `types.ts` · 저장소 `data-source.ts`)

### 4.3 계약 삭제 (단건 · 일괄)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 잘못 만든 계약을 지운다 |
| 실행 조건 | 삭제 권한이 있어 컨트롤이 보일 때 |
| 사용자 동작 | 행 휴지통 → `DLG-COMMON-DELETE-ONE` · 선택 후 일괄 삭제 → `DLG-COMMON-DELETE-BULK` |
| 시스템 처리 | 확인 시 삭제 요청을 보낸다 |
| 데이터 처리 | 그 계약이 사라진다. **그 계약을 지우면 원 견적의 '계약 만들기' 버튼이 다시 살아난다** — 견적 상세가 계약 존재 여부를 렌더 시점에 다시 조회하기 때문이다. 막다른 골목이 생기지 않는다 |
| Validation | 선택 0건이면 확인해도 요청을 만들지 않는다 |
| API 호출 여부 | Y — 대상 건수만큼 |
| 성공 처리 | 목록이 갱신되고 공통 토스트(`MSG-COMMON-02`·`MSG-COMMON-03`)가 뜬다 |
| 실패 처리 | 다이얼로그가 닫히지 않고 배너로 이유를 말한다: `이미 삭제된 계약이에요.` — `apps/admin/src/pages/sales/contracts/data-source.ts:132` |
| 예외 처리 | 취소·Esc·딤 클릭은 진행 중 요청을 중단하고 실패로 치지 않는다 |
| 화면 변경 사항 | 다이얼로그 · 목록영역 · 선택 바 |
| 후속 동작 | 견적 목록의 '계약' 열이 `—` 로 돌아간다 |
| 로그 기록 여부 | 확인하지 못했다 |

**⚠ 계약을 지워도 앞뒤 칸이 정리되지 않는다** — ① 그 견적의 상태는 `수주(계약 진행)` 그대로 남는다(되돌리는 전이가 없다). 견적 상세는 그 상태를 종점으로 보고 아무 경고도 띄우지 않는다 — 계약이 사라진 `ordered` 견적은 조용히 남는다. ② 그 계약에서 만든 프로젝트의 `contractId` 도 그대로 남아 '원 계약 보기' 가 없는 계약을 가리킨다. 이 정리를 하는 코드가 없다.
