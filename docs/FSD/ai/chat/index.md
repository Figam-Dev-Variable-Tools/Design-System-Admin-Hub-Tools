# 새 채팅

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 새 채팅 |
| 화면 ID | `SCR-AI-CHAT` |
| 메뉴 경로 | AI > AI 에이전트 > 새 채팅 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 운영자가 `@데이터` 를 멘션하고 조건을 적으면, 그 조건으로 **실제 데이터를 조회해** 표로 돌려준다. 목록 화면을 열고 필터를 여러 번 누르는 대신 한 줄로 묻는다.

**업무 배경** — "이번달 가입한 VIP 몇 명이야?" 같은 질문은 회원 관리 화면에서 등급 필터와 기간 필터를 각각 세팅해야 답이 나온다. 그 세팅을 문장 한 줄로 접는 것이 이 화면의 존재 이유다. **답을 짓는 언어 모델은 이 앱에 없다** — 못 알아들은 요청은 알아듣지 못했다고 말하고 할 수 있는 것을 대신 보여 준다 — `apps/admin/src/pages/ai/NewChatPage.tsx:5-8`.

**화면 설명** — 왼쪽에 대화 기록 레일(검색 · 새 채팅 · 오늘/이전), 오른쪽에 메시지 목록과 하단의 한 줄 입력줄이 있다. 입력줄에는 첨부 · 질문 입력 · 응답 모드 선택 · 음성 입력 · 보내기가 한 줄로 놓인다.

**주요 사용자** — 최상위 관리자 · 운영자. 이 화면 자체에 역할별 분기는 없다(§9).

**사용 시나리오**

1. 화면을 열면 `무엇을 조회할까요?` 와 예시 세 줄이 뜬다. 예시를 눌러 그대로 묻는다.
2. `@` 를 치면 데이터 후보가 뜨고, 고르면 입력에 별칭이 채워진다. 뒤에 조건을 이어 쓴다.
3. 보내면 조회가 돌고 표와 건수가 답으로 온다. 조회에 걸린 실제 시간이 위에 표시된다.
4. 표 아래 `↳` 제안을 눌러 조건을 넓히거나 좁혀 다시 묻는다.
5. 좌측 레일에서 지난 대화를 골라 이어 묻는다.

**선행 조건** — 로그인. 이 화면의 읽기 권한. **AI 모델 연동은 선행 조건이 아니다** — 연동 없이도 `규칙 기반 조회` 모드가 동작한다(§4.6).

**후행 처리** — 대화와 메시지가 만들어진다. **다만 그 대화는 브라우저 메모리에만 있고 새로고침하면 사라진다**(§7). 조회 대상 데이터는 아무것도 바뀌지 않는다.

**관련 화면** — `SCR-AI-CONVERSATIONS`(대화 목록 — 여기서 대화를 열거나 지운다). 답변의 표에서 이어지는 목록 화면: `SCR-USERS-MEMBERS` · `SCR-PRODUCTS` · `SCR-SUPPORT-TICKETS` · `SCR-PROGRAMS`. 응답 모드를 여는 화면: `SCR-SETTINGS-API-KEYS`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `AI > AI 에이전트` 의 첫 항목 — `apps/admin/src/shared/layout/nav-config.ts:255-256`. **플랜으로 잠긴다** — `ai.agent` 모듈이 `/ai` 가지와 두 잎을 지배하며 최소 티어는 `엔터프라이즈` 다 — `apps/admin/src/shared/entitlements/module-resources.ts:81-85` · `apps/admin/src/shared/entitlements/plan.ts:243-248`.

**관련 기능** — API Key 설정(`SCR-SETTINGS-API-KEYS`)에서 프로바이더를 연동하면 잠겨 있던 응답 모드 셋이 열린다. 조회 대상 데이터의 값 목록(등급·판매상태 등)은 각 도메인이 소유하고 이 화면은 배선으로 받아 쓴다.

**관련 API** — 목적 넷. ① 대화 목록 조회 ② 대화 1건 조회 ③ 질문 1건 처리(대화가 없으면 새로 연다) ④ 대화 삭제(대화 목록 화면이 쓴다). **백엔드가 없다** — 지금은 결정적 파서가 로컬 데이터를 걸러 답하며, 언어 모델이 붙을 자리는 질문 처리 함수 안쪽이다. 그때도 지켜야 할 계약 셋이 코드에 명시되어 있다: 답변 종류가 값으로 구분될 것 · 결과의 숫자는 조회 결과일 것 · 알아듣지 못한 요청은 안내로 남을 것 — `apps/admin/src/pages/ai/data-source.ts:7-21`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

| 연동 있음 | 네 모드가 모두 열린다 | 전부 ○ | N | `{프로바이더 이름들} 연동됨 — 모든 응답 모드를 고를 수 있어요.` |
| 저장 중 | 해당 없음 — 저장 기능이 없다 | — | N | — |

| 플랜 잠김 | 본문 대신 잠금 화면 | `플랜 보기` ○ · `현재 플랜 확인` ○ | N | `AI 에이전트 기능은 엔터프라이즈 플랜부터 사용할 수 있어요. …` |

**`0건`과 `모름`을 가르는 자리** — 결과 0건은 조회가 돌려준 사실이고, 데이터가 아직 연결되지 않은 경우는 **미배선 답변**이라는 다른 종류로 갈린다(§4.3). 0 을 그리는 대신 연결되지 않았다고 말한다.

**⚠ 조회 대상 데이터의 권한이 이 화면에 반영되는지 확인하지 못했다** — 이 화면은 회원·상품·문의·프로그램을 조회해 표로 보여 주는데, 그 화면들의 읽기 권한이 없는 역할이 이 화면을 통해 같은 데이터를 볼 수 있는지 막는 코드를 찾지 못했다. 조회기는 도메인 배선에서 오고 그 배선에 권한 판정이 없다.

**판정 순서는 인증 → 플랜 → 권한이다** — `plan.ts:10-17`.

### 10.1 `?c=` 가 라우트가 아닌 이유

**열린 대화는 주소 쿼리에 실리는 화면 상태이지 다른 화면이 아니다.** 그래서 `SCR-AI-CHAT-DETAIL` 같은 화면 ID 를 만들지 않는다.

| 판정 근거 | 사실 |
|---|---|
| 라우트가 하나다 | `/ai/chat` 하나이며 `/ai/chat/:id` 같은 경로가 없다. 대화 id 는 `?c=` 로만 실린다 — `NewChatPage.tsx:20-21,36` |
| 같은 컴포넌트다 | 대화가 열려 있든 없든 같은 화면이 그려진다. 레일·입력줄·응답 상태 영역이 그대로 있고, 본문의 목록만 비어 있거나 차 있다 |
| 화면의 구성이 바뀌지 않는다 | 영역(§2)·컴포넌트(§3)·이벤트(§5)·권한(§9)이 전부 같다. 대화를 여는 것은 `새 채팅` 을 누르는 것과 같은 층의 동작이다 |
| 인벤토리도 그렇게 센다 | `?c=` 는 '탭이 아닌 URL 상태'로 분류되어 있고, 이 항목의 문서는 **부모의 §3·§8** 이 갖는다 — [인벤토리](../../inventory.md) 의 '탭이 아닌 URL 상태' 표 |

**그럼에도 주소에 싣는 이유는 라우트의 이유와 다르다** — 새로고침·뒤로가기·링크 공유에서 열린 대화를 잃지 않기 위해서다 — `NewChatPage.tsx:11-12`. 같은 이유로 레일 검색어도 `?q=` 에 싣는다. 즉 `?c=` 는 **'다른 화면으로 간다'가 아니라 '이 화면의 상태를 복원한다'** 는 장치다.

**그 구분이 실제로 만드는 차이** — `?c=` 가 라우트였다면 대화를 열 때마다 히스토리가 쌓이고 뒤로가기가 대화 사이를 오갔을 것이다. 지금은 대화를 열면 히스토리에 쌓이지만(`?c=` 는 교체가 아니라 추가다 — `NewChatPage.tsx:201-205`) 검색어는 쌓이지 않는다(교체다 — `NewChatPage.tsx:161`). 두 상태가 같은 주소에 실리면서 다른 히스토리 정책을 갖는 것 자체가, 이것이 '화면'이 아니라 '상태'라는 증거다.

**이 화면에는 팝업도 확인 다이얼로그도 없고, 결과 통지 토스트도 없다** — 인벤토리 §9.3 의 화면별 토스트 표에 `pages/ai/NewChatPage.tsx` 가 올라 있지 않다. 성공도 실패도 **대화 안의 메시지**로 말하기 때문이다.
