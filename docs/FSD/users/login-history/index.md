# 로그인 이력

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 로그인 이력 |
| 화면 ID | `SCR-USERS-LOGIN-HISTORY` |
| 메뉴 경로 | 사용자 관리 > 로그인 이력 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 누가 언제 어디서 로그인했고 무엇이 실패했는지를 본다. **읽기 전용 감사 로그**이며, 연속 실패를 배지로 드러내 계정 탈취 시도를 눈에 띄게 한다.

**업무 배경** — 계정 사고는 대개 '어느 IP 가 무엇을 두드렸나'로 시작한다. 그 질문에 답하는 화면이 여기 하나여야 한다. 동시에 **감사 기록은 불변이어야 한다** — 지울 수 있는 감사 로그는 감사 로그가 아니다. 침입자가 가장 먼저 지우는 것이 자기 흔적이기 때문이다.

**화면 설명** — 좌측은 필터 셋(결과 · 계정 유형 · 기간)과 안내문, 우측은 검색·내보내기 툴바 → 요약 줄(총 건수 + 실패 건수) → 표 → 페이지네이션이다. 기간에서 '직접 지정'을 고르면 시작일·종료일 입력이 그 아래에 열린다.

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. '실패'만 걸고 최근 30일을 훑어 연속 실패 배지가 붙은 계정을 찾는다.
2. 특정 IP 로 검색해 그 주소가 무엇을 두드렸는지 본다.
3. 행을 눌러 그 계정의 회원 상세 또는 운영자 상세로 건너뛴다.
4. 감사 자료로 제출할 구간을 직접 지정해 CSV 로 받는다.

**선행 조건** — 로그인 · 이 화면의 `read` 권한.

**후행 처리** — 이 화면은 **아무것도 바꾸지 않는다.** 조회·필터·검색·내보내기 넷이 전부다.

**관련 화면** — `SCR-USERS-MEMBERS-DETAIL` · `SCR-USERS-ADMINS-DETAIL`(행 클릭 목적지 둘) · `SCR-LOGIN`(여기 기록되는 사건이 일어나는 곳). 이 화면에는 팝업이 없다. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `사용자 관리 > 로그인 이력`(`/users/login-history`) — `apps/admin/src/shared/layout/nav-config.ts:132`.

**관련 기능** — 로그인 화면은 미등록 계정에도 계정 존재 여부를 알리지 않는 문구를 보이지만, **이 화면은 진짜 사유를 기록한다** — 그것을 보라고 이 화면이 있다 — `apps/admin/src/pages/login-history/types.ts:34-36`.

**관련 API** — 목적 둘. ① 이력 조회(결과·계정 유형·기간·검색어·페이지) ② 내보내기용 전체 조회. **쓰기 API 가 계약에 하나도 없다 — 그것이 설계다.** **백엔드가 없다** — 연동 지점은 `TODO(backend): GET /api/login-history` · `GET /api/login-history/export` — `apps/admin/src/pages/login-history/data-source.ts:112,135`. 서버가 지켜야 할 계약 둘이 주석에 적혀 있다: 응답은 이미 최신순으로 정렬돼 있어야 하고, **연속 실패 횟수는 서버가 계산해 내려준다**(페이지네이션된 목록에서 프런트가 세면 페이지 경계에서 값이 왜곡된다) — `data-source.ts:112-114`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/login-history/LoginHistoryPage.tsx` (필터 `components/LoginHistoryFilters.tsx` · 툴바 `components/LoginHistoryToolbar.tsx` · 표 `components/LoginHistoryTable.tsx` · 기간 `period.ts` · 검증 `validation.ts` · 모델 `types.ts` · 어댑터 `data-source.ts`)

## 4. 기능 명세

### 4.1 이력 조회 · 필터 · 검색

| 항목 | 내용 |
|---|---|
| 기능 목적 | 결과 · 계정 유형 · 기간 · 검색어로 좁힌 로그인 시도를 본다 |
| 실행 조건 | 화면 진입 또는 조건 변경. **구간이 유효할 때만 조회한다** |
| 사용자 동작 | 좌측 필터 클릭 · 검색어 입력 · 페이지 이동 |
| 시스템 처리 | 세 축은 AND 로 걸린다. 검색은 계정·이름·IP 를 훑는다 — `data-source.ts:90-110`. 조회 조건은 **화면 안 상태**이고 주소에 실리지 않는다 |
| 데이터 처리 | 없음(읽기만) |
| Validation | 프리셋은 코드가 만들어 틀릴 수 없다. **직접 지정만 검증을 거친다**(§6) |
| API 호출 여부 | Y — 이력 조회 1건. 좌측 배지 숫자가 같은 응답에 실려 온다 |

| 실패 처리 | 표·요약·페이지네이션을 그리지 않고 danger 배너 + `다시 시도`: `로그인 이력을 불러오지 못했어요.` — `LoginHistoryPage.tsx:277` |
| 예외 처리 | ① 조건이 바뀌면 1페이지로 돌아간다 ② 조건을 좁혀 총 페이지가 줄면 **마지막 페이지로 보정**한다 — `LoginHistoryPage.tsx:177-181` ③ 재조회 중에는 이전 행을 유지한다(스켈레톤은 최초 로드에만) ④ **입력이 유효하지 않으면 조회 자체를 걸지 않는다** — `LoginHistoryPage.tsx:159-161` |
| 화면 변경 사항 | 목록영역 · 요약 줄 · 좌측 배지 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N — 이 화면은 아무것도 쓰지 않는다 |

**보이는 문구와 기록되는 사유는 다르다** — 로그인 화면은 미등록 계정에도 계정 존재 여부를 알리지 않는 문구를 보인다(계정 열거를 막기 위해서다). 그러나 **관리자용 감사 이력은 진짜 사유를 기록한다** — `미등록 계정` 이 실패 사유 목록에 있는 이유다 — `types.ts:25-36`.

**좌측 배지의 스코프** — 결과·유형 배지는 **기간 안에서만** 센다(결과·유형·검색어와 무관하다). 기간이 이 화면의 스코프이고, 나머지 둘은 그 안을 나누는 축이기 때문이다. 그래서 '이 기간에 성공 N · 실패 M'으로 읽힌다 — `types.ts:156-161`.

### 4.3 CSV 내보내기

| 항목 | 내용 |
|---|---|
| 기능 목적 | 현재 조건에 걸린 **전체**(현재 페이지가 아니다)를 감사 자료로 받는다 |

| 후속 동작 | 없음 |
| 로그 기록 여부 | 확인하지 못했다 — 개인정보 반출을 관리자 로그에 남기는 배선을 코드에서 찾지 못했다 |

**CSV 열은 10개다** — `시각` · `계정` · `이름` · `유형` · `결과` · `실패 사유` · `연속 실패` · `IP` · `브라우저` · `OS` — `data-source.ts:154-165`. **실패를 성공 톤으로 옮겨 적지 않는다** — 색은 파일로 옮겨지지 않으므로 문자열이 스스로 말해야 한다.

## 6. Validation

**프리셋은 검증하지 않는다** — 코드가 만들어 틀릴 수 없다. **직접 지정만 사람이 친다.** 규칙의 정본은 스키마다 — `validation.ts:52-97`. 감사 로그에는 두 가지 불가능이 있다: 미래는 없고, 시작이 끝보다 뒤일 수 없다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 시작일 | **실재하는 날짜**일 것(형식만 맞는 `2026-02-31` 은 통과하지 못한다) | 값이 바뀔 때마다 | `날짜를 YYYY-MM-DD 형식으로 입력하세요.` — `apps/admin/src/pages/login-history/validation.ts:46` | 그 칸 아래 인라인 · **조회하지 않는다** |
| 종료일 | 〃 | 〃 | 〃 | 〃 |
| 종료일 | **오늘(서울 기준)을 넘지 않을 것** | 〃 | `미래 날짜는 조회할 수 없어요. 감사 기록에 미래는 없어요.` — `apps/admin/src/pages/login-history/validation.ts:73` | 종료일 칸 아래 인라인 |
| 시작일 | 종료일보다 늦지 않을 것 | 〃 | `시작일은 종료일보다 늦을 수 없어요.` — `apps/admin/src/pages/login-history/validation.ts:82` | 시작일 칸 아래 인라인 |
| 구간 | **최대 90일** | 〃 | `조회 기간은 최대 90일이에요. (선택한 기간 {N}일)` — `apps/admin/src/pages/login-history/validation.ts:93` | **두 칸 아래 따로** 뜬다(어느 한 칸의 잘못이 아니다) |

**한쪽이라도 날짜가 아니면 비교하지 않는다** — 이미 그 칸에 이슈가 붙었기 때문이다 — `validation.ts:65-66`.

**'오늘'의 기준은 서울이다** — 실행 시각이 아니라 주입받은 기준 시각으로 판정하므로 테스트가 오늘을 고정할 수 있다 — `validation.ts:50-51,108`.

**저장을 막는 검증은 없다** — 이 화면에는 저장이 없다. 위 다섯은 전부 **조회를 막는** 규칙이다.

**입력 오류로 조회를 막은 상태는 '조회 결과 없음'과 다르다** — 그때 요약 줄이 `조회 기간을 확인해 주세요.` 로 바뀌고 표는 **스켈레톤도 빈 상태도 아닌** 이전 상태로 남는다(`LoginHistoryPage.tsx:250,263`). 빈 표를 보여 주면 사용자가 *기록이 지워졌다*고 의심하기 때문이다.

## 8. 화면 상태(State)

| 상태 | 화면 표시 | 버튼 활성화 여부 | Loading 여부 | 사용자 안내 메시지 |
|---|---|---|---|---|
| 최초 로딩 | 표가 스켈레톤 10행. 좌측 필터와 툴바는 그대로 있다 | 내보내기 ○ | Y | `불러오는 중…` |
| 데이터 있음 | 표 + 요약 줄 + 페이지네이션 | 전부 ○ | N | `전체 {N}건` |
| 데이터 있음(실패 포함) | 실패 행이 danger 색조 + ✕ 아이콘 + 사유 + (2회 이상이면) 연속 실패 배지 | 전부 ○ | N | `전체 {N}건` · `이 기간의 로그인 실패 {M}건` |
| 데이터 없음 | 표 골격 + `조회된 로그인 이력이 없어요.` | 내보내기 ○(0건 파일이 나간다) | N | `전체 0건` |
| 조회 실패 | 표·요약·페이지네이션을 그리지 않고 danger 배너 | `다시 시도` ○ | N | `로그인 이력을 불러오지 못했어요.` |
| **입력 오류(구간 무효)** | **조회 자체를 걸지 않는다.** 표는 스켈레톤이 되지 않는다. 두 날짜 칸에 오류 문구가 뜬다 | 내보내기 ○(누르면 아무 요청도 만들지 않는다) | N | `조회 기간을 확인해 주세요.` + 각 칸의 §6 문구 |
| 재조회 중 | **이전 행을 그대로 유지한다** — 조건을 만질 때마다 표가 깜빡이지 않는다 | 전부 ○ | Y(표시는 없다) | 이전 건수를 계속 보여 준다 |
| 내보내는 중 | 내보내기 버튼이 `내보내는 중…` 으로 잠긴다 | 그 버튼만 × | Y | 라벨이 곧 안내다 |
| 저장 중 | **해당 없음 — 이 화면에는 저장이 없다** | — | N | — |
| 읽기 전용(권한) | **이 화면은 언제나 읽기 전용이다.** 쓰기 컨트롤이 존재하지 않는다. 권한 차이는 내보내기 버튼의 유무로만 나타난다 | — | N | 없음(부재로 말한다) |
| 미등록 계정 행 | 계정이 링크가 아니고 이름 자리가 `—`. 행을 눌러도 아무 일도 일어나지 않는다 | — | N | 없음 |

**`null`(모름) 과 0 을 가르는 자리 둘**

| 자리 | 모름 | 0 |
|---|---|---|
| 좌측 배지 숫자 | 응답이 오기 전에는 `null` 로 넘겨 배지를 **그리지 않는다** — `LoginHistoryPage.tsx:226-227` | 응답이 온 뒤 0건이면 `0` 을 그린다 |
| 이름 | 미등록 계정은 가리킬 사람이 **없다** — `—` 로 그린다 | 해당 없음 |

**연속 실패 0과 1을 배지로 구분하지 않는다** — 1회 실패는 오타이고 **2회부터가 신호**이므로, 그 아래는 배지를 달지 않는다. 값 자체는 CSV 의 `연속 실패` 열에 그대로 나간다.

## 9. 권한

| 권한 | 조회 | 등록 | 수정 | 삭제 | 다운로드 | 업로드 | 승인 | 반려 |
|---|---|---|---|---|---|---|---|---|
| 슈퍼어드민(전체권한) | ○ | — | — | — | ○ | — | — | — |
| 운영자 | 역할의 `page:/users/login-history` 판정을 따른다 | — | — | — | 〃 | — | — | — |
| 뷰어 | ○ | — | — | — | ○ | — | — | — |
| 권한 없음 | × | — | — | — | × | × | × | × |

**등록·수정·삭제·업로드·승인·반려는 이 화면에 존재하지 않는 동작이다.** 권한 매트릭스에서 `create`·`update`·`remove` 를 켜도 이 화면에 나타나는 것이 없다 — 감사 기록은 불변이기 때문이다.

| 액션 | 이 화면에서 무엇을 여는가 | 근거 |
|---|---|---|
| `read` | 화면 진입 · 조회 · 행 클릭 이동 | `RequirePermission` 이 라우트를 덮는다 |
| `export` | 내보내기 버튼 | `LoginHistoryPage.tsx:109` → `components/LoginHistoryToolbar.tsx:82` |

**`export` 만 별도로 게이팅한 이유** — 이 화면의 유일한 쓰기성 액션이 내보내기인데, 그 하나가 **계정·이름·IP 가 담긴 파일을 뽑아낸다**. 로그 화면과 같은 규칙으로 권한이 없으면 버튼을 **비활성이 아니라 부재**로 둔다 — `LoginHistoryPage.tsx:105-108`.

**권한이 없을 때 화면이 어떻게 되는가**

| 없는 것 | 결과 |
|---|---|
| 화면 읽기(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다. 사이드바에서도 이 잎이 사라진다 |
| `export` | 내보내기 버튼이 **사라진다**(비활성이 아니다). 조회·검색·필터는 그대로다 |
| `create`·`update`·`remove` | 화면이 달라지지 않는다 — 그 액션에 걸린 컨트롤이 애초에 없다 |
| 플랜(엔타이틀먼트) | 해당 없음 — `/users/**` 는 어떤 모듈에도 매핑되어 있지 않다 |
