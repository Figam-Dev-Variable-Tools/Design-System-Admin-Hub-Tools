# 동의 이력

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 동의 이력 |
| 화면 ID | `SCR-USERS-CONSENTS` |
| 메뉴 경로 | 사용자 관리 > 동의 이력 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 동의에 관한 세 질문에 한 화면에서 답한다. ① 우리는 무엇에 동의를 받는가, 그중 무엇이 필수인가 ② 누가 언제 무엇에 동의하고 철회했는가 ③ 지금 다시 물어야 할 사람과 지워야 할 기록은 누구인가.

**업무 배경** — 법이 정한 두 가지가 이 화면의 근거다. ① 동의는 **항목마다 구분해** 알리고 받아야 한다 — 마케팅 수신 동의를 필수로 강제하면 최대 1,000만 원의 과태료 대상이다. ② 동의 없이 이용하면 5년 이하 징역 또는 5천만 원 이하 벌금이므로 **'동의했는가'를 언제든 증명할 수 있어야** 하고, 그 증명은 이력이 한다 — `apps/admin/src/pages/users/consents/types.ts:9-14`.

**화면 설명** — 상단에 설명 한 줄, (실패 시) 항목 조회 실패 배너, 그 아래 탭 셋과 그 탭의 패널이다. 세 질문의 답이 **같은 이력에서 나오기 때문에** 한 화면에 두었다 — 나누면 어느 화면이 정본인지 흐려지고, 항목 정의를 바꾼 결과가 다른 화면에서만 보인다.

**주요 사용자** — 최상위 관리자 · 운영자 · 법무 담당자.

**사용 시나리오**

1. 마케팅 문자 수신 동의를 필수로 바꾸려다 잠긴 토글과 그 사유를 읽는다.
2. 약관이 개정된 뒤 '재동의 · 파기' 탭에서 다시 물어야 할 대상을 뽑는다.
3. 특정 회원의 동의·철회 이력을 검색해 분쟁에 대비한 근거를 확인한다.
4. 보관 기간이 지난 철회 이력이 몇 건인지 세어 서버 파기 작업을 요청한다.

**선행 조건** — 로그인 · 이 화면의 `read` 권한.

**후행 처리** — 항목 정의를 저장하면 그 정의가 가입 화면의 동의 구성에 반영된다(고객 화면은 이 리포 밖이다). **이력은 이 화면에서 만들어지지도 고쳐지지도 않는다.**

**관련 화면** — `SCR-CONTENT-TERMS`(약관 **본문**의 정본 — 이 화면은 버전만 읽는다) · `SCR-CONTENT-PRIVACY`. 탭 3건은 별도 문서다: [동의 항목](tab-items.md) · [동의 이력](tab-history.md) · [재동의 · 파기](tab-compliance.md). 이 화면에는 팝업이 없다. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `사용자 관리 > 동의 이력`(`/users/consents`) — `apps/admin/src/shared/layout/nav-config.ts:134`. 조건부 노출이 아니다.

**관련 기능** — 약관 관리와 무엇이 다른가: **그쪽은 문서 본문을 관리한다.** 문서가 있다는 것과 누가 그것에 동의했다는 것은 다른 사실이고, 분쟁에서 요구되는 것은 후자다. 이 화면은 문서를 쓰지 않고 시행 중 약관 버전을 **조회기로 읽기만** 한다 — `apps/admin/src/pages/users/consents/ConsentsPage.tsx:12-14`.

**관련 API** — 목적 셋. ① 동의 항목 정의 조회 ② 항목 정의 저장 ③ 동의 이력 조회. **백엔드가 없다** — 연동 지점은 `TODO(backend): GET/PUT /api/users/consent-items` · `GET /api/users/consent-events` — `apps/admin/src/pages/users/consents/data-source.ts:80,356`. 서버 계약으로 두 가지가 함께 적혀 있다: 이력 조회는 **커서 페이지네이션**을 쓴다(이력은 지워지지 않으므로 오프셋이 계속 밀린다), 그리고 이력 추가는 서버가 세션·서버 시각으로 기록해야 한다(**프런트가 보내는 시각을 신뢰하면 안 된다**) — `data-source.ts:357,371-372`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/users/consents/ConsentsPage.tsx` (패널 `components/ConsentItemsCard.tsx` · `components/ConsentHistoryPanel.tsx` · `components/CompliancePanel.tsx` · 규칙 `rules.ts` · 모델 `types.ts` · 어댑터 `data-source.ts` · 약관 버전 이음매 `apps/admin/src/shared/domain/terms-version.ts`)

| 성공 처리 | 해당 없음 |
| 실패 처리 | 해당 없음 |
| 예외 처리 | 어떤 역할에게도 이력의 수정·삭제 표면이 열리지 않는다. 권한 매트릭스에서 `remove` 를 켜도 **화면에 나타나는 것이 없다** |
| 화면 변경 사항 | 없음 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | 해당 없음 |

**현재 동의 상태는 저장되지 않는다** — 이력의 마지막 줄에서 **파생**된다. 저장하면 이력과 파생값이 갈라지는 날이 오고, 그때 어느 쪽이 사실인지 아무도 모른다 — `rules.ts:3-7`.

## 5. 이벤트 정의

| 이벤트 | 발생 조건 | 처리 내용 | 대상 컴포넌트 | 결과 화면 |
|---|---|---|---|---|
| 화면 진입 | `/users/consents` 도달 | 항목 정의와 이력을 **각각 한 번씩** 조회한다 | 화면 전체 | 동의 항목 탭 |
| 항목 조회 실패 | 그 조회 오류 | 상단에 danger 배너 + `다시 시도`. **기본 정의로 계속 그린다** | 실패 배너 | 배너 + 정상 패널 |
| 이력 조회 실패 | 그 조회 오류 | **탭마다 다르게 말한다**(§4.2) | 이력 탭 · 재동의·파기 탭 | 각 탭의 실패 표시 |
| 탭 클릭 | 탭 줄 클릭 | 활성 탭 교체. **재조회하지 않는다** | 탭 줄 · 패널 | 패널 교체 |
| 알 수 없는 탭 id | 컴포넌트가 목록에 없는 값을 받음 | **무시한다** — 탭이 바뀌지 않는다 | 탭 줄 | 변화 없음 |
| 항목 토글 변경 | 동의 항목 탭에서 토글 | 초안만 바뀐다(저장 전) | 동의 항목 탭 | 저장 버튼 활성화 |
| 항목 저장 | '저장' 클릭 | 마지막 관문 통과 → 저장 → 성공/실패 토스트 | 동의 항목 탭 | 토스트 |
| 항목 저장 성공 | 서버 수락 | 항목 정의 재조회 → **이력 탭의 필터 선택지와 재동의 판정 입력도 갱신** | 화면 전체 | 즉시 반영 |
| 새로고침 | 브라우저 새로고침 | **첫 탭(`동의 항목`)으로 돌아간다** — 탭이 주소에 실리지 않는다 | 탭 줄 | 첫 탭 |
| 권한 변경(다른 화면에서) | 역할의 `update` 가 켜지거나 꺼짐 | 리로드 없이 항목 탭의 토글·저장 컨트롤이 나타나거나 사라진다 | 동의 항목 탭 | 즉시 반영 |

## 6. Validation

이 화면 껍데기에는 입력 칸이 없다. 사용자가 정하는 값은 활성 탭 하나이고, **정해진 목록에서 되찾아 좁히므로** 목록 밖의 값은 조용히 버려진다 — `ConsentsPage.tsx:49-51`.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 활성 탭 id | `items` · `history` · `compliance` 중 하나 | 탭 변경 콜백에서 | (문구 없음) | 목록 밖의 값은 무시하고 탭을 바꾸지 않는다 |

**탭 안의 검증은 각 탭 문서가 갖는다** — 필수/선택 전환 규칙은 [동의 항목](tab-items.md) §6, 조회 조건은 [동의 이력](tab-history.md) §6, 재동의·파기 탭은 입력이 없다.

## 7. 예외 처리

| 발생 상황 | 사용자 메시지 | 시스템 처리 | 복구 방법 |
|---|---|---|---|
| 조회 결과 없음 | **탭마다 다르다.** 항목 0건은 성립하지 않는다(실패해도 기본 정의로 그린다). 이력 0건과 대상 0건은 각 탭 문서가 갖는다 | 화면 골격은 유지된다 | 탭 문서 참조 |
| 저장 실패 | `동의 항목을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `ConsentsPage.tsx:103` | 실패 토스트(자동으로 사라지지 않는다). 초안은 그대로 남는다 | 다시 저장 |
| 수정 실패 | 저장 실패와 같다 — 이 화면의 유일한 쓰기가 항목 정의 저장이다 | 〃 | 〃 |
| 삭제 실패 | 해당 없음 — **이 화면에 삭제 기능이 없다.** 어댑터에 이력 삭제 함수가 존재하지 않는다 | — | — |
| API 오류 | 항목 조회는 상단 배너, 이력 조회는 탭별 표시, 저장은 실패 토스트다. 서버 응답 원문을 문장으로 노출하지 않는다 | 항목 조회 실패는 화면을 막지 않는다 | 배너·패널의 `다시 시도` 또는 재저장 |
| 서버 오류 | 위와 같다 — 코드가 5xx 에 별도 문구를 두지 않는다 | 〃 | 〃 |
| 네트워크 오류 | 위와 같다 | **자동 재시도를 하지 않는다** | 사용자가 재시도 |
| 권한 없음 | `read` 가 없으면 본문 대신 `접근 권한이 없어요` 화면 — `apps/admin/src/shared/errors/ErrorScreens.tsx:73`. `update` 만 없으면 항목 탭에 info 배너가 뜨고 저장 컨트롤이 사라진다(탭 문서 §9) | 사이드바·헤더는 남는다 | 관리자에게 권한 요청 |
| 세션 만료 | 이 화면에 자기 문구가 없다. 401 이 관측되면 로그인 화면이 알린다 | 세션을 지우고 원래 경로를 보존한 채 이동 | 재로그인하면 이 화면으로 돌아온다 — **탭은 첫 탭으로 돌아간다** |
| 데이터 충돌 | 이 화면에 **충돌 다이얼로그가 없다.** 다른 관리자가 그 사이 항목 정의를 바꿨어도 나중에 저장한 쪽이 이긴다. 다만 재조회로 서버 값이 바뀌면 **초안이 그 위에서 다시 시작한다** — `components/ConsentItemsCard.tsx:99-104` | 덮어쓰기 경고가 없다 | 저장 후 화면을 새로 고쳐 최신 값 확인 |
| 중복 데이터 | 해당 없음 — 이 화면은 아무것도 새로 만들지 않는다(항목을 추가하는 손잡이도 없다) | — | — |
| 파일 업로드 실패 | 해당 없음 — 이 화면에 파일 입력이 없다 | — | — |

**`null`(모름)과 `[]`(진짜 0건)이 갈리는 두 축** — 이 화면의 핵심 규율이다.

| 축 | 모름(`null`) | 진짜 0건 |
|---|---|---|
| 동의 이력 | 조회 중이거나 실패했을 때. 재동의·파기 탭이 **판정 자체를 하지 않고** '판정하지 못했어요'라고 말한다 — `rules.ts:139-148` | 이력이 실제로 0건이면 각 탭이 '없어요'라고 완결되게 말한다 |
| 시행 중 약관 버전 | 조회기가 배선되지 않았을 때. 재동의 산출이 `null` 을 돌려주고 화면이 `시행 중인 약관 버전을 확인할 수 없어 재동의 대상을 판정하지 못했어요.` 를 띄운다 — `apps/admin/src/shared/domain/terms-version.ts:56-64` | 대상이 0명이면 `모든 동의가 시행 중인 버전 기준이에요.` |

**왜 이 구분에 집착하는가** — 빈 결과를 주면 화면이 '재동의 대상 0명'이라는 **완결된 문장**을 그리고, 운영자는 개정 공지를 보내지 않는다. **배선 사고가 법적 사고로 번지는 경로**다 — `rules.ts:126-137`.

**등록·삭제·다운로드·업로드·승인·반려는 이 화면에 존재하지 않는 동작이다.** 항목을 새로 추가하는 손잡이도, 이력을 지우는 손잡이도, 내보내기도 없다. 권한 매트릭스에서 `create`·`remove`·`export` 를 켜도 이 화면에 나타나는 것이 없다.

| 액션 | 이 화면에서 무엇을 여는가 | 근거 |
|---|---|---|
| `read` | 화면 진입 · 탭 셋 · 두 조회 | `RequirePermission` 이 라우트를 덮는다 |
| `update` | **동의 항목 탭의 토글과 저장 컨트롤만** | `ConsentsPage.tsx:79` → `components/ConsentItemsCard.tsx:174,187` |

**이력은 조회뿐이다 — 어떤 역할에게도 수정·삭제 표면이 없다.** 항목 정의만 `update` 로 게이팅된다 — `ConsentsPage.tsx:16-17`.

**권한이 없을 때 화면이 어떻게 되는가**

| 없는 것 | 결과 |
|---|---|
| 화면 읽기(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다. 사이드바에서도 이 잎이 사라진다 |
| `update` | 동의 항목 탭에서 **저장 컨트롤 자체가 사라지고**(눌러 보고 거절당하는 자리를 만들지 않는다) 토글이 잠긴다. 그리고 그 사실을 info 배너로 밝힌다 |
| 플랜(엔타이틀먼트) | 해당 없음 — `/users/**` 는 어떤 모듈에도 매핑되어 있지 않다 |
