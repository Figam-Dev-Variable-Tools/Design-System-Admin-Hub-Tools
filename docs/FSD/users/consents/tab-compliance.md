# 동의 이력 — 재동의 · 파기 탭

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 재동의 · 파기 |
| 화면 ID | `TAB-USERS-CONSENTS-COMPLIANCE` |
| 메뉴 경로 | 사용자 관리 > 동의 이력 > 재동의 · 파기 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 두 가지를 **세어 보여 준다.** ① 약관이 개정돼 다시 물어야 할 이용자가 누구인가 ② 철회 후 보관 기간이 지나 파기해야 할 이력이 무엇인가.

**업무 배경** — 약관이 개정되면 옛 버전에 동의한 채 남아 있는 사람에게 다시 물어야 한다. 그리고 철회한 사람의 기록은 법정 보관 기간이 지나면 지워야 한다 — **지정된 기간을 넘겨 들고 있는 것 자체가 위반**이다. 두 판정 모두 이력에서 매번 **계산**되며 저장되지 않는다 — 저장하면 이력과 갈라지는 날이 오고, 그때 어느 쪽이 사실인지 아무도 모른다 — `apps/admin/src/pages/users/consents/rules.ts:3-7`.

**화면 설명** — 카드 둘이 세로로 놓인다. 위는 `재동의 대상`(항목별 묶음 + 대상 표), 아래는 `보관 기간 경과 (파기 대상)`(대상 표)이다. **버튼이 하나도 없다** — 이 탭은 세어 보여 줄 뿐 지우지 않는다.

**주요 사용자** — 최상위 관리자 · 운영자 · 법무 담당자.

**사용 시나리오**

1. 약관을 개정한 뒤 이 탭을 열어 몇 명에게 재동의 공지를 보내야 하는지 확인한다.
2. 파기 대상 표를 근거로 서버 파기 작업을 요청한다.
3. 약관 연동이 끊긴 상태에서 이 탭을 열면 **숫자 대신 '판정하지 못했어요'가 뜬다** — 0명이라는 거짓을 보지 않는다.

**선행 조건** — 부모 화면(`SCR-USERS-CONSENTS`)이 열려 있고 이 탭이 활성일 것. **판정에는 셋이 모두 필요하다**: 동의 이력 · 항목 정의 · 시행 중 약관 버전.

**후행 처리** — 없다. **이 탭은 아무것도 바꾸지 않는다.** 실제 파기는 원장과 백업까지 함께 다뤄야 하는 서버의 일이다.

**관련 화면** — `SCR-USERS-CONSENTS`(부모) · `TAB-USERS-CONSENTS-ITEMS`(항목 정의와 보관 기간의 출처) · `TAB-USERS-CONSENTS-HISTORY`(같은 이력을 본다) · `SCR-CONTENT-TERMS`(시행 중 약관 버전의 정본). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 탭이라 사이드바에 자기 자리가 없다.

**관련 기능** — 시행 중 약관 버전은 이 화면이 직접 읽지 않고 **공통 층의 조회기**에서 온다. 동의 화면이 약관 화면을 직접 참조하면 페이지 간 결합이 되고, 무엇보다 두 화면이 서로 다른 버전을 말하게 된다 — `apps/admin/src/shared/domain/terms-version.ts:10-17`.

**관련 API** — 없음(이 탭 자체는). 판정 입력 셋 중 둘(이력·항목 정의)은 부모가 조회하고, 셋째(시행 중 약관 버전)는 **네트워크가 아니라 앱 안 조회기**에서 온다. 그 조회기가 꽂혀 있지 않으면 `null` 이 돌아오고 이 탭은 판정을 하지 않는다 — `terms-version.ts:56-64`.

### 4.2 재동의 대상 산출

| 항목 | 내용 |
|---|---|
| 기능 목적 | 지금 시행 중인 약관 버전과 **다른 버전**에 동의한 채 남아 있는 이용자를 뽑는다 |
| 실행 조건 | 이력이 '준비됨' 일 때 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | (회원 × 항목)마다 **가장 최근 사실**을 뽑고, 그중 `동의 중`이면서 버전이 시행 중 버전과 다른 것을 모은다 — `../rules.ts:177-217` |
| 데이터 처리 | 없음 — **저장하지 않는다.** 매번 계산한다 |
| Validation | 해당 없음 |
| API 호출 여부 | N |
| 성공 처리 | 항목별 묶음마다 제목·배지 둘·표가 뜬다. 대상이 0명인 묶음은 그리지 않는다 |
| 실패 처리 | **약관 버전을 모르면 판정하지 않는다** — 숫자 대신 warning 배너로 그 사실을 말한다(§3 No.5) |
| 예외 처리 | ① **철회한 이용자는 대상이 아니다** — 개정을 이유로 다시 권유하는 것은 철회 의사를 무시하는 재권유이고, 그 자체가 수신거부 위반이 된다 — `../rules.ts:197-198` ② 약관이 없는 항목(`약관 종류` 가 비어 있는 항목)은 판정에서 제외한다 ③ **시행 중 버전을 찾지 못한 항목은 조용히 건너뛰지 않고** 그 이름을 배너로 밝힌다 — '대상 0명'과 '판정 못 함'은 다른 사실이다 — `../rules.ts:163-168` |
| 화면 변경 사항 | 재동의 대상 카드 |
| 후속 동작 | 없음 — **공지를 보내는 기능이 이 화면에 없다** |
| 로그 기록 여부 | N |

**현재 동의 상태는 저장되지 않는다** — 이력의 마지막 줄에서 파생된다. **동의만 세면 현재 상태를 알 수 없다** — 동의했다가 철회한 사람과 계속 동의 중인 사람이 똑같이 '동의 1건'으로 보이기 때문이다 — `../rules.ts:88-93`.

**이 탭에서 나가는 링크가 하나도 없다** — 대상 회원의 상세로도, 약관 관리로도, 발송 화면으로도 이어지지 않는다. 배너가 `약관 관리 연동을 확인해 주세요` 라고 말하지만 그 화면으로 가는 링크는 없다.

**이 탭에는 팝업도 확인 다이얼로그도 없다.** 인벤토리의 팝업 19건·확인 다이얼로그 56건 어디에도 이 탭이 없다 — 되돌릴 수 없는 동작이 하나도 없기 때문이다.

**이 탭에는 결과 통지 토스트가 없다** — 조회·계산 전용이라 통지할 결과가 없고, 판정 불가는 인라인 배너가 말한다.
