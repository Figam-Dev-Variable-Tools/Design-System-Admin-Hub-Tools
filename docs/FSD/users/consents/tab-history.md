# 동의 이력 — 동의 이력 탭

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 동의 이력 |
| 화면 ID | `TAB-USERS-CONSENTS-HISTORY` |
| 메뉴 경로 | 사용자 관리 > 동의 이력 > 동의 이력 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 누가 언제 무엇에 동의하고 철회했는지를 **최신순으로** 본다. 항목·구분으로 좁히고 대상·회원 id·약관 버전으로 검색한다.

**업무 배경** — 분쟁에서 요구되는 것은 '문서가 있다'가 아니라 '이 사람이 그때 이것에 동의했다'는 사실이다. 그 증명이 이력이며, **이력은 덧붙이기만 한다** — 고칠 수 있는 기록은 증거가 되지 못한다. 마음이 바뀐 것은 수정이 아니라 **철회 이벤트를 한 줄 더 쌓아** 표현한다 — `apps/admin/src/pages/users/consents/types.ts:15-19`.

**화면 설명** — 상단 툴바(검색 + 항목 필터 + 구분 필터) → 총 건수 → 표 → 빈 상태다. **행을 눌러도 아무 일도 일어나지 않는다** — 이력 한 줄에는 열어 볼 하위 문서가 없다.

**주요 사용자** — 최상위 관리자 · 운영자 · 법무 담당자.

**사용 시나리오**

1. '마케팅 문자 + 철회'만 걸어 놓은 화면을 법무 담당자에게 **링크로 그대로 넘긴다**(조회 조건이 주소에 실린다).
2. 특정 회원의 표시값으로 검색해 그 사람의 동의·철회 흐름을 시간순으로 확인한다.
3. 약관 버전으로 검색해 개정 전 문구에 동의한 기록을 찾는다.

**선행 조건** — 부모 화면(`SCR-USERS-CONSENTS`)이 열려 있고 이 탭이 활성일 것.

**후행 처리** — 없다. **이 탭은 아무것도 바꾸지 않는다.**

**관련 화면** — `SCR-USERS-CONSENTS`(부모) · `TAB-USERS-CONSENTS-ITEMS`(항목 이름과 필터 선택지의 출처) · `TAB-USERS-CONSENTS-COMPLIANCE`(같은 이력으로 판정한다). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 탭이라 사이드바에 자기 자리가 없다.

**관련 기능** — 이력 한 줄은 회원 명부를 조회해 이름을 붙이지 않고 **사건 시점의 표시값을 스스로 들고 있다**. 회원이 탈퇴하면 명부에서 사라지지만 그 사람의 동의 사실은 보관 기간 동안 남아야 하고, 그때 이력이 이름을 잃으면 '누구의 동의였는지 모르는 동의'가 된다 — `types.ts:120-127`.

**관련 API** — 목적 하나 — 이력 조회. 부모가 소유한다(부모 문서 §4.2). 연동 지점은 `TODO(backend): GET /api/users/consent-events?from=&to=&itemId=&action=` 이며 서버는 **커서 페이지네이션**을 써야 한다(이력은 지워지지 않으므로 오프셋이 계속 밀린다) — `apps/admin/src/pages/users/consents/data-source.ts:356-357`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [동의 이력](index.md)

**구현 파일** — `apps/admin/src/pages/users/consents/components/ConsentHistoryPanel.tsx` (좁히기 규칙 `../rules.ts:305-338` · 모델 `../types.ts:84-142` · 목록 껍데기 `apps/admin/src/shared/crud/CrudReadListShell.tsx`)

## 4. 기능 명세

### 4.1 이력 조회와 좁히기

| 항목 | 내용 |
|---|---|
| 기능 목적 | 항목 · 구분 · 검색어로 좁힌 이력을 최신순으로 본다 |
| 실행 조건 | 부모가 이력을 조회했을 때 |
| 사용자 동작 | 검색어 입력 · 항목 필터 선택 · 구분 필터 선택 |
| 시스템 처리 | 부모가 받아 온 이력에 **화면에서** 필터와 검색을 적용한다(서버 재조회가 아니다) — `ConsentHistoryPanel.tsx:78-81`. 두 필터는 AND 로 걸리고, 검색은 **표시값 · 회원 id · 약관 버전** 셋을 훑는다 — `../rules.ts:308-333` |
| 데이터 처리 | 없음(읽기만) |
| Validation | 주소로 들어온 알 수 없는 값은 조용히 `전체` 로 되돌린다(§6) |
| API 호출 여부 | N — **조건을 바꿔도 서버를 다시 부르지 않는다.** 조회는 부모가 화면 진입 시 한 번 한다 |
| 성공 처리 | 표와 요약 줄이 바뀐다. **토스트를 띄우지 않는다** |
| 실패 처리 | 부모의 이력 조회가 실패했으면 표 대신 danger 배너 + `다시 시도` 가 뜬다 |
| 예외 처리 | 정렬은 **최신순 고정**이다 — 조사 화면이 먼저 보아야 하는 것은 방금 일어난 일이다. 같은 시각이면 id 로 안정 비교해 순서가 흔들리지 않는다 — `../rules.ts:80-86,335-338` |
| 화면 변경 사항 | 목록영역 · 요약 줄 · 라이브 통지 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

## 6. Validation

이 탭에는 저장이 없다. 검증 대상은 **주소로 들어온 조회 조건** 둘뿐이며, 규칙은 '목록에서 되찾아 좁히는 것'이다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 주소의 `item` | `all` 또는 **지금 존재하는 항목의 id** | 주소를 읽을 때 | (문구 없음) | 목록 밖의 값은 조용히 `all` 로 되돌린다 — `ConsentHistoryPanel.tsx:60-66`. 손으로 고친 값이나 **삭제된 항목을 가리키는 옛 링크**가 조회를 깨뜨리지 않게 한다 |
| 주소의 `action` | `all` · `granted` · `withdrawn` 중 하나 | 주소를 읽을 때 | (문구 없음) | 〃 — `ConsentHistoryPanel.tsx:67-71` |
| 검색어 | 제약 없음 | — | — | 한글 조합이 끝난 뒤에만 반영된다 |

**되돌림을 사용자에게 알리지 않는다** — 조용히 전체 목록을 보여 준다. 0건짜리 빈 화면보다 낫다는 판단이다.

**등록·수정·삭제·다운로드·업로드·승인·반려는 이 탭에 존재하지 않는 동작이다.** 권한 매트릭스에서 `create`·`update`·`remove`·`export` 를 켜도 이 탭에 나타나는 것이 없다 — 이력은 조회뿐이고, 내보내기도 없다.
