# 동의 이력 — 동의 항목 탭

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 동의 항목 |
| 화면 ID | `TAB-USERS-CONSENTS-ITEMS` |
| 메뉴 경로 | 사용자 관리 > 동의 이력 > 동의 항목 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 우리가 무엇에 동의를 받는지, 그중 **무엇이 필수이고 무엇이 선택인지**를 정한다. 이 탭이 이 앱에서 필수/선택을 정하는 유일한 자리다.

**업무 배경** — 동의는 항목마다 구분해 알리고 받아야 한다. 그런데 **마케팅·선택 개인정보·제3자 제공은 서비스 이용의 조건이 될 수 없다** — 묶는 순간 그 동의는 자유로운 의사가 아니라 이용 조건이 되고, 마케팅 수신 동의의 경우 최대 1,000만 원의 과태료 대상이다. 그래서 화면이 그 셋을 **필수로 세우지 못하게 잠그고, 왜 잠겼는지를 그 자리에서 말한다** — `apps/admin/src/pages/users/consents/rules.ts:24-31`.

**화면 설명** — 카드 하나에 안내 한 줄 + (거절이 있었다면) 배너 + (권한이 없다면) 배너 + 항목 목록 + (권한이 있다면) 저장 줄이다. 항목마다 이름·배지 둘·설명·보관 기간 안내와 필수/선택 토글이 있다.

**주요 사용자** — 최상위 관리자 · 운영자 · 법무 담당자.

**사용 시나리오**

1. 새 서비스 정책에 맞춰 '개인정보 수집·이용 동의 (선택)'을 필수로 바꾸려 한다 → 토글이 잠겨 있고 그 이유가 옆에 적혀 있다.
2. 서비스 이용 성격의 항목을 선택으로 내렸다가 다시 필수로 되돌린다.
3. 조회 권한만 있는 담당자가 지금 정의가 어떤지 확인한다 → 토글이 잠기고 그 사실이 배너로 뜬다.

**선행 조건** — 부모 화면(`SCR-USERS-CONSENTS`)이 열려 있고 이 탭이 활성일 것. 항목 정의 조회가 끝났을 것(아직이면 스켈레톤이 뜬다).

**후행 처리** — 저장한 정의가 가입 화면의 동의 구성 근거가 된다(고객 화면은 이 리포 밖이다). 같은 화면의 **이력 탭 필터 선택지**와 **재동의 판정 입력**도 이 정의를 읽는다.

**관련 화면** — `SCR-USERS-CONSENTS`(부모) · `TAB-USERS-CONSENTS-HISTORY` · `TAB-USERS-CONSENTS-COMPLIANCE`(같은 정의를 읽는 형제 탭) · `SCR-CONTENT-TERMS`(항목이 가리키는 약관 종류의 정본). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 탭이라 사이드바에 자기 자리가 없다.

**관련 기능** — 항목이 가리키는 `약관 종류` 는 콘텐츠 관리 약관의 종류 id 다. 동의를 받을 때 **어느 문서의 어느 버전**을 보여 줬는지가 그 참조로 이어진다 — `apps/admin/src/pages/users/consents/types.ts:56-60`.

**관련 API** — 목적 둘 — 항목 정의 조회와 저장. 부모가 소유한다(부모 문서 §1 · §4.1 · §4.4). 연동 지점은 `TODO(backend): GET/PUT /api/users/consent-items` — `apps/admin/src/pages/users/consents/data-source.ts:80`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [동의 이력](index.md)

**구현 파일** — `apps/admin/src/pages/users/consents/components/ConsentItemsCard.tsx` (규칙 `../rules.ts:41-64` · 모델 `../types.ts` · 기본 정의 `../data-source.ts:31-77`)

## 4. 기능 명세

### 4.1 필수/선택 전환

| 항목 | 내용 |
|---|---|
| 기능 목적 | 어느 항목이 서비스 이용의 조건인지를 정한다 |
| 실행 조건 | `update` 권한 + 저장 중이 아님 + 그 항목이 잠기지 않았을 것 |
| 사용자 동작 | 항목의 토글 클릭 |
| 시스템 처리 | 규칙에 물어 막을 이유가 있으면 **초안을 바꾸지 않고** 사유를 배너로 남긴다. 없으면 초안의 그 항목만 바꾼다 — `ConsentItemsCard.tsx:108-119` |
| 데이터 처리 | 초안만 바뀐다. 실제 반영은 저장이 한다 |
| Validation | §6 |
| API 호출 여부 | N |
| 성공 처리 | 배지가 바뀌고 저장 줄이 `저장하지 않은 변경 사항이 있어요.` 로 바뀐다. **토스트를 띄우지 않는다** |
| 실패 처리 | 카드 상단에 danger 배너로 사유가 뜬다. **토글은 원래 상태로 남는다** |
| 예외 처리 | 잠긴 토글은 애초에 눌리지 않지만 **거절의 책임은 화면이 아니라 규칙이 진다** — 같은 술어가 `disabled` 와 거절을 함께 정한다. 판단이 두 벌이면 언젠가 버튼은 눌리는데 저장이 막힌다 — `ConsentItemsCard.tsx:7-8` |
| 화면 변경 사항 | 그 항목의 배지·토글 · 저장 줄 · (거절 시) 배너 |
| 후속 동작 | 4.2 |
| 로그 기록 여부 | N |

**선택으로 내리는 방향은 언제나 열려 있다** — 규칙이 보는 것은 `필수` 로 세우는 방향뿐이다 — `../rules.ts:41-43`.

### 4.2 항목 정의 저장

| 항목 | 내용 |
|---|---|
| 기능 목적 | 초안을 실제 정의로 확정한다 |
| 실행 조건 | `update` 권한 + 변경이 있고 저장 중이 아닐 때 |
| 사용자 동작 | '저장' 클릭 |
| 시스템 처리 | **저장 직전 마지막 관문**이 묶음 전체를 다시 본다 — 토글이 이미 막고 있어도, 폼 값은 주소·개발자도구·다른 탭에서 바뀔 수 있고 거절의 책임은 저장 경로가 진다 — `../rules.ts:52-64` |
| 데이터 처리 | 항목 정의 문서가 통째로 갈린다. **이력은 건드리지 않는다** |
| Validation | §6 |
| API 호출 여부 | Y — 부모가 소유한다. 성공하면 항목 조회를 무효화한다 |
| 성공 처리 | 부모가 토스트를 띄운다: `동의 항목을 저장했어요.` — `../ConsentsPage.tsx:101`(부모 문서의 `MSG-USERS-CONSENTS-01`). 재조회로 서버 값이 바뀌면 **초안이 그 위에서 다시 시작하고** 거절 배너가 지워진다 |
| 실패 처리 | 부모가 실패 토스트를 띄운다: `동의 항목을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `../ConsentsPage.tsx:103`(`MSG-USERS-CONSENTS-02`). **초안은 그대로 남는다** |
| 예외 처리 | 마지막 관문에 걸리면 **요청 자체를 만들지 않고** 카드 안 배너로 사유를 말한다 — `ConsentItemsCard.tsx:121-129` |
| 화면 변경 사항 | 저장 줄 · (거절 시) 배너 · 토스트 |
| 후속 동작 | 항목 정의 재조회 → 이력 탭의 필터 선택지와 재동의 판정 입력도 갱신 |
| 로그 기록 여부 | 확인하지 못했다 — 항목 정의 변경을 관리자 로그로 남기는 배선을 코드에서 찾지 못했다 |

**확인 다이얼로그를 거치지 않는다** — 되돌릴 수 없는 파괴가 아니라 덮어쓰기이고, 잘못 바꾼 값은 다시 토글해 저장하면 된다.

## 6. Validation

규칙의 정본은 순수 함수다 — `apps/admin/src/pages/users/consents/rules.ts:41-64`. **토글의 잠금과 저장의 거절이 같은 술어를 읽는다** — 판단이 두 벌이면 언젠가 갈라진다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 필수 여부 토글 | 성격이 `광고성 정보 수신` · `개인정보 (선택)` · `제3자 제공` 인 항목은 **필수로 세울 수 없다** | ① 토글을 그릴 때(잠금 판정) ② 토글을 누를 때(거절) ③ 저장 직전(마지막 관문) | `'{항목 이름}' 은 {성격} 동의라 필수로 바꿀 수 없어요. 이 동의를 강제하면 미동의 이용자의 가입이 막히고, 마케팅 수신 동의의 경우 최대 1,000만 원의 과태료 대상이에요.` — `rules.ts:48` | ①은 토글을 잠그고 옆에 이 문구를 글자로 띄운다 ②·③은 초안을 바꾸지 않고 카드 상단 danger 배너 |
| 필수 여부 토글 | **선택으로 내리는 방향은 제한이 없다** | — | — | 언제나 허용된다 — `rules.ts:41-43` |
| 항목 묶음 전체 | 저장하려는 목록에 위 규칙을 어긴 항목이 없을 것 | 저장 클릭 시 | 위와 같은 문구(첫 위반 항목의 사유) | **요청 자체를 만들지 않고** 배너로 알린다 |
| (권한) 저장 | `update` 권한이 있을 것 | 렌더 시 | (문구 없음) | 저장 컨트롤 자체를 그리지 않는다. 대신 info 배너가 사실을 밝힌다 |

**왜 `boolean` 이 아니라 사유 문자열인가** — 버튼을 흐리게 만드는 것과 저장을 거절하는 것이 서로 다른 판단을 읽으면 언젠가 갈라진다. 하나의 술어가 '막힘 + 이유'를 함께 돌려주고, 화면은 그것으로 `disabled` 도 정하고 안내도 그린다. **이유를 말하지 않는 잠긴 컨트롤은 운영자에게 '고장'으로 보인다** — `rules.ts:36-40`.

**이 탭에 길이·형식 검증은 없다** — 자유 입력 칸이 하나도 없기 때문이다.
