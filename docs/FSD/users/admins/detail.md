# 운영자 상세

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 운영자 상세 |
| 화면 ID | `SCR-USERS-ADMINS-DETAIL` |
| 메뉴 경로 | 사용자 관리 > 관리자 관리 > (행 클릭) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 운영자 한 명의 기본 정보 · 소속 그룹(발신 프로필) · 역할(권한) · 관리자 메모를 보고, 수정 화면으로 가거나 계정을 지운다.

**업무 배경** — 이 화면은 원래 없었다. 운영자 상세가 **회원 상세를 재사용**하던 시절에는 운영자에게 '회원 유형: 일반회원'이 붙고 적립금·쿠폰·동의정보 카드가 빈 채로 떴으며, 정작 목록이 이미 갖고 있던 부서·직급은 어디에도 없었다. 더 나쁜 것은 ⋯ 메뉴의 삭제·알림이 **회원 엔드포인트**를 때린 것이다. 화면을 아낀 대가로 도메인이 섞였고, 그 섞임이 가장 위험한 조작에서 터졌다 — `AdminDetailPage.tsx:4-11`.

**화면 설명** — '목록으로' + 제목(닉네임) + 우측 액션 둘(수정 · 삭제), 그 아래 카드 3장(기본 정보 · 소속·권한 · 관리자 메모)이다. **삭제가 막히면 그 이유가 카드 위 info 배너로 뜬다.**

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. 어느 운영자의 역할이 무엇인지 확인하고 수정 화면으로 넘어간다.
2. 퇴사한 운영자의 계정을 지운다.
3. 자기 자신의 상세를 열면 '현재 로그인 계정' 배지가 뜨고 삭제 버튼이 사유와 함께 잠긴다.
4. 시스템 관리자 역할을 가진 마지막 사람을 지우려 하면 그 사실을 먼저 알려 준다.

**선행 조건** — 로그인 · `page:/users/admins` 의 `read` 권한 · 유효한 운영자 id.

**후행 처리** — 삭제하면 목록으로 되돌아가고(히스토리 대체) 목록 캐시가 무효화되며, **이 상세의 캐시는 버려진다** — 사라진 것에 대해서는 다시 묻지 않는다.

**관련 화면** — `SCR-USERS-ADMINS`(목록 · 돌아갈 곳) · `SCR-USERS-ADMINS-FORM`(수정) · `SCR-USERS-ROLES`(역할의 정본). 이 화면에는 팝업이 없다. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. 권한은 잎 `/users/admins` 가 덮는다.

**관련 기능** — ① 역할 이름은 **권한 스토어**에서 되찾는다(운영자 레코드는 역할 id 만 든다) ② 삭제 가드가 전체 운영자 명부와 세션의 계정을 함께 읽는다 ③ 그룹은 메시지 템플릿의 **발신 프로필**과 같은 실체다.

**관련 API** — 목적 셋. ① 운영자 상세 조회 ② 가드 판정용 전체 명부 조회 ③ 운영자 삭제. **백엔드가 없다** — 연동 지점은 `TODO(backend): GET /api/admins/:id`(200 / 404) · `GET /api/admins/roster` · `DELETE /api/admins/:id`(204 / 404 / 422 — 본문 message 를 그대로 표시) — `apps/admin/src/pages/admins/data-source.ts:293-294,315,387-388`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/admins/AdminDetailPage.tsx` (카드 `components/AdminProfileCards.tsx` · 가드 `guards.ts` · 가드 사실 수집 `useAdminGuardContext.ts` · 조회 `queries.ts`)

## 4. 기능 명세

### 4.1 상세 조회 + 가드 판정 준비

| 항목 | 내용 |
|---|---|
| 기능 목적 | 이 운영자의 정보를 가져오고, **동시에** 삭제 가능 여부를 판정할 사실들을 모은다 |
| 실행 조건 | 화면 진입 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 조회가 **둘**이다: 이 운영자 1건과 **가드 판정용 전체 명부**. 뒤엣것은 '시스템 관리자가 몇 명 남았나'에 답하기 위한 것으로, 페이지네이션된 목록으로는 답할 수 없다 — `apps/admin/src/pages/admins/data-source.ts:306-315` |
| 데이터 처리 | 없음(읽기만) |
| Validation | 라우트 파라미터가 없으면 **조회 자체를 걸지 않는다** — 빈 문자열을 조회해 스스로 404 를 만들면 진짜 404 와 구분할 수 없다 — `apps/admin/src/pages/admins/queries.ts:66-72` |
| API 호출 여부 | Y — 상세 1건 + 명부 1건 |
| 성공 처리 | 카드 3장이 채워지고 삭제 버튼의 잠금 여부가 정해진다. **토스트를 띄우지 않는다** |
| 실패 처리 | **상세 조회가 실패하면 화면 전체를 배너로 대체한다.** 404 와 그 밖을 문구와 **복구 수단으로 모두** 가른다(§3 No.6) |
| 예외 처리 | **명부 조회가 실패하면 삭제를 막는다** — 판정 자체가 불가능하다는 사실을 화면이 말한다(§4.3). 모르는 것을 '막을 이유 없음'으로 읽으면 마지막 시스템 관리자가 조용히 지워진다 |
| 화면 변경 사항 | 화면 본문 전체 · 삭제 버튼 상태 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

### 4.3 운영자 삭제 — 가드가 셋을 막는다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 운영자 계정을 지운다. 되돌릴 수 없다 |
| 실행 조건 | `remove` 권한 + **가드 판정이 섰고** 차단 사유가 없을 때만 버튼이 눌린다 |
| 사용자 동작 | '삭제' → 확인 다이얼로그에서 '운영자 삭제' |
| 시스템 처리 | 화면이 미리 판정해 버튼을 잠그지만 **거절의 책임은 어댑터(백엔드 자리)에 있다** — 확인을 본 뒤 다른 탭에서 다른 관리자가 시스템 관리자를 강등할 수 있기 때문이다. 문구는 양쪽 모두 같은 곳에서 온다 — `apps/admin/src/pages/admins/data-source.ts:267-274` |
| 데이터 처리 | 운영자가 사라지고 목록 캐시가 무효화되며 **이 상세 캐시는 버려진다** — `queries.ts:159-169` |
| Validation | 아래 차단 사유 표 |
| API 호출 여부 | Y — 삭제 1건 |
| 성공 처리 | 토스트 `MSG-USERS-ADMINS-DETAIL-01`: `'{닉네임}' 운영자를 삭제했어요.` — `AdminDetailPage.tsx:154`. 그 뒤 **목록으로 되돌아간다(히스토리 대체)** — 뒤로가기로도 돌아오지 않는다 |
| 실패 처리 | **다이얼로그를 닫지 않는다.** 그 안에 danger 배너로 이유가 뜬다 — 가드에 걸렸다면 **그 문장이 그대로**, 그 밖에는 `운영자를 삭제하지 못했어요. 잠시 후 다시 시도해 주세요.` — `AdminDetailPage.tsx:162-166` |

**확인 다이얼로그** — 제목 `운영자 삭제` · 본문 `'{닉네임}'({계정}) 운영자를 삭제할까요? 이 계정으로는 더 이상 로그인할 수 없으며, 삭제는 되돌릴 수 없어요.` · 확인 라벨 `운영자 삭제` — `AdminDetailPage.tsx:266-269`.
