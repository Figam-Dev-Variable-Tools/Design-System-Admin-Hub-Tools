# 운영자 등록 · 수정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 운영자 등록 · 수정 |
| 화면 ID | `SCR-USERS-ADMINS-FORM` |
| 메뉴 경로 | 사용자 관리 > 관리자 관리 > (운영자 등록 / 수정) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 운영자 한 명의 계정·소속·**역할(권한)** 을 정한다. 등록(`/users/admins/new`)과 수정(`/users/admins/:id/edit`)이 **같은 화면**이다.

**업무 배경** — 역할 모델은 '권한만 다룬다'로 재설계되면서 운영자 배정을 스스로에게서 제거하고, "누구에게 어떤 역할을 주는지는 관리자 관리 화면의 몫"이라고 못박았다. 그런데 그 화면이 만들어진 적이 없었다 — 역할은 있는데 배정할 곳이 없는 상태였다. **이 폼이 그 자리다** — `AdminFormPage.tsx:4-7`.

**화면 설명** — '목록으로' + 제목 + 설명 → 카드 하나(운영자 정보) 안에 입력 여덟 칸 → 하단에 취소·저장이다. 그 위에 **상세 로딩·404 분기·저장 실패 시 입력 보존·충돌 다이얼로그·미저장 이탈 가드**가 공통 폼 껍데기로 이미 붙어 있다.

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. 새 운영자를 등록하고 역할을 준다 → 그 사람이 로그인하면 그 역할의 권한으로 앱을 쓴다.
2. 부서가 바뀐 운영자의 정보를 고친다.
3. **자기 계정을 수정하면 역할 칸이 잠긴다** — 자기 권한을 스스로 낮출 수 없다.
4. 이미 등록된 계정으로 저장하면 계정 칸에 인라인 오류가 꽂히고 포커스가 그리로 간다.

**선행 조건** — 로그인 · 등록은 `page:/users/admins` 의 `create`, 수정은 `update` 권한. 권한이 없으면 폼 대신 `접근 권한이 없어요` 화면이 뜬다.

**후행 처리** — 저장하면 목록으로 돌아가고 목록·상세 캐시가 함께 무효화된다. 배정한 역할은 그 사람의 다음 로그인부터 유효 권한이 된다.

**관련 화면** — `SCR-USERS-ADMINS`(목록 · 돌아갈 곳) · `SCR-USERS-ADMINS-DETAIL`(수정 진입점) · `SCR-USERS-ROLES`(역할 목록의 정본) · `SCR-MARKETING-TEMPLATES`(소속 그룹 = 발신 프로필의 소비처). 이 화면에는 팝업이 없다. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. 권한은 잎 `/users/admins` 가 덮는다.

**관련 기능** — ① 역할 셀렉트는 **권한 스토어의 목록을 그대로 읽는다** — 복제하면 새로 만든 역할이 뜨지 않거나 지운 역할이 남아 배정 가능해진다 ② 소속 그룹 셀렉트는 운영진 그룹 목록을 읽는다(= 메시지 템플릿의 발신 프로필 목록) ③ 자기 역할 잠금 가드가 세션의 계정과 **서버 원본**을 비교한다.

**관련 API** — 목적 넷. ① 상세 조회(수정 시) ② 가드 판정용 전체 명부 ③ 등록 ④ 수정. **백엔드가 없다** — 연동 지점은 `TODO(backend): GET /api/admins/:id` · `GET /api/admins/roster` · `POST /api/admins`(201 / 409 계정 중복) · `PUT /api/admins/:id`(200 / 404 / 409 / 422 역할 변경 차단) — `apps/admin/src/pages/admins/data-source.ts:293,315,341-342,355-357`. 소속 그룹 목록은 `GET /api/admin-groups` 다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/admins/AdminFormPage.tsx` (껍데기 `apps/admin/src/shared/crud/FormPageShell.tsx` · 컨트롤러 `apps/admin/src/shared/crud/useCrudForm.ts` · 검증 `apps/admin/src/pages/admins/validation.ts:106-156` · 가드 `guards.ts` · 어댑터 `data-source.ts`)

**등록과 수정이 한 화면인 근거** — 두 라우트가 같은 컴포넌트를 쓴다(`apps/admin/src/App.tsx:253,255`). 차이는 제목·저장 버튼 라벨·요구 권한·상세 조회 여부뿐이다.

| 실행 조건 | 등록은 `create`, 수정은 `update` 권한. 저장 중이 아니고 상세 로딩 중이 아닐 때 |
| 사용자 동작 | '등록'(또는 '저장') 클릭 또는 Enter |
| 시스템 처리 | 값의 앞뒤 공백을 털어 보낸다 — 보이는 값과 저장되는 값이 같게 하기 위해서다 — `AdminFormPage.tsx:66-82`. **렌더를 기다리지 않는 동기 잠금**과 제출 시도 단위 멱등키가 두 번째 요청을 막는다 |
| 데이터 처리 | 등록은 새 레코드를 만들고(id·가입일은 저장소가 붙인다), 수정은 기존 레코드를 갈아 끼운다 |
| Validation | §6 |
| API 호출 여부 | Y — 등록 또는 수정 1건. 저장 성공 후 목록·상세 캐시를 함께 무효화한다 |
| 성공 처리 | **목록으로 돌아간다.** 공통 성공 토스트 `MSG-COMMON-01` 이 뜬다 — 등록/수정 동사와 엔티티 이름(`운영자`)이 그 문구에 실린다 |
| 실패 처리 | **목록으로 이동하지 않고 입력을 유지한다.** 폼 상단에 서버 오류 배너가 뜨고, 예상 밖 실패면 짧은 참조 코드가 함께 붙는다. **필드 단위 거절(422)은 배너가 아니라 그 입력의 인라인 오류로 꽂히고 포커스가 옮겨간다** |
| 예외 처리 | 사용자가 취소한 요청은 실패로 치지 않는다(토스트도 배너도 없다) |
| 화면 변경 사항 | 폼 전체 · 목록(성공 시) |
| 후속 동작 | 목록·상세 캐시 무효화 — 돌아갔을 때 새 값이 보인다 |
| 로그 기록 여부 | 확인하지 못했다 — 운영자 등록·수정을 관리자 로그로 남기는 배선을 코드에서 찾지 못했다 |

**서버가 거절하는 세 가지** — 화면 스키마가 판정할 수 없는 것들이라 어댑터가 맡는다.

| 상황 | 응답 | 화면에서의 모습 | 근거 |
|---|---|---|---|
| 계정 중복 | 422(필드 위반 `account`) | **계정 칸의 인라인 오류** `이미 같은 계정의 운영자가 있어요.` + 그 칸으로 포커스 | `data-source.ts:322-331` · `apps/admin/src/pages/admins/fixtures.ts:155` |
| 없는 id 를 수정 | 409 | 충돌 다이얼로그 `DLG-COMMON-FORM-CONFLICT` — 서버 사유 `다른 사용자가 먼저 삭제한 운영자예요.` | `data-source.ts:367-370` |
| 역할 변경이 가드에 걸림 | 422(필드 위반 `roleId`) | **역할 칸의 인라인 오류** + 그 칸으로 포커스 | `data-source.ts:372-377` |

**계정 중복을 폼 스키마가 보지 않는 이유** — 스키마는 명부를 모른다. 지금 화면이 든 목록으로 판정하면 **다른 탭에서 방금 만들어진 계정을 통과시킨다.** 중복은 명부를 쥔 쪽이 거절한다 — `apps/admin/src/pages/admins/validation.ts:97-99`.

### 4.3 자기 역할 잠금 — 운영자 고유 규칙

| 항목 | 내용 |
|---|---|
| 기능 목적 | 자기 권한을 스스로 낮춰 되돌릴 수 없게 되는 것을 막는다 |
| 실행 조건 | 수정 모드에서, 편집 대상의 계정이 지금 로그인한 계정과 같을 때 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | **폼 값이 아니라 서버 원본**으로 판정한다 — 폼의 계정 칸을 남의 이메일로 고쳐 적는 것만으로 잠금이 풀리면, 그 뒤 자기 역할을 마음대로 올릴 수 있다 — `AdminFormPage.tsx:132-143` |
| 데이터 처리 | 없음 |
| Validation | 값이 그대로면 판정하지 않는다 — 이름·연락처만 고치는 저장까지 막으면 본인 계정은 아무것도 수정하지 못하는 계정이 된다 — `guards.ts:110-115` |
| API 호출 여부 | N(판정만) |
| 성공 처리 | 역할 셀렉트가 잠기고 그 아래에 사유가 뜬다. 나머지 일곱 칸은 정상 편집된다 |
| 실패 처리 | 잠금을 우회해 역할을 바꾼 요청은 **어댑터가 422 로 거절**하고 그 문장이 역할 칸의 인라인 오류로 꽂힌다 |
| 예외 처리 | **세션의 계정을 알 수 없으면 이 규칙이 걸리지 않는다** — 모르는 것은 '자기 자신이 아니다'와 다르다. 반면 마지막 시스템 관리자 보호는 세션과 무관하므로 그대로 산다 — `guards.ts:40-46` |
| 화면 변경 사항 | 역할 셀렉트 · 그 아래 안내 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**왜 승격까지 막는가** — '더 낮은 역할'을 매트릭스 전수 비교로 판정할 수도 있지만, 그 계산은 경계에서 틀리면 **그 실패가 곧 잠금**이다. 그래서 더 단순하고 방어 가능한 규칙을 쓴다: **자기 역할은 자기가 바꾸지 않는다.** 승격도 다른 시스템 관리자가 해 주면 되는 일이고 잠금을 만들지 않는다 — `guards.ts:12-17`.

**가드가 막는 두 번째 상황** — 마지막 시스템 관리자의 역할을 시스템 역할이 아닌 것으로 내리는 것도 거절된다: `'{닉네임}' 은(는) 시스템 관리자 역할을 가진 마지막 운영자예요. 역할을 내리면 역할·권한을 관리할 수 있는 사람이 아무도 남지 않아요. 다른 운영자에게 시스템 관리자 역할을 먼저 준 뒤 바꿔 주세요.` — `guards.ts:121`. 이것은 **화면이 미리 막지 않고** 저장 시점에 어댑터가 거절한다.

### 4.4 저장 충돌

| 항목 | 내용 |
|---|---|
| 기능 목적 | 내가 폼을 여는 사이 다른 관리자가 같은 운영자를 먼저 바꿨을 때 덮어쓰지 않는다 |

| 연락처 | **비어 있거나**, 숫자와 `-` 로만 9~20자일 것 | 제출 시 | `연락처는 숫자와 '-' 로만 입력하세요. 예: 010-1234-5678` — `apps/admin/src/pages/admins/validation.ts:146` | 〃 |
| 관리자 메모 | 500자 이하 | 제출 시 | `메모는 500자를 넘을 수 없어요.` — `apps/admin/src/pages/admins/validation.ts:151` | 〃 |

**필수는 넷뿐이다** — 닉네임 · 계정 · 그룹 · 역할. 이 넷이 없으면 만들어진 레코드가 뜻을 갖지 못한다: 계정은 이 사람이 누구인지를 정하고(세션의 이메일과 맞춰 자기 자신을 찾는다), 그룹은 소속이자 발신 프로필이며, 역할은 이 사람이 무엇을 할 수 있는지다. 부서·직급·연락처·메모는 있으면 좋은 정보라 요구하지 않는다 — `validation.ts:91-95`.

**연락처는 선택이지만 적었다면 형식을 본다** — 문자 발송·비상 연락이 이 값으로 나가므로, 형식이 깨진 번호는 저장은 되고 발송만 실패해서 실패한 뒤에야 잘못을 알게 된다 — `validation.ts:139-143`.

**그룹·역할은 '골랐는가'만 본다** — 두 값 모두 셀렉트가 목록에서 고르게 하므로, 값 자체의 유효성을 스키마가 다시 판정하면 목록과 규칙이 어긋날 수 있다.

**화면 검증은 UX 이지 보증이 아니다** — 서버가 같은 값을 다시 검증해야 한다.

## 7. 예외 처리

| 발생 상황 | 사용자 메시지 | 시스템 처리 | 복구 방법 |
|---|---|---|---|
| 조회 결과 없음 | `{엔티티}{조사} 찾을 수 없어요. 이미 삭제되었을 수 있어요.` — `FormPageShell.tsx:147` | **본문 전체를 배너로 대체**하고 `다시 시도` 를 그리지 않는다 | `목록으로` |
| 저장 실패 | 폼 상단 배너: `저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `apps/admin/src/shared/crud/useCrudForm.ts:221`. 예상 밖이면 그 아래 `오류 코드 {참조}` | **목록으로 이동하지 않고 입력을 유지한다** | 다시 저장. 계속 실패하면 오류 코드와 함께 신고 |

| API 오류 | 위 저장 실패 문구를 그대로 쓴다. 서버 응답 원문·스택·상태 코드를 문장으로 노출하지 않고 짧은 참조 코드만 남긴다 | 배너 | 재시도 또는 오류 코드와 함께 신고 |
| 서버 오류 | 위와 같다 | 〃 | 〃 |
| 네트워크 오류 | 위와 같다 | **자동 재시도를 하지 않는다** | 사용자가 재시도 |

| 세션 만료 | 이 화면에 자기 문구가 없다. 401 이 관측되면 로그인 화면이 알린다 | 세션을 지우고 원래 경로를 보존한 채 이동 | 재로그인하면 이 화면으로 돌아온다 — **다만 입력한 내용은 사라진다** |
| 데이터 충돌 | `DLG-COMMON-FORM-CONFLICT`. 서버 사유 `다른 사용자가 먼저 삭제한 운영자예요.` — `apps/admin/src/pages/admins/data-source.ts:369` | **덮어쓰지 않는다.** 성공 토스트도 화면 이동도 없고 입력은 살아 있다 | '최신 내용 불러오기' 또는 '이어서 편집' |

**중복 제출** — 저장 버튼은 진행 중 잠기고, 공통 폼은 **렌더를 기다리지 않는 동기 잠금**을 따로 둔다. 그래서 빠른 두 번째 Enter 도 두 번째 요청을 만들지 않는다. 재시도는 **같은 멱등키**를 재사용해 서버가 최초 응답을 재생한다.

삭제·다운로드·업로드·승인·반려는 이 화면에 존재하지 않는 동작이다.

**요구 권한이 모드에 따라 갈린다** — 등록 라우트는 `create`, 수정 라우트는 `update` 다. 이 판정은 화면이 아니라 **폼 껍데기가 소유한다**: 폼이 25개고 계속 늘어나는데 화면마다 적게 하면 다음에 추가되는 폼이 그것을 빠뜨린다(목록의 등록 CTA 가 실제로 12개 화면에서 빠져 있었다). 껍데기가 소유하면 새 폼은 껍데기를 쓰는 것만으로 게이팅된 채 태어난다 — `apps/admin/src/shared/crud/FormPageShell.tsx:120-127`.

**권한이 없을 때 화면이 어떻게 되는가** — **폼 자체를 열지 않는다.** 목록에서 '운영자 등록' 버튼이 사라지는 것에 더해, URL 로 `/users/admins/new` 나 `/users/admins/:id/edit` 에 직접 들어와도 본문이 `접근 권한이 없어요` 화면으로 대체된다. `read` 만 가진 역할에게 폼 라우트가 열리고 제출까지 되던 것이 이 가드가 없애겠다고 선언한 구멍이다 — `apps/admin/src/shared/permissions/RequirePermission.tsx:54-70`.

**플랜(엔타이틀먼트)** — 해당 없음. `/users/**` 는 어떤 모듈에도 매핑되어 있지 않다.
