# 새 그룹 만들기 (팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 새 그룹 만들기 |
| 화면 ID | `POP-USERS-MEMBERS-CREATE-GROUP` |
| 메뉴 경로 | 사용자 관리 > 회원 관리 (팝업 — 사이드바에 자리가 없다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 회원을 묶을 **그룹**을 만든다. 그룹명 · 그룹 유형 · 배송비 혜택 셋을 받는다.

**업무 배경** — 회원을 등급 말고 다른 축으로도 묶어야 할 때가 있다(예: 무료 배송을 주는 결제 그룹). 그 묶음에 배송비 정책을 붙이는 실체가 그룹이다. **이건 회원을 만드는 것이 아니다** — 고객은 여전히 회원가입으로만 유입된다.

**화면 설명** — 그룹명 한 칸 + 셀렉트 둘(각각 도움말 아이콘 동반) + 오류 한 줄 + 안내 한 줄 + 버튼 둘이다. 제출하면 곧바로 만들지 않고 **생성 확인 다이얼로그**를 한 번 더 세운다.

**주요 사용자** — 회원 관리 화면에 들어올 수 있는 관리자.

**사용 시나리오**

1. `무료 배송 결제 그룹` 을 '일반 회원 그룹 + 무료 배송'으로 만든다.
2. '운영진 그룹'으로 만들면 그 그룹은 회원 필터가 아니라 **관리자 관리 화면**의 목록에 나타난다.
3. 반쯤 채운 채 딤을 잘못 눌렀다 → 이탈 가드가 뜨고 '취소'를 고르면 입력이 그대로 남는다.

**선행 조건** — `SCR-USERS-MEMBERS` 좌측 그룹 필터의 '+ 새 그룹 만들기' 를 눌렀을 때만 열린다.

**후행 처리** — 성공하면 회원 화면의 그룹 목록이 다시 조회된다. **유형이 '운영진 그룹'이면** 그 그룹은 운영진 그룹 정본 저장소로 들어가 관리자 관리 화면의 좌측 목록에 나타나며, 회원 그룹 필터에는 뜨지 않는다.

**관련 화면** — `SCR-USERS-MEMBERS`(부모) · `SCR-USERS-ADMINS`(운영진 그룹이 도착하는 곳) · `SCR-PRODUCTS-SHIPPING`('조건부 무료 배송'의 기준 금액을 정하는 곳). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — 관리자 관리 화면에도 그룹 만들기 팝업이 있으나 **다른 폼이다**([운영진 그룹 만들기](../admins/pop-create-group.md)) — 그쪽은 대표 발신번호·발신 이메일·발신 자격을 받는다. 두 팝업이 공유하는 것은 그룹명 길이 규칙과 모달 껍데기·이탈 가드뿐이다.

**관련 API** — 목적 하나 — 그룹 생성. **백엔드가 없다**. 연동 지점은 `TODO(backend): POST /api/member-groups (type='staff' 는 POST /api/admin-groups 로 갈린다)` — `apps/admin/src/pages/members/data-source.ts:200`. ⚠ **회원 그룹(`type='member'`)은 지금 저장되지 않는다** — 회원 그룹 픽스처가 읽기 전용이라 성공 통지만 뜨고 목록은 늘지 않는다 — `data-source.ts:198`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [회원 관리](index.md)

**구현 파일** — `apps/admin/src/pages/members/components/CreateGroupModal.tsx` (검증 `apps/admin/src/pages/members/validation.ts:19-28` · 선택지 `apps/admin/src/pages/members/types.ts:35-49` · 어댑터 `data-source.ts:201-213`)

### 4.2 그룹 생성 (확인 후)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 실제로 그룹을 만든다 |
| 실행 조건 | 생성 확인 다이얼로그에서 '그룹 만들기' 를 눌렀을 때 |
| 사용자 동작 | 확인 클릭 |
| 시스템 처리 | 이름·유형·배송비 혜택을 실어 생성 요청을 보낸다 — `CreateGroupModal.tsx:107-132` |
| 데이터 처리 | **유형에 따라 도착지가 갈린다.** `운영진 그룹` 은 운영진 그룹 정본 저장소로 들어가며 **발신 자격이 꺼진 채** 태어난다(이 팝업에는 발신번호·발신 이메일 칸이 없고, 없는 값을 지어내지 않는다). `일반 회원 그룹` 은 지금 저장되지 않는다 — `apps/admin/src/pages/members/data-source.ts:186-213` |
| Validation | 이미 통과했다 |
| API 호출 여부 | Y — 그룹 생성 1건. 성공하면 회원 그룹 목록 조회를 무효화한다 |
| 성공 처리 | 팝업과 확인 다이얼로그가 함께 닫히고 부모가 토스트를 띄운다: `'{그룹명}' 그룹을 만들었어요.` — `apps/admin/src/pages/members/MembersPage.tsx:466`(부모 문서의 `MSG-USERS-MEMBERS-05`) |
| 실패 처리 | **확인 다이얼로그를 닫고 폼으로 돌려보낸다** — 값을 고쳐 다시 시도할 수 있어야 하기 때문이다. 오류 한 줄: `그룹을 만들지 못했어요. 잠시 후 다시 시도해 주세요.` — `CreateGroupModal.tsx:128` |
| 예외 처리 | 취소로 중단된 요청은 **실패로 치지 않는다** — 아무것도 알리지 않는다 — `CreateGroupModal.tsx:124-125`. 팝업이 언마운트될 때도 진행 중 요청을 중단한다 — `CreateGroupModal.tsx:85` |
| 화면 변경 사항 | 팝업(닫힘) · 부모의 좌측 그룹 필터 |
| 후속 동작 | 회원 그룹 목록 재조회. 운영진 그룹이면 관리자 관리 화면의 좌측 목록에 나타난다 |
| 로그 기록 여부 | 확인하지 못했다 |

**⚠ 회원 그룹은 성공 통지가 사실과 어긋난다** — `일반 회원 그룹` 으로 만들면 토스트는 성공을 알리지만 좌측 그룹 필터는 늘지 않는다. 회원 그룹 픽스처가 읽기 전용이기 때문이며(`data-source.ts:198`), 백엔드가 붙기 전까지의 상태다.

## 6. Validation

규칙의 정본은 스키마다 — `apps/admin/src/pages/members/validation.ts:19-28`. 체크 순서가 곧 오류 우선순위이며, 필드당 첫 이슈 하나만 뜬다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 그룹명 | 앞뒤 공백을 턴 값이 비어 있지 않을 것 | 제출 시 | `그룹명을 입력하세요.` — `apps/admin/src/pages/members/validation.ts:21` | 팝업 유지 · 오류 한 줄 · 그룹명 칸으로 포커스 |
| 그룹명 | 앞뒤 공백을 턴 길이가 **30자 이하** | 제출 시 | `그룹명은 30자를 넘을 수 없어요.` — `apps/admin/src/pages/members/validation.ts:23` | 〃 |
| 그룹 유형 | `일반 회원 그룹` · `운영진 그룹` 중 하나 | 제출 시 | (문구 없음 — 셀렉트라 다른 값이 들어올 수 없다) | 스키마가 거절한다 |
| 배송비 혜택 | `없음` · `무료 배송` · `조건부 무료 배송` 중 하나 | 제출 시 | (문구 없음 — 같은 이유) | 〃 |

**30자 상한은 운영진 그룹 만들기와 같은 상수를 본다** — `apps/admin/src/shared/domain/admin-group.ts:64`. 한쪽만 늘리면 같은 화면의 두 모달이 서로 다른 길이를 허용하게 되기 때문이다.

**제출 전에는 오류를 띄우지 않는다** — 첫 제출 전까지는 빨간 문구가 나오지 않고, 새 제출 시도마다 직전 오류가 먼저 지워진다.
