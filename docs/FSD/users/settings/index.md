# 고객 설정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 고객 설정 |
| 화면 ID | `SCR-USERS-SETTINGS` |
| 메뉴 경로 | 사용자 관리 > 고객 설정 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 회원 **등급 정책**을 정한다. 등급을 추가·이름 수정·삭제하고, 등급마다 승급 조건(누적 구매금액)과 할인율을 정하며, 등급을 언제 어떤 구간으로 산정할지를 고른다.

**업무 배경** — 등급은 회원 목록의 필터이자 주문 시 적용되는 할인율의 근거다. 그 값을 화면마다 따로 두면 갈라지므로, 등급을 만들고 고치는 자리가 여기 하나여야 한다. 동시에 정책 변경은 **전 회원의 등급과 할인율을 움직이므로** 저장 전에 그 결과를 미리 보여 준다.

**화면 설명** — 2단 격자다. 좌측은 등급 정책 표(등급 · 승급 조건 · 할인율 · 관리)와 등급 산정 기준 카드, 우측은 현재 등급 분포(저장 전 미리보기)와 승급 시 발급 쿠폰 요약이다. 하단 우측에 저장 툴바가 있고, **저장은 확인 다이얼로그를 거친다.**

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. VIP 승급 조건을 100만 원에서 80만 원으로 낮춘다 → 우측 미리보기가 즉시 '저장하면 N명이 VIP로 승급돼요'로 바뀐다.
2. `골드` 등급을 새로 추가한다 → 승급 조건이 비어 있어 저장이 막히고, 표에 그 이유가 뜬다.
3. 쓰지 않는 등급을 지우려 한다 → 그 등급을 쓰는 회원이 남아 있으면 삭제가 사유와 함께 잠긴다.
4. 강등 허용을 켜고 미리보기에서 몇 명이 내려가는지 확인한 뒤 저장한다.

**선행 조건** — 로그인 · 이 화면의 `read` 권한. 등급 정책은 조회로 받아 온다.

**후행 처리** — 저장하면 등급 카탈로그가 갱신되어 다른 화면이 그 목록을 읽는다. **다만 지금 이 저장은 서버가 아니라 모듈 안 픽스처 저장소를 갱신한다** — 새로고침하면 초기값으로 돌아간다(`apps/admin/src/pages/customer-settings/data-source.ts:17-27`).

**관련 화면** — `SCR-USERS-MEMBERS`(등급 필터·회원 CSV 가 등급 라벨을 읽는다) · `SCR-PRODUCTS-COUPONS-FORM`(승급 시 발급 쿠폰의 정본) · `SCR-PRODUCTS-SHIPPING`(관련 없음 — 배송비 혜택은 그룹의 축이다). 팝업 1건은 별도 문서다: [등급 추가·이름 수정](pop-tier-form.md). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `사용자 관리 > 고객 설정`(`/users/settings`) — `apps/admin/src/shared/layout/nav-config.ts:129`. 관리자 관리 좌측 안내문의 `[가입 및 그룹설정]` 링크도 이 화면을 가리킨다 — `apps/admin/src/pages/admins/components/AdminGroupPanel.tsx:91`.

**관련 기능** — ① 등급 삭제 가드가 **회원 명부**에 '이 등급을 쓰는 회원이 몇 명인가'를 되묻는다 ② 승급 쿠폰 요약이 **쿠폰**의 발급 기준을 읽어 온다(고치지는 않는다) ③ 저장된 등급 목록을 다른 화면이 카탈로그 조회기로 읽는다.

**관련 API** — 목적 둘. ① 등급 정책 조회 ② 등급 정책 저장. **백엔드가 없다** — 연동 지점은 `TODO(backend): GET /api/member-tiers` · `PUT /api/member-tiers` — `apps/admin/src/pages/customer-settings/data-source.ts:55,71`. 저장 계약은 **목록이 곧 등급의 전부**라는 규약이다: 빠진 등급은 삭제, 새 id 는 추가, 바뀐 라벨은 개명. 서버도 같은 규칙을 다시 검증해야 한다고 표시되어 있다(`data-source.ts:72-77`).

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/customer-settings/CustomerSettingsPage.tsx` (카드 `components/TierPolicyCard.tsx` · `components/TierCriteriaCard.tsx` · `components/TierDistributionCard.tsx` · `components/TierUpCouponCard.tsx` · 등급 편집 `useTierEditing.tsx` · 팝업 `components/TierFormModal.tsx` · 모델 `types.ts` · 검증 `validation.ts` · 미리보기 `distribution.ts`)

### 4.2 승급 조건 · 할인율 편집

| 항목 | 내용 |
|---|---|
| 기능 목적 | 등급마다 언제 올라가고 얼마를 깎아 주는지 정한다 |
| 실행 조건 | `update` 권한이 있고 저장 중이 아닐 때 |
| 사용자 동작 | 표의 두 입력에 타이핑 |
| 시스템 처리 | **입력을 막지 않는다.** 승급 조건은 입력 중 숫자만 남기고, 포커스를 잃으면 천 단위 구분을 붙인다. 잘못된 값은 저장 시점에 모델이 거부한다 — `CustomerSettingsPage.tsx:248-266` |
| 데이터 처리 | 초안만 바뀐다. **표를 다시 정렬하지 않는다** — 금액을 한 자리 지울 때마다 행이 뛰면 편집할 수 없다. 정렬은 저장할 때 한 번 한다 — `apps/admin/src/pages/customer-settings/types.ts:119-122` |
| Validation | §6 (오류는 **저장을 한 번 시도한 뒤에만** 표에 뜬다) |
| API 호출 여부 | N |
| 성공 처리 | 저장 툴바가 `저장하지 않은 변경 사항이 있어요.` 로 바뀌고 저장 버튼이 살아난다. 우측 미리보기가 즉시 다시 계산된다 |
| 실패 처리 | 실패 경로가 없다(입력일 뿐이다) |
| 예외 처리 | **일반회원의 승급 조건은 잠겨 있다** — 항상 0원이며 정책이 아니라 정의다. 할인율은 일반회원도 열려 있다 |
| 화면 변경 사항 | 등급 정책 카드 · 분포 카드 · 저장 툴바 |

| 로그 기록 여부 | N |

**변경 감지는 포맷을 무시한다** — `1000000` 을 쳐서 `1,000,000` 으로 포맷됐다는 이유만으로 '변경됨'이 되지 않는다. 값을 파싱해서 비교하고, 파싱되지 않는 원문은 그대로 실어 '잘못된 값으로 바뀐 상태'도 변경으로 잡는다 — `types.ts:355-379`.

| 기본 제공 등급 | `'{등급명}' 은(는) 기본 제공 등급이라 이름 변경·삭제를 할 수 없어요. 회원 목록·회원 CSV·쿠폰 발급 기준이 이 이름과 id 를 그대로 읽어요. 승급 조건과 할인율은 수정할 수 있어요.` |
| 회원 수를 셀 수 없다(`null`) | `'{등급명}' 등급을 쓰는 회원을 확인하지 못해 삭제할 수 없어요. 잠시 후 다시 시도해 주세요.` |
| 그 등급을 쓰는 회원이 1명 이상 | `'{등급명}' 등급을 쓰는 회원이 {N}명 있어 삭제할 수 없어요. 회원 관리(/users/members)에서 이 회원들의 등급을 먼저 바꾼 뒤 다시 삭제해 주세요.` |

**확인 다이얼로그** — 제목 `등급 삭제` · 확인 라벨 `등급 삭제` — `useTierEditing.tsx:121-124`. 본문은 차단 사유가 있으면 **그 사유를 그대로** 보여 주고(경합으로 열릴 수 있다), 없으면 `'{등급명}' 등급을 목록에서 지워요. 하단의 저장을 눌러야 실제로 반영돼요.` — `useTierEditing.tsx:149-153`. 진행 표시가 없다(`busy={false}` — 초안 조작이라 즉시 끝난다).

### 4.6 정책 저장

| 항목 | 내용 |
|---|---|
| 기능 목적 | 초안을 실제 정책으로 반영한다 |
| 실행 조건 | 변경이 있고 저장 중이 아니며 `update` 권한이 있을 때 |
| 사용자 동작 | '저장' → 확인 다이얼로그에서 확인 |
| 시스템 처리 | **저장 버튼을 눌러도 저장하지 않는다** — 검증만 하고 확인 다이얼로그를 세운다. 정책 변경은 전 회원의 등급/할인율을 움직이기 때문이다 — `CustomerSettingsPage.tsx:287-313` |
| 데이터 처리 | 검증을 통과한 초안이 정책이 된다. **목록이 곧 등급의 전부다** — 빠진 등급은 삭제, 새 id 는 추가, 바뀐 라벨은 개명 |
| Validation | §6 — 오류가 하나라도 있으면 저장하지 않는다 |
| API 호출 여부 | Y — 정책 저장 1건 |
| 성공 처리 | 다이얼로그가 닫히고 **이 상태가 새 기준선이 된다**(저장 버튼이 다시 비활성). 토스트 `MSG-USERS-SETTINGS-03`: `등급 정책을 저장했어요.` — `CustomerSettingsPage.tsx:340` |
| 실패 처리 | **다이얼로그를 닫지 않는다.** 그 안에 danger 배너: `등급 정책을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `CustomerSettingsPage.tsx:346`. 확인 재클릭이 곧 재시도다 |
| 예외 처리 | 취소·Esc·딤 클릭은 진행 중 요청을 중단하고 **실패로 치지 않는다** — `CustomerSettingsPage.tsx:352-360`. 화면을 떠나도 진행 중 저장이 취소된다 |
| 화면 변경 사항 | 다이얼로그 · 저장 툴바 |
| 후속 동작 | 저장된 등급 목록을 다른 화면이 카탈로그로 읽는다. ⚠ **지금은 모듈 안 저장소만 갱신된다** — 새로고침하면 초기값으로 돌아간다 |
| 로그 기록 여부 | 확인하지 못했다 |

**저장 전 두 관문**

| 관문 | 걸리면 | 문구 |
|---|---|---|
| 권한 | 실패 토스트 `MSG-USERS-SETTINGS-04` | `이 화면의 수정 권한이 없어 저장할 수 없어요.` — `CustomerSettingsPage.tsx:55,297` |
| 검증 | 실패 토스트 `MSG-USERS-SETTINGS-05` + 표 안 인라인 오류가 켜진다 | `입력값이 규칙에 맞지 않아 저장하지 않았어요. 표의 오류 문구를 확인해 주세요.` — `CustomerSettingsPage.tsx:307` |

**저장 확인 다이얼로그** — 제목 `등급 정책 저장` · 본문 `등급 정책을 저장하면 이후 등급 산정부터 새 기준이 적용돼요. 저장할까요?` — `CustomerSettingsPage.tsx:443-444,52`.

**버튼을 잠그는 술어와 저장을 거절하는 술어가 같다** — 버튼만 잠그면 URL 로 걸어 들어온 사람이 그대로 저장한다.

## 6. Validation

**저장을 막는 오류의 정본은 스키마다**(`apps/admin/src/pages/customer-settings/validation.ts:173-198`). 경고는 스키마 밖에 있다 — 이 도구에는 에러 채널만 있어서, 경고를 스키마에 넣으면 지금까지 저장되던 정책이 저장 거부로 바뀌기 때문이다(`validation.ts:8-17`).

**오류 — 저장을 막는다**

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 기본 제공 등급 존재 | `일반회원`·`VIP`·`VVIP` 가 목록에 있을 것 | 저장 시도 시 | `기본 제공 등급 '{등급명}' 이(가) 목록에 없어요. 이 등급을 쓰는 회원이 존재하므로 지울 수 없어요.` — `apps/admin/src/pages/customer-settings/validation.ts:82` | 표 밖의 danger 배너 |
| 등급 이름 | 비어 있지 않을 것 | 〃 | `이름이 비어 있는 등급이 있어요. 등급 이름을 입력하세요.` — `apps/admin/src/pages/customer-settings/validation.ts:95` | 〃 |
| 승급 조건 | 0 이상의 정수(원) | 〃 | `{등급명} 승급 조건은 0 이상의 정수(원)로 입력하세요.` — `apps/admin/src/pages/customer-settings/validation.ts:102` | 그 칸 아래 인라인 |
| 할인율 | 0~100 정수(%) | 〃 | `{등급명} 할인율은 0~100 사이의 정수(%)로 입력하세요.` — `apps/admin/src/pages/customer-settings/validation.ts:109` | 〃 |
| 기본 등급 서열 | 일반회원 < VIP < VVIP 순으로 승급 조건이 커질 것 | 〃 | `{상위 등급} 승급 조건({N}원)은 {하위 등급} 승급 조건({M}원)보다 커야 해요.` — `validation.ts:137-139` | 상위 등급 칸 아래 인라인 |
| 승급 조건 중복 | 등급마다 승급 조건이 **달라야** 한다 | 〃 | `{등급명} 승급 조건({N}원)이 {다른 등급명} 과(와) 같아요. 등급마다 다른 금액이어야 어느 등급이 위인지 정해져요.` — `validation.ts:164-166` | 뒤에 나온 쪽 칸 아래 인라인 |

**경고 — 저장을 막지 않는다**

| 대상 항목 | 검증 조건 | 검증 시점 | 메시지 | 처리 방식 |
|---|---|---|---|---|
| 할인율 역전 | 승급 조건이 큰 등급의 할인율이 작은 등급보다 낮음 | 초안이 바뀔 때마다(저장 시도 전에도) | `{상위 등급} 할인율({N}%)이 {하위 등급} 할인율({M}%)보다 낮아요. 등급이 올라가는데 혜택이 줄어들어요 — 의도한 정책인지 확인하세요.` — `validation.ts:222-226` | 카드 하단 warning 배너. 저장은 그대로 가능하다 |

**등급 이름의 검증은 팝업이 받는다** — 빈 값·20자 상한·중복은 [등급 추가·이름 수정](pop-tier-form.md) §6 이 갖는다.

**오류는 저장을 한 번 시도한 뒤에만 표에 뜬다** — 타이핑 도중에 빨간 문구가 먼저 튀어나오지 않게 한다(`CustomerSettingsPage.tsx:169-170`). 경고는 그 규칙을 따르지 않는다.

**서열은 저장하지 않는다** — 등급의 순서는 승급 조건 금액에서 파생된다. 별도의 순번 칸을 두면 금액과 순서가 서로를 배신하는 상태가 저장될 수 있다.

다운로드·업로드·승인·반려는 이 화면에 존재하지 않는 동작이다.

| 액션 | 이 화면에서 무엇을 여는가 | 근거 |
|---|---|---|
| `read` | 화면 진입 · 카드 넷 · 미리보기 · 쿠폰 링크 | `RequirePermission` 이 라우트를 덮는다 |
| `create` | '등급 추가' 버튼과 그 팝업의 제출 | `CustomerSettingsPage.tsx:156,407` · `useTierEditing.tsx:164` |
| `update` | 승급 조건·할인율 입력 · 산정 기준 셋 · 행의 연필 · **저장 버튼** | `components/TierPolicyCard.tsx:207,268,317` · `components/TierCriteriaCard.tsx` 의 `disabled` · `CustomerSettingsPage.tsx:143,294-299` |
| `remove` | 행의 휴지통과 삭제 확인의 실제 삭제 | `TierPolicyCard.tsx:333` · `useTierEditing.tsx:91-93` |

**버튼을 잠그는 술어와 거절하는 술어가 같다** — 이 화면은 세 액션 모두에서 그 규율을 지킨다. 저장은 버튼이 잠겨 있어도 URL 로 들어와 누를 수 있는데, 그때도 `requestSave` 가 같은 판정으로 막고 실패 토스트로 사유를 말한다.

**권한이 없을 때 화면이 어떻게 되는가**

| 없는 것 | 결과 |
|---|---|
| 화면 읽기(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다 |
| `create` | '등급 추가' 버튼이 **사라진다**(비활성이 아니다) |
| `update` | 입력·셀렉트·체크박스·연필이 **잠기고**(사라지지 않는다 — 값은 읽어야 한다), 저장 버튼이 잠기며 사유가 접근성 이름과 툴팁에 실린다 |
| `remove` | 행의 휴지통이 **사라진다** |
| `update`·`remove` 둘 다 | 표의 `관리` 열 자체가 사라진다 |
| 플랜(엔타이틀먼트) | 해당 없음 — `/users/**` 는 어떤 모듈에도 매핑되어 있지 않다 |
