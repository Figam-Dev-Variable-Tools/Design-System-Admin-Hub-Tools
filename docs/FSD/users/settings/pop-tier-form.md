# 등급 추가 · 등급 이름 수정 (팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 등급 추가 · 등급 이름 수정 |
| 화면 ID | `POP-USERS-SETTINGS-TIER-FORM` |
| 메뉴 경로 | 사용자 관리 > 고객 설정 (팝업 — 사이드바에 자리가 없다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 등급의 **이름 한 칸**만 받는다. 추가 모드는 새 등급을 목록에 얹고, 수정 모드는 기존 등급의 이름만 바꾼다.

**업무 배경** — 등급을 만들 때 승급 조건까지 받으면 모달과 표가 같은 값을 두 곳에서 편집하게 된다. 더 나쁜 것은 **여기서 금액을 지어내는 것**이다 — 지어낸 숫자는 그대로 정책이 되어 회원 등급을 움직인다. 그래서 새 등급의 승급 조건은 비워 두고, 채우지 않으면 검증이 저장을 막는다.

**화면 설명** — 제목 + 입력 한 칸 + (실패 시) 오류 한 줄 + 모드별 안내 한 줄 + 버튼 둘이 전부다.

**주요 사용자** — 고객 설정 화면에서 `create`(추가) 또는 `update`(수정) 권한을 가진 관리자.

**사용 시나리오**

1. '등급 추가'로 `골드` 를 만든다 → 표에 행이 늘고, 승급 조건이 비어 있어 저장이 막힌 채 그 이유가 표에 뜬다.
2. 추가했던 등급의 이름을 `골드 회원` 으로 바꾼다 → 그 등급을 든 회원은 그대로 유지된다.
3. 이름을 반쯤 치다 딤을 잘못 눌렀다 → 이탈 가드가 뜨고 '취소'를 고르면 입력이 그대로 남는다.

**선행 조건** — `SCR-USERS-SETTINGS` 에서 '등급 추가' 또는 행의 연필을 눌렀을 때만 열린다. 연필은 **기본 제공 등급(일반회원·VIP·VVIP)에서는 잠겨 있다.**

**후행 처리** — 성공해도 **아직 반영되지 않는다.** 바뀌는 것은 편집 초안뿐이고, 실제 반영은 부모 화면 하단의 저장이 한다 — 토스트가 그 사실을 직접 말한다.

**관련 화면** — `SCR-USERS-SETTINGS`(부모). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — 부모 화면의 등급 추가(§4.3)·등급 이름 수정(§4.4). 검증은 팝업이 아니라 **모델**이 한다. 관례는 권한 관리의 [역할 추가·수정](../roles/pop-role-form.md) 과 같다 — 이름만 받고 나머지는 만든 뒤 본문에서 채운다.

**관련 API** — 없음. 이 팝업은 **초안만 바꾼다.** 서버로 나가는 요청은 부모의 저장 하나이며 그 연동 지점은 부모 문서가 갖는다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [고객 설정](index.md)

**구현 파일** — `apps/admin/src/pages/customer-settings/components/TierFormModal.tsx` (검증 `apps/admin/src/pages/customer-settings/types.ts:204-220` · 초안 연산 `types.ts:247-296` · 권한·상태 `apps/admin/src/pages/customer-settings/useTierEditing.tsx`)

## 4. 기능 명세

### 4.1 등급 만들기 (추가 모드)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 이름만 받아 새 등급 행을 초안에 얹는다 |
| 실행 조건 | 부모에서 '등급 추가'를 눌러 팝업이 열린 상태 |
| 사용자 동작 | 이름 입력 → '등급 만들기' 또는 Enter |
| 시스템 처리 | 부모가 넘긴 제출 함수가 **권한 → 모델** 순으로 통과시킨다 — `useTierEditing.tsx:65-83,157-172` |
| 데이터 처리 | 새 행: **승급 조건 비어 있음** · 할인율 `0` · 시스템 아님 · id 는 자동 생성 — `types.ts:247-259`. 여기서 금액을 지어내면 그 숫자가 정책이 되기 때문이다 |
| Validation | §6 |
| API 호출 여부 | N — 초안만 바뀐다 |
| 성공 처리 | **팝업이 닫힌다.** 부모가 토스트를 띄운다 — `'{이름}' 등급을 추가했어요. 승급 조건을 채우고 저장하세요.` — `useTierEditing.tsx:77`(부모 문서의 `MSG-USERS-SETTINGS-01`). 표에 행이 늘고 저장 버튼이 살아난다 |
| 실패 처리 | **팝업이 닫히지 않는다.** 입력 아래에 사유가 뜨고 **포커스가 이름 칸으로 돌아간다** — `TierFormModal.tsx:64-66`. 문구는 §6 |
| 예외 처리 | 이름은 앞뒤 공백이 제거된 채 초안에 들어간다 — `types.ts:254` |
| 화면 변경 사항 | 팝업(닫힘) · 부모의 등급 정책 카드 · 분포 카드(승급 조건이 비어 계산 불가로 바뀐다) · 저장 툴바 |
| 후속 동작 | **승급 조건을 채우고 부모에서 저장해야 실제로 만들어진다** |
| 로그 기록 여부 | 확인하지 못했다 |

**추가한 등급의 한계를 부모가 말한다** — 쿠폰의 '회원 등급 승급 시' 트리거는 기본 제공 3종만 알아서, 새로 만든 등급은 쿠폰 발급 기준에서 고를 수 없다. 그 사실은 부모의 승급 쿠폰 카드가 두 자리에서 안내한다.

**목록에 `VIP` 가 둘이면 어느 쪽이 회원의 등급인지 화면에서 구분할 길이 없다** — 중복 금지가 기본 제공 등급까지 덮는 이유다.

**제출 전에는 오류를 띄우지 않는다** — 타이핑 도중에 빨간 문구가 먼저 나오지 않고, 한 글자라도 고치면 직전 오류가 사라진다.

**승급 조건·할인율은 이 팝업이 받지 않는다** — 그 검증은 부모 화면 §6 이 갖는다.

삭제·다운로드·업로드·승인·반려는 이 팝업에 존재하지 않는 동작이다.

**권한이 없을 때 화면이 어떻게 되는가** — **두 겹으로 막힌다.** ① 팝업을 여는 컨트롤이 사라진다: '등급 추가' 버튼은 `create` 없이 그려지지 않고(`components/TierPolicyCard.tsx:485`), 행의 연필은 `update` 없이 그려지지 않는다(`TierPolicyCard.tsx:317`). ② 그럼에도 제출 경로가 권한을 **다시** 본다 — 버튼을 잠그는 술어와 거절하는 술어가 같아야 하기 때문이다(`useTierEditing.tsx:157-172`). 부모 화면의 `read` 가 없으면 본문이 `접근 권한이 없어요` 화면으로 대체되어 도달할 수 없다.

이 팝업의 게이팅은 같은 섹션의 [회원 그룹 만들기](../members/pop-create-group.md)(게이팅 없음)와 대비된다.
