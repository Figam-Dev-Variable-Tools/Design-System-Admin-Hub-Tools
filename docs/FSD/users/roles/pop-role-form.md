# 역할 추가 · 역할명 수정 (팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 역할 추가 · 역할명 수정 |
| 화면 ID | `POP-USERS-ROLES-ROLE-FORM` |
| 메뉴 경로 | 사용자 관리 > 권한 관리 (팝업 — 사이드바에 자리가 없다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 역할의 **이름 한 칸**만 받는다. 추가 모드는 새 역할을 만들고, 수정 모드는 기존 역할의 이름만 바꾼다.

**업무 배경** — 역할을 만들 때 권한까지 함께 받으면 모달이 매트릭스를 통째로 품게 된다. 그래서 이름만 받고 권한은 만든 뒤 본문의 매트릭스에서 켠다 — 새 역할은 **전 권한 OFF** 로 태어난다. 그 사실을 이 팝업이 직접 말한다.

**화면 설명** — 제목 + 입력 한 칸 + (실패 시) 오류 한 줄 + (추가 모드에서만) 안내 한 줄 + 버튼 둘이 전부다.

**주요 사용자** — 권한 관리 화면에 들어올 수 있는 관리자.

**사용 시나리오**

1. '추가'를 눌러 `콘텐츠 운영` 을 입력하고 '역할 만들기' → 새 역할이 선택된 채로 본문이 바뀐다.
2. 이미 있는 이름을 입력하면 팝업이 닫히지 않고 이유가 뜬다.
3. 이름을 반쯤 치다 딤을 잘못 눌렀다 → 이탈 가드가 뜨고 '취소'를 고르면 입력이 그대로 남는다.

**선행 조건** — `SCR-USERS-ROLES` 에서 '추가' 또는 '수정'을 눌렀을 때만 열린다. 수정은 시스템 역할이 아닐 때만 눌린다.

**후행 처리** — 성공하면 역할 목록이 바뀌고, 추가한 경우 그 역할이 선택 상태가 된다. 관리자 관리의 역할 셀렉트·역할 필터에도 즉시 반영된다.

**관련 화면** — `SCR-USERS-ROLES`(부모) · `SCR-USERS-ADMINS`(만든 역할을 사람에게 배정하는 곳). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — 부모 화면의 역할 추가(§4.2)·역할명 수정(§4.3). 검증은 팝업이 아니라 **모델**이 한다.

**관련 API** — 없음. 역할 상태는 브라우저 저장소에 있다. 연동 지점 표시는 부모 화면 문서와 같다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [권한 관리](index.md)

**구현 파일** — `apps/admin/src/pages/permissions/components/RoleFormModal.tsx` (검증 `apps/admin/src/shared/permissions/roles.ts:293-309` · 저장 `permission-store.ts:234-268`)

## 4. 기능 명세

### 4.1 역할 만들기 (추가 모드)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 이름만 받아 새 역할을 만든다 |
| 실행 조건 | 부모 화면에서 '추가'를 눌러 팝업이 열린 상태 |
| 사용자 동작 | 이름 입력 → '역할 만들기' 또는 Enter |
| 시스템 처리 | 부모가 넘긴 제출 함수를 부른다. 검증은 모델이 하고, 통과하면 역할을 만들어 목록 끝에 붙인 뒤 **그 역할을 선택 상태로** 바꾼다 — `permission-store.ts:234-251` |
| 데이터 처리 | 새 역할: 전 리소스 × 전 액션 OFF · 위젯 7개 전부 OFF · 범위 `전체` · 시스템 아님 |
| Validation | §6 |
| API 호출 여부 | N |
| 성공 처리 | **팝업이 닫힌다.** 부모가 토스트를 띄운다 — `'{입력한 이름}' 역할을 만들었어요.` — `PermissionsPage.tsx:169`(부모 문서의 `MSG-USERS-ROLES-02`). 우측 카드 셋이 새 역할(전부 꺼진 상태)로 바뀐다 |
| 실패 처리 | **팝업이 닫히지 않는다.** 입력 아래에 사유가 뜨고 **포커스가 이름 칸으로 돌아간다** — `RoleFormModal.tsx:59-62`. 문구는 §6 |
| 예외 처리 | 이름은 앞뒤 공백이 제거된 채 저장된다 — `permission-store.ts:241`. 토스트에 뜨는 이름도 같은 값이다 |
| 화면 변경 사항 | 팝업(닫힘) · 부모의 좌측 목록 · 우측 카드 셋 |
| 후속 동작 | 관리자 관리의 역할 셀렉트·역할 필터에 새 역할이 나타난다 |
| 로그 기록 여부 | 확인하지 못했다 — 역할 변경을 관리자 로그로 남기는 배선을 코드에서 찾지 못했다 |

### 4.2 역할명 저장 (수정 모드)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 역할의 이름만 바꾼다 |

| 성공 처리 | 팝업이 닫히고 부모가 토스트를 띄운다 — `역할명을 '{입력한 이름}' 으로 바꿨어요.` — `PermissionsPage.tsx:182`(`MSG-USERS-ROLES-03`) |

## 6. Validation

규칙의 정본은 팝업이 아니라 **모델**이다 — `apps/admin/src/shared/permissions/roles.ts:293-309`. 체크 순서가 곧 오류 우선순위이며, 한 번에 한 문장만 뜬다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 역할명 | 앞뒤 공백을 턴 값이 비어 있지 않을 것 | 제출 시 | `역할명을 입력하세요.` — `roles.ts:299` | 팝업 유지 · 입력 아래 인라인 · 이름 칸으로 포커스 |
| 역할명 | 30자 이하 | 제출 시(입력에서도 30자로 잘린다) | `역할명은 30자를 넘을 수 없어요.` — `roles.ts:301` | 〃 |
| 역할명 | 다른 역할과 겹치지 않을 것(**대소문자 무시**, 앞뒤 공백 무시. 수정 모드는 자기 자신 제외) | 제출 시 | `이미 같은 이름의 역할이 있어요.` — `roles.ts:308` | 〃 |
| 역할 존재(수정) | 대상 역할이 지금도 있을 것 | 제출 시 | `역할을 찾을 수 없어요.` — `permission-store.ts:256` | 〃 |
| 시스템 역할(수정) | 시스템 역할이 아닐 것 | 제출 시 | `시스템 역할은 이름을 바꿀 수 없어요.` — `permission-store.ts:257` | 〃 (정상 경로에서는 버튼이 잠겨 도달하지 않는다) |

**제출 전에는 오류를 띄우지 않는다** — 타이핑 도중에 빨간 문구가 먼저 나오지 않는다. 그리고 한 글자라도 고치면 직전 오류가 사라진다.

삭제·다운로드·업로드·승인·반려는 이 팝업에 존재하지 않는 동작이다.

**권한이 없을 때 화면이 어떻게 되는가** — 이 팝업은 **자기 권한 판정을 갖지 않는다.** 부모 화면의 `read` 로 들어온 사람에게는 '추가'·'수정' 버튼이 보이고 이 팝업도 그대로 열린다. 즉 뷰어 역할이 역할을 만들고 이름을 바꿀 수 있다 — 부모 문서 §9 가 기록한 격차의 일부다. 부모 화면의 `read` 가 없으면 애초에 이 팝업을 여는 버튼에 도달할 수 없다(본문이 `접근 권한이 없어요` 화면으로 대체된다).
