# 권한 관리

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 권한 관리 |
| 화면 ID | `SCR-USERS-ROLES` |
| 메뉴 경로 | 사용자 관리 > 권한 관리 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 이 어드민의 **권한 모델 자체를 정의한다.** 역할(Role)을 만들고 지우며, 역할마다 리소스 × 액션 매트릭스와 대시보드 위젯 노출, 데이터 접근 범위를 정한다. 그중 하나를 '적용 중'으로 세우면 그 역할의 권한이 곧 이 앱 전체의 유효 권한이 된다.

**업무 배경** — 여러 사람이 한 어드민을 나눠 쓰면 '누가 무엇을 할 수 있는가'를 어딘가에서 한 벌로 정해야 한다. 그 자리가 여기다. 이 화면이 없으면 모든 운영자가 삭제·내보내기를 포함해 전권을 갖거나, 반대로 화면마다 사람이 손으로 통제해야 한다. **다른 15개 묶음의 §9(권한)는 전부 이 화면이 정한 모델을 참조한다.**

**화면 설명** — 좌측은 역할 목록 패널(추가·수정·삭제)이고, 우측은 위에서부터 역할 헤더 카드(적용 상태 · 배정 운영자 · 데이터 접근 범위) → 권한설정 매트릭스(리소스 행 × 액션 5열) → 대시보드 위젯 카드다. **저장 버튼이 없다** — 체크를 바꾸면 그 자리에서 저장되고, 그 역할이 '적용 중'이면 사이드바와 대시보드가 리로드 없이 따라 바뀐다.

**주요 사용자** — 최상위 관리자(시스템 역할 보유자). 운영자·상담원은 이 화면의 read 권한이 꺼져 있는 것이 보통이다.

**사용 시나리오**

1. 새 팀이 생겨 '콘텐츠 운영' 역할을 추가한다 → 전 권한 OFF 로 태어난 역할에서 콘텐츠 관리 그룹의 조회·등록·수정만 켠다.
2. 어느 역할이 지금 몇 명에게 붙어 있는지 확인한다 → 헤더의 '이 역할 운영자 N명' 링크로 관리자 관리 목록의 그 역할 필터로 착지한다.
3. 쓰지 않게 된 역할을 지우려 한다 → 그 역할을 쓰는 운영자가 남아 있으면 삭제 버튼이 사유와 함께 잠긴다.
4. 대시보드에서 매출 위젯을 감추고 싶다 → 위젯 카드에서 해당 토글을 끈다. 대시보드가 즉시 그 위젯을 지운다.

**선행 조건** — 로그인 · 이 화면의 `read` 권한. 역할 목록은 브라우저 저장소에서 읽으며, 저장값이 없거나 깨졌으면 기본 역할 3종(슈퍼어드민(전체권한) · 운영자 · 뷰어)으로 시작한다 — `apps/admin/src/shared/permissions/permission-store.ts:64-99`.

**후행 처리** — 변경은 즉시 저장되고, **적용 중인 역할을 고쳤다면** 사이드바 메뉴 가시성 · 각 화면의 쓰기 컨트롤 노출 · 대시보드 위젯이 리로드 없이 바뀐다. 열려 있는 다른 탭에도 저장소 이벤트로 전파된다 — `permission-store.ts:337-346`.

**관련 화면** — `SCR-USERS-ADMINS`(역할을 사람에게 **배정하는** 화면 · `?role=` 필터로 착지) · `SCR-USERS-ADMINS-FORM`(운영자 1명의 역할 선택) · `SCR-DASHBOARD`(위젯 7개가 이 화면의 토글에 걸린다). 팝업 1건은 별도 문서다: [역할 추가·수정](pop-role-form.md). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `사용자 관리 > 권한 관리`(`/users/roles`) — `apps/admin/src/shared/layout/nav-config.ts:131`. 조건부 노출이 아니다.

**관련 기능** — ① 관리자 관리가 이 화면의 역할 목록을 셀렉트 선택지로 그대로 읽는다(목록을 복제하지 않는다). ② 역할 삭제 가드가 관리자 관리의 명부에 '이 역할을 쓰는 운영자가 몇 명인가'를 되묻는다. ③ 이 화면의 매트릭스 행은 **사이드바 잎에서 파생**되므로 메뉴가 늘거나 줄면 행도 함께 바뀐다 — `apps/admin/src/shared/permissions/resources.ts:75-104`.

**관련 API** — 없음(관측된 네트워크 호출 0건). 역할 상태는 브라우저 저장소에 있고, 백엔드 연동 지점은 주석으로만 표시되어 있다: `TODO(backend): GET /api/roles · PUT /api/roles/:id/permissions` — `apps/admin/src/shared/permissions/roles.ts:11` · `PermissionProvider.tsx:23` · `PermissionsPage.tsx:15`. 데이터 접근 범위(scope)도 저장만 하고 실제 필터링은 서버 몫으로 표시되어 있다 — `apps/admin/src/pages/permissions/components/RoleHeaderCard.tsx:151-152`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/permissions/PermissionsPage.tsx` (좌측 패널 `components/RolePanel.tsx` · 헤더 `components/RoleHeaderCard.tsx` · 매트릭스 `components/PermissionMatrixTable.tsx` · 위젯 `components/DashboardWidgetsCard.tsx` · 팝업 `components/RoleFormModal.tsx`). 모델은 화면 밖이다: `apps/admin/src/shared/permissions/resources.ts` · `roles.ts` · `permission-store.ts` · `feature-registry.ts` · `RequirePermission.tsx` · `route-resource.ts`.

## 4. 기능 명세

### 4.1 역할 고르기 · 적용

| 항목 | 내용 |
|---|---|
| 기능 목적 | 편집할 역할을 고르고, 그중 하나를 앱의 유효 권한으로 세운다 |
| 실행 조건 | 화면이 열려 있고 역할이 1개 이상 |
| 사용자 동작 | 좌측 목록에서 역할 클릭 → (필요하면) 헤더의 '이 역할 적용' 클릭 |
| 시스템 처리 | 고른 역할 id 를 세션 상태로 들고(영속하지 않는다), 적용 시에만 저장 상태의 `activeRoleId` 를 바꾼다 — `permission-store.ts:224-232` |
| 데이터 처리 | 적용은 앱의 유효 권한을 바꾼다. 고르기만 하면 아무것도 바뀌지 않는다 |
| Validation | 존재하지 않는 역할 id 로는 적용되지 않는다(목록에 있는 id 만 통과) — `permission-store.ts:229-231` |
| API 호출 여부 | N — 브라우저 저장소만 다룬다 |
| 성공 처리 | 우측 카드 셋이 그 역할의 값으로 다시 그려진다. 적용 시 토스트 `MSG-USERS-ROLES-01`: `'{역할명}' 역할을 적용했어요.` — `PermissionsPage.tsx:125`. 배지가 `현재 적용 중` 으로 바뀌고 좌측 항목에 `적용 중` 배지가 옮겨 붙는다 |
| 실패 처리 | 실패 경로가 없다 — 네트워크를 타지 않고, 목록 밖의 id 는 조용히 무시된다 |
| 예외 처리 | 다른 탭에서 그 역할이 지워졌다면 저장소 동기화 후 첫 역할로 폴백한다 — `permission-store.ts:148-150` |
| 화면 변경 사항 | 좌측 목록 · 역할 헤더 · 매트릭스 · 위젯 카드. **적용했다면 사이드바·대시보드까지** |
| 후속 동작 | 적용 중인 역할을 바꾸면 앱 전체의 메뉴·버튼 가시성이 즉시 재판정된다 |
| 로그 기록 여부 | 확인하지 못했다 — 역할 변경을 관리자 로그로 남기는 배선을 코드에서 찾지 못했다 |

| 성공 처리 | 다이얼로그가 닫히고 목록에서 사라진다. 토스트 `MSG-USERS-ROLES-04`: `'{역할명}' 역할을 삭제했어요.` — `PermissionsPage.tsx:208` |

**삭제 차단 사유 — 숫자가 문장의 핵심이다.** 문구의 정본은 도메인이 갖는다(`roles.ts:331-338`). 같은 문장이 두 곳에서 쓰이기 때문이다: 화면이 버튼을 잠그며 다는 사유(`aria-label`·`title`)와, 저장소가 삭제를 거절하며 돌려주는 오류.

| 상태 | 문구 | 출처 |
|---|---|---|
| 배정 인원을 셀 수 없다(`null`) | `'{역할명}' 역할을 쓰는 운영자를 확인하지 못해 삭제할 수 없어요. 잠시 후 다시 시도해 주세요.` | `roles.ts:333` |
| 배정된 운영자가 1명 이상 | `'{역할명}' 역할을 쓰는 운영자가 {N}명 있어 삭제할 수 없어요. 관리자 관리에서 이 운영자들의 역할을 먼저 바꾼 뒤 다시 삭제해 주세요.` | `roles.ts:336` |
| 막을 이유 없음(0명) | 차단하지 않는다 | `roles.ts:338` |

**확인 다이얼로그는 배정 0명일 때만 열린다.** 그래서 본문은 '아무도 이 역할을 쓰지 않는다'는 사실 확인이 된다 — 두 경우로 갈린다(`PermissionsPage.tsx:195-199`).

| 경우 | 본문 |
|---|---|
| 적용 중인 역할 | `'{역할명}' 은 지금 적용 중인 역할이에요. 이 역할을 쓰는 운영자는 없으며, 삭제하면 남아 있는 첫 역할이 대신 적용돼요. 삭제할까요?` |
| 그 밖 | `'{역할명}' 역할을 삭제할까요? 이 역할을 쓰는 운영자는 없어요. 되돌릴 수 없어요.` |

렌더 지점은 `PermissionsPage.tsx:189` 이며, 제목 `역할 삭제` — `PermissionsPage.tsx:190` · 확인 라벨 `역할 삭제` — `PermissionsPage.tsx:200`. 진행 표시가 없다(`busy={false}` — 저장소가 브라우저 안이라 즉시 끝난다) — `PermissionsPage.tsx:202`.

| Validation | 입력 칸이 없으므로 §6 의 대상이 아니다. 대신 아래 **의존 규칙 3가지**가 모델 층에서 강제된다 |
| API 호출 여부 | N |
| 성공 처리 | 표가 즉시 다시 그려진다. **토스트를 띄우지 않는다** — 체크마다 통지가 뜨면 매트릭스 편집이 불가능해진다. 적용 중인 역할이면 사이드바·대시보드가 함께 바뀐다 |
| 실패 처리 | 실패 경로가 없다(네트워크를 타지 않는다). 브라우저 저장이 실패해도 **현재 세션 동안은 메모리 상태로 계속 동작한다** — `permission-store.ts:49-55`. 즉 실패가 화면에 드러나지 않는다(§7) |
| 예외 처리 | 시스템 역할을 보고 있으면 모든 칸이 비활성이고 편집이 무시된다 — `permission-store.ts:200-205` |
| 화면 변경 사항 | 매트릭스 표(그리고 적용 중이면 사이드바·전 화면의 쓰기 컨트롤) |
| 후속 동작 | 열려 있는 다른 탭도 저장소 이벤트로 따라온다 |
| 로그 기록 여부 | 확인하지 못했다 |

**모델이 강제하는 의존 규칙 3가지** — `resources.ts:17-25`.

| # | 규칙 | 화면에서 보이는 모습 |
|---|---|---|
| 1 | `조회`를 끄면 나머지 4개도 함께 꺼진다 | 조회 체크를 풀면 그 행의 다섯 칸이 전부 풀린다 |
| 2 | 나머지 중 하나라도 켜면 `조회`가 함께 켜진다 | 등록만 눌렀는데 조회도 켜진다 |
| 3 | 그룹 행의 값 = 자식(잎)들의 **합집합** | 자식 일부만 켜면 그룹은 mixed(중간 상태)로 보인다. 그룹 체크를 누르면 자식 전체가 따라간다 |

**그룹 행은 직접 저장되지 않는다** — 자식에서 파생된다. 그래서 자식이 전부 조회 OFF 면 그룹도 OFF 가 되고, 사이드바에서 그 묶음이 통째로 사라진다.

| 데이터 있음 | 좌측 목록 + 우측 카드 3장 | 추가 ○ · 수정/삭제는 고른 역할에 따라 | N | 하단 안내문 3문단 |
| 데이터 없음 | 성립하지 않는다 — 목록이 빌 수 없다(§7) | — | N | — |
| 조회 실패 | 저장값이 깨졌으면 **오류를 띄우지 않고** 기본 역할 3종으로 되돌아간다 — `permission-store.ts:95-98` | 전부 ○ | N | 없음 ⚠ 저장값이 사라졌다는 사실을 화면이 알리지 않는다 |
| 저장 중 | 성립하지 않는다 — 즉시 저장이라 진행 상태가 없다. 삭제 다이얼로그도 `busy={false}` 다 | 전부 ○ | N | — |
| 읽기 전용(권한) | 이 화면의 `read` 가 없으면 본문이 `접근 권한이 없어요` 로 대체된다. **read 만 있는 역할에게는 이 화면이 편집 가능한 채로 열린다**(§9 의 알려진 격차) | — | N | §7 '권한 없음' |
| 시스템 역할 선택 | 자물쇠 + 전 컨트롤 비활성. 매트릭스는 전 칸 체크된 채 잠긴다 | 추가 ○ · 수정 × · 삭제 × · 매트릭스 × · 위젯 × · 범위 × | N | `슈퍼어드민(전체권한)은 시스템 역할이라 …` |
| 배정 인원 0명 | 링크(`이 역할 운영자 0명`) + `이 역할을 쓰는 운영자가 없어요.` | 삭제 ○ | N | 위 문구 |
| 배정 인원 1명 이상 | 링크(`이 역할 운영자 {N}명`) + 삭제 불가 안내 | 삭제 × (사유가 `aria-label`·`title` 에 실린다) | N | `배정된 운영자가 있는 역할은 삭제할 수 없어요 — 먼저 이 운영자들의 역할을 바꿔 주세요.` |
| 배정 인원 **모름**(`null`) | 링크도 숫자도 그리지 않고 문구만 | 삭제 × | N | `운영자 명부를 확인할 수 없어 배정 인원을 셀 수 없어요. 이 상태에서는 역할을 삭제할 수 없어요.` |
| 비활성 역할 편집 중 | 매트릭스는 정상 편집되지만 화면에는 반영되지 않는다 | 전부 ○ | N | `체크를 바꾸면 즉시 저장돼요. 다만 지금 적용 중인 역할은 '{활성 역할명}' 이라 …` |

**`null`(모름) 과 0 을 가르는 자리** — 배정 인원이다. 0명은 '지워도 된다'는 사실이고, `null` 은 '판정할 수 없다'는 사실이다. 화면은 둘을 다른 문장·다른 컨트롤 상태로 그리며, 삭제는 **둘 중 후자에서도** 막힌다(fail-closed).

## 9. 권한

### 9.1 이 화면 자체의 권한

| 권한 | 조회 | 등록 | 수정 | 삭제 | 다운로드 | 업로드 | 승인 | 반려 |
|---|---|---|---|---|---|---|---|---|
| 슈퍼어드민(전체권한) | ○ | ○ | ○ | ○ | — | — | — | — |
| 운영자 | 역할의 `page:/users/roles` 판정을 따른다 | 〃 | 〃 | 〃 | — | — | — | — |
| 뷰어 | ○ | × | × | × | — | — | — | — |
| 권한 없음 | × | × | × | × | × | × | × | × |

다운로드·업로드·승인·반려는 **이 화면에 존재하지 않는 동작**이다(내보내기 버튼도, 파일 입력도 없다).

⚠ **알려진 격차** — 이 화면은 `RequirePermission` 이 거는 `read` 게이팅만 받는다. 역할 추가·이름 수정·삭제·매트릭스 편집·위젯 토글 어디에도 `useRouteWritePermissions` 판정이 없다(`PermissionsPage.tsx` 전체에서 그 훅을 부르지 않는다). 즉 **`read` 만 가진 역할이 이 화면에 들어오면 권한 모델 전체를 고칠 수 있다.** 같은 섹션의 회원 관리·고객 설정·관리자 관리는 전부 쓰기 게이팅을 갖고 있으므로, 이것은 규칙의 예외가 아니라 누락으로 읽힌다. 서버가 없어 실제 차단도 걸리지 않는다.

### 9.2 이 앱의 권한 모델 — 다른 묶음이 참조하는 정본

**권한의 단위는 역할(Role)이다.** 역할 하나 = 리소스 × 액션 매트릭스 + 대시보드 위젯 7토글 + 데이터 접근 범위. 그중 `activeRoleId` 가 가리키는 역할이 앱의 유효 권한을 결정한다 — `roles.ts:3-6`.

**액션은 다섯이다** — `resources.ts:31`.

| 액션 키 | 화면 라벨 | 설명(화면에 함께 뜬다) |
|---|---|---|
| `read` | `조회` | `메뉴 노출 + 목록/상세 조회` |
| `create` | `등록` | `신규 생성` |
| `update` | `수정` | `기존 항목 변경` |
| `remove` | `삭제` | `항목 삭제` |
| `export` | `내보내기` | `CSV/엑셀 다운로드` |

출처는 `resources.ts:41-47`. **승인·반려 액션은 권한 모델에 없다** — 어느 화면의 명세도 그 축을 쓸 수 없다.

**리소스는 하드코딩하지 않는다** — 사이드바 트리에서 파생된다(`resources.ts:75-104`).

| 종류 | 리소스 id | 무엇인가 |
|---|---|---|
| 가지(branch) | `group:{basePath}` | 메뉴 묶음. 예: `group:/users` |
| 가지의 잎 | `page:{to}` | 실제 화면. 예: `page:/users/members` |
| 최상위 잎 | `page:{to}` | 자식이 없는 단독 행. 예: `page:/dashboard` |

접두(`group:` / `page:`)를 붙이는 이유는 경로가 겹치는 쌍이 실재하기 때문이다 — '상품 관리'(basePath `/products`)와 그 첫 잎(`/products`) — `resources.ts:8-9`.

**라우트 → 리소스 규칙: 자기를 감싸는 가장 구체적인 잎이 곧 자기 리소스다** — `apps/admin/src/shared/permissions/route-resource.ts:9-13`.

| 라우트 | 리소스 |
|---|---|
| `/users/members/12` | `page:/users/members` |
| `/users/admins/A-00002/edit` | `page:/users/admins` |
| `/products/9/edit` | `page:/products` (`/products/categories` 가 아니다 — 접두가 아니라 세그먼트 경계로 본다) |
| `/products/categories` | `page:/products/categories` (정확히 일치하는 더 긴 잎이 이긴다) |
| 어떤 잎에도 속하지 않는 경로 | `null` = **권한 대상 아님** → 게이팅하지 않는다(`RequirePermission.tsx:32-33`) |

그래서 **상세·등록·수정 화면은 자기 잎의 권한을 쓴다.** 화면이 자기 리소스 이름을 들고 다니지 않으므로, 같은 컴포넌트가 두 라우트에서 각자 옳게 판정된다.

**의존 규칙 3가지**는 §4.5 표와 같다(모델 층 강제 · UI 가 흉내 내지 않는다).

**모르는 리소스의 기본 판정 — 만장일치 상속.** 저장된 역할이 그 화면이 생기기 전에 만들어졌을 때 어떤 권한을 줄 것인가. 판정 규칙은 `resources.ts:341-366` 이며, 두 번의 교훈이 그 근거로 코드에 기록되어 있다.

| 상황 | 판정 |
|---|---|
| 아는 리소스가 하나도 없다 | 가장 조심스러운 값 — 전 액션 OFF + `read` 만 ON |
| 아는 리소스가 있다 | 액션마다 **아는 리소스 전부에서 그 권한을 갖고 있었을 때만** 새 리소스에서도 갖는다(만장일치) |
| `read` | **예외적으로 항상 ON** — 발견 자체를 막지 않기 위해서다 |

즉 부분 제한 역할이 새 기능으로 **권한을 얻는 일은 없다**. 근거로 남은 두 교훈: ① 전부 OFF 로 읽으면 새 기능이 기존 세션 전부에서 조용히 사라진다(실제로 AI 에이전트 메뉴가 그렇게 사라졌다) ② 읽기만 열면 화면은 보이는데 등록·수정·삭제 버튼만 사라진다('새 메뉴를 연 다음 날 등록 버튼이 없다'로 되돌아온 증상).

**저장값 방어의 방향은 좁은 쪽이다(fail-closed)** — 모순된 저장값(`read=false` 인데 `update=true`)은 전 액션 OFF 로 되돌린다(`resources.ts:192-199`). 이것은 **플랜(엔타이틀먼트)과 정반대**다 — 그쪽은 모르면 연다(fail-open).

**기본 역할 3종** — `roles.ts:98-139`.

| 역할 id | 이름 | 시스템 | 범위 | 매트릭스 | 위젯 |
|---|---|---|---|---|---|
| `role-super-admin` | `슈퍼어드민(전체권한)` | Y — 이름 변경·삭제·권한 수정 불가, 항상 전 권한 ON | 전체 | 전 리소스 × 전 액션 ON | 7개 전부 ON |
| `role-operator` | `운영자` | N | 전체 | 저장값이 없으면 전 권한 ON | 전부 ON |
| `role-viewer` | `뷰어` | N | **본인** | 전 리소스 `read` 만 | 전부 ON |

첫 실행의 적용 역할은 `운영자` 다 — `roles.ts:141-150`. 슈퍼어드민이 목록에서 사라지면 정규화가 되살린다(전 권한을 잃는 잠금 상태를 막는다) — `roles.ts:190-191`.

**새 역할은 전 권한 OFF 로 태어난다** — 최소 권한(`permission-store.ts:243-246`).

### 9.3 권한이 없을 때 화면이 어떻게 되는가

세 층이 서로 다르게 움직인다. **이 표가 다른 15묶음의 §9 가 참조하는 기준이다.**

| 축 | 없을 때 | 근거 |
|---|---|---|
| 화면 읽기(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다. 사이드바·헤더는 남아 다른 메뉴로 걸어 나갈 수 있다. 사이드바에서는 그 잎이 아예 사라진다 | `apps/admin/src/shared/errors/ErrorScreens.tsx:73` · `RequirePermission.tsx:79-82` |
| 쓰기(`create`/`update`/`remove`/`export`) | 그 컨트롤이 **사라진다**(비활성이 아니라 부재). 폼 라우트는 등록이면 `create`, 수정이면 `update` 를 요구하며 URL 로 들어와도 폼 대신 403 화면이 뜬다 | `RequirePermission.tsx:37-70` · `apps/admin/src/shared/crud/FormPageShell.tsx:127-132` |
| 플랜(엔타이틀먼트) | 3상태다: `granted` 정상 · `locked` 잠금 화면 · `absent` 대시보드로 되돌림. **`/users/**` 는 어떤 모듈에도 매핑되지 않아 언제나 `granted` 다** | `apps/admin/src/shared/entitlements/module-resources.ts:16-18`(매핑 없음 = 해당 없음) |

**판정 순서는 인증 → 플랜 → 권한이다.** 뒤집으면 사지 않은 기능에 '권한이 없다'고 말하게 되고, 운영자는 켜 줄 수 없는 관리자에게 권한을 요청한다.

**프런트 게이팅은 보안 경계가 아니라 UX 다** — 위조된 브라우저 저장값으로 우회된다. 실제 차단은 서버의 몫이며, 지금 서버는 없다 — `RequirePermission.tsx:8-11`.
