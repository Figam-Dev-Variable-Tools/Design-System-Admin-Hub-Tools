# 로고 등록 · 수정 (공유 팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 로고 등록 · 수정 |
| 화면 ID | `POP-COMMON-LOGO-FORM` |
| 메뉴 경로 | 기업 관리 > 파트너사 · 기업 관리 > 고객사 (두 화면이 같은 팝업을 쓴다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 홈페이지에 노출할 파트너사·고객사 로고 한 건을 등록하거나 고친다. 이름 · 로고 이미지 · (선택) 링크 세 값을 받는다.

**업무 배경** — 파트너사·고객사 로고 띠는 회사 소개 페이지의 신뢰 장치다. 로고가 늘고 줄 때마다 개발자를 부를 수 없으므로 운영자가 직접 올린다. 두 메뉴는 데이터 모양(로고·이름·링크·정렬·노출)과 화면이 같아 **한 목록 화면과 한 팝업을 공유**하고, 다른 것은 자원 이름과 라벨뿐이다 — `apps/admin/src/pages/company/logo-list/types.ts:3-6`.

**화면 설명** — 목록 위에 뜨는 모달이다. 등록과 수정을 한 팝업이 겸하며, 무엇인지는 제목이 말한다(`파트너사 추가` / `파트너사 수정`). 정렬 순서와 노출 여부는 **여기서 받지 않는다** — 목록에서 드래그와 토글로 다룬다.

**주요 사용자** — 최상위 관리자 · 기업 정보를 다루는 운영자.

**사용 시나리오**

1. 운영자가 파트너사 목록에서 '파트너사 추가'를 누른다 → 이 팝업이 열리고 이름 칸에 커서가 놓인다 → 이름·로고를 채우고 '추가'를 누른다 → 팝업이 닫히고 목록 맨 뒤에 새 행이 생긴다.
2. 운영자가 목록 행의 수정을 누른다 → 같은 팝업이 기존 값으로 채워져 열린다 → 로고 이미지를 교체하고 '저장'을 누른다.
3. 운영자가 이름만 치다 마음이 바뀌어 Esc 를 누른다 → 저장하지 않은 변경 확인이 뜬다.

**선행 조건** — 로그인 · 파트너사(또는 고객사) 화면의 등록·수정 권한. 수정이면 대상 행이 아직 살아 있어야 한다.

**후행 처리** — 저장된 값은 파트너사·고객사 목록에 반영되고, 그 목록은 고객이 보는 홈페이지의 로고 띠에 쓰인다. 홈페이지는 이 리포 밖에 있어 반영 경로를 코드에서 확인하지 못했다.

**관련 화면** — `SCR-COMPANY-PARTNERS` · `SCR-COMPANY-CLIENTS`(둘 다 같은 목록 컴포넌트를 쓴다) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](index.md).

**관련 메뉴** — 기업 관리 아래 두 잎. 조건부 노출이 아니다.

**관련 기능** — 목록의 정렬 드래그 · 노출 여부 토글 · 단건/일괄 삭제. 이 팝업은 그중 어느 것도 다루지 않는다.

**관련 API** — 등록 1건 · 수정 1건. 목적은 '로고 한 건을 만든다/고친다'뿐이다. **백엔드가 없다** — 지금은 픽스처 어댑터가 응답하며, 실제 엔드포인트는 각 화면의 데이터 소스에 `TODO(backend)` 로 남아 있다(`apps/admin/src/pages/company/logo-list/adapter.ts:1-6`). 이미지 업로드 엔드포인트도 아직 없다 — `apps/admin/src/shared/crud/validation.ts:56-58`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [고객 화면 대응](../../reference/user-view.md)

**구현 파일** — `apps/admin/src/pages/company/logo-list/LogoFormModal.tsx` (검증 `validation.ts` · 상수 `types.ts` · 데이터 `adapter.ts`·`queries.ts`)

## 4. 기능 명세

| 항목 | 내용 |

### 4.2 저장(등록 · 수정)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 입력한 로고 한 건을 만들거나 고친다 |
| 실행 조건 | 필수 두 칸(이름 · 로고 이미지)이 채워지고 링크가 형식에 맞을 때 |
| 사용자 동작 | '추가'(등록) 또는 '저장'(수정) 클릭, 또는 폼에서 Enter |
| 시스템 처리 | 검증 → 통과하면 요청 1건. 진행 중에는 입력·버튼이 전부 잠긴다 |
| 데이터 처리 | 등록: 목록 맨 뒤(현재 최대 순서 + 1)에 붙고 **노출 상태로 시작**한다 — `apps/admin/src/pages/company/logo-list/adapter.ts:59-61`. 수정: 이름·이미지·링크만 바뀌고 순서·노출은 그대로다 |
| Validation | §6 전부 |
| API 호출 여부 | Y — 로고 1건 등록 또는 수정. 성공하면 목록 조회가 무효화되어 다시 불린다 — `apps/admin/src/pages/company/logo-list/queries.ts:34,50` |
| 성공 처리 | 팝업이 닫히고 목록이 갱신되며 토스트가 뜬다. 등록: `'{이름}'을(를) 추가했어요.` · 수정: `'{이름}'을(를) 저장했어요.` — `LogoListPage.tsx:255-258` (조사는 이름의 받침이 고른다) |
| 실패 처리 | **팝업이 닫히지 않는다.** 본문 맨 위에 danger 배너: `저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `LogoFormModal.tsx:97`. 입력은 그대로 남는다 |
| 예외 처리 | 검증 실패면 요청을 보내지 않고 **이름 칸으로 포커스**를 옮긴다 — `LogoFormModal.tsx:131`. 팝업이 사라지는 중 도착한 응답은 무시한다(요청이 중단된다) — `LogoFormModal.tsx:85,96` |
| 화면 변경 사항 | 팝업 전체 → 닫힘 · 부모 목록영역 |
| 후속 동작 | 목록 재조회 · 토스트 |
| 로그 기록 여부 | 확인하지 못했다 — 이 경로에서 관리자 로그를 남기는 배선이 코드에 없다 |

### 4.3 이미지 올리기 · 교체 · 제거

| 항목 | 내용 |
|---|---|
| 기능 목적 | 로고 이미지를 고른다 |
| 실행 조건 | 팝업이 열려 있고 저장 중이 아닐 때 |
| 사용자 동작 | 드롭존 클릭(파일 선택) · 파일 끌어다 놓기 · '이미지 교체' · 제거 |
| 시스템 처리 | 고른 파일이 이미지인지, 5MB 이하인지 그 자리에서 검사한다 — `ImageUploadField.tsx:37-41` |
| 데이터 처리 | **업로드하지 않는다.** 지금은 브라우저 안에서만 미리보기가 만들어지고, 저장 값은 그 임시 참조다 — `apps/admin/src/shared/crud/validation.ts:44-50` |
| Validation | §6 의 이미지 항목 |
| API 호출 여부 | N — 업로드 엔드포인트가 아직 없다(`TODO(backend)` · `apps/admin/src/shared/crud/validation.ts:57`) |
| 성공 처리 | 미리보기가 그려지고 안내가 `업로드 완료 — 아래에서 이미지를 교체하거나 제거할 수 있어요.` 로 바뀐다 — `ImageUploadField.tsx:279` |
| 실패 처리 | 필드 아래 인라인 오류: `이미지 파일만 올릴 수 있어요.` 또는 `파일 용량은 5MB 를 넘을 수 없어요.` — `ImageUploadField.tsx:38,40`. 값은 바뀌지 않는다 |
| 예외 처리 | 고른 이미지를 브라우저가 그리지 못하면 `이미지를 불러오지 못했어요. 다시 선택하세요.` — `ImageUploadField.tsx:116` |
| 화면 변경 사항 | 입력영역의 이미지 칸 |
| 후속 동작 | 폼이 dirty 가 되어 닫기 가드가 걸린다 |
| 로그 기록 여부 | N |

**⚠ 지금 확인되는 사실** — 이미지가 서버로 올라가지 않으므로, 팝업을 떠난 뒤 목록 썸네일이 깨진다. 코드가 이것을 '아직 갚지 못한 빚'이라고 명시하고 있으며(`apps/admin/src/shared/crud/validation.ts:41-62`), 검증을 조여도 해결되지 않는다(조이면 저장 자체가 불가능해진다). 업로드 엔드포인트가 붙으면 사라지는 문제다.

| 팝업 닫힘 | 위 경로 또는 저장 성공 | 진행 중 요청 중단 · 포커스 복귀 | 팝업 | 부모 목록 |

## 6. Validation

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 이름 | 공백만이면 안 된다 | 제출 시 | `이름을 입력하세요.` — 공용 헬퍼 `requiredText('이름', 60)` 이 조립한다(`apps/admin/src/shared/crud/validation.ts:22`) | 제출 차단 · 이름 칸으로 포커스 · 인라인 오류 |
| 이름 | 60자 이하 | 제출 시 | `이름은 60자를 넘을 수 없어요.` — 같은 헬퍼의 두 번째 규칙(`apps/admin/src/shared/crud/validation.ts:25`) | 〃 (입력 자체도 60자에서 막힌다) |
| 로고 이미지 | 값이 있어야 한다 | 제출 시 | `로고를 등록하세요.` — 공용 헬퍼 `requiredImage('로고')` 가 조립한다(`apps/admin/src/shared/crud/validation.ts:68`) | 제출 차단 · 인라인 오류 |
| 로고 이미지 | 이미지 파일이어야 한다 | 파일을 고른 즉시 | `이미지 파일만 올릴 수 있어요.` — `packages/ui/src/molecules/ImageUploadField/ImageUploadField.tsx:38` | 값을 바꾸지 않고 인라인 오류 |
| 로고 이미지 | 5MB 이하 | 파일을 고른 즉시 | `파일 용량은 5MB 를 넘을 수 없어요.` — `packages/ui/src/molecules/ImageUploadField/ImageUploadField.tsx:40` | 〃 |
| 링크 URL | 비어 있거나, `http://`·`https://` 로 시작 | 제출 시 | `링크 URL 은 http:// 또는 https:// 로 시작해야 해요.` — 공용 헬퍼 `optionalHttpUrl('링크 URL')` 이 조립한다(`apps/admin/src/shared/crud/validation.ts:34`) | 제출 차단 · 인라인 오류 |

**규칙의 정본과 문장의 정본이 다르다 — 파일이 둘이라 헷갈리기 쉽다.**

| 무엇 | 어디 |
|---|---|
| **어떤 규칙이 걸리는가**(필드 3개에 어떤 헬퍼를 붙였는가) | `apps/admin/src/pages/company/logo-list/validation.ts` — 13줄짜리 스키마 파일이다. 여기에는 **문장이 한 줄도 없다** |
| **문장이 어떻게 만들어지는가** | `apps/admin/src/shared/crud/validation.ts` — 세 헬퍼(`requiredText` · `requiredImage` · `optionalHttpUrl`)가 라벨을 받아 문장을 조립한다 |

조사(`을`/`를`·`은`/`는`)도 코드가 고른다 — 라벨의 받침을 보고 붙이므로 `로고` → `로고를`, `이름` → `이름을` 이 된다. 그래서 이 문서는 문장 인용의 출처로 **헬퍼가 있는 파일**을 적는다(§1.1 규약 4).

화면이 따로 규칙을 갖지 않는다. 이름 오류는 '필수'가 '길이'보다 먼저 걸린다(체크 순서가 곧 우선순위다).

삭제·다운로드는 이 팝업에 없다(부모 목록의 기능이다). 업로드는 등록·수정 권한에 딸린다 — 별도 액션이 아니다(권한 액션은 `read`·`create`·`update`·`remove`·`export` 다섯뿐이다 — `apps/admin/src/shared/permissions/resources.ts:31`).
