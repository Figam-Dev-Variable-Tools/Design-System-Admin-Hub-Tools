# 저장 충돌 — 시스템 설정 (공유 팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 저장 충돌 (시스템 설정) |
| 화면 ID | `POP-COMMON-SETTINGS-CONFLICT` |
| 메뉴 경로 | 시스템 설정 > 사이트 설정 · OAuth 설정(+상세) · 결제 설정(+상세) · API Key 설정 상세 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 내가 설정 화면을 연 뒤 다른 관리자가 같은 설정을 먼저 저장했을 때, **덮어쓰기 전에** 무엇이 달라졌는지 보여 주고 세 갈래 중 하나를 고르게 한다.

**업무 배경** — 시스템 설정은 여러 관리자가 같은 문서를 만진다. 마지막에 저장한 사람이 이기는 구조라면 앞사람의 변경이 아무 흔적 없이 사라진다. 그래서 저장 요청이 충돌로 거절되면 화면은 조용히 다시 시도하지 않고 이 팝업을 세운다.

**화면 설명** — 목록이 아니라 편집 중인 설정 화면 위에 뜨는 모달이다. 무슨 일이 일어났는지 한 문단, 마지막 저장자와 시각, 값이 갈린 항목 목록, 두 선택이 각각 무엇을 버리는지에 대한 안내, 그리고 버튼 둘(최신 내용 불러오기 · 내 변경으로 덮어쓰기)이 있다. 세 번째 선택(닫고 생각하기)은 Esc · 딤 · × 다.

**주요 사용자** — 최상위 관리자 · 시스템 설정을 다룰 권한이 있는 운영자.

**사용 시나리오**

1. 운영자 A 가 사이트 설정을 열어 사이트명을 고치는 동안 운영자 B 가 같은 화면에서 다른 값을 저장한다 → A 가 저장을 누르면 이 팝업이 뜨고, 갈린 항목이 무엇인지 목록으로 보인다.
2. A 가 '최신 내용 불러오기'를 고른다 → 폼이 B 의 값으로 갈아끼워지고, A 의 입력은 사라진다.
3. A 가 '내 변경으로 덮어쓰기'를 고른다 → A 의 값으로 저장되고 B 의 변경이 사라진다.
4. A 가 판단을 미루고 팝업을 닫는다 → 입력은 그대로 살아 있고 아무것도 저장되지 않는다.

**선행 조건** — 로그인 · 해당 설정 화면의 수정 권한 · 저장을 시도했고 그 요청이 충돌로 거절됐을 것.

**후행 처리** — 고른 선택에 따라 설정 문서가 갱신되거나 그대로 남는다. 결제 설정처럼 **다른 화면의 표시를 바꾸는** 설정이 있으므로(상품·프로그램의 구매 버튼, 대시보드의 위젯 구성), 덮어쓰기의 영향은 이 화면 밖으로 나간다.

**관련 화면** — `SCR-SETTINGS-SITE` · `SCR-SETTINGS-OAUTH` · `SCR-SETTINGS-OAUTH-DETAIL` · `SCR-SETTINGS-PAYMENT` · `SCR-SETTINGS-PAYMENT-DETAIL` · `SCR-SETTINGS-API-KEYS-DETAIL`. 폼 화면의 충돌은 다른 것이다 — [공통 문서](index.md) §4.5 `DLG-COMMON-FORM-CONFLICT`.

**관련 메뉴** — 시스템 설정 아래. 조건부 노출이 아니다.

**관련 기능** — 각 설정 화면의 저장(확인 다이얼로그를 거친다) · 감사 표기(마지막 저장자·시각).

**관련 API** — 목적 둘: ① 최신 설정 문서를 다시 조회한다 ② 개정 토큰 검사를 건너뛰고 저장한다(덮어쓰기). **백엔드가 없다** — 지금은 설정 저장소가 픽스처로 충돌을 흉내 낸다.

**왜 확인 다이얼로그가 아닌가** — 확인 다이얼로그는 '확인/취소' 이지선다다. 충돌은 세 갈래이고, 이지선다에 억지로 끼우면 어느 버튼이 무엇을 버리는지 흐려진다. 파괴적 선택일수록 라벨이 결과를 말해야 한다 — `apps/admin/src/pages/settings/_shared/ConflictDialog.tsx:5-10`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [품질 기준](../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/_shared/ConflictDialog.tsx` (갈린 항목 계산 `_shared/diff.ts` · 호출부 6곳: `settings/site/SiteSettingsPage.tsx:653` · `settings/oauth/OAuthPage.tsx:487` · `settings/oauth/OAuthProviderPage.tsx:521` · `settings/payment/PaymentSettingsPage.tsx:531` · `settings/payment/PgProviderPage.tsx:865` · `settings/api-keys/AiConnectionPage.tsx:661`)

## 4. 기능 명세

### 4.1 충돌 알림 띄우기

| 항목 | 내용 |
|---|---|
| 기능 목적 | 저장이 충돌로 거절된 사실을, **덮어쓰지 않은 채** 사용자에게 넘긴다 |
| 실행 조건 | 설정 저장 요청이 충돌로 거절됐을 때 — 예: `settings/site/SiteSettingsPage.tsx:262-266` |
| 사용자 동작 | 없음(저장을 누른 결과로 자동으로 뜬다) |
| 시스템 처리 | 저장 확인 다이얼로그를 닫고 이 팝업을 세운다. 서버가 준 최신 문서를 쥐고, 내 폼 값과 비교해 갈린 항목의 이름을 만든다 |
| 데이터 처리 | **아무것도 바뀌지 않는다.** 폼 입력은 그대로 살아 있다 — `ConflictDialog.tsx:12-13` |
| Validation | 해당 없음 — 입력 칸이 없다 |
| API 호출 여부 | N — 팝업을 띄우는 것 자체는 서버를 부르지 않는다(거절 응답이 최신 문서를 함께 준다) |
| 성공 처리 | 팝업이 뜨고 제목과 상황 설명이 함께 읽힌다. 갈린 항목이 없으면 차이 목록은 그리지 않는다 |
| 실패 처리 | 이 단계에는 실패가 없다 |
| 예외 처리 | 사용자가 저장 중 취소한 경우(요청 중단)는 충돌로 치지 않고 아무 팝업도 띄우지 않는다 — `SiteSettingsPage.tsx:258-259` |
| 화면 변경 사항 | 딤 + 팝업 전체 |
| 후속 동작 | 사용자의 선택을 기다린다 |
| 로그 기록 여부 | 확인하지 못했다 |

### 4.2 최신 내용 불러오기

| 항목 | 내용 |
|---|---|
| 기능 목적 | 다른 관리자의 최신 값을 받아들이고 내 입력을 버린다 |
| 실행 조건 | 팝업이 떠 있고 덮어쓰기가 진행 중이 아닐 때 |
| 사용자 동작 | '최신 내용 불러오기' 클릭 |
| 시스템 처리 | 팝업이 쥐고 있던 최신 문서로 폼을 갈아끼우고 팝업을 닫은 뒤, 설정을 다시 조회한다 |
| 데이터 처리 | **내 입력이 사라진다.** 서버 문서는 바뀌지 않는다(저장하지 않는다) |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 해당 설정 문서 재조회 |
| 성공 처리 | 팝업이 닫히고 폼이 최신 값으로 바뀐다. 성공 토스트가 뜬다 — 문구는 화면마다 다르다: `최신 기본 설정을 불러왔어요.`(`SiteSettingsPage.tsx:306`) · `최신 OAuth 설정을 불러왔어요.`(`OAuthPage.tsx:278` · `OAuthProviderPage.tsx:342`) · `최신 결제 설정을 불러왔어요.`(`PaymentSettingsPage.tsx:324` · `PgProviderPage.tsx:496`) · `최신 연동 설정을 불러왔어요.`(`AiConnectionPage.tsx:406`) |
| 실패 처리 | 폼은 이미 최신 값으로 바뀌어 있고, 재조회가 실패하면 그 화면의 조회 실패 표시가 대신 뜬다 |
| 예외 처리 | 이 선택은 되돌릴 수 없다 — 버려진 입력을 되살리는 수단이 없다. 라벨이 그 사실을 말한다 |
| 화면 변경 사항 | 팝업 → 닫힘 · 입력영역 전체 |
| 후속 동작 | 설정 재조회 · 토스트. 폼이 저장된 값과 같아지므로 이탈 가드도 함께 내려간다 |
| 로그 기록 여부 | N — 저장이 아니다 |

## 6. Validation

해당 없음 — 이 팝업에는 입력 칸이 없다. 저장할 값의 검증은 **팝업이 뜨기 전에** 부모 설정 화면의 폼 검증이 이미 끝냈다. 이 팝업이 다루는 것은 값의 옳고 그름이 아니라 **누구의 값을 남길 것인가**다.

**권한이 없을 때 화면이 어떻게 되는가** — 팝업이 **뜨지 않는다.** 수정 권한이 없으면 부모 설정 화면의 저장 컨트롤이 사라지고, 저장을 시작하지 못하므로 충돌도 일어나지 않는다. 화면 읽기 권한마저 없으면 부모 화면 자리에 `접근 권한이 없어요` 화면이 뜬다 — `apps/admin/src/shared/errors/ErrorScreens.tsx:73`.

**플랜(엔타이틀먼트) 축** — 청구 상태가 정상이 아니면(결제 지연·이용 정지) 전 화면이 조회 전용으로 내려앉아 저장 컨트롤이 잠긴다. 그때도 이 팝업은 뜨지 않는다 — `apps/admin/src/shared/entitlements/plan.ts:403-405`.
