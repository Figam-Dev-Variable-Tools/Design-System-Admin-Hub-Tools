# 공통 확인 다이얼로그 · 공통 안내 토스트

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 공통 확인 다이얼로그 · 공통 안내 토스트 |
| 화면 ID | 해당 없음 — 라우트를 가진 화면이 아니다. 이 문서가 정의하는 개별 요소의 ID 는 아래 **B0 식별자 규약**이 정한다 |
| 메뉴 경로 | 해당 없음 — 사이드바에 자리가 없다. 모든 메뉴 아래에서 함께 뜬다 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 여러 화면이 **같은 모양·같은 문구·같은 동작**으로 쓰는 확인 다이얼로그 5종과 결과 통지 토스트 6종을 한곳에 정의한다. 각 화면 문서는 이것을 다시 적지 않고 참조한다.

**업무 배경** — 삭제 확인, 저장하지 않은 입력을 버리는 확인, 동시 편집 충돌 알림, 그리고 "됐다/안 됐다"를 알리는 통지는 화면마다 필요하다. 화면마다 따로 적으면 같은 규칙이 154벌로 갈라지고 한 벌만 고쳐진다. 코드에서도 이 다섯 다이얼로그와 여섯 토스트는 **공통 훅 한 벌**이 만든다 — 문서도 같은 자리에 한 벌만 둔다.

**화면 설명** — 확인 다이얼로그는 모달 위에 뜨는 이지선다(확인/취소)다. 아이콘 + 제목 + 본문 한 줄 + (실패했다면) 빨간 배너 + 버튼 둘이 전부다. 토스트는 화면 하단 중앙에 쌓이는 통지로, 최대 3개까지 보이고 종류에 따라 스스로 사라지거나 사용자가 닫을 때까지 남는다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원 — 즉 이 어드민에 로그인하는 전원.

**사용 시나리오**

1. 운영자가 목록에서 한 건을 삭제한다 → 단건 삭제 확인이 뜨고, 확인하면 목록이 갱신되며 성공 토스트가 뜬다.
2. 운영자가 폼을 반쯤 채운 채 다른 메뉴를 누른다 → 이탈 가드가 뜨고, '취소'를 고르면 입력이 그대로 남는다.
3. 두 관리자가 같은 폼을 동시에 편집한다 → 나중에 저장한 쪽이 저장 충돌 다이얼로그를 만나고, 내 입력을 지킬지 최신 값을 받을지 고른다.

**선행 조건** — 로그인되어 있어야 한다(로그인 화면에는 이 층이 없다 — 아래 §9). 확인 다이얼로그는 부모 화면이 그것을 여는 동작을 갖고 있어야 뜬다.

**후행 처리** — 확인 다이얼로그의 결과는 부모 화면의 데이터 변경으로 이어지고, 그 결과를 토스트가 통지한다. 토스트 자체는 아무 데이터도 바꾸지 않는다.

**관련 화면** — `CrudListShell` 을 쓰는 모든 목록 화면 · 모든 폼 화면 · 입력을 가진 모든 팝업. 공유 팝업 3건은 별도 문서다: [로고 등록·수정](pop-logo-form.md) · [로그 원문 보기](pop-log-payload.md) · [저장 충돌(설정)](pop-settings-conflict.md).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — 목록의 단건/일괄 삭제 · 폼의 저장 · 목록 행에서 바로 값을 바꾸는 인라인 갱신.

**관련 API** — 없음. 다이얼로그와 토스트 자체는 서버를 부르지 않는다. **부모 기능**이 부르는 쓰기 요청의 결과를 표시할 뿐이다. 이 앱에는 아직 백엔드가 없고, 쓰기 경로는 픽스처 어댑터가 흉내 낸다.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [품질 기준](../../reference/quality-bar.md)

### 1.1 B0 식별자 규약 — 뒤 묶음이 이것으로 참조한다

[ID 규약](../id-convention.md) 은 접두 넷(`SCR-` · `POP-` · `TAB-` · `MSG-`)을 정했다. 확인 다이얼로그는 그중 어디에도 속하지 않는다 — 화면이 아니라 **부모 문서의 절**이기 때문이다. 그런데 공통 다이얼로그는 부모가 16묶음에 흩어져 있어 가리킬 이름이 필요하다. 그래서 B0 가 접두 하나를 더 정한다.

| 접두 | 무엇 | 어디에 산다 |
|---|---|---|
| `DLG-COMMON-` | 여러 화면이 공유하는 확인 다이얼로그 | 이 문서 §4 |
| `MSG-COMMON-` | 여러 화면이 공유하는 안내 토스트 | 이 문서 §4 |

**규약 1.** 화면 문서는 이 요소들의 **제목·본문·버튼 라벨을 옮겨 적지 않는다.** §5 이벤트 · §7 예외 처리 · §10 화면 이동에서 `DLG-COMMON-DELETE-ONE` 처럼 ID 로만 가리키고, 그 화면에서 달라지는 값(엔티티 이름 · 항목 이름 · 건수)만 적는다.

**규약 2.** 화면 고유 확인 다이얼로그 51건은 이 규약을 쓰지 않는다. 부모가 하나뿐이므로 부모 문서 안에서 제목으로 부르면 된다.

**규약 3.** 화면 문서의 토스트 건수를 [인벤토리](../inventory.md) 의 화면별 표와 대조할 때, 이 문서가 정의하는 여섯 건은 **세지 않는다.** 인벤토리도 그것을 공통 층 6 으로 따로 세고 있다.

**규약 4 — 문구 인용의 출처 표기.** §3 Label·Placeholder · §6 오류 메시지 · §7 사용자 메시지 · §8 안내 메시지에 화면 문구를 옮길 때는 그 뒤에 출처를 `— 파일:줄` 형태로 단다. 사용자 문구는 지금도 다듬어지는 중이라, 출처가 없으면 문구가 바뀐 뒤 어느 문서를 고쳐야 하는지 기계적으로 찾을 수 없다.

출처는 **가리키기만 하면 되는 것이 아니라 실재해야 한다.** 아래 넷을 지킨다.

| # | 규칙 | 왜 |
|---|---|---|
| 4-1 | 인용은 **그 문자열이 실제로 있는 파일**을 가리킨다. 소비처가 아니라 **문장이 만들어지는 곳**이다 | 소비처를 적으면 문구가 바뀔 때 아무도 찾지 못한다. 실제로 이 문서 묶음에서 `로고를 등록하세요.` 를 스키마 파일(문장이 한 줄도 없는 13줄짜리)로 잘못 가리킨 일이 있었다 |
| 4-2 | 템플릿·헬퍼가 조립하는 문구는 **조립하는 함수의 위치**를 적고, **어떤 인자로 그 결과가 나오는지** 한 줄 덧붙인다. 예: `로고를 등록하세요.` — 공용 헬퍼 `requiredImage('로고')` 가 조립한다(`apps/admin/src/shared/crud/validation.ts:68`) | 같은 헬퍼를 쓰는 다른 화면이 이 문서를 근거로 삼을 수 있다. 조사(`을`/`를`)까지 코드가 고른다는 사실도 그 한 줄에서 드러난다 |
| 4-3 | 자리표시자는 `{엔티티}` 처럼 **중괄호 표기**로 적는다. 코드의 `${...}` 를 그대로 옮기지 않는다 | 명세는 기획 문서다. 코드 문법을 옮기면 문장이 무엇으로 채워지는지가 오히려 흐려진다 |
| 4-4 | 줄 번호는 **그 파일의 길이 안에** 있어야 한다. 같은 문장이 여러 자리에 있으면 `파일:63,80` 처럼 전부 적는다 | 파일 길이를 넘는 줄 번호는 검증 없이는 드러나지 않는다 — 이 규약이 '출처를 단다'까지만 정했을 때 실제로 그런 인용이 하나 살아남았다 |

파일 이름이 흔한 경우(`validation.ts` · `types.ts` · `api.ts` 처럼 여러 폴더에 같은 이름이 있는 경우)에는 **리포 루트 기준 전체 경로**를 적는다. 짧게 줄이면 읽는 사람이 다른 파일로 해소한다.

**규약 5 — `null` 은 '모른다'이고 `[]`·0 과 다르다.** 화면이 값을 받지 못했을 때(조회기가 배선되지 않았거나 서버가 답하지 않았을 때) 0 을 그리지 않는다. 이 앱은 그 자리에 `—` 를 쓰거나 그 줄을 그리지 않는다. 명세는 §7(예외 처리)과 §8(화면 상태)에서 **'0건'과 '모름'을 다른 행으로** 적는다.

**이 층 자체에는 권한 축이 없다.** 확인 다이얼로그와 토스트는 **부모 기능의 권한을 물려받는다** — 삭제 확인이 뜬다는 것은 이미 삭제 컨트롤이 보였다는 뜻이고, 그 컨트롤의 노출을 정하는 것은 부모 화면의 리소스×액션 판정이다(`apps/admin/src/shared/permissions/RequirePermission.tsx:45-52`). 위 표는 그 판정의 뿌리인 기본 역할 3종(`apps/admin/src/shared/permissions/roles.ts:98-139`)을 옮긴 것이다. 액션은 다섯이다: `read` · `create` · `update` · `remove` · `export` — `apps/admin/src/shared/permissions/resources.ts:31`. 승인·반려 액션은 권한 모델에 없다.

**권한이 없을 때 화면이 어떻게 되는가** — 세 갈래가 서로 다르게 다뤄진다.

| 축 | 없을 때 | 근거 |
|---|---|---|
| 화면 읽기 권한(`read`) | 화면 본문 대신 `접근 권한이 없어요` 화면. 사이드바·헤더는 남아 다른 메뉴로 걸어 나갈 수 있다 | `apps/admin/src/shared/errors/ErrorScreens.tsx:73` |
| 쓰기 권한(`create`/`update`/`remove`) | 그 컨트롤이 **사라진다**(비활성이 아니다). 그래도 URL 로 폼에 들어오면 저장 시 §7 '권한 없음' 배너가 막는다 | `RequirePermission.tsx:1-6` · `useCrudForm.ts:290-298` |
| 플랜(엔타이틀먼트) | 3상태다: `granted` 정상 · `locked` 잠금 화면(상위 플랜 안내) · `absent` 대시보드로 되돌림(메뉴에서도 사라진다) | `apps/admin/src/shared/entitlements/plan.ts:109-112` · `RequireEntitlement.tsx:107-114` |

판정 순서는 **인증 → 플랜 → 권한** 이다 — 사지 않은 기능에 '권한이 없다'고 말하면 운영자가 켜 줄 수 없는 관리자에게 권한을 요청하게 되기 때문이다(`apps/admin/src/shared/entitlements/plan.ts:10-17`).
