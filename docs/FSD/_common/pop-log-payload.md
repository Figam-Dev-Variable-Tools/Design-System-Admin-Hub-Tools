# 로그 원문 보기 (공유 팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 로그 원문 보기 |
| 화면 ID | `POP-COMMON-LOG-PAYLOAD` |
| 메뉴 경로 | 로그 관리 > 관리자 로그 · 회원 활동 로그 · API 로그 · 오류 로그 (네 화면이 같은 팝업을 쓴다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 목록에서 고른 로그 한 건의 상세 항목과 원문 페이로드(요청/응답 본문 · 변경 전후 · 스택)를 **읽기만** 하는 창이다.

**업무 배경** — 감사 로그는 '무엇이 일어났는가'를 확인하는 기록이다. 목록 표에는 한 줄만 담기므로 원문을 봐야 판단이 서는 순간이 있다. 그러나 그 원문에는 비밀번호·토큰·카드번호가 실제로 들어 있어, 그대로 그리면 이 화면은 감사 도구가 아니라 자격증명 열람 창구가 된다 — `apps/admin/src/pages/logs/masking.ts:1-13`. 그래서 원문은 **가려진 뒤에만** 화면에 닿는다.

**화면 설명** — 목록 위에 뜨는 모달이다. 상단에 라벨/값 쌍 목록과 시각 기준 안내, 그 아래에 페이로드가 있으면 마스킹 안내와 원문 블록이 온다. **푸터에 '닫기' 하나뿐이다** — 저장도 삭제도 없다.

**주요 사용자** — 최상위 관리자 · 로그를 볼 권한이 있는 운영자.

**사용 시나리오**

1. 오류 로그 목록에서 어제 실패한 발송 건의 행을 누른다 → 이 팝업이 열려 오류 맥락과 스택을 보여 준다. 수신자 메일 주소는 가려져 있다.
2. API 로그에서 어느 요청이 400 을 받았는지 확인하려고 첫 칸을 키보드로 눌러 연다 → 요청 본문에서 `password` 필드가 있었다는 사실은 보이고 값은 가려져 있다.

**선행 조건** — 로그인 · 해당 로그 화면의 조회 권한 · 목록에 행이 하나 이상 있을 것.

**후행 처리** — 없다. **감사 기록을 열어 본 것은 아무것도 바꾸지 않는다** — `apps/admin/src/pages/logs/components/LogPayloadDialog.tsx:4-8`.

**관련 화면** — `SCR-LOGS-ADMIN` · `SCR-LOGS-MEMBER-ACTIVITY` · `SCR-LOGS-API` · `SCR-LOGS-ERRORS`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](index.md).

**관련 메뉴** — 로그 관리 아래 네 잎. 조건부 노출이 아니다.

**관련 기능** — 로그 목록의 필터·검색·기간·페이지네이션·CSV 내보내기. 이 팝업은 그중 어느 것도 다루지 않는다.

**관련 API** — 없음. 이 팝업은 **자기 조회를 하지 않는다** — 목록이 이미 들고 있는 항목을 표시 형태로 옮길 뿐이다(`spec.detailOf(entry)` · `apps/admin/src/pages/logs/components/LogListShell.tsx:299`). 로그 데이터를 가져오는 조회는 부모 목록의 것이고, 백엔드가 없어 지금은 픽스처가 응답한다.

**왜 라우트가 아니라 팝업인가** — 상세에서 할 일이 '읽는 것'뿐이고, 목록의 맥락(어느 필터로 좁힌 결과의 몇 번째 행인가)이 읽는 동안에도 필요하기 때문이다. 전용 라우트를 만들면 목록으로 돌아올 때마다 조회가 다시 돈다 — `LogPayloadDialog.tsx:15-18`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [품질 기준](../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/logs/components/LogPayloadDialog.tsx` (마스킹 `apps/admin/src/pages/logs/masking.ts` · 타입 `apps/admin/src/pages/logs/types.ts` · 여는 쪽 `components/LogListShell.tsx`·`components/LogTable.tsx`)

## 4. 기능 명세

### 4.1 로그 상세 열기

| 항목 | 내용 |
|---|---|
| 기능 목적 | 목록에서 고른 한 건의 상세와 원문을 본다 |
| 실행 조건 | 로그 목록에 행이 있고, 그 행을 활성화했을 때 |
| 사용자 동작 | ① 행 아무 곳이나 클릭(마우스) ② 첫 칸의 버튼을 키보드로 활성화 — `apps/admin/src/pages/logs/components/LogTable.tsx:181,190-199` |
| 시스템 처리 | 목록이 들고 있던 항목을 표시 형태(제목·항목 목록·페이로드)로 옮겨 팝업에 넘긴다. **페이로드는 화면에 닿기 전 마스킹을 통과한다** |
| 데이터 처리 | 없음 — 읽기만 한다 |
| Validation | 해당 없음 — 입력이 없다 |
| API 호출 여부 | N — 별도 상세 조회를 하지 않는다 |
| 성공 처리 | 팝업이 열리고 포커스가 첫 포커스 가능 요소(닫기 ×)로 간다. 배경은 스크롤이 잠기고 보조기술에서 격리된다 |
| 실패 처리 | 여는 것이 실패하는 경로가 코드에 없다(네트워크를 타지 않는다) |

### 4.2 민감정보 마스킹

| 항목 | 내용 |
|---|---|
| 기능 목적 | 감사에 필요한 사실(어떤 필드가 있었는가)은 남기고, 필요 없는 것(그 값)은 가린다 |
| 실행 조건 | 페이로드가 있을 때 항상 — 끌 수 있는 손잡이가 화면에 없다 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 키 이름을 규칙 목록에 위에서부터 대조해 **처음 걸리는 방식**으로 가린다. 가장 위험한 것(비밀번호·토큰)이 맨 위에 있다 — `masking.ts:44-49` |
| 데이터 처리 | 표시만 바뀐다. 저장된 로그는 그대로다 |
| Validation | 해당 없음 |
| API 호출 여부 | N |
| 성공 처리 | 값 자리에 `●●●●●● [마스킹됨]` 또는 뒤 네 자리만 남은 형태가 표시되고, 그 위에 왜 가려졌는지가 한 문단으로 적혀 있다 |
| 실패 처리 | 규칙에 없는 키는 가리지 않는다 — 다 가리면 감사가 불가능해지기 때문이다 — `masking.ts:44-46`. 즉 '모르는 키'는 노출 쪽으로 열린다 |
| 예외 처리 | 카드 정보처럼 **객체로 감싸져 오는** 값은 감싼 가지 자체를 가려 그 아래 전부가 함께 사라진다 — `masking.ts:56-62` |
| 화면 변경 사항 | 원문영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**⚠ 지금 위험한 사실** — 클라이언트 마스킹은 **두 번째 방어선**이다. 진짜 방어선은 비밀을 로그에 쓰지 않는 것이고, 그다음이 서버가 저장 시점에 가리는 것이다 — `masking.ts:20-25`. 백엔드가 붙으면 이 규칙은 사라지지 않고 서버 규칙과 이중으로 남는다.

| 파일 업로드 실패 | 해당 없음 — 이 팝업에 파일 업로드가 없다 | — | — |

**이 팝업 고유의 위험은 예외가 아니라 노출이다** — 규칙에 없는 키 이름은 가려지지 않는다(§4.2 실패 처리). 새 페이로드 필드가 생기면 마스킹 규칙에 걸리는지 확인해야 한다.
