# 성공 사례 등록·수정

> **[포트폴리오 등록·수정](../items/form.md) 과 같은 폼이다** — 공용 폼 껍데기 · 카드 하나 ·
> 첫 줄 2열 · 공유 하단부(대표 이미지 · 본문 이미지 · 노출 토글) · 저장/이탈/충돌 흐름이 같다.
> 이 문서는 **다른 곳만** 적는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 성공 사례 등록·수정 |
| 화면 ID | `SCR-PORTFOLIO-CASE-STUDIES-FORM` |
| 메뉴 경로 | 비즈니스 > 포트폴리오 관리 > 성공 사례 > 등록·수정 (사이드바에 자리가 없다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 성공 사례 한 건의 **업종·일자·제목·고객사·과제·해결·성과·이미지·노출 여부**를 입력해 만들거나 고친다.

**업무 배경** — 성공 사례는 **세 문단(과제 → 해결 → 성과)이 갖춰져야 이야기가 된다.** 셋 중 하나가 비면 읽는 쪽이 맥락을 잃으므로 셋 다 필수다.

**화면 설명** — 첫 줄에 업종·일자, 그 아래 제목 · 고객사 · **과제 · 해결 · 성과(여러 줄 셋)** · 대표 이미지 · 본문 이미지 · 노출 토글.

**주요 사용자** — 최상위 관리자 · 마케팅 담당 운영자.

**사용 시나리오**

1. 목록의 `성공 사례 등록` 으로 들어온다.
2. 업종을 고르고 일자를 넣는다.
3. 제목·고객사에 이어 과제·해결·성과를 각각 쓴다.
4. 이미지를 올리고 노출을 정해 저장한다.

**선행 조건** — 로그인. 등록은 `create`, 수정은 `update` 권한. **미리 만들어 둘 선행 데이터가 없다** — 업종이 고정 목록이다.

**후행 처리** — 저장하면 목록으로 돌아간다.

**관련 화면** — [성공 사례](index.md)(`SCR-PORTFOLIO-CASE-STUDIES`) · [포트폴리오 등록·수정](../items/form.md)(구조 정본). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. 권한 리소스는 잎 `/portfolio/case-studies` 가 덮는다.

**관련 기능** — 업종 목록·배지 색·정렬·필터가 모두 한 파일에 있다 — `apps/admin/src/pages/portfolio/case-studies/types.ts`.

**관련 API** — 목적 셋: 단건 조회(수정) · 등록 · 수정. **선택지 조회가 없다.** 백엔드가 없다 — 경계는 `apps/admin/src/pages/portfolio/case-studies/data-source.ts:66` 의 `TODO(backend)` 다.

**참고 문서** — [포트폴리오 등록·수정](../items/form.md) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/portfolio/case-studies/CaseStudyFormPage.tsx` · 검증 `case-studies/validation.ts` · 공유 하단부 `apps/admin/src/pages/portfolio/_shared/PortfolioMediaFields.tsx` · 상한값 `case-studies/types.ts:40-43`

## 4. 기능 명세

### 4.1 등록·수정 저장

[포트폴리오 §4.1](../items/form.md) 과 같다 — 같은 공용 폼 컨트롤러를 쓰고, 저장 직전에 문자열의 앞뒤 공백을 턴다. **다른 것은 담기는 항목뿐이다**(과제·해결·성과 셋이 더 있고 분류 라벨 비정규화가 없다).

| 항목 | 내용 |
|---|---|
| 기능 목적 | 한 건을 만들거나 고친다 |
| 실행 조건 | 등록·수정 권한 |
| 사용자 동작 | 입력 후 저장 |
| 시스템 처리 | 공용 폼 컨트롤러가 검증 → 변환 → 저장 |
| 데이터 처리 | **분류 이름을 항목에 박는 처리가 없다** — 업종이 고정 목록이라 id 만 저장하고 이름은 볼 때 찾는다 — `apps/admin/src/pages/portfolio/case-studies/types.ts:60-62` |
| Validation | §6 |
| API 호출 여부 | Y |
| 성공 처리 | 목록으로 돌아가고 공통 성공 토스트 — `MSG-COMMON-01` · `MSG-COMMON-02` |
| 실패 처리 | 입력을 유지한 채 폼 배너 |
| 예외 처리 | 검증에 걸리면 첫 오류로 포커스 |
| 화면 변경 사항 | 화면 전체(성공) · 폼 배너(실패) |
| 후속 동작 | 목록 재조회 |
| 로그 기록 여부 | 확인하지 못했다 |

### 4.2 선택지 채우기 — **이 화면에는 없다**

포트폴리오 폼의 §4.2(분류 선택지 조회)에 해당하는 동작이 없다. 업종이 코드 안에 있어 **조회 실패도, 미도착도, '분류가 하나도 없어 저장할 수 없는' 상태도 생기지 않는다.**

### 4.3 이미지 다루기

[포트폴리오 §4.3](../items/form.md) 과 같다 — 같은 하단부 컴포넌트를 쓰고, **업로드 엔드포인트가 없다**(전역 제약). 최대 장수도 10장으로 같다.

### 4.4 저장하지 않고 떠나기 · 저장 충돌

공통 층의 것이다 — `DLG-COMMON-LEAVE-FORM` · `DLG-COMMON-FORM-CONFLICT`. 안내 문구만 다르다: `성공 사례에 저장하지 않은 변경 사항이 있어요. 이 화면을 벗어나면 입력한 내용이 사라져요.` — `CaseStudyFormPage.tsx:28-29`.
