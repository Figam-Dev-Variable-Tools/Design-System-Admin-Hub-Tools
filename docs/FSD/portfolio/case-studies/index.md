# 성공 사례

> **[포트폴리오](../items/index.md) 와 같은 목록이다** — 툴바(필터 + 등록) · 노출 인라인 토글 ·
> 5열 표 · 빈 목록 · 조회 실패 · 삭제 흐름이 같은 구조다.
> 이 문서는 **다른 곳만** 적는다. 같은 것의 설명은 그 문서가 갖는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 성공 사례 |
| 화면 ID | `SCR-PORTFOLIO-CASE-STUDIES` |
| 메뉴 경로 | 비즈니스 > 포트폴리오 관리 > 성공 사례 — `apps/admin/src/shared/layout/nav-config.ts:246` |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 고객 성공 사례(과제 → 해결 → 성과)를 목록으로 관리한다: 업종으로 거르고, 노출을 표에서 바로 켜고 끄고, 등록·수정·삭제로 넘어간다.

**업무 배경** — 포트폴리오가 **무엇을 만들었나**라면, 성공 사례는 **무엇을 해결했나**다. 영업 자료로 쓰이므로 **업종**으로 찾는다 — 상담 상대와 같은 업종의 사례를 먼저 보여 주기 위해서다.

**화면 설명** — 위에 업종 셀렉트 + `성공 사례 등록` 버튼, 아래에 5열 표(업종 · 제목 · 고객사 · 성과 · 노출)다. **이미지 열은 없다** — `apps/admin/src/pages/portfolio/case-studies/CaseStudyListPage.tsx:4`.

**주요 사용자** — 최상위 관리자 · 마케팅 담당 운영자 · 영업 지원.

**사용 시나리오**

1. 상담 업종에 맞춰 `제조` 만 남기고 훑는다.
2. 더 이상 쓰지 않는 사례의 노출을 끈다.
3. 새 사례는 `성공 사례 등록` 으로 만든다.

**선행 조건** — 로그인. 이 화면의 읽기 권한. **포트폴리오와 달리 선행 데이터가 필요 없다** — 업종이 고정 목록이라 미리 만들어 둘 것이 없다.

**후행 처리** — 노출을 켠 사례가 홈페이지에 실린다(백엔드 없음).

**관련 화면** — [성공 사례 등록·수정](form.md)(`SCR-PORTFOLIO-CASE-STUDIES-FORM`) · [포트폴리오](../items/index.md)(구조 정본). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `비즈니스 > 포트폴리오 관리 > 성공 사례` — `nav-config.ts:246`.

**관련 기능** — 업종은 **코드 안의 고정 6종**이다. 포트폴리오의 분류와 달리 관리 화면이 없다.

**관련 API** — 목적 셋: 목록 조회 · 한 건 수정(노출 토글) · 삭제. **선택지 조회가 없다.** 백엔드가 없다 — 경계는 `apps/admin/src/pages/portfolio/case-studies/data-source.ts:66` 의 `TODO(backend)` 다.

**참고 문서** — [포트폴리오](../items/index.md) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/portfolio/case-studies/CaseStudyListPage.tsx` · 타입·순수 규칙 `case-studies/types.ts` · 공유 열 `apps/admin/src/pages/portfolio/_shared/publishColumn.tsx`

## 4. 기능 명세

### 4.1 업종 필터 — 고정 목록이라 **값을 대조한다**

| 항목 | 포트폴리오 | 성공 사례 |
|---|---|---|
| 거르는 축 | 분류(운영자가 만든다) | **업종(고정 6종)** — `case-studies/types.ts:51-58` |
| 주소 파라미터 | `?category=` | **`?industry=`** — `CaseStudyListPage.tsx:129` |
| 주소의 값 검증 | **하지 않는다**(모르는 값 → 빈 목록) | **한다** — 손으로 고친 값은 `전체` 로 되돌린다 — `CaseStudyListPage.tsx:76-80` |
| 거르는 곳 | 클라이언트 | 클라이언트 — `case-studies/types.ts:80-86` |
| 필터를 바꾸면 | 선택이 풀린다 | 같다 — `CaseStudyListPage.tsx:91-93` |
| 기본값이면 | 주소에서 지운다 | 같다 — `CaseStudyListPage.tsx:67-68` |

**왜 여기서는 대조하는가** — 업종은 **닫힌 목록**이라 허용값을 이미 알고 있고, 그 목록이 비동기로 늦게 오지도 않는다. 포트폴리오가 대조하지 않는 이유(운영자가 만드는 값 + 늦게 도착)가 여기서는 성립하지 않는다.

### 4.2 노출 토글 — 같은 동작, 같은 문구 모양

동작·잠금 규칙·실패 시 되돌림은 [포트폴리오 §4.2](../items/index.md) 와 같고, **같은 열 구현을 공유한다.** 토스트도 같은 모양이다: `'{제목}'을(를) 게시했어요.` / `'{제목}'을(를) 숨겼어요.` — 조립은 `CaseStudyListPage.tsx:116-118`(조사는 제목의 받침을 보고 고른다).

### 4.3 등록·수정·삭제로 넘어가기

[포트폴리오 §4.3](../items/index.md) 과 같다. 경로만 `/portfolio/case-studies/new` · `/portfolio/case-studies/{id}/edit` 다 — `CaseStudyListPage.tsx:142,163`. **상세 화면이 없다**는 것도 같다.

## 6. Validation

입력 폼이 없다. 검증할 사용자 입력은 업종 필터 하나뿐이다.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 주소의 `?industry=` | 허용 6종 + `전체` 인지 본다 | 화면이 그려질 때 | 없다 | 모르는 값은 조용히 `전체` 로 되돌린다 — `CaseStudyListPage.tsx:76-80` |

## 7. 예외 처리

[포트폴리오 §7](../items/index.md) 과 같다(0건 · 조회 실패 · 저장 실패 · 삭제 실패 · 권한 · 세션 만료 · 중복 · 업로드). **다른 줄은 둘이다.**

## 10. 화면 이동(Screen Flow)

[포트폴리오 §10](../items/index.md) 과 같다 — 사이드바 → 목록 → 폼, 공통 삭제 다이얼로그, 401 → 로그인. **다른 것은 경로와 진입 경로 하나다.**
