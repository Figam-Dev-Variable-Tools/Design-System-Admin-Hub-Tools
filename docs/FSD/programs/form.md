# 프로그램 등록·수정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 프로그램 등록 / 프로그램 수정 |
| 화면 ID | `SCR-PROGRAMS-FORM` |
| 메뉴 경로 | 프로그램 관리 > 프로그램 > (등록 · 수정) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 후원형 펀딩 프로그램을 만들고 고친다. 기본 정보 · **펀딩 설정(목표 금액 · 기간)** · 스토리 · 상세 설명 · 이미지 · 옵션 · **리워드** 일곱 구획을 한 폼에서 받는다.

**업무 배경** — 후원형 펀딩은 **목표 금액 · 기간 · 리워드** 세 가지가 계약의 전부다. 모금액과 후원자수는 후원이 만드는 값이라 이 폼에 없다 — `apps/admin/src/pages/programs/ProgramFormPage.tsx:8-9`. ⚠ 그리고 **후원 금액 입력은 결제(PG) 연동이 없으면 잠긴다**(§4.6).

**화면 설명** — `/new` 와 `/:id/edit` 가 **같은 컴포넌트**다. 좌측 구획 목차 레일 + 가운데 입력 카드 7장 + 우측 실시간 미리보기 3단이다. 목차의 붉은 점이 오류가 남은 구획을 가리킨다.

**주요 사용자** — 최상위 관리자 · 콘텐츠/펀딩 담당 운영자.

**사용 시나리오**

1. `프로그램 등록` 으로 들어와 제목·창작자·카테고리·소개를 채우고, 목표 금액과 기간을 정한 뒤 리워드를 두세 종 만든다.
2. 옵션(색상·용량)을 정의하고, **리워드마다 어떤 옵션을 고르게 할지** 체크한다.
3. 결제 연동이 없는 상태에서 리워드 표를 연다 → **후원 금액 칸만 잠기고** 리워드명·설명·수량·옵션은 계속 편집된다.

**선행 조건** — 로그인 · 등록은 `create`, 수정은 `update` 권한. **권한이 없으면 폼 자체가 열리지 않는다**(§4.1).

**후행 처리** — 저장하면 프로그램 목록으로 돌아간다. 새 프로그램의 모금액·후원자수는 0 에서 시작한다 — `apps/admin/src/pages/programs/_shared/store.ts:392-404`.

**관련 화면** — [프로그램 목록](index.md) · [프로그램 상세](detail.md) · [카테고리](categories/index.md)(대·중분류를 만드는 곳) · `SCR-SETTINGS-PAYMENT`(후원 금액 잠금을 푸는 곳) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. **메뉴에 '등록'을 따로 걸지 않은 이유**는 목록을 거치지 않는 우회로가 생겨 수정 경로와 갈라지기 때문이다 — `apps/admin/src/shared/layout/nav-config.ts:198-203`.

**관련 기능** — 카테고리 셀렉트는 카테고리 관리 화면이 만드는 2단계 분류를 읽는다. **후원 금액 잠금은 상품 폼의 판매가와 같은 술어**(`resolvePriceDisplay`)를 부른다 — 두 화면이 다른 답을 말할 수 없게 하는 유일한 방법이다.

**관련 API** — 프로그램 상세 조회 1건(수정 시) · 카테고리 목록 조회 1건 · 등록/갱신 1건. **백엔드가 없다** — `TODO(backend): GET/POST /api/programs · PUT/DELETE /api/programs/:id` — `apps/admin/src/pages/programs/data-source.ts:27`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [품질 기준](../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/programs/ProgramFormPage.tsx` (검증 `validation.ts` · 저장소·정규화 `_shared/store.ts` · 표시 규칙 `types.ts` · 레이아웃 `program-form.css` · 잠금 규칙 `apps/admin/src/shared/commerce/price-display.ts`)

| 기능 목적 | 결제 연동이 없으면 **후원 금액 칸을 잠근다** — 표시(가격문의)와 입력 잠금이 같은 사실의 두 얼굴이기 때문이다 |
| 실행 조건 | 렌더할 때마다 공통 규칙(`resolvePriceDisplay`)을 다시 부른다 — 결제 설정이 바뀌면 다음 렌더가 곧바로 새 답을 쓴다 — `apps/admin/src/pages/programs/ProgramFormPage.tsx:927-933` |

| 성공 처리 | 잠기지 않았으면 금액을 그대로 입력·저장한다 |

| 로그 기록 여부 | N |

**축이 둘이었다가 하나가 됐다** — 예전에는 상품마다 사람이 고르는 '금액 노출 / 가격문의로 대체' 라디오가 있었다. **운영 요구로 그 축을 없앴다**: *"이렇게 선택하는 게 아니라, 결제 연동이 안 돼 있으면 전부 가격 문의로 대체"* — 금액이 이미 설정돼 있어도 그렇고 프로그램도 같다. 라디오가 사라지면 아무도 설정할 수 없는 저장값만 남고, 그것이 이 리포가 가장 싫어하는 '어느 쪽이 진실인지 모르는 값'이다 — `apps/admin/src/shared/commerce/price-display.ts:3-19`.

**잠금 사유 문구(원인별)** — PG 미사용 `PG 결제를 쓰지 않도록 설정되어 있어 모든 금액이 가격문의로 대체돼요.` · 자격증명 거절 `저장된 결제 자격증명이 PG 에서 거절되었어요. 값을 다시 확인해 주세요. 그동안 모든 금액은 가격문의로 대체돼요.` · 자격증명 미완 `필수 자격증명이 아직 채워지지 않아` 로 시작하는 문장(PG 이름과 빠진 값 목록이 함께 들어간다) — `apps/admin/src/shared/commerce/payment-settings.ts` 의 `unhealthyCredentialReason`.

**이 폼은 권한이 없으면 아예 열리지 않는다** — 형제 화면들(견적·계약 폼)이 폼을 열되 저장을 배너로 막는 것과 다르다. 저장할 수 없는 폼을 여는 것 자체를 하지 않는다(§4.1).

**⚠ 플랜(엔타이틀먼트) 매핑이 없다** — 모듈 표에 `/programs` 가 없어 fail-open 으로 언제나 열린다(§ [프로그램 목록](index.md) §9).

**권한이 없을 때 화면이 어떻게 되는가** — 본문이 통째로 `접근 권한이 없어요` 화면으로 바뀐다. 사이드바·헤더는 남아 다른 메뉴로 걸어 나갈 수 있다.
