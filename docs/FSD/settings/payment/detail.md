# 결제 설정 — PG 상세

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | PG 연동 설정 (화면 제목은 PG 이름, 카드 제목은 `연동 설정`) |
| 화면 ID | `SCR-SETTINGS-PAYMENT-DETAIL` |
| 메뉴 경로 | 시스템 설정 > 결제 설정 > (PG) — 사이드바에 자기 자리는 없다 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 결제 대상 하나의 **자격증명 · 연동 모드 · 계약 상태 · 결제수단**을 넣고, 우리가 그 PG 콘솔에 **등록해 줘야 하는 주소**를 보여 준다. 저장은 연동 설정만 쓴다 — 마스터 스위치와 문의 전환 안내는 목록 화면의 것이다.

**업무 배경** — 붙일 수 있는 대상이 다섯이고 PG 마다 받아야 할 값이 2~5개씩 다르다. 한 화면에 다 펼치면 지금 무엇을 채워야 하는지 보이지 않는다. 더 중요한 것은 **절반 이상이 요청 서명의 재료, 즉 비밀**이라는 사실이다 — `signKey` · `site_key` 처럼 이름이 '키'라서 식별자로 읽히는 값도 유출되면 결제 요청을 위조할 수 있다.

**화면 설명** — `목록으로` 링크 → 브랜드 마크가 붙은 PG 이름 → 카드 하나. 카드 안은 자격증명 칸 → 연동 모드·계약 상태 → 결제수단 목록 → 콘솔에 등록할 주소 → 연결 테스트가 없는 이유 순이다. 저장은 확인 다이얼로그를 한 번 거친다.

**주요 사용자** — 최상위 관리자 · 시스템 설정 수정 권한을 가진 운영자.

**사용 시나리오**

1. 토스페이먼츠 계약이 끝났다 → 개발자센터에서 받은 세 값을 넣고, 모드를 `운영`, 계약 상태를 `실결제 오픈` 으로 바꿔 저장한다.
2. 이니시스로 갈아탄다 → 이니시스 화면에 들어가면 카드 위 경고가 '지금은 토스페이먼츠이고 저장하면 갈아탄다'고 말한다.
3. 웹훅을 등록해야 한다 → 콘솔에 등록할 주소를 복사한다. 그 주소가 **아직 서버에 없다**는 경고를 함께 읽는다.
4. 시크릿을 잃어버렸다 → 저장된 값은 `••••••••••••` 로만 보이고, 안내가 콘솔에서 재발급하라고 말한다.

**선행 조건** — 로그인 · `/settings/payment` 잎의 조회 권한. 저장하려면 수정 권한이 필요하다. 주소의 대상 id 가 우리가 아는 다섯 중 하나여야 한다.

**후행 처리** — 저장된 자격증명이 완비이고 목록의 마스터 스위치가 켜져 있으면 결제가 열린다. 그 결과가 상품·프로그램의 금액 표시·입력 잠금·구매 CTA·여섯 섹션의 잠금·사이드바의 문의 잎·대시보드 구성을 바꾼다 — 관계의 전문은 [결제 설정 목록](index.md) §1.1 이 갖는다. **이 화면은 스위치를 켤 수 없으므로**, 꺼져 있으면 값을 넣어도 고객 화면은 그대로 문의하기다(카드 위 info 배너가 그 사실을 말한다).

**관련 화면** — `SCR-SETTINGS-PAYMENT`(목록 · `목록으로` 로 돌아간다) · 그리고 목록 문서 §1.1 이 나열하는 여섯 자리. 저장 충돌 팝업은 공유 문서다: [저장 충돌(설정)](../../_common/pop-settings-conflict.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 항목이 없다. 메뉴 강조와 권한 리소스는 잎 `/settings/payment` 가 덮는다 — `apps/admin/src/App.tsx:216-219`.

**관련 기능** — 목록의 마스터 스위치와 문의 전환 안내는 여기서 다루지 않는다. 반대로 자격증명·모드·계약 상태·결제수단은 **여기서만** 다룬다.

**관련 API** — 목적 둘: ① 결제 설정 문서 1건 조회(목록과 같은 문서) ② 그 문서 저장(개정 토큰 동반). **백엔드가 없다** — 계약은 `pages/settings/payment/data-source.ts:41-45` 와 `shared/commerce/payment-settings.ts:290-304` 의 `TODO(backend)` 가 갖는다. ⚠ **비밀 필드의 값은 응답에 실리지 않는다** — 이름 목록만 온다. 저장의 비밀은 '새로 넣은 값'만 보내고 빈 문자열이면 서버가 기존 값을 유지한다. 결제창 세션 생성(서명) · 승인과 금액 재검증 · 웹훅 수신은 **전부 서버의 일이며 브라우저에서 만들 수 없다.**

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/payment/PgProviderPage.tsx` (자격증명 칸 `components/PgCredentialFields.tsx` · 검증 `validation.ts` · 폼↔도메인 변환 `types.ts` · 데이터 `data-source.ts` · 경로 `paths.ts` · 브랜드 마크 `components/PgMark.tsx` · 시크릿 표시·복사 `../_shared/secret.ts`). 카탈로그의 정본은 `shared/commerce/pg-catalog.ts` 다.

### 1.1 시크릿 취급 — 이 화면의 규율

**저장된 비밀은 화면에 채워지지 않는다.** 화면이 아는 것은 '저장돼 있는가' 하나이고, 그것을 고정 길이 글리프로 표시한다. **그 글리프는 원본을 가린 표시가 아니라 우리가 가진 정보의 전부다** — `../_shared/secret.ts:3-14`.

| 축 | 규칙 | 근거 |
|---|---|---|
| 무엇이 비밀인가 | **칸의 성질**이다. 카탈로그의 `visibility` 가 필드마다 필수로 정하고, 화면은 그것을 읽기만 한다 — 항목마다 다시 판단하지 않는다 | `pg-catalog.ts:108-116` · `components/PgCredentialFields.tsx:3-7` |
| 왜 필수 필드인가 | PG 를 추가할 때 **"이 값이 비밀인가"를 한 번은 답하게** 만들기 위해서다. `signKey` · `site_key` · `MerchantKey` 는 이름이 '키'라 식별자로 읽히지만 전부 서명 재료다 | `pg-catalog.ts:3-13` |
| 저장된 비밀의 표시 | `••••••••••••` — 마지막 4자도 자릿수도 남기지 않는다 | `../_shared/secret.ts:28` |
| 저장된 비밀이 있을 때의 입력 요소 | **아예 렌더하지 않는다.** 가리는 것이 아니라 평문이 들어갈 자리를 두지 않는 것이 방어다 | `components/PgCredentialFields.tsx:9-19,138-153` |
| 새로 넣을 값의 뜻 | 빈 문자열 = 기존 값 유지 · 값이 있음 = 교체 | `validation.ts:3-16` |
| 저장 직후 | 새로 넣은 평문 자리를 **비운다.** 저장은 시크릿이 화면에서 사라지는 시점이기도 하다 | `types.ts:154-162` |
| 공개 식별자 | 값으로 저장되고 되읽힌다. 클라이언트 키처럼 브라우저 SDK 에 심겨 배포되는 값은 애초에 비밀이 될 수 없다 | `pg-catalog.ts:293-297` |
| 타입이 계약을 못박는다 | 연결 모델의 어느 갈래에도 **비밀의 값이 없다** — 직접 연동은 이름 목록만, 게이트웨이는 저장 여부 불리언만 든다. 그래서 화면이 실수로 평문을 그릴 자리가 없다 | `pg-catalog.ts:627-650` |
| 연결 테스트 | **없다.** 브라우저에서 PG 를 부르려면 시크릿이 브라우저로 내려와야 하고, 그 순간 위 규약이 통째로 거짓이 된다. 화면이 그 판단을 문장으로 적는다 | `PgProviderPage.tsx:841-848` · `components/PgCredentialFields.tsx:21-24` |
| 그래서 유효성은 늘 '모름' | 자격증명이 실제로 유효한지 판정하는 조회기가 **배선되어 있지 않다**(등록 함수의 호출부가 앱 코드에 0건이다). 판정은 언제나 `'확인한 적 없음'` 으로 돌아온다 | `payment-settings.ts:158-184` |

### 1.2 콘솔에 등록할 주소 — 방향이 반대다

| 축 | 자격증명 | 콘솔에 등록할 주소 |
|---|---|---|
| 방향 | **PG → 우리** | **우리 → PG** |
| 화면 요소 | 입력칸 | **읽고 복사하는 값**(`<output>` + `복사` 버튼) |
| 값의 출처 | 운영자가 콘솔에서 받아 붙여넣는다 | **사이트 오리진에서 파생**한다 — 손으로 적어 두면 도메인이 바뀌는 날 조용히 낡는다 |
| 등록 주체 | 우리 | PG 콘솔(이니시스는 셀프서비스가 아니라 영업담당자에게 요청해야 한다) |

둘을 같은 모양으로 그리면 운영자는 이 칸도 어딘가에서 받아 와 채워야 하는 줄 알고 콘솔을 뒤진다 — 실제로는 우리가 알려 주는 값인데 — `pg-catalog.ts:139-159` · `PgProviderPage.tsx:790-794`.

## 6. Validation

**마스터 스위치가 필수 여부를 가른다** — 꺼 둔 채로 미리 값을 넣어 두는 것은 정상적인 운영이고(계약 전에 콘솔 값을 받아 적어 두는 일이 흔하다), 그때 빈 칸으로 저장을 막으면 아무것도 저장할 수 없다. **켜져 있을 때만 '다 채웠는가'를 묻는다** — `validation.ts:130-139`.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 공개 자격증명(필수) | 스위치가 켜져 있으면 비울 수 없다 | 제출 시 | `{PG 라벨} {필드 라벨}를 입력하세요.` — `apps/admin/src/pages/settings/payment/validation.ts:171` | 저장 차단 · 첫 위반 필드로 포커스 |
| 공개 자격증명(전부) | 200자 이하 | 〃 | `{필드 라벨}는 200자를 넘을 수 없어요.` — `apps/admin/src/pages/settings/payment/validation.ts:176` | 〃. **길이는 켜짐과 무관하게 본다** — 꺼진 채로도 저장되는 값이라 폭주를 그대로 두면 안 된다 |
| 비밀 자격증명(필수) | 스위치가 켜져 있으면 저장된 값도 새로 넣은 값도 없으면 안 된다 | 〃 | `{PG 라벨} {필드 라벨}를 입력하세요. 저장된 값이 없어요.` — `apps/admin/src/pages/settings/payment/validation.ts:155` | 〃 |
| 비밀 자격증명(전부) | 200자 이하 | 〃 | `{필드 라벨}는 200자를 넘을 수 없어요.` — `apps/admin/src/pages/settings/payment/validation.ts:160` | 〃 |
| 결제수단 | 이 PG 가 지원하지 않는 수단이 켜져 있으면 안 된다 | 〃 | `{PG 라벨}는 {수단 이름들}을(를) 지원하지 않아요. 체크를 풀어 주세요.` — `apps/admin/src/pages/settings/payment/validation.ts:203` | 〃. 화면의 체크박스는 지원 목록에서 파생되므로 **정상 경로로는 생기지 않는다** — PG 를 바꿨을 때만 생긴다 |
| 결제수단 | 스위치가 켜져 있으면 하나 이상 | 〃 | `결제수단을 하나 이상 선택하세요.` — `apps/admin/src/pages/settings/payment/validation.ts:210` | 〃 |

**형식은 검사하지 않는다** — 상점 ID·클라이언트 키·CID 의 길이·문자셋·접두어를 판정하지 않는다. PG 마다 다르고 어느 형식도 문서로 보장받지 못했다 — 정규식으로 박아 두면 PG 사가 채번 규칙을 바꾸는 날 **멀쩡한 값이 거절된다.** 막는 것은 **비어 있음**과 **길이 폭주** 둘뿐이다 — `validation.ts:18-24`.

### 6.1 경고 — 저장을 막지 않는다

| 대상 항목 | 조건 | 문구 | 왜 막지 않는가 |
|---|---|---|---|
| 공개 자격증명 | 값 안에 공백이 있을 때 | `값 안에 공백이 있어요. 붙여넣기할 때 앞뒤가 잘렸는지 확인하세요.` — `apps/admin/src/pages/settings/payment/validation.ts:316` | '틀렸다'가 아니라 '확인해 보라'다 — `validation.ts:304-311` |

**계약 상태는 검증하지 않고 결제도 막지 않는다** — 운영자가 손으로 적는 값이라 오타 하나가 상점 전체를 닫아 버릴 수 있다. 계약이 아직 열리지 않았다는 사실은 막는 대신 **CTA 의 사유 문장이 말한다** — `payment-settings.ts:103-108`.
