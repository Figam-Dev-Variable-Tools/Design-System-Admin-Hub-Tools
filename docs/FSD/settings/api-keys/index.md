# API Key 설정 (연동 목록)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | API Key 설정 (화면 안 카드 제목은 `연동 목록`) |
| 화면 ID | `SCR-SETTINGS-API-KEYS` |
| 메뉴 경로 | 시스템 설정 > API Key 설정 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 이 어드민이 **밖으로 걸 수 있는 연동 7종**을 한자리에 모아 보여 주고, 각각이 지금 붙어 있는지(연동 완료 / 연동 해제)를 말한다. 자격증명을 넣는 일은 이 화면이 하지 않는다 — 이름을 누르면 그 연동의 상세로 간다.

**업무 배경** — 연동마다 요구하는 값의 개수와 이름이 다르다(키 한 칸으로 끝나는 곳도 있고, 리소스 주소·배포명·리전·고객코드까지 필요한 곳도 있다). 일곱을 한 화면에 펼치면 지금 무엇을 채워야 하는지 보이지 않는다. 그래서 진열대(목록)와 자격증명 화면(상세)을 갈랐다 — OAuth 설정이 목록과 상세로 갈린 것과 같은 관례다.

**⚠ 이 화면은 AI 전용이 아니다** — 분류 축에 `shipping`(배송)이 있고 CJ대한통운이 그 자리에 서 있다. 화면 문구가 스스로를 'AI 프로바이더 진열대'라고 부르던 자리는 전부 정리됐다. 진열대가 담은 것보다 좁게 말하면, 배송 연동을 찾아온 운영자가 '여기가 아니구나' 하고 되돌아간다 — `apps/admin/src/pages/settings/api-keys/ApiKeysPage.tsx:8-10`.

**화면 설명** — 안내문 한 줄 아래에 카드 하나가 있고, 그 안에 탭 여섯과 표가 있다. 표의 열은 상태 · 이름 · 연동 시작일 · 설정 넷이다. 이름과 `앱 설정`은 둘 다 그 연동의 상세로 가는 **링크**이며, 행 끝의 더보기 메뉴에 `연동 해제` · `연동 방법 안내` 둘이 있다.

**주요 사용자** — 최상위 관리자 · 시스템 설정 조회 권한을 가진 운영자.

**사용 시나리오**

1. 운영자가 `배송` 탭을 눌러 CJ대한통운이 있는지 확인하고, 이름을 눌러 상세로 가서 인증키와 고객코드를 넣고 켠다 → 목록으로 돌아오면 그 행이 `연동 완료`가 되어 있고 `연동 시작일`이 찍혀 있다.
2. 어떤 모델을 붙일 수 있는지 훑어보려고 기본 탭(`모델`)에서 넷을 확인하고, `연동 방법 안내`로 그 제공자의 공식 문서를 새 창에 연다.
3. 조회가 실패한 채로 화면에 들어온다 → 표는 그려지되 전 행이 `연동 해제`이고, 위쪽 배너가 **그것이 확인된 사실이 아니라는 것**을 말한다.

**선행 조건** — 로그인 · 이 화면의 조회 권한. 이 화면에는 저장 동작이 없어 수정 권한이 필요한 자리가 없다.

**후행 처리** — 이 화면 자체는 아무것도 바꾸지 않는다. 상세에서 저장한 결과가 이 목록의 상태·연동 시작일·탭 건수로 돌아온다. 모델 4종의 연동 상태는 AI 화면의 응답 모드 잠금이 그대로 읽는다 — `apps/admin/src/pages/settings/api-keys/data-source.ts:328-345`.

**관련 화면** — `SCR-SETTINGS-API-KEYS-DETAIL`(이름·`앱 설정`을 누르면 간다) · `SCR-PRODUCTS-SHIPPING`(배송 연동이 가리키는 택배사 원장. 대조는 상세가 한다) · `SCR-SETTINGS-OAUTH`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md). 이 화면으로 리다이렉트되는 옛 경로는 없다.

**관련 메뉴** — 사이드바 `시스템 설정` 가지의 두 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:299`. 조건부 노출이 아니다. 이 라벨은 한때 `AI 모델 연동`이었고 지금 이름으로 되돌아왔다 — 경로는 그 사이 바뀌지 않았다.

**관련 기능** — 상세 화면의 저장이 이 목록의 상태를 바꾼다. 상태는 이 화면이 지어내지 않고 **저장된 자격증명에서 해소한다** — `apps/admin/src/pages/settings/api-keys/integrations.ts:843-857`.

**관련 API** — 목적 하나: 저장된 연동 문서 1건 조회(어느 연동에 어느 칸이 저장돼 있고 켜져 있는가). **백엔드가 없다** — 계약은 `apps/admin/src/pages/settings/api-keys/data-source.ts:53-58` 의 `TODO(backend)` 가 갖는다. ⚠ **응답에 평문 자격증명은 실리지 않는다** — 저장 여부만 온다. 카탈로그(이름·설명·요구 자격증명)는 서버가 주지 않고 프론트가 소유한다 — `apps/admin/src/pages/settings/api-keys/integrations.ts:460-462`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/api-keys/ApiKeysPage.tsx` (목록 카드 `components/IntegrationsCard.tsx` · 카탈로그·탭·필터 `integrations.ts` · 저장된 연동 `data-source.ts` · 연동 성립 판정 `ai-connections.ts` · 로고 `components/ServiceGlyph.tsx` · 주소 `paths.ts`)

### 1.1 카탈로그 7종과 분류 3축

**분류 축은 셋이다** — `model` · `foundation` · `shipping` — `apps/admin/src/pages/settings/api-keys/integrations.ts:95`.

**식별자와 화면 문구가 같은 낱말이어야 한다**는 규약이 이 축에 걸려 있다. 직전 기준에서는 식별자가 `cloud` 인데 화면은 '파운데이션 모델'이라 불렀고, 그러면 코드를 읽는 사람과 화면을 보는 사람이 같은 분류를 다른 이름으로 부르게 된다('클라우드 탭이…'라는 제보를 받은 사람이 화면에서 그 탭을 찾지 못한다). 그래서 `foundation` 으로 개명했고, 새로 들어온 `shipping`('배송')도 같은 규칙을 따른다 — `apps/admin/src/pages/settings/api-keys/integrations.ts:69-84`.

**카탈로그가 비밀 여부를 적지 않는다** — 어느 칸이 비밀인지는 칸 이름 하나가 답하고(`CREDENTIAL_SECRECY`), 카탈로그 항목에는 그것을 적을 자리가 타입에 없다. 항목마다 적을 수 있으면 같은 칸이 한 연동에서는 비밀이고 다른 연동에서는 공개가 될 수 있고, 그 갈라짐은 **그 연동에서만 키가 평문으로 저장되는** 형태로 나타난다 — `apps/admin/src/pages/settings/api-keys/integrations.ts:157-162`. 규율의 전문은 [연동 상세](detail.md)가 갖는다.

## 4. 기능 명세

### 4.1 목록 해소 — 카탈로그(항상 있다) + 저장된 연동(조회해야 안다)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 무엇을 붙일 수 있는지(카탈로그)와 지금 붙어 있는지(저장된 자격증명)를 합쳐 표 한 벌을 만든다 |
| 실행 조건 | 화면 진입 시 자동 |
| 사용자 동작 | 없음(자동). 실패 후 `다시 시도` 클릭 |
| 시스템 처리 | 카탈로그를 돌면서 같은 id 의 저장 기록을 찾고, **켜져 있고 필수 칸이 빠짐없이 저장돼 있을 때만** `연동 완료`로 해소한다 — `apps/admin/src/pages/settings/api-keys/ai-connections.ts:254-262`. 상태를 카탈로그에 적어 두지 않는 이유는 적는 순간 그것이 사실처럼 보이고 아무도 확인하지 않은 채 화면에 나가기 때문이다 |
| 데이터 처리 | 없음(읽기만). 조회 응답에 평문 자격증명은 오지 않는다 |
| Validation | 해당 없음 — 입력 칸이 없다 |
| API 호출 여부 | Y — 저장된 연동 문서 1건 조회 |
| 성공 처리 | 탭 건수와 표가 그려진다. **토스트를 띄우지 않는다** |
| 실패 처리 | 표를 지우지 않는다. 전 항목이 `연동 해제` 쪽으로 붙고(fail-closed), 그 사실을 danger 배너가 말한다: `저장된 연동을 불러오지 못했어요.` + `다시 시도` — `apps/admin/src/pages/settings/api-keys/ApiKeysPage.tsx:118` |
| 예외 처리 | **확인하지 못한 것을 `연동 완료`로 그리지 않는다.** 다만 조용히 해제로 그리지도 않는다 — 그러면 진짜 해제와 구분되지 않는다 — `apps/admin/src/pages/settings/api-keys/ApiKeysPage.tsx:87-93` |
| 화면 변경 사항 | 목록영역 전체 · 오류영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**저장된 것이 없으면 7/7 이 `연동 해제` 다** — 픽스처로 연동된 척 채우지 않는다. 붙지도 않은 것을 완료라 말하면 운영자는 되지도 않는 기능을 믿고 판다 — `apps/admin/src/pages/settings/api-keys/data-source.ts:44-51`.

**재현 손잡이** — `?fail=load` · `?fail=api-keys:load` — `apps/admin/src/pages/settings/_shared/store.ts:53-58`.

### 4.2 탭으로 거르기

| 항목 | 내용 |
|---|---|
| 기능 목적 | 같은 목록을 분류·상태 두 축으로 거른다 |

**축 둘이 각각 목록을 남김없이 나눈다** — 분류 셋의 합도, 상태 둘의 합도 `전체`와 같다. 이 불변식과 **빈 분류 탭을 남기지 않는다**는 규칙은 테스트가 고정한다 — `apps/admin/src/pages/settings/api-keys/api-keys.test.ts:355,375,381,386`.

**빈 탭을 만들지 않는다** — 한때 있던 `gateway` 분류는 그 분류를 가진 항목이 0이 되자 축에서 지웠다. 남겨 두면 탭이 `게이트웨이 (0)` 로 남고, 그것은 없는 것을 '아직 안 채운 것'처럼 보이게 한다. 반대로 **항목이 하나뿐인 분류는 탭을 갖는다**(`배송 (1)`) — 이 규칙이 금지하는 것은 0건이지 1건이 아니다 — `apps/admin/src/pages/settings/api-keys/integrations.ts:86-93,899-901`.

**`0건`과 `모름`이 다른 자리** — '저장된 연동이 0건'은 조회가 돌려준 **사실**이라 화면이 단정해 `연동 해제`로 그린다. 반면 **조회 자체가 실패한 상태**는 같은 모양으로 그리되 배너가 '확인하지 못했다'고 말한다 — 두 상태의 표가 같아 보이므로 그 배너가 유일한 구분점이다. 같은 이유로 카탈로그 밖 레코드 경고는 조회 실패 시 뜨지 않는다.
