# OAuth 설정 (소셜 로그인 목록)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | OAuth 설정 (화면 안 카드 제목은 `소셜 로그인`) |
| 화면 ID | `SCR-SETTINGS-OAUTH` |
| 메뉴 경로 | 시스템 설정 > OAuth 설정 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 소셜 로그인 제공자 여섯을 **사용 중 / 이용 가능** 두 묶음으로 보여 주고, 이 화면에서만 정할 수 있는 두 가지를 저장한다: **로그인 버튼 순서**와 **로그인 화면 표시 정책**. 켜고 끄기와 자격증명은 제공자를 눌러 상세 화면에서 한다.

**업무 배경** — 제공자마다 받아야 할 값의 개수와 이름이 다르고(Apple 은 정적 시크릿조차 없다) 한 화면에 전부 펼치면 지금 무엇을 채워야 하는지 보이지 않는다. 그래서 목록과 상세로 갈랐다. 반면 '버튼을 어떤 순서로 보여 줄까'와 '카카오톡 인앱 브라우저에서 무엇을 감출까'는 **제공자 하나의 사실이 아니라 로그인 화면 전체의 규칙**이라 목록이 소유한다.

**화면 설명** — 안내문 아래 카드 하나에 두 타일 묶음이 세로로 쌓이고, 그 아래에 표시 정책 카드가 하나 더 있다. 사용 중 묶음의 제목 오른쪽에는 순서 변경 버튼이 있고, 그 모드를 켜면 타일마다 위/아래 이동 버튼이 붙는다. 타일을 누르면 **주소가 바뀌어** 그 제공자의 자격증명 화면으로 간다.

**주요 사용자** — 최상위 관리자 · 시스템 설정 수정 권한을 가진 운영자.

**사용 시나리오**

1. 운영자가 카카오 타일을 눌러 상세로 가서 자격증명을 넣고 켠 뒤 돌아온다 → 카카오가 '사용하고 있는 서비스' 묶음으로 옮겨 와 있다.
2. 사용 중 제공자가 둘 이상이 되자 `로그인 버튼 순서 변경`을 눌러 카카오를 맨 위로 올리고 저장한다.
3. 카카오톡 인앱 브라우저에서 구글 로그인이 막힌다는 제보를 받고 표시 정책 체크박스를 켠다 → 카카오가 꺼져 있으면 저장이 막히고, 목록에서 카카오가 어느 묶음에 있는지 눈으로 확인한다.

**선행 조건** — 로그인 · 이 화면의 조회 권한. 순서와 정책을 저장하려면 수정 권한이 필요하다.

**후행 처리** — 저장된 배열 순서가 **로그인 화면의 버튼 순서**가 되고, 표시 정책이 인앱 브라우저에서 어떤 버튼을 감출지 정한다. **이 반영은 아직 서버 쪽 계약으로만 존재한다** — `apps/admin/src/pages/settings/oauth/OAuthPage.tsx:29` 의 `TODO(backend)`. 이 어드민의 로그인 화면은 소셜 버튼을 그리지 않는다.

**관련 화면** — `SCR-SETTINGS-OAUTH-DETAIL`(타일을 누르면 간다) · `SCR-SETTINGS-API-KEYS`(연동 목록이 '연동 완료' 판정에 이 화면의 자격증명 판정 함수를 쓴다 — `validation.ts:876-898`). 저장 충돌 팝업은 공유 문서다: [저장 충돌(설정)](../../_common/pop-settings-conflict.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md). 이 화면으로 리다이렉트되는 옛 경로는 없다.

**관련 메뉴** — 사이드바 `시스템 설정` 가지의 세 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:300`. 조건부 노출이 아니다.

**관련 기능** — 상세 화면의 켜고 끄기가 이 화면의 두 묶음을 바꾼다(두 목록은 저장되지 않고 `enabled` 에서 파생된다). 표시 정책의 교차 규칙이 **카카오의 켜짐 여부**를 읽으므로 두 화면이 같은 문서 한 벌을 공유한다.

**관련 API** — 목적 둘: ① OAuth 설정 문서 1건 조회 ② 그 문서 저장(개정 토큰 동반). **백엔드가 없다** — 계약은 `apps/admin/src/pages/settings/oauth/data-source.ts:61-73` 의 `TODO(backend)` 가 갖는다. ⚠ **조회 응답에 시크릿은 실리지 않는다** — 저장 여부만 온다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/oauth/OAuthPage.tsx` (제공자 메타·검증 `validation.ts` · 데이터 `data-source.ts` · 경로 `paths.ts` · 타일 목록 `components/ProviderTileList.tsx` · 브랜드 마크 `components/provider-marks.tsx` · 공용 껍데기 `../_shared/SettingsFormShell.tsx` · 개정 토큰 저장소 `../_shared/store.ts`)

### 4.2 로그인 버튼 순서 바꾸기

| 항목 | 내용 |

| API 호출 여부 | N(저장 시점에 함께 나간다) |

| 예외 처리 | 순서를 바꾸다 사용 중이 1개로 줄면 버튼이 비활성이 되는데, 그때 라벨이 `순서 변경 완료` 로 굳으면 빠져나올 수 없는 모드가 된다. 그래서 **조건이 사라지면 모드도 함께 풀린다** — `:360-365` |

### 4.3 로그인 화면 표시 정책

| 항목 | 내용 |
|---|---|
| 기능 목적 | 카카오톡 인앱 브라우저에서 **카카오 로그인만** 보여 줄지 정한다 |
| 실행 조건 | 조회가 끝났고 수정 권한이 있을 때 |
| 사용자 동작 | 체크박스를 켜거나 끈다 |
| 시스템 처리 | 이 값은 제공자 배열 밖의 별도 자리에 산다 — 자격증명 옆에 두면 '카카오 콘솔에서 받아오는 값'으로 오해받기 때문이다 — `validation.ts:636-647` |
| 데이터 처리 | 켜짐/꺼짐 한 값. 저장해야 반영된다 |
| Validation | **교차 규칙**: 카카오가 꺼져 있으면 이 정책을 켤 수 없다(§6) |
| API 호출 여부 | N(저장 시점에 함께 나간다) |

| 실패 처리 | 교차 규칙 위반이면 저장이 막히고 정책 카드 안 danger 배너가 이유와 다음 행동을 말한다 |
| 예외 처리 | 카카오의 켜짐 여부는 **다른 화면(상세)에서 바뀌지만**, 두 화면이 같은 문서 한 벌을 읽으므로 이 화면의 폼에도 최신 값이 실려 있다 — `validation.ts:761-764` |
| 화면 변경 사항 | 입력영역 |
| 후속 동작 | 저장 |

### 4.4 저장 — 순서와 정책만 쓴다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 로그인 버튼 순서와 표시 정책을 저장한다. **자격증명은 건드리지 않는다** |
| 실행 조건 | 변경 사항이 있고, 저장 중이 아니며, 수정 권한이 있을 때 |
| 사용자 동작 | 저장 클릭 → 확인 다이얼로그에서 확인 또는 취소 |
| 시스템 처리 | 저장 페이로드를 **폼이 아니라 서버가 준 최신 문서에서** 만든다: 순서는 폼이 정하고 각 항목의 내용은 서버가 정한다 — `OAuthPage.tsx:113-124`. 폼 사본을 그대로 보내면 그 사이 상세 화면에서 바뀐 시크릿을 낡은 값으로 덮어쓴다 |
| 데이터 처리 | 제공자 배열의 **순서**와 표시 정책만 바뀐다. 각 제공자의 자격증명·켜짐 여부는 서버 값 그대로다 |
| Validation | 표시 정책의 교차 규칙 하나만 본다 — 자격증명은 검증하지 않는다. **고칠 입력칸이 이 화면에 없기 때문이다** — `validation.ts:813-830` |
| API 호출 여부 | Y — OAuth 설정 문서 저장(개정 토큰 동반) |
| 성공 처리 | 다이얼로그가 닫히고 성공 토스트 `MSG-SETTINGS-OAUTH-01`: `로그인 버튼 순서와 표시 정책을 저장했어요.` — `OAuthPage.tsx:229` |
| 실패 처리 | 다이얼로그가 닫히지 않고 그 안 danger 배너: `OAuth 설정을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `:239` |
| 예외 처리 | 충돌 거절은 실패가 아니라 §4.5 로 간다. 취소·Esc·딤은 진행 중 요청을 중단하고 실패로 치지 않는다 — `:263-270`. 빠른 더블 클릭의 두 번째는 동기 잠금이 막는다 — `:214` |
| 화면 변경 사항 | 확인 다이얼로그 · 카드 · 버튼영역 |
| 후속 동작 | 저장된 순서가 로그인 화면의 버튼 순서가 된다(서버 계약) |
| 로그 기록 여부 | 확인하지 못했다 |

**확인 문구는 바뀐 것을 조립해 말한다** — `저장할까요?` 만 물으면 자격증명까지 함께 저장되는 줄 알고 확인하게 된다 — `OAuthPage.tsx:126-163`.

| 무엇이 바뀌었나 | 본문 조각 |
|---|---|
| 순서가 바뀌었고 켜진 제공자가 있다 | `로그인 버튼을 {이름} → {이름} 순서로 보여줘요.` — `:147` |
| 순서가 바뀌었으나 켜진 제공자가 없다 | `로그인 버튼 순서를 바꿔요.` — `:145` |
| 정책을 켰다 | `카카오톡 인앱 브라우저에서는 카카오 로그인만 보여줘요.` — `:154` |
| 정책을 껐다 | `카카오톡 인앱 브라우저에서도 모든 소셜 로그인을 보여줘요.` — `:155` |
| 위 조각이 하나 이상 | 뒤에 `각 제공자의 자격증명은 바뀌지 않아요. 저장할까요?` 가 붙는다 — `:162` |
| 바뀐 것을 짚어낼 수 없을 때 | `로그인 버튼 순서와 표시 정책을 저장할까요?` — `:134,159` |

**재현 손잡이** — `?fail=save` — `apps/admin/src/pages/settings/oauth/data-source.ts:9`.

### 4.5 저장 충돌 — 이 화면이 쓰는 것만 짚는다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 다른 관리자가 먼저 저장했을 때 덮어쓰기 전에 사람이 고르게 한다 |

| Validation | 이미 통과한 상태다 |
| API 호출 여부 | 불러오기는 재조회 1건 · 덮어쓰기는 토큰 검사를 건너뛴 저장 1건 |

| 예외 처리 | 닫기는 판단 보류다 — 입력이 남고 토스트도 배너도 없다. 덮어쓰기도 폼이 아니라 **서버 최신 문서 + 내 순서·정책**으로 만들어 보낸다 — `:281-285` |
| 화면 변경 사항 | 팝업 · (불러오기 시) 목록영역·입력영역 |
| 후속 동작 | 불러오기는 재조회를 함께 건다 |
| 로그 기록 여부 | 확인하지 못했다 |

## 6. Validation

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |
|---|---|---|---|---|
| 카카오톡 인앱 정책 | 정책을 켜려면 **카카오 로그인이 켜져 있어야 한다** | 값이 바뀔 때 + 제출 시 | `카카오 로그인이 꺼져 있어 이 정책을 켤 수 없어요. 켜면 카카오톡 인앱 브라우저 방문자에게 로그인 수단이 남지 않아요.` — `apps/admin/src/pages/settings/oauth/validation.ts:772`. 화면은 그 뒤에 `위 '이용 가능한 서비스'에서 카카오 로그인을 열어 먼저 켜세요.` 를 붙인다 — `OAuthPage.tsx:465-466` | 저장 차단 · 정책 카드 안 danger 배너 |

**이 화면은 자격증명을 검증하지 않는다.** 다른 제공자의 오류가 이 화면의 저장을 막는데, **그 오류를 고칠 입력칸이 여기 없기 때문**이다 — 저장 버튼이 이유 없이 죽은 것처럼 보인다. 자격증명 규칙은 그것을 그리는 화면이 갖는다: [제공자 상세](detail.md) — 근거는 `validation.ts:813-819`.

**순서에는 규칙이 없다.** 어떤 순서든 유효하고, 사용 중이 1개 이하이면 순서 개념 자체가 성립하지 않아 컨트롤이 잠긴다(그 이유를 문구가 말한다).

**교차 규칙이 필드가 아니라 문서 수준인 이유** — 두 값(카카오 켜짐 · 정책 켜짐)은 각자 보면 멀쩡하고 **함께 볼 때만** 틀린다. 켜진 채 저장되면 인앱 브라우저 방문자에게는 로그인 수단이 하나도 남지 않는다 — `validation.ts:755-759`.
