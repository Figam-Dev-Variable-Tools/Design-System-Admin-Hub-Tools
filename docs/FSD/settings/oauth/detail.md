# OAuth 설정 — 제공자 상세

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 소셜 로그인 제공자 설정 (화면 제목은 제공자 이름, 카드 제목은 `자격증명`) |
| 화면 ID | `SCR-SETTINGS-OAUTH-DETAIL` |
| 메뉴 경로 | 시스템 설정 > OAuth 설정 > (제공자) — 사이드바에 자기 자리는 없다 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 제공자 한 명의 **자격증명과 Redirect URI** 를 넣고, 그 제공자의 로그인을 켜거나 끈다. 저장은 **이 제공자 자리만** 쓴다 — 다른 제공자와 표시 정책은 서버 값 그대로 남는다.

**업무 배경** — 제공자마다 받아야 할 값의 모양이 다르다. 다섯은 `client_id + client_secret` 짝이지만 **Apple 에는 정적 시크릿이 아예 없다** — 그 자리에 서명 재료 넷(Services ID · Team ID · Key ID · `.p8` 개인키)이 온다. 한 화면에 여섯을 다 펼치면 지금 무엇을 채워야 하는지 보이지 않고, 시크릿 한 칸에 Apple 을 욱여넣으면 나머지 세 값이 갈 곳을 잃는다.

**화면 설명** — `목록으로` 링크 → 브랜드 마크가 붙은 제공자 제목 → 카드 하나. 카드 맨 위 오른쪽에 사용 토글이 있고, 그 아래 발급 위치 안내 · (있으면) 콘솔 준비물 안내 · 자격증명 칸 · Redirect URI · (카카오만) 카카오싱크 안내 · 비활성 연결 테스트 순으로 놓인다. 저장은 확인 다이얼로그를 한 번 거친다.

**주요 사용자** — 최상위 관리자 · 시스템 설정 수정 권한을 가진 운영자.

**사용 시나리오**

1. 목록에서 카카오 타일을 눌러 들어와 REST API 키와 Client Secret 을 넣고, Redirect URI 를 콘솔에 등록한 값과 한 글자도 다르지 않게 적은 뒤 토글을 켜고 저장한다.
2. Google 의 시크릿을 잃어버렸다 → 저장된 값은 `••••••••••••` 로만 보이고, 안내가 '콘솔에서도 다시 볼 수 없으니 회전(재발급)하라'고 말한다 → `변경` 을 눌러 새 값을 넣는다.
3. Apple 을 설정한다 → 시크릿 칸이 없고 서명 재료 넷을 받는 것을 확인한다. `.p8` 을 고르면 **이름만** 저장된다는 안내를 읽는다.

**선행 조건** — 로그인 · `/settings/oauth` 잎의 조회 권한. 저장하려면 수정 권한이 필요하다. 주소의 제공자 id 가 우리가 아는 여섯 중 하나여야 한다.

**후행 처리** — 저장된 켜짐 여부가 목록 화면의 두 묶음을 바꾸고, 표시 정책의 교차 규칙(카카오가 켜져 있는가)에도 그대로 반영된다. 자격증명이 완비인지에 대한 판정은 연동 목록(`SCR-SETTINGS-API-KEYS`)의 '연동 완료' 표시가 그대로 쓴다 — `validation.ts:876-898`.

**관련 화면** — `SCR-SETTINGS-OAUTH`(목록 · `목록으로` 로 돌아간다) · `SCR-SETTINGS-API-KEYS`. 저장 충돌 팝업은 공유 문서다: [저장 충돌(설정)](../../_common/pop-settings-conflict.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 항목이 없다. 메뉴 강조와 권한 리소스는 잎 `/settings/oauth` 가 덮는다 — `apps/admin/src/App.tsx:213-214`.

**관련 기능** — 목록 화면의 순서 변경·표시 정책은 여기서 다루지 않는다. 반대로 켜고 끄기와 자격증명은 **여기서만** 다룬다.

**관련 API** — 목적 셋: ① OAuth 설정 문서 1건 조회(목록과 같은 문서) ② 그 문서 저장(개정 토큰 동반) ③ Apple `.p8` 파일 업로드 전용 통로. **백엔드가 없다** — 계약은 `apps/admin/src/pages/settings/oauth/data-source.ts:61-84` 의 `TODO(backend)` 가 갖는다. ⚠ **조회 응답에 시크릿과 `.p8` 내용은 실리지 않는다** — 저장 여부만 온다. 저장 요청의 시크릿은 '새로 넣은 값'만 보내고, 비어 있으면 서버가 기존 값을 유지한다. 연결 테스트용 엔드포인트도 계약으로만 있다 — `components/OAuthProviderCard.tsx:900-903`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/oauth/OAuthProviderPage.tsx` (자격증명 카드 `components/OAuthProviderCard.tsx` · 제공자 메타·검증 `validation.ts` · 데이터 `data-source.ts` · 시크릿 표시 규약 `../_shared/secret.ts` · 공용 껍데기 `../_shared/SettingsFormShell.tsx`)

### 1.1 시크릿 취급 — 이 화면의 규율

**저장된 비밀은 화면에 채워지지 않는다.** 화면이 아는 것은 '저장돼 있는가' 하나이고, 그것을 고정 길이 글리프로 표시한다. **그 글리프는 원본을 가린 표시가 아니라 우리가 가진 정보의 전부다** — `../_shared/secret.ts:3-14`.

| 축 | 규칙 | 근거 |
|---|---|---|
| 무엇이 비밀인가 | **칸의 성질**이다. 제공자 메타가 갈래(`client-secret` / `apple-key`)를 정하고, 갈래가 어느 칸이 비밀인지 정한다 — 항목마다 따로 판단하지 않는다 | `validation.ts:86-89,581-625` |
| 저장된 비밀의 표시 | `••••••••••••` — 마지막 4자도 자릿수도 남기지 않는다 | `../_shared/secret.ts:28` |
| 저장된 비밀이 있을 때의 입력 요소 | **아예 렌더하지 않는다.** 가리는 것이 아니라 평문이 들어갈 자리를 두지 않는 것이 방어다 | `components/OAuthProviderCard.tsx:5-7,416-424` |
| 새로 넣을 값의 뜻 | 빈 문자열 = 기존 값 유지 · 값이 있음 = 교체 | `validation.ts:5-7` |
| 저장 직후 | 새로 넣은 평문 자리를 **비운다.** 저장은 시크릿이 화면에서 사라지는 시점이기도 하다 | `data-source.ts:91-121` |
| 비밀이 아닌 값 | 네이티브 앱 키(카카오)와 Apple 의 세 식별자는 **평문으로 왕복하고 마스킹하지 않는다** — 비밀이 아닌 값을 비밀처럼 다루면 진짜 비밀의 취급이 헐거워진다 | `validation.ts:589-596` · `components/OAuthProviderCard.tsx:529-531` |
| 연결 테스트 | **버튼은 있으나 비활성이다.** 브라우저에서 제공자를 부르려면 시크릿이 브라우저에 있어야 하고, 게다가 교차 출처도 막힌다. 가짜 성공을 보여 주느니 없는 편이 정직하다 | `components/OAuthProviderCard.tsx:36-37,899-908` |
| Apple `.p8` | **내용은 폼 상태에 들어오지 않는다** — 고른 파일의 **이름만** 담는다. 업로드 통로는 아직 없고 화면이 그 사실을 숨기지 않는다 | `components/OAuthProviderCard.tsx:31-34,658-661,694-699` |
