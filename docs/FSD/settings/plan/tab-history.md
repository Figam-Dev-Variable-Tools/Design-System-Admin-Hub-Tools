# 플랜·이용 현황 — 결제내역 탭

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 결제내역 (탭) |
| 화면 ID | `TAB-SETTINGS-PLAN-HISTORY` |
| 메뉴 경로 | 시스템 설정 > 플랜·이용 현황 > 결제내역 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — **우리가 사내에 낸 구독료의 지난 청구**를 보여 준다. 청구일 · 이용 기간 · 그때의 계약명 · 금액 · 상태 · 영수증 여섯 열이 전부다.

**업무 배경** — '얼마를 냈나'는 이용현황(무엇을 쓰고 있나)·결제(어떻게 내고 있나) 다음에 오는 질문이다. 세 탭의 순서가 곧 읽는 순서다 — `tabs.ts:20-25`.

**화면 설명** — 카드 하나 안에 범위 안내 한 줄과 표 하나가 있다. 아직 목록을 받지 못했으면 표 대신 info 배너가, 받았는데 실제로 0건이면 빈 상태 안내가 온다. **이 표에는 행 액션이 없고, 앞으로도 없어야 한다.**

**주요 사용자** — 최상위 관리자 · 이 화면의 조회 권한을 가진 운영자.

**사용 시나리오**

1. 지난달 구독료를 확인하려 한다 → 표에서 청구일과 금액을 본다. 지금은 **아직 목록을 받지 못했다**는 배너가 대신 뜬다.
2. 세금계산서를 찾는다 → 그 행의 영수증 링크로 나간다. 취소·환불 건에는 링크가 없어 `—` 가 온다.
3. 고객에게 청구한 건을 찾으려다 여기로 왔다 → 범위 안내 한 줄이 `/sales/billing` 으로 안내한다.

**선행 조건** — 로그인 · `/settings/plan` 의 조회 권한. 부모 화면의 탭 하나다.

**후행 처리** — 없다. 이 탭은 아무것도 바꾸지 않는다.

### 1.1 `/sales/billing` 과 무엇이 다른가 — 이 문서에서 가장 중요한 문단

두 화면은 낱말이 거의 같다(청구 · 금액 · 상태 · 미납). 그래서 한쪽을 만들다 보면 다른 쪽을 흉내 내게 되고, **흉내 낸 순간 운영자는 돈의 방향을 잃는다** — `components/InvoiceHistoryPanel.tsx:3-18`.

| 축 | `/sales/billing` (영업 관리 > 청구·입금) | 이 탭 (`/settings/plan` · 결제내역) |
|---|---|---|
| 돈의 방향 | 우리가 **고객에게** 청구한 건 — 받을 돈 | 우리가 **사내에** 낸 구독료 — 낼 돈 |
| 누가 만드나 | 수주 전환된 견적에서 생긴다 | 사내 어드민이 만든다 |
| 쓰기 | **있다** — 운영자가 입금을 확인해 상태를 바꾼다 | **없다** — 결과를 받아 보여줄 뿐이다 |
| 필터·검색·상세 | 있다 | **없다** |
| 바깥으로 나가는 것 | — | 영수증 링크 하나 |

그래서 이 표에는 '입금 확인' 같은 **행 액션이 없다.** 모델의 타입 이름도 `Billing*` 이 아니라 `Subscription*` 으로 두어 import 한 줄만 봐도 어느 쪽 돈인지 드러나게 했다 — `subscription.ts:11-13`.

**관련 화면** — 부모 [플랜·이용 현황](index.md) · 나머지 탭 [이용현황](tab-usage.md) · [결제](tab-billing.md) · `SCR-SALES-BILLING`(혼동 금지 대상).

**관련 메뉴** — 사이드바에 자기 항목이 없다. 부모 화면의 `?tab=history` 상태다.

**관련 기능** — 다음 청구일은 이 탭이 아니라 결제 탭이 갖는다 — 빈 상태 안내가 그리로 보낸다.

**관련 API** — 목적 하나: 지난 청구 목록 조회. **조회만 한다** — `subscription.ts:29-32`. **지금 이 조회기는 `null` 을 돌려준다**(빈 배열이 아니다).

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/plan/components/InvoiceHistoryPanel.tsx` (모델·포맷 `../subscription.ts` · 공용 스타일 `../styles.ts`)

### 1.2 세 상태를 반드시 가른다

**모름을 '내역이 없습니다'로 그리면, 3년째 결제 중인 고객에게도 같은 문장이 나온다.** 이 화면에서 가장 비싼 결함이 그것이라 상태를 뭉치지 않는다 — `InvoiceHistoryPanel.tsx:20-26`.

| 상태 | 뜻 | 화면 | 운영자가 할 일 |
|---|---|---|---|
| `null` | 이 어드민이 **청구 목록을 받지 못했다** | info 배너 하나 | 사내 홈페이지에서 확인 |
| `[]`(빈 배열) | 받았고 지난 청구가 **정말 0건이다**(계약 첫 달) | 빈 상태 안내 + 다음 청구 안내 | 없음 — 정상이다 |
| 목록 | 받았고 건이 있다 | 표 | 필요하면 영수증을 연다 |

**0건일 때는 단정해도 된다** — 받았고, 정말 없다. 그래서 그때만 완결된 문장을 쓴다 — `InvoiceHistoryPanel.tsx:157-159`.
