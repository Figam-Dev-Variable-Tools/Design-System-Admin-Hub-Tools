# 사이트 설정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 사이트 설정 (화면 안 카드 제목은 `기본 설정`) |
| 화면 ID | `SCR-SETTINGS-SITE` |
| 메뉴 경로 | 시스템 설정 > 사이트 설정 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 사이트의 **이름 · 설명 · 표시 이미지 · 공개 범위 · 이용 옵션**을 한 문서로 정한다. 여기서 정하는 값의 대부분은 이 화면 밖에서만 눈에 띄므로(브라우저 탭 · 공유 카드 · 문자 본문 앞), 값을 받는 자리 옆에 **결과를 함께 그린다**.

**업무 배경** — 사이트 이름 하나가 브라우저 탭·검색 결과·카카오톡 공유 카드·문자 발신 표시에 동시에 쓰인다. 그 값들이 화면마다 흩어져 있으면 어디를 고쳐야 하는지 알 수 없고, 고친 결과를 확인하려면 실제로 공유해 봐야 한다. 이 화면은 값의 자리를 하나로 모으고 결과를 미리 보여 준다.

**화면 설명** — 안내문 아래 카드 하나에 네 섹션이 세로로 쌓인다: 사이트 기본 정보 → 사이트 표시 이미지 → 공개 범위 → 사이트 이용 옵션. 각 섹션은 왼쪽에 라벨·설명, 오른쪽에 컨트롤을 두는 같은 축을 쓴다. 카드 아래에 마지막 변경자·시각과 저장 버튼이 있고, 저장은 확인 다이얼로그를 한 번 거친다.

**주요 사용자** — 최상위 관리자 · 시스템 설정 수정 권한을 가진 운영자. 수정 권한이 없으면 조회만 된다.

**사용 시나리오**

1. 운영자가 사이트 이름을 고친다 → 오른쪽 브라우저 탭 목업과 공유 카드 목업의 글자가 입력과 동시에 바뀐다 → 저장을 누르고 확인한다.
2. 리뉴얼 기간 동안 사이트를 닫는다 → 공개 범위를 '비공개'로 바꾸면 카드 위에 경고가 뜨고, 저장 확인 문구가 "방문자는 접속할 수 없어요"를 앞세운다.
3. 문자 발송의 발신 표시를 짧게 만들려고 메일·SMS 전용 이름을 켠다 → 카운터가 글자 수가 아니라 **바이트**를 세는 것을 보고 한글 길이를 조정한다.

**선행 조건** — 로그인 · 이 화면의 조회 권한. 저장하려면 수정 권한이 필요하다.

**후행 처리** — 저장된 값 중 셋이 이 화면 밖에서 실제로 읽힌다.

| 저장 값 | 어디에서 읽는가 |
|---|---|
| 사이트 이름 · 메일·SMS 전용 이름 | SMS 발송 본문과 이메일·뉴스레터 제목의 접두. 전용 이름 스위치가 꺼져 있으면 사이트 이름이 대신 쓰인다 — `apps/admin/src/wiring.ts:337-346` |
| 로그인 상태 유지 | 로그인 화면 '로그인 유지' 체크박스의 **기본값** — `apps/admin/src/shared/domain/site-policy.ts:96-98` |
| 복사 방지 · 모바일 확대 허용 | **이 리포 안에 소비자가 없다.** 방문자가 보는 공개 사이트의 규칙이고, 그 사이트는 리포 밖에 있다 |

**관련 화면** — `SCR-LOGIN`(로그인 유지 기본값을 읽는다) · `SCR-MARKETING-SMS` · `SCR-MARKETING-EMAIL` · `SCR-MARKETING-NEWSLETTERS`(발신 표시 이름을 읽는다). 저장 충돌 팝업은 공유 문서다: [저장 충돌(설정)](../../_common/pop-settings-conflict.md) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md). 이 화면으로 리다이렉트되는 옛 경로는 없다.

**관련 메뉴** — 사이드바 `시스템 설정` 가지의 첫 항목. 조건부 노출이 아니다 — `apps/admin/src/shared/layout/nav-config.ts:289-290`.

**관련 기능** — 권한 관리(`SCR-USERS-ROLES`)가 이 화면의 조회·수정 가능 여부를 정한다. 발송 화면과 로그인 화면은 이 화면을 **모른다** — 값은 공통 층의 조회기를 통해 건너간다.

**관련 API** — 목적 둘: ① 설정 문서 1건 조회 ② 그 문서 저장(내가 읽은 개정 토큰을 함께 보낸다). 파일 업로드용 통로가 하나 더 필요하다. **백엔드가 없다** — 계약은 `apps/admin/src/pages/settings/site/data-source.ts:77-82`(조회·저장)와 `:92`(업로드)의 `TODO(backend)` 가 갖는다. 저장 응답은 성공 / 충돌 / 필드 검증 실패 셋으로 갈린다.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/site/SiteSettingsPage.tsx` (검증 `validation.ts` · 데이터 `data-source.ts` · 업로드 흐름 `useAssetUpload.ts` · 골격 `components/SettingLayout.tsx` · 자산 칸 `components/AssetField.tsx` · 카운터 입력 `components/CountedInput.tsx` · 미리보기 `components/Previews.tsx` · 공용 껍데기 `../_shared/SettingsFormShell.tsx` · 개정 토큰 저장소 `../_shared/store.ts`)

## 4. 기능 명세

### 4.1 설정 조회

| 항목 | 내용 |
|---|---|
| 기능 목적 | 지금 저장된 사이트 설정 1건과 그 개정 토큰·감사 정보를 가져온다 |
| 실행 조건 | 화면 진입 시 자동. 조회 신선도(30초) 안에는 캐시가 답한다 |
| 사용자 동작 | 없음(자동). 실패 후 `다시 시도` 클릭 |
| 시스템 처리 | 문서가 도착하면 **폼을 그 값으로 채우고**, 그 값이 곧 변경 여부(dirty) 판정의 기준선이 된다 — `SiteSettingsPage.tsx:182-185` |
| 데이터 처리 | 없음(읽기만). 개정 토큰을 함께 쥔다 — 다음 저장이 그 토큰을 되돌려 보낸다 |
| Validation | 해당 없음 — 조회 결과를 검증하지 않는다 |

### 4.2 저장 — 확인을 한 번 거친다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 네 섹션의 값을 한 문서로 저장한다. **사이트를 내리는 값이 섞여 있어 클릭 한 번으로 바뀌지 않는다** |
| 실행 조건 | 변경 사항이 있고, 저장 중이 아니며, 수정 권한이 있을 때 |
| 사용자 동작 | 저장 클릭 → 확인 다이얼로그에서 확인 또는 취소 |
| 시스템 처리 | 제출 시점에 스키마가 값을 판정한다. 통과하면 **저장하지 않고** 그 값을 쥔 채 확인 다이얼로그를 세운다 — `SiteSettingsPage.tsx:276-280`. 확인을 누르면 내가 읽은 개정 토큰과 함께 저장 요청 1건을 보낸다 |
| 데이터 처리 | 설정 문서 전체가 폼 값으로 치환된다. 성공하면 그 값이 새 기준선이 되어 변경 표시와 이탈 가드가 함께 내려간다 |
| Validation | §6 전부. 첫 위반 필드로 포커스가 옮겨간다 |
| API 호출 여부 | Y — 설정 문서 저장(개정 토큰 동반) |
| 성공 처리 | 다이얼로그가 닫히고 성공 토스트 `MSG-SETTINGS-SITE-01` 이 뜬다: `기본 설정을 저장했어요.` — `SiteSettingsPage.tsx:254` |
| 실패 처리 | **다이얼로그가 닫히지 않는다.** 다이얼로그 안 danger 배너로 알린다: `기본 설정을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `SiteSettingsPage.tsx:268`. 확인 재클릭이 곧 재시도다 |
| 예외 처리 | 저장이 충돌로 거절되면 실패가 아니라 §4.3 으로 간다. 취소·Esc·딤 클릭은 진행 중이던 요청을 중단하고 실패로 치지 않는다 — `:288-295`. 빠른 더블 클릭의 두 번째는 동기 잠금이 막는다 — `:237` · `../_shared/queries.ts:58-75` |
| 화면 변경 사항 | 확인 다이얼로그 · 카드(오류 배너) · 버튼영역 |
| 후속 동작 | 저장된 값이 발송 화면의 발신 표시 이름과 로그인 화면의 '로그인 유지' 기본값에 반영된다(§1 후행 처리) |
| 로그 기록 여부 | 확인하지 못했다 — 이 화면의 저장을 관리자 로그로 남기는 배선을 코드에서 찾지 못했다 |

**확인 문구는 세 갈래다** — 무엇이 일어나는지 모르고 확인을 누르지 않게, 공개 범위 전환이면 그 사실을 앞세운다 — `SiteSettingsPage.tsx:138-146`.

|---|---|
| 전체 공개 → 비공개 | `사이트를 비공개로 바꿔요. 저장하는 즉시 관리자를 제외한 방문자는 사이트에 접속할 수 없어요. 저장할까요?` — `:140` |
| 비공개 → 전체 공개 | `사이트를 전체 공개로 바꿔요. 저장하는 즉시 누구나 사이트에 접속할 수 있어요. 저장할까요?` — `:143` |
| 그 밖 | `기본 설정을 저장하면 사이트 전반에 즉시 반영돼요. 저장할까요?` — `:145` |

### 4.3 저장 충돌 — 덮어쓰지 않는다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 내가 화면을 연 뒤 다른 관리자가 같은 설정을 먼저 저장했을 때, **내 값으로 덮어쓰기 전에** 사람이 고르게 한다 |
| 실행 조건 | 저장 요청이 개정 토큰 불일치로 거절됐을 때 — `SiteSettingsPage.tsx:262-266` · `../_shared/store.ts:143-146` |
| 사용자 동작 | 최신 내용 불러오기 · 내 변경으로 덮어쓰기 · 닫기(Esc·딤·×) 셋 중 하나 |
| 시스템 처리 | 저장 확인 다이얼로그를 닫고 `POP-COMMON-SETTINGS-CONFLICT` 를 세운다. 서버가 함께 준 최신 문서와 내 폼 값을 비교해 **값이 갈린 항목의 이름 목록**을 만든다 — `:322-325` |
| 데이터 처리 | 이 시점에는 **아무것도 바뀌지 않는다.** 폼 입력이 그대로 살아 있다 |
| Validation | 이미 통과한 상태다(저장을 누를 때 걸렀다) |
| API 호출 여부 | 팝업을 띄우는 것 자체는 N. '불러오기'는 재조회 1건, '덮어쓰기'는 토큰 검사를 건너뛴 저장 1건이다 |
| 성공 처리 | '불러오기' → 폼이 최신 값으로 바뀌고 토스트 `MSG-SETTINGS-SITE-02`: `최신 기본 설정을 불러왔어요.` — `:306`. '덮어쓰기' → 팝업이 닫히고 `MSG-SETTINGS-SITE-01` 이 뜬다 |
| 실패 처리 | 덮어쓰기가 실패하면 팝업이 닫히지 않고 그 안 danger 배너로 알린다(§4.2 와 같은 문구) |
| 예외 처리 | 팝업을 닫는 것은 취소가 아니라 **판단 보류**다 — 토스트도 배너도 뜨지 않고 입력이 남는다. 다시 저장하면 서버 값이 그대로이므로 같은 충돌을 다시 만난다 |
| 화면 변경 사항 | 팝업 · (불러오기 시) 입력영역 전체 |
| 후속 동작 | 불러오기는 재조회를 함께 건다 — `:305` |
| 로그 기록 여부 | 확인하지 못했다 |

**갈린 항목의 이름은 필드 라벨 표에서 온다** — 12개 필드가 라벨과 1:1 로 묶여 있다(`사이트 이름` · `사이트 설명` · `메일·SMS 전용 사이트 이름 사용` · `메일·SMS 전용 사이트 이름` · `사이트 주소` · `파비콘` · `대표 이미지` · `공개 범위` · `비공개용 이미지` · `복사 방지` · `모바일 확대 허용` · `로그인 상태 유지`) — `data-source.ts:106-119`.

**재현 손잡이** — `?fail=conflict` 로 충돌을 재현한다 — `../_shared/store.ts:53-58`.

등록·삭제·다운로드·승인·반려는 이 화면에 **존재하지 않는 동작**이다. 설정 문서는 자원당 하나뿐이라 만들거나 지우는 개념이 없고, 내보내기 컨트롤도 없다. 업로드는 별도 액션이 아니라 수정의 일부다 — 저장하지 않으면 반영되지 않는다.

**권한이 없을 때 화면이 어떻게 되는가** — 두 층이 다르다.

| 없는 것 | 결과 | 근거 |
|---|---|---|
| 조회 권한(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다. 사이드바·헤더는 남는다 | `apps/admin/src/shared/errors/ErrorScreens.tsx:73` |
| 수정 권한(`update`) | 저장 버튼과 상태 문구가 **사라지고**(비활성이 아니다) 모든 입력이 잠긴다. info 배너가 이유를 말한다 | `../_shared/SettingsFormShell.tsx:167-168` |

**플랜(엔타이틀먼트) 축** — 시스템 설정 경로에는 모듈 매핑이 없어 잠금 판정이 `granted` 로 열린다 — `apps/admin/src/shared/entitlements/module-resources.ts:15-17`. **청구 상태(결제 지연·이용 정지)가 이 화면의 저장을 잠그는 배선은 코드에 없다** — 판정 함수(`usePlanWriteBlock`)는 있으나 호출부가 0건이다.
