# 알림 설정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 알림 설정 |
| 화면 ID | `SCR-SETTINGS-NOTIFICATIONS` |
| 메뉴 경로 | 시스템 설정 > 알림 설정 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 알림 종류마다 **받을지 말지**를 정한다. 그것 하나가 이 화면이 하는 일의 전부다.

**업무 배경** — 종류를 만들거나 문구를 고치는 자리가 아니다. 알림 종류를 자유 문자열로 두면 같은 사건이 '신규문의' · '새 문의' · 영문 키 세 이름으로 쌓이고, 그 순간 '문의 알림만 끄기' 같은 설정은 만들 수 없게 된다. **종류는 코드가 아는 사실이지 운영자가 짓는 이름이 아니다** — `shared/notifications/catalog.ts:4-7`.

**화면 설명** — 안내문 아래 카드 하나. 종류 여덟이 한 줄씩 쌓이고 각 줄에 이름·설명·이동 위치와 토글이 있다. 카드 아래에 상태 문구와 저장 버튼이 있다. **저장 확인 다이얼로그가 없다** — 되돌리기 쉬운 값이고 이 화면 밖을 바꾸지 않기 때문이다.

**주요 사용자** — 최상위 관리자 · 시스템 설정 수정 권한을 가진 운영자.

**사용 시나리오**

1. 주문 알림이 너무 잦다 → `주문 접수` 토글을 끄고 저장한다. 그 뒤로 그 알림은 벨에 쌓이지 않는다.
2. 설정을 못 불러온 상태에서 토글을 만지려 한다 → **토글이 잠겨 있고** 배너가 왜 지금 저장할 수 없는지 말한다.
3. 몇 개를 껐는지 잊었다 → 카드 위 경고가 `{n}개 종류를 받지 않도록 설정했어요` 로 알려 준다.

**선행 조건** — 로그인 · 이 화면의 조회 권한. 저장하려면 수정 권한이 필요하다. **설정 문서를 한 번은 읽어야 저장할 수 있다**(§4.3).

### 1.1 끈 알림은 만들어지지도 않는다

목록에서 **감추는 것이 아니라 애초에 넣지 않는다.** 감추기만 하면 안읽음 수는 계속 오르고, 벨에 점이 떠 있는데 열면 아무것도 없는 상태가 된다 — `NotificationSettingsPage.tsx:10-11` · `shared/notifications/useNotifications.ts:72-73`.

**권한이 언제나 먼저다** — 여기서 켜 두어도 그 화면의 조회 권한이 없으면 알림은 오지 않는다. 수신 설정은 '보고 싶은가'이고 권한은 '볼 수 있는가'다 — 순서가 바뀌면 알림이 권한 우회로가 된다. 종류마다 권한 리소스가 달려 있고, 리소스를 풀지 못하는 종류도 **버린다**(모르는 것은 보여 주지 않는다) — `useNotifications.ts:3-11,75-77`.

**후행 처리** — 저장된 설정이 헤더의 알림 벨이 만드는 목록을 거른다. 안읽음 수는 그 목록에서 세므로 함께 달라진다 — `useNotifications.ts:86-88`.

**관련 화면** — 알림이 데려가는 여덟 잎: `SCR-PRODUCTS-INQUIRIES` · `SCR-PROGRAMS-INQUIRIES` · `SCR-ORDERS` · `SCR-ORDERS-CLAIMS` · `SCR-USERS-MEMBERS` · `SCR-SUPPORT-TICKETS`. 알림 벨은 라우트를 가진 화면이 아니라 셸의 일부다. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `시스템 설정` 가지의 마지막 항목 — `nav-config.ts:305`. 조건부 노출이 아니다.

**관련 기능** — 권한 관리(`SCR-USERS-ROLES`)가 알림의 실제 노출을 정한다. **결제 설정이 문의 잎 둘을 메뉴에서 감추면** 그 잎의 권한 리소스가 살아 있는 한 알림은 계속 오지만, 이 화면의 목록은 카탈로그가 정하므로 항목 수는 변하지 않는다.

**관련 API** — 목적 둘: ① 수신 설정 문서 1건 조회 ② 그 문서 저장. **사람에 매인 설정이라 서버가 갖는다** — 계약은 `shared/notifications/preferences.ts:74` 의 `TODO(backend)` 다. 알림 목록 자체는 실시간 수신 경로(서버 이벤트)로 오는 것이 계약이고, 지금은 픽스처를 한 번 읽는다 — `shared/notifications/store.ts:10-14`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [품질 기준](../../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/settings/notifications/NotificationSettingsPage.tsx` (종류 카탈로그 `shared/notifications/catalog.ts` · 수신 설정 모델·저장소 `shared/notifications/preferences.ts` · 알림 목록 `shared/notifications/store.ts` · 필터 `shared/notifications/useNotifications.ts`)

### 1.2 이 화면은 다른 설정 화면과 구조가 다르다

| 축 | 사이트·OAuth·결제 설정 | 이 화면 |

| 저장 확인 다이얼로그 | 있다(사이트를 내리고 인증을 끊는 값이라) | **없다** |

| 대신 있는 것 | — | **읽기 전 저장 차단**(§4.3) |

### 3.1 알림 종류 여덟

고른 기준은 **운영자가 모르면 손해가 나는 것**만이다. '상품이 수정되었습니다'처럼 운영자가 직접 한 일은 알림이 아니다 — 자기가 방금 한 일을 다시 알려 주는 것은 소음이고, 소음이 쌓이면 정말 중요한 알림도 함께 무시된다 — `catalog.ts:37-43`.

| 종류 이름 | 설명 | 이동 위치 | 심각도 |
|---|---|---|---|
| `상품 문의 접수` | `새 상품 문의가 들어오면 알려요.` | `/products/inquiries` | 정보 |
| `상품 문의 응답 기한 임박` | `약속한 응답 기한(3일)을 넘길 상품 문의를 알려요.` | `/products/inquiries` | **경고** |
| `프로그램 문의 접수` | `새 프로그램 문의가 들어오면 알려요.` | `/programs/inquiries` | 정보 |
| `프로그램 문의 응답 기한 임박` | `약속한 응답 기한(2일)을 넘길 프로그램 문의를 알려요.` | `/programs/inquiries` | **경고** |
| `주문 접수` | `새 주문이 들어오면 알려요.` | `/orders` | 정보 |
| `취소·교환·반품 요청` | `클레임이 접수되면 알려요.` | `/orders/claims` | **경고** |
| `신규 회원가입` | `새 회원이 가입하면 알려요.` | `/users/members` | 정보 |
| `1:1 문의 접수` | `새 1:1 문의가 들어오면 알려요.` | `/support/tickets` | 정보 |

| 시스템 처리 | 문서가 도착하면 그 값이 초안의 기준선이 된다. 서버 값이 바뀌면(재조회) 초안을 그 위에서 다시 시작한다 — `NotificationSettingsPage.tsx:120-130` |
| 데이터 처리 | 없음(읽기만) |
| Validation | **저장값 방어**: 알 수 없는 키는 버리고, 빠진 키는 **켜짐**으로 채운다 — `preferences.ts:31-49` |
| API 호출 여부 | Y — 수신 설정 문서 1건 조회 |
| 성공 처리 | 여덟 줄이 저장된 값으로 채워진다. **토스트를 띄우지 않는다** |
| 실패 처리 | **목록을 감추지 않는다.** 위에 danger 배너를 얹고 `다시 시도` 를 준다. 배너 문구는 '한 번이라도 읽었는가'로 갈린다(§4.3) |
| 예외 처리 | 재조회 중에는 이전 값을 유지한다 — 스켈레톤은 첫 조회에서만 — `:143` |
| 화면 변경 사항 | 입력영역 · 오류영역 · 버튼영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**빠진 키를 켜짐으로 채우는 이유** — 새 알림 종류가 카탈로그에 추가됐을 때 옛 저장값에는 그 키가 없다. 꺼짐으로 채우면 새 알림이 **아무에게도 도착하지 않고, 그 사실은 아무도 모른다**(알림이 안 오는 것을 알아채는 방법이 없다). 반대로 켜짐이면 최악의 경우 소음이 늘고, 소음은 운영자가 끌 수 있다 — `preferences.ts:31-37`.

**재현 손잡이(백엔드 연동 전)** — `?fail=load` 또는 `?fail=notification-prefs:load` 로 조회 실패를 재현한다 — 스코프 이름은 `notification-prefs` 다 — `preferences.ts:75-78` · `shared/crud/dev.ts`.

### 4.3 읽기 전 저장 차단 — 이 화면의 핵심 규율

| 항목 | 내용 |
|---|---|
| 기능 목적 | **읽기 실패가 쓰기 사고가 되는 경로를 막는다** |
| 실행 조건 | 설정 문서를 **한 번도 읽지 못한** 상태 |
| 사용자 동작 | 없음(자동으로 잠긴다). `다시 시도` 로 벗어난다 |
| 시스템 처리 | 문서를 못 읽은 상태에서는 화면이 기본값(전부 받음)을 그린다. 그 상태로 토글 하나를 만지고 저장하면 문서가 「기본값 + 그 하나」로 치환되고 **운영자가 예전에 꺼 둔 종류가 조용히 켜진다** — `preferences.ts:54-66` |
| 데이터 처리 | 아무것도 저장하지 않는다 |
| Validation | 판정은 술어 하나가 갖고 **토글의 잠금 · 저장 버튼의 잠금 · 배너 문구 · 클릭 시 재확인이 전부 같은 술어를 읽는다** — `preferences.ts:70-72` · `NotificationSettingsPage.tsx:154,223,247,251` |
| API 호출 여부 | N — 저장 요청을 만들지 않는다 |
| 성공 처리 | 해당 없음(막는 것이 목적이다) |
| 실패 처리 | 배너가 이유를 말한다: `알림 설정을 불러오지 못했어요. 지금 보이는 값은 실제 설정이 아니라 기본값(전부 받음)이라, 이대로 저장하면 예전에 꺼 둔 종류가 다시 켜져요. 다시 시도해 설정을 불러온 뒤 저장해 주세요.` — `preferences.ts:51-52`. 상태 문구도 `설정을 불러오지 못해 저장할 수 없어요.` 로 바뀐다 — `:239` |
| 예외 처리 | **한 번이라도 읽은 뒤의 재조회 실패는 막지 않는다** — 화면의 값이 진짜이기 때문이다. 그때 배너 문구가 달라진다: `알림 설정을 새로 불러오지 못했어요. 아래는 마지막으로 읽은 설정이에요.` — `:170-171`. 버튼의 잠금과 같은 술어를 **클릭 시점에 한 번 더 읽어** Enter·재렌더 틈으로 들어온 클릭도 멈춘다 — `:249-252` |
| 화면 변경 사항 | 오류영역 · 입력영역(잠금) · 버튼영역 |
| 후속 동작 | `다시 시도` 로 읽으면 잠금이 풀린다 |
| 로그 기록 여부 | N |

**왜 부분 저장이 아니라 차단인가** — 부분 저장을 하려면 '이 키만 바꿔라'는 계약이 필요한데 지금 저장소는 문서를 **통째로 치환**한다. 없는 계약을 화면이 흉내 내면 서버가 붙는 날 그 흉내와 실제 의미가 갈린다. 그래서 목적('모르는 것을 켜짐으로 덮어쓰지 않는다')만 지키고 수단은 가장 정직한 것을 고른다 — 읽지 못한 동안에는 저장하지 않고 그 이유를 말한다 — `preferences.ts:62-66`.

### 4.4 저장

| 항목 | 내용 |
|---|---|
| 기능 목적 | 초안을 수신 설정 문서로 저장한다 |
| 실행 조건 | 변경 사항이 있고, 저장 중이 아니며, 첫 조회가 끝났고, 저장이 막히지 않았고, 수정 권한이 있을 때 |
| 사용자 동작 | 저장 클릭 |
| 시스템 처리 | **확인 다이얼로그를 거치지 않고 바로 보낸다.** 클릭 시점에 차단 술어를 한 번 더 읽고, 막혀 있으면 아무것도 하지 않는다 — `:248-252` |
| 데이터 처리 | 문서가 초안으로 통째로 치환된다. 성공하면 조회를 무효화해 서버 값을 다시 읽는다 — `:135` |
| Validation | 해당 없음(값이 전부 켬/끔이다) |
| API 호출 여부 | Y — 수신 설정 문서 저장 |
| 성공 처리 | 토스트 `MSG-SETTINGS-NOTIFICATIONS-01`: `알림 설정을 저장했어요.` — `:137` |
| 실패 처리 | 토스트 `MSG-SETTINGS-NOTIFICATIONS-02`: `알림 설정을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` — `:139`. **초안은 그대로 남아 다시 저장할 수 있다** |
| 예외 처리 | **동시 편집 충돌을 다루지 않는다** — 이 문서에는 개정 토큰이 없고 저장이 문서를 통째로 치환한다(마지막 저장이 이긴다). 다른 관리자가 그 사이 저장했다면 그 변경은 조용히 사라진다 |
| 화면 변경 사항 | 버튼영역 · 입력영역(잠금) · 토스트 |
| 후속 동작 | 헤더 알림 벨의 목록과 안읽음 수가 새 설정으로 걸러진다 |
| 로그 기록 여부 | 확인하지 못했다 |

**저장 실패를 토스트로 알리는 것이 이 섹션의 다른 화면과 다르다** — 사이트·OAuth·결제 설정은 저장 실패를 인라인 배너로 말하지만, 이 화면은 확인 다이얼로그가 없어 배너를 얹을 자리가 없다. 대신 실패 토스트는 **자동으로 사라지지 않는다** — [공통 문서](../../_common/index.md) §4.6.

**재현 손잡이** — `?fail=save` 또는 `?fail=notification-prefs:save`.

## 6. Validation

해당 없음 — **사용자 입력 검증이 없다.** 이 화면의 값은 종류마다 켬/끔 하나뿐이라 잘못된 값이 들어올 자리가 없다.

대신 두 가지 방어가 있다. 둘 다 사용자 입력이 아니라 **저장값과 조회 상태**에 대한 것이다.

| 대상 | 조건 | 처리 | 근거 |
|---|---|---|---|
| 저장값 | 알 수 없는 키는 버린다 | 카탈로그에 없는 키는 무시된다 | `preferences.ts:38-49` |
| 저장값 | 빠진 키는 **켜짐**으로 채운다 | 새 종류가 조용히 꺼지지 않게(§4.1) | `preferences.ts:31-37` |
| 조회 상태 | 문서를 한 번도 못 읽었으면 **저장을 막는다** | 토글·저장 버튼이 잠기고 배너가 이유를 말한다(§4.3) | `preferences.ts:54-72` |

| 데이터 충돌 | **이 화면에는 충돌 화면이 없다.** 이 문서에는 개정 토큰이 없고 저장이 문서를 통째로 치환한다 — 다른 관리자가 그 사이 저장했다면 그 변경은 **조용히 사라진다** | 마지막 저장이 이긴다 — `shared/crud/document.ts:22-35` | 해당 없음 |
| 중복 데이터 | 해당 없음 — 종류가 고정 여덟이고 문서가 자원당 1건이라 중복이 성립하지 않는다 | — | — |
| 파일 업로드 실패 | 해당 없음 — 파일 입력이 없다 | — | — |

**⚠ 지금 확인되는 격차 셋**

| 무엇 | 사실 | 근거 |
|---|---|---|
| 동시 편집 보호가 없다 | 같은 섹션의 다른 네 화면은 개정 토큰으로 충돌을 막는데 이 화면만 마지막 저장이 이긴다. 알림 설정이 **사람에 매인 설정**이라면 문제가 작지만, 지금 저장소는 계정 단위 문서 하나다 | `preferences.ts:74-78` · `shared/crud/document.ts:22-35` |
| 미저장 이탈 가드가 없다 | 토글을 여럿 바꾼 뒤 다른 메뉴를 누르면 **초안이 조용히 사라진다.** 같은 섹션의 다른 화면은 전부 가드가 있다 | `NotificationSettingsPage.tsx` 에 이탈 가드 배선이 없다 |
| 알림 목록이 실시간이 아니다 | 픽스처를 **한 번** 읽는다. 폴링으로 실시간인 척하지 않는 것은 의도이고, 실시간 수신 경로는 서버의 몫이다 | `shared/notifications/store.ts:10-14` |

## 8. 화면 상태(State)

| 상태 | 화면 표시 | 버튼 활성화 여부 | Loading 여부 | 사용자 안내 메시지 |
|---|---|---|---|---|
| 최초 로딩 | 안내문과 카드 제목은 즉시 그려지고 목록 자리가 스켈레톤 | 저장 × · 토글 × | Y | 접근 이름 `알림 설정을 불러오는 중` |
| 데이터 있음 | 여덟 줄이 저장된 값으로 채워진다 | 저장 ×(변경 없음) · 토글 ○ | N | `변경 사항이 없어요.` |
| 데이터 없음 | **성립하지 않는다** — 종류 여덟은 카탈로그가 고정하고 저장값이 없으면 전부 켜짐으로 채워진다 | — | N | — |
| **값을 모름(한 번도 못 읽음)** | 목록은 기본값(전부 받음)으로 그려지지만 **토글과 저장이 잠긴다.** 위에 danger 배너 | 저장 × · 토글 × · 다시 시도 ○ | N | `알림 설정을 불러오지 못했어요. 지금 보이는 값은 실제 설정이 아니라 기본값(전부 받음)이라, 이대로 저장하면 예전에 꺼 둔 종류가 다시 켜져요. …` |
| 조회 실패(재조회 · 값이 있음) | 목록은 마지막으로 읽은 값 그대로. 위에 danger 배너 | 저장 ○ · 토글 ○ · 다시 시도 ○ | N | `알림 설정을 새로 불러오지 못했어요. 아래는 마지막으로 읽은 설정이에요.` |
| 변경 있음 | 저장 버튼이 살아난다 | 저장 ○ | N | `저장하지 않은 변경 사항이 있어요.` |
| 저장 중 | 토글이 잠기고 버튼 라벨이 `저장 중…` | 저장 × · 토글 × | Y | `저장하는 중이에요…` |
| 저장 실패 | **인라인 배너가 아니라 토스트**로 알린다. 초안은 그대로 | 저장 ○ | N | `알림 설정을 저장하지 못했어요. 잠시 후 다시 시도해 주세요.` |
| 끈 종류가 있음 | 카드 위 warning 경고 | 그대로 | N | `{n}개 종류를 받지 않도록 설정했어요. 그 사건들은 목록을 직접 새로고침해야 알 수 있어요.` |
| 읽기 전용(권한) | 저장 버튼·상태 문구가 사라지고 토글이 잠긴다 | 저장 컨트롤 없음 · 토글 × | N | `조회 권한만 있어 알림 설정을 변경할 수 없어요.` |

**`null`(모름)과 꺼짐을 가르는 자리** — 이 화면에서 그 구분이 저장 차단의 근거다.

| 값 | 화면이 아는 것 | 그리는 것 | 무엇을 막는가 |
|---|---|---|---|
| 문서를 한 번도 못 읽음 | **모른다** | 기본값(전부 받음)을 그리되 **토글과 저장을 잠근다** | 모르는 값을 켜짐으로 덮어쓰는 저장 |
| 저장값에 그 키가 없음 | 새 종류라 옛 저장값에 없다 | **켜짐**으로 읽는다 | 새 알림이 아무에게도 도착하지 않는 상태 |
| 저장값의 그 키가 꺼짐 | **사실**이다 | 꺼짐 | — |
| 알 수 없는 키가 저장값에 있음 | 카탈로그에 없는 이름이다 | 버린다 | 화면에 없는 항목이 문서에 남는 것 |

등록·삭제·다운로드·업로드·승인·반려는 이 화면에 존재하지 않는 동작이다. 알림 종류는 카탈로그가 고정하므로 만들거나 지우는 개념이 없다.

**권한이 없을 때 화면이 어떻게 되는가**

| 없는 것 | 결과 | 근거 |
|---|---|---|
| 조회 권한(`read`) | 본문이 `접근 권한이 없어요` 화면으로 대체된다 | `apps/admin/src/shared/errors/ErrorScreens.tsx:73` |
| 수정 권한(`update`) | 저장 버튼과 상태 문구가 **사라지고**(비활성이 아니다) 토글이 잠긴다. info 배너가 이유를 말한다 | `NotificationSettingsPage.tsx:232-233,223` |

**이 화면의 설정이 권한을 이기지 않는다** — 여기서 켜 두어도 그 화면의 조회 권한이 없으면 알림은 오지 않는다. 알림 목록을 만들 때가 아니라 **보여 줄 때** 권한으로 거른다 — `shared/notifications/catalog.ts:118-126` · `useNotifications.ts:75-77`.

**플랜(엔타이틀먼트) 축** — 시스템 설정 경로에는 모듈 매핑이 없어 이 화면은 늘 열린다 — `shared/entitlements/module-resources.ts:15-17`. 다만 알림이 데려가는 잎 중 일부는 모듈에 매여 있어, 그 모듈이 없으면 그 잎의 권한도 성립하지 않는다.
