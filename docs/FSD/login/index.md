# 로그인

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 로그인 |
| 화면 ID | `SCR-LOGIN` |
| 메뉴 경로 | 해당 없음 — 사이드바(셸) **밖**의 단독 화면이다 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 어드민 계정으로 인증하고, 인증에 성공하면 원래 가려던 화면(없으면 대시보드)으로 보낸다.

**업무 배경** — 이 어드민은 회원 개인정보·주문·결제 설정·시스템 시크릿을 다룬다. 아무나 들어올 수 없어야 하고, 반대로 자기 자리를 잃은 운영자가 무엇을 해야 할지 모르는 상태로 빈 화면 앞에 남겨져서도 안 된다. 이 화면은 그 두 요구가 만나는 유일한 자리다.

**화면 설명** — 좌우 5:5 화면이다. 왼쪽은 브랜드 비주얼(장식이라 보조기술에서 제외된다), 오른쪽은 이메일·비밀번호·체크박스 둘·제출 버튼으로 된 폼이다. 실패는 폼 안 배너로 말하고, 계정이 잠기면 그 이메일로는 제출 버튼이 비활성된다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원 — 이 어드민에 계정을 가진 전원.

**사용 시나리오**

1. 운영자가 어드민 주소를 열면 세션이 없어 이 화면으로 온다 → 이메일·비밀번호를 넣고 로그인한다 → 대시보드가 열린다.
2. 회원 목록을 필터로 좁혀 보던 중 세션이 만료된다 → 이 화면으로 밀려나며 `세션이 만료되었어요. 다시 로그인해 주세요.` 가 보인다 → 다시 로그인하면 **필터가 걸린 그 목록**으로 되돌아온다.
3. 비밀번호를 연달아 틀린다 → 실패 횟수가 문구에 함께 뜨고, 5회째에 계정이 잠긴다.

**선행 조건** — 없다. 이 화면만이 로그인 없이 열린다.

**후행 처리** — 인증에 성공하면 세션이 저장되고(로그인 유지 여부에 따라 저장 위치가 갈린다) 원래 목적지로 이동한다. '이메일 저장'을 켜 두면 다음 방문에 이메일 칸이 채워진다.

**관련 화면** — `SCR-DASHBOARD`(기본 이동처) · 모든 셸 안 화면(세션이 없으면 여기로 밀려난다) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 해당 없음 — 사이드바에 없다.

**관련 기능** — 시스템 설정 > 사이트 설정의 **'로그인 상태 유지'** 값이 이 화면의 '로그인 유지' 체크박스 **기본값**을 정한다 — `apps/admin/src/pages/login/LoginPage.tsx:200-203`. 로그인 이력(`SCR-USERS-LOGIN-HISTORY`)과 관리자 로그(`SCR-LOGS-ADMIN`)가 로그인·로그아웃을 다루지만, 이 화면이 그 기록을 남기는 배선은 코드에서 확인하지 못했다.

**관련 API** — 인증 요청 1건(이메일 + 비밀번호 → 세션 또는 실패 사유). **백엔드가 없다** — `apps/admin/src/pages/login/api.ts` 가 로컬 mock 이며, 실제 API 가 붙으면 이 모듈만 교체한다(호출부 시그니처 유지). 실패 카운트·잠금 판정도 지금은 이 mock 이 브라우저 안에서 흉내 내며 새로고침하면 초기화된다 — `apps/admin/src/pages/login/api.ts:100-101`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [품질 기준](../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/login/LoginPage.tsx` (폼 조립 `components/LoginForm.tsx` · 검증 `validation.ts` · 인증 `api.ts` · 세션 저장 `session.ts` · 레이아웃 `login.css`)

---

**⚠ 지금 화면에 보이는 사실** — 폼 아래에 mock 계정 안내가 그대로 노출되어 있다: `mock 계정 — admin@tds.local / password123 (백엔드 연동 전 로컬 mock: ./api.ts)` — `LoginPage.tsx:426`. 백엔드가 붙기 전까지의 개발용 표시다.

## 4. 기능 명세

### 4.1 이미 로그인한 사용자의 진입

| 항목 | 내용 |
|---|---|
| 기능 목적 | 이미 세션이 있는 사람에게 로그인 폼을 보여 주지 않는다 |
| 실행 조건 | `/login` 진입 시점에 유효한 세션이 저장되어 있을 때 |
| 사용자 동작 | 주소 입력 · 북마크 열기 |
| 시스템 처리 | 폼을 그리지 않고 즉시 대시보드로 보낸다(히스토리를 남기지 않는 이동) — `LoginPage.tsx:372-374` |
| 데이터 처리 | 없음 |
| Validation | 해당 없음 |
| API 호출 여부 | N — 저장된 세션만 읽는다 |
| 성공 처리 | 대시보드가 열린다. 폼은 한 프레임도 보이지 않는다 |
| 실패 처리 | 저장된 값의 형태가 깨져 있으면 세션이 없는 것으로 읽고 폼을 그린다 — `session.ts:52-62` |
| 예외 처리 | 브라우저가 저장소 접근을 막으면(사생활 보호 모드 등) 세션이 없는 것으로 읽는다 — `session.ts:15-22` |
| 화면 변경 사항 | 화면 전체 |
| 후속 동작 | 대시보드 진입 |
| 로그 기록 여부 | N |

**세션을 두 서랍에서 찾는다** — 탭 수명 저장소를 먼저 보고, 없으면 영속 저장소를 본다. '로그인 유지'를 끄고 방금 한 로그인이 예전 영속 세션을 이긴다 — `session.ts:64-72`.

### 4.2 인증 요청(로그인)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 입력한 자격 증명으로 세션을 얻는다 |
| 실행 조건 | 이메일·비밀번호가 검증을 통과하고, 그 이메일이 잠긴 상태가 아닐 때 |
| 사용자 동작 | '로그인' 클릭 또는 폼에서 Enter |
| 시스템 처리 | 이메일을 앞뒤 공백 제거 + 소문자로 정규화해 보낸다 — `apps/admin/src/pages/login/api.ts:130-132`. 요청은 **10초**를 넘기면 중단한다 — `apps/admin/src/pages/login/api.ts:22` · `LoginPage.tsx:321-322` |
| 데이터 처리 | 성공 시 세션을 저장한다. '로그인 유지'가 켜져 있으면 영속 저장소, 꺼져 있으면 탭 저장소이며 **반대편 서랍은 반드시 비운다** — `session.ts:84-100`. '이메일 저장'이 켜져 있으면 이메일만 보관하고, 꺼져 있으면 저장된 값을 지운다 — `LoginPage.tsx:334-336` |
| Validation | §6 전부 |
| API 호출 여부 | Y — 인증 1건. 지금은 로컬 mock 이 응답한다 |
| 성공 처리 | **토스트를 띄우지 않는다** — 이 화면에는 결과 통지 토스트가 하나도 없다(화면이 통째로 바뀌는 것이 곧 결과다). 원래 가려던 경로가 있으면 그곳으로, 없으면 대시보드로 이동한다(히스토리를 남기지 않는다) — `LoginPage.tsx:338-340` |
| 실패 처리 | 실패 종류에 따라 배너 문구가 달라진다(§7). 자격 증명 불일치일 때만 **비밀번호 칸이 비워지고** 이메일은 남는다 — `LoginPage.tsx:281-287`. 그 밖의 실패는 입력을 전부 유지한다 |
| 예외 처리 | 10초 타임아웃과 네트워크 오류는 같은 문구로 말한다 — `LoginPage.tsx:344-346`. 응답·타임아웃·예외 어느 경로로 끝나도 잠금과 로딩 표시가 함께 풀린다 — `LoginPage.tsx:347-352` |
| 화면 변경 사항 | 입력영역 · 안내영역 · 버튼영역 |
| 후속 동작 | 화면 이동 또는 재시도 |
| 로그 기록 여부 | 확인하지 못했다 — 로그인 성공·실패를 관리자 로그로 남기는 배선이 이 화면에 없다 |

**중복 제출을 두 겹으로 막는다** — 렌더를 기다리지 않는 동기 잠금과 버튼 비활성 둘 다 쓴다. 그래서 빠른 두 번째 Enter 도 두 번째 요청을 만들지 않는다 — `LoginPage.tsx:246-248,310`.

**'전송 중'과 '검증 중'을 가른다** — 잠기는 것은 전송 중뿐이다. 검증 중에 입력을 잠그면 위반 필드로 포커스를 옮길 수 없다(비활성 요소는 포커스를 받지 못한다) — `LoginPage.tsx:219-231`.

### 4.4 세션 만료로 밀려온 진입

| 항목 | 내용 |
|---|---|
| 기능 목적 | 세션이 끊겨 되돌려 보내진 사람에게 이유를 말하고, 재인증 뒤 원래 자리로 되돌린다 |

| API 호출 여부 | N — 진입 자체는 서버를 부르지 않는다 |
| 성공 처리 | 배너 `세션이 만료되었어요. 다시 로그인해 주세요.` 가 정보 톤으로 뜬다 — `LoginPage.tsx:58`. 재인증에 성공하면 **쿼리까지 보존된 원래 경로**로 돌아간다 — `RequireAuth.tsx:19-22` |

**⚠ 코드에서 확인한 미해결 지점** — 만료 통지에는 중복 방지 래치가 있고(여러 쿼리가 동시에 401 을 받아도 한 번만 이동한다), 그 래치를 푸는 함수가 있다. 그러나 **재인증 성공 시 그것을 부르는 호출부가 앱 코드에 없다**(`apps/admin/src/shared/auth/session-expiry.ts:44-50` 의 `resetSessionExpiry` — 호출부 검색 결과 0건). 페이지를 새로 고치지 않은 채 두 번째 만료가 오면 재인증 경로로 보내지지 않을 수 있다.

| 읽기 전용(권한) | 성립하지 않는다 — 이 화면은 인증 **이전**이라 권한 축이 없다 | — | N | — |
| 검증 실패 | 위반한 칸 아래 인라인 오류 + 첫 위반 칸에 포커스 | 제출 ○ | N | §6 의 해당 문구 |
| 인증 실패 | 체크박스 위 배너. danger 면 배너로 포커스가 이동한다 | 제출 ○ | N | §7 의 해당 문구 |
| 계정 잠금 | 배너 + 제출 버튼 비활성 | 제출 × | N | 잠금 문구 |

**`null` 과 빈 값의 구분** — 저장된 이메일은 없으면 `null` 이고, 화면은 그것을 빈 문자열로 옮기면서 **체크박스의 기본값 판정에 그대로 쓴다**(`null` 이면 체크 해제) — `LoginPage.tsx:206-207` · `session.ts:120-130`. 즉 '저장된 적 없음'과 '빈 이메일이 저장됨'을 같은 것으로 읽지 않는다.

## 9. 권한

| 권한 | 조회 | 등록 | 수정 | 삭제 | 다운로드 | 업로드 | 승인 | 반려 |
|---|---|---|---|---|---|---|---|---|
| 인증 전(전원) | ○ | — | — | — | — | — | — | — |
| 인증 후(전원) | 화면이 열리지 않는다 — 즉시 대시보드로 이동 | — | — | — | — | — | — | — |

**이 화면에는 권한 축이 없다.** 인증 가드와 권한 가드는 모두 셸 안쪽에 붙어 있고 `/login` 은 그 바깥이다 — `apps/admin/src/App.tsx:552-563`. 플랜(엔타이틀먼트) 축도 마찬가지로 셸 안쪽이라 여기에 걸리지 않는다.

**권한이 없을 때 화면이 어떻게 되는가** — 성립하지 않는다. 다만 **계정 상태**는 화면을 바꾼다: 비활성 계정은 배너를 받고, 잠긴 계정은 제출 버튼이 비활성된다. 둘 다 화면 자체는 그대로 보인다.

**인증 후 무엇을 볼 수 있는가는 이 화면이 정하지 않는다** — 역할(권한 매트릭스)이 정하고, 그것은 `SCR-USERS-ROLES` 가 다룬다. mock 인증은 계정마다 역할을 함께 돌려주지만(`apps/admin/src/pages/login/api.ts:24,67-81`), 그 값이 앱의 권한 판정에 꽂히는 배선은 코드에서 확인하지 못했다 — 권한은 별도 저장소가 갖는다.

**이 화면에는 팝업도 확인 다이얼로그도 없다.** 인벤토리의 팝업 19건·확인 다이얼로그 56건 어디에도 로그인 화면이 없다.

**이 화면에는 결과 통지 토스트가 없다.** 인벤토리 §9.3 의 화면별 토스트 표에 로그인 화면이 올라 있지 않다 — 성공은 화면 이동으로, 실패는 폼 안 배너로 말한다.
