# 상품 문의 처리 (상세)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 상품 문의 처리 |
| 화면 ID | `SCR-PRODUCTS-INQUIRIES-DETAIL` |
| 메뉴 경로 | 상품 관리 > 문의 > (행 클릭) — 부모 메뉴는 **조건부 노출**이다 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 문의 한 건의 내용과 문의자 정보를 보고, **답변을 쓰고 상태를 옮기고**, 필요하면 견적을 발행한다.

**업무 배경** — **상태는 버튼이 아니라 규칙이 정한다.** '답변 완료' 를 직접 고르는 셀렉트를 두지 않는다 — 답변을 저장하면 상태가 따라 넘어가고, 종결은 답변이 나간 뒤에만 열린다. 화면이 상태를 자유롭게 고르게 두면 '답변 없이 답변 완료' 가 만들어지고, 그 순간 목록의 미답변 집계와 경과 문구가 전부 거짓말이 된다 — `apps/admin/src/pages/products/inquiries/ProductInquiryDetailPage.tsx:6-9`.

**화면 설명** — 2단이다: 왼쪽 본문 카드(제목·상태 배지·채널·경과·요청 내역·답변·버튼 넷), 오른쪽에 `문의자 정보` 와 `처리 이력` 카드.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. 접수된 문의를 연다 → `답변 착수` 로 담당자가 잡았음을 표시한다 → 답변을 쓰고 `답변 저장` → 상태가 '답변 완료' 로 넘어간다.
2. 답변을 보낸 뒤 `문의 종결` → 그 뒤로는 답변을 고칠 수 없다.
3. 고객이 견적을 원한다 → `견적 발행` → 견적이 만들어지고 상태가 '견적 발행' 이 되며 `견적 보기` 링크가 생긴다.

**선행 조건** — 로그인 · `/products/inquiries` 리소스 조회 권한. 답변·상태 전환·견적 발행은 `update` 권한.

**후행 처리** — 답변 저장은 목록의 상태·경과 배지를 바꾸고, 대시보드·통계의 미답변 집계에도 반영된다.

**관련 화면** — `SCR-PRODUCTS-INQUIRIES`(목록) · `SCR-SALES-QUOTES-DETAIL`(발행된 견적) · `SCR-SETTINGS-PAYMENT`(부모 메뉴 노출의 원인). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다. **부모 잎이 조건부로 사라져도 이 라우트는 살아 있다**([문의 목록](index.md)의 조건부 메뉴 규칙).

**관련 기능** — 견적 발행기는 공통 층의 이음매가 들고 있다 — 상품 화면이 영업 화면을 직접 부르면 페이지 간 결합이 된다 — `ProductInquiryDetailPage.tsx:213-221`.

**관련 API** — 상세 조회 1건 · 문의 갱신 1건(답변·상태 전환·견적 발행이 한 통로). **백엔드가 없다** — `apps/admin/src/pages/products/inquiries/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [문의 비기능 명세서](../../../NFRS/products/inquiries.md)

**구현 파일** — `apps/admin/src/pages/products/inquiries/ProductInquiryDetailPage.tsx` (상태 규칙 `_shared/store.ts` · 검증 `validation.ts` · 표시 헬퍼 `types.ts` · 견적 이음매 `apps/admin/src/shared/domain/quote-issue.ts`)

### 4.2 상태 규칙 — 버튼의 존재 조건이 곧 전이 가드다

**상태 흐름** — `접수 → 답변 중 → 견적 발행 → 답변 완료 → 종결` — `types.ts:44-51`.

**세 조작의 가능 조건** — `_shared/store.ts:95-119`.

| 조작 | 가능한 상태 | 규칙 |
|---|---|---|
| `답변 착수` | **`접수` 일 때만** | 접수 직후 한 번만. 고객에게는 아직 아무것도 나가지 않는다 |
| `답변 저장` | **종결이 아닐 때** | 종결된 문의는 기록이라 손대지 않는다 |
| `문의 종결` | **`답변 완료` 일 때만** | 답변이 나간 뒤에만 닫는다 |
| `견적 발행` | **종결이 아닐 때** + 공통 가드 통과 | 답변을 보낸 뒤에 '그럼 견적 주세요' 가 오는 것은 흔한 일이라 그 문을 닫지 않는다 |

**누를 수 없는 것을 보여 주지 않는다** — 위 조건에 맞지 않으면 **버튼이 아예 렌더되지 않는다**(비활성이 아니다) — `ProductInquiryDetailPage.tsx:364-391`.

**저장소도 같은 술어로 막는다** — 화면과 저장소가 같은 함수를 부르므로 '눌리는데 거부당하는 버튼' 이 생기지 않는다. 어긋나면 도메인이 던진다.

**빈 답변으로 '답변 완료' 가 되지 않는다** — 공백만 남은 입력은 검증이 막고, 저장소도 같은 규칙으로 한 번 더 막는다 — `validation.ts:4-6`.

### 4.5 견적 발행

| 항목 | 내용 |
|---|---|
| 기능 목적 | 이 문의를 영업 관리의 견적 한 장으로 만든다 |
| 실행 조건 | `update` 권한 + 공통 발행 가드 통과(종결이 아니고 이미 발행되지 않았고 배선이 있을 때) |
| 사용자 동작 | `견적 발행` |
| 시스템 처리 | 공통 층의 발행기를 부르고, 나온 견적 id 를 문의에 얹는다. **견적 id 와 상태가 함께 옮겨 간다** — 둘이 갈라지면 '상태는 견적 발행인데 견적이 없는' 문의가 생긴다 — `_shared/store.ts:152-165` |
| 데이터 처리 | 견적 1건(영업 모듈) + 문의의 견적 id · 상태 `견적 발행` |
| Validation | 공통 발행 가드 |
| API 호출 여부 | Y — 문의 갱신 1건 |
| 성공 처리 | 토스트 `MSG-PRODUCTS-INQUIRIES-DETAIL-01`: `견적 {견적번호}{조사} 발행했어요.` — `ProductInquiryDetailPage.tsx:230` · 요청 내역에 `견적 보기` 링크가 생긴다 |
| 실패 처리 | 카드 안 danger 배너 |
| 예외 처리 | **두 번 눌러도 견적은 하나다** — 견적 id 가 멱등키라 이미 발행된 문의는 그대로 돌려준다. 버튼도 발행된 문의에서는 사라진다 |
| 화면 변경 사항 | 본문 카드(상태·발행 견적 줄·버튼 목록) |
| 후속 동작 | 목록의 담기 열이 `견적 보기` 로 바뀐다 |
| 로그 기록 여부 | 확인하지 못했다 |

**목록의 발행과 다른 점** — 목록은 여러 문의를 **한 견적**으로 합치고 발행 후 견적 화면으로 **이동한다**. 상세는 이 문의 하나만 발행하고 **화면에 머문다**.

| 대상 항목 | 검증 조건 | 검증 시점 | 오류 메시지 | 처리 방식 |

| 상태 전이 | §4.2 의 세 조건 | 버튼 렌더 · 저장 | 버튼이 없어 사유를 띄울 자리가 없다. 저장소가 던지면 배너로 온다 | 버튼을 그리지 않는다 |

**규칙의 정본은 zod 스키마다** — `apps/admin/src/pages/products/inquiries/validation.ts`. 필드가 하나뿐이라 폼 라이브러리를 얹지 않고 화면이 오류를 직접 들되, **판정은 스키마가 한다** — 화면이 자기 조건문으로 다시 판단하면 규칙이 둘로 갈라진다.

**저장소도 같은 규칙으로 한 번 더 막는다** — 빈 답변이 '답변 완료' 로 상태를 넘기면 고객은 아무것도 받지 못한 채 처리된 것으로 집계된다.
