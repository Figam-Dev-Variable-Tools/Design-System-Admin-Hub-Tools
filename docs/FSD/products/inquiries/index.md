# 문의 (상품 · 조건부 노출)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 문의 |
| 화면 ID | `SCR-PRODUCTS-INQUIRIES` |
| 메뉴 경로 | 상품 관리 > 문의 — **조건부 노출**(§1 관련 메뉴 · §10) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 결제를 쓰지 않는 동안 '문의하기' 버튼으로 들어온 고객 글을 **얼마나 오래 답을 못 받았는가** 기준으로 보고, 여러 건을 묶어 견적 한 장으로 만든다.

**업무 배경** — 결제대행(PG)을 끈 상품의 스토어프론트 버튼은 '구매하기' 대신 '문의하기' 다. 그 버튼을 누른 고객의 글이 도착하는 곳이 여기다. 그래서 이 목록의 중심 열은 상품도 제목도 아니고 **경과**다 — 운영자가 여기서 하는 판단은 '무엇부터 답할까' 하나뿐이다 — `apps/admin/src/pages/products/inquiries/ProductInquiryListPage.tsx:3-6`.

**화면 설명** — 좌측은 처리 상태 필터(전체 + 5) + 안내 3~4줄. 우측은 검색 · (담은 것이 있으면) 견적 바구니 막대 · 읽기 전용 표다. 표에는 선택 체크박스가 없고, 대신 **'담기' 열**이 있다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. 아침에 연다 → 좌측 안내가 `답변을 기다리는 문의가 6건 있어요.` 를 말한다 → '접수' 를 눌러 붉은 배지(`3일째 미답변`)부터 연다.
2. 한 고객이 세 상품을 따로 문의했다 → 세 줄을 `담기` 로 바구니에 넣고 `견적 발행` → 세 문의가 **같은 견적**을 가리키고 그 견적 화면으로 이동한다.
3. 답변 권한이 없는 상담원이 연다 → 좌측에 `답변 권한이 없어 조회만 가능해요.` 가 뜨고 담기 열이 사라진다.

**선행 조건** — 로그인 · `/products/inquiries` 리소스 조회 권한. 견적 발행은 `update` 권한.

**후행 처리** — 견적 발행은 문의의 상태를 `견적 발행` 으로 옮기고 견적 id 를 붙인다. 그 뒤 영업 관리의 견적 화면으로 이동한다.

**관련 화면** — `SCR-PRODUCTS-INQUIRIES-DETAIL`(행 클릭) · `SCR-SALES-QUOTES-DETAIL`(발행된 견적) · `SCR-PRODUCTS`(문의 건수 배지가 이 화면을 가리킨다) · `SCR-SETTINGS-PAYMENT`(메뉴 노출의 원인) · `SCR-DASHBOARD`(문의 현황 카드). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — ⚠ **조건부다.** 사이드바 `상품 관리 > 문의` 는 아래 세 상태 중 하나로 나타난다 — `apps/admin/src/shared/layout/nav-config.ts:326-336`.

| 상태 | 언제 | 사이드바 |
|---|---|---|
| `open` | **결제로 판매할 수 없을 때**(PG 미사용·자격증명 미완·거절) | 평소 그대로 `문의` |
| `archive` | 결제로 판매할 수 있는데 **남은 문의가 있거나 잔여를 모를 때** | `문의 · 읽기 전용` (라벨에 꼬리표가 붙는다) |
| `hidden` | 결제로 판매할 수 있고 **잔여가 0건일 때** | **사라진다** |

**메뉴에서 사라져도 라우트는 살아 있다** — 메뉴 해소 함수는 라우트 생성 경로를 지나지 않는다. 감추는 것과 없애는 것은 다른 결정이다 — `nav-config.ts:320-325`.

**관련 기능** — 상품 목록의 `문의 {N}건` 배지가 이 화면을 가리킨다. 대시보드의 문의 현황과 통계의 문의 지표도 같은 조회기를 읽는다.

**관련 API** — 문의 목록 조회 1건 · 문의 갱신 N건(견적 발행 시 담은 건수만큼). **백엔드가 없다** — `apps/admin/src/pages/products/inquiries/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [문의 비기능 명세서](../../../NFRS/products/inquiries.md)

**구현 파일** — `apps/admin/src/pages/products/inquiries/ProductInquiryListPage.tsx` (도메인·상태 규칙 `_shared/store.ts` · 표시 헬퍼 `types.ts` · 메뉴 규칙 `apps/admin/src/shared/commerce/inquiry-backlog.ts` · 견적 이음매 `apps/admin/src/shared/domain/quote-issue.ts`)

| 미답변 · 1~2일 | `{N}일째 미답변` | 주의 |
| 미답변 · **3일 이상** | `{N}일째 미답변` | **위험** — 사흘은 국내 커머스 응대 관례의 마지노선이다 |
| 답변됨 · 당일 | `당일 답변` | 중립 |
| 답변됨 · N일 | `{N}일 만에 답변` | 중립 |
| 날짜를 읽을 수 없음 | `—` | 중립 |

### 4.3 견적 바구니 · 견적 발행

| 항목 | 내용 |
|---|---|
| 기능 목적 | '이거랑 이거랑 저거 합쳐서 얼마인가요' 를 **견적 한 장**으로 만든다 |
| 실행 조건 | `update` 권한 · 담은 것이 1건 이상 · 발행 가드 통과 |
| 사용자 동작 | 행의 `담기` → 막대의 `견적 발행` |
| 시스템 처리 | 담은 문의들을 **한 견적**으로 만들고, 그 견적 id 를 **담은 문의 전부**에 붙인다 — 하나라도 빠지면 그 문의는 견적 없이 남는다 — `ProductInquiryListPage.tsx:208-221` |
| 데이터 처리 | 견적 1건 생성(영업 모듈) + 문의 N건 갱신(견적 id · 상태 `견적 발행`) |
| Validation | 발행 가드(공통 층의 술어) — 버튼의 비활성 조건과 발행의 거절 조건이 **같은 것**을 읽는다 |
| API 호출 여부 | Y — 문의 갱신 N건 |
| 성공 처리 | 바구니가 비워지고 목록이 갱신된다. 토스트 `MSG-PRODUCTS-INQUIRIES-01`: `견적 {견적번호}{조사} 발행했어요.` — `ProductInquiryListPage.tsx:225` · 그 뒤 **견적 화면으로 이동한다** |
| 실패 처리 | 실패 토스트 `MSG-PRODUCTS-INQUIRIES-02`: `견적을 발행하지 못했어요. 잠시 후 다시 시도해 주세요.` — `ProductInquiryListPage.tsx:230` |
| 예외 처리 | 이미 견적이 발행된 문의는 담기 버튼 대신 **`견적 보기` 링크**가 그 자리에 온다. 발행 이음매가 배선되지 않았으면 가드가 먼저 막으므로 여기 도달하지 않는다 |
| 화면 변경 사항 | 바구니 막대 · 목록영역 · 화면 이동 |
| 후속 동작 | 영업 관리의 견적 상세로 이동 |
| 로그 기록 여부 | 확인하지 못했다 |

**왜 바구니가 필요한가** — 상품 문의는 상품을 **하나**만 가리킨다. 그런데 실제 요청은 여러 상품을 합친 것이라, 문의마다 견적을 한 장씩 내면 합계가 없는 견적 세 장이 되고 합치는 일이 앱 밖에서 손으로 벌어진다 — `ProductInquiryListPage.tsx:182-188`.

### 4.5 조건부 메뉴 노출 — 이 화면 고유의 규칙

| 항목 | 내용 |
|---|---|
| 기능 목적 | 결제를 켜면 새 문의가 생기지 않으므로 메뉴에서 내리되, **과거 문의로 가는 길은 끊지 않는다** |
| 실행 조건 | 사이드바를 그릴 때마다 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 결제 설정과 **잔여 문의 건수**를 함께 읽어 세 상태 중 하나를 낸다 — `apps/admin/src/shared/commerce/inquiry-backlog.ts:114-123` |
| 데이터 처리 | 없음(파생값) |
| Validation | 해당 없음 |
| API 호출 여부 | N — 조회기가 답한다 |
| 성공 처리 | §1 '관련 메뉴' 의 세 상태 |
| 실패 처리 | 조회기가 배선되지 않으면 `null`(모른다)이고, 그때는 **남긴다**(fail-open) |
| 예외 처리 | **메뉴에서 사라져도 라우트는 살아 있다** — URL 로 들어온 과거 문의 화면은 그대로 열린다 |
| 화면 변경 사항 | 사이드바(이 화면 자신은 바뀌지 않는다) |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**왜 지우지 않고 '읽기 전용' 을 두는가** — 그대로 지우면 **어제까지 받은 문의가 접근 불가**가 된다: 답변 중이던 건, 고객이 전화로 물을 문의번호, 감사 대상 기록이 전부 URL 을 아는 사람만 볼 수 있는 자리로 밀린다. 그래서 사라지는 조건에 '남은 것이 없을 때' 를 더한다 — 잔여가 0이 되는 날 메뉴는 스스로 사라진다 — `inquiry-backlog.ts:100-110`.

**잔여는 종결 포함 전체 건수다** — 미종결만 세면 안 된다. 답변을 끝낸 문의도 고객이 전화로 문의번호를 대며 다시 물을 기록이고 감사 대상이다. 그걸 0으로 읽고 메뉴를 지우면 과거 문의가 통째로 접근 불가가 된다 — `inquiry-backlog.ts:19-25`.

**배선을 잊으면 지우는 쪽으로 실패하지 않는다** — 조회기가 없을 때 0 을 돌려주면 메뉴가 사라진다. 그 결과는 과거 문의로 가는 길이 통째로 끊기는 것이고 원인은 데이터가 없어서가 아니라 배선을 잊어서다. 그래서 모를 때는 `null` 이고 메뉴 규칙은 그것을 '남긴다' 로 읽는다 — `inquiry-backlog.ts:10-13`.

**라벨의 꼬리표** — `archive` 상태의 사이드바 라벨에는 ` · 읽기 전용` 이 붙는다. 사이드바의 하위 항목은 배지 슬롯이 없어 **라벨 자체가** 상태를 말한다 — `inquiry-backlog.ts:125-135`.

**등록·삭제 액션은 존재하지 않는다** — 문의는 고객이 만들고 감사 성격이라 지우지 않는다.

**메뉴가 숨겨져도 권한은 그대로다** — 조건부 노출은 **메뉴 규칙**이지 권한 규칙이 아니다. 사이드바에서 사라져도 리소스 키와 판정은 살아 있다.
