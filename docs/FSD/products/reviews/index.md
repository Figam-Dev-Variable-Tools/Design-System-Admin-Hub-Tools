# 리뷰

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 리뷰 |
| 화면 ID | `SCR-PRODUCTS-REVIEWS` |
| 메뉴 경로 | 상품 관리 > 리뷰 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 고객이 남긴 리뷰를 별점으로 좁혀 보고, 노출 여부를 그 자리에서 바꾸고, 답변을 달러 상세로 들어간다.

**업무 배경** — 이 화면의 업무는 **루프**다: 낮은 별점 훑기 → 답변 → 다음. 그래서 조회 조건이 주소에 남는 것이 특히 중요하다 — 1점 리뷰만 골라 한 건을 열어 답변하고 뒤로 오면, 예전에는 필터가 풀린 전체 목록에 착지해 방금 처리하던 줄을 처음부터 다시 찾아야 했다 — `apps/admin/src/pages/products/reviews/ReviewListPage.tsx:6-11`.

**화면 설명** — 툴바(검색 + 별점 셀렉트) 위에 리뷰 표가 선다. 표에는 선택 체크박스와 행 액션이 있고, **행 클릭은 상세로 간다**(형제 목록 대부분이 수정 폼으로 가는 것과 다르다).

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원.

**사용 시나리오**

1. 별점 `★ 1점` 으로 거른다 → 신고 배지가 붙은 행부터 연다 → 상세에서 답변을 달거나 숨긴다.
2. 악성 리뷰를 목록에서 바로 숨긴다 → 토글을 끄면 즉시 저장된다.
3. 포토 리뷰만 눈으로 훑는다 → 표시 열의 `포토` 배지로 구분한다.

**선행 조건** — 로그인 · `/products/reviews` 리소스 조회 권한. 삭제는 `remove` 권한.

**후행 처리** — 노출 토글과 상세의 답변이 고객 화면의 리뷰 목록에 그대로 반영된다.

**관련 화면** — `SCR-PRODUCTS-REVIEWS-DETAIL`(행 클릭) · `SCR-PRODUCTS`(리뷰가 달린 상품). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `상품 관리 > 리뷰`. 조건부 노출이 아니다.

**관련 기능** — **이 화면은 결제 잠금과 무관하다.** 리뷰는 결제 연동 여부와 상관없이 쌓이고 관리된다(잠금 표에 리뷰 섹션이 없다).

**관련 API** — 리뷰 목록 조회 1건 · 리뷰 갱신 1건(노출 토글) · 삭제 N건. **백엔드가 없다** — `apps/admin/src/pages/products/reviews/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [리뷰 비기능 명세서](../../../NFRS/products/reviews.md)

**구현 파일** — `apps/admin/src/pages/products/reviews/ReviewListPage.tsx` (도메인·표시 헬퍼 `types.ts` · 어댑터 `data-source.ts`)

## 4. 기능 명세

### 4.1 목록 조회 · 별점 필터 · 검색

| 항목 | 내용 |
|---|---|
| 기능 목적 | 손볼 리뷰를 찾는다 |
| 실행 조건 | 화면 진입 · 필터·검색 변경 |
| 사용자 동작 | 셀렉트 선택 · 검색어 타이핑 |
| 시스템 처리 | 목록을 한 번 조회하고 별점 → 검색어 순으로 좁힌다. 조회 조건은 주소 쿼리가 소유한다(`?rating=` · 검색어) — `ReviewListPage.tsx:98-118` |
| 데이터 처리 | 없음(읽기). 조건이 바뀌면 **선택을 푼다** — `ReviewListPage.tsx:111-113` |
| Validation | 주소의 모르는 별점 값은 조용히 '전체' 로 되돌린다 |
| API 호출 여부 | Y — 목록 1건 |
| 성공 처리 | 표가 그려진다. 토스트 없음 |
| 실패 처리 | 표 대신 danger 배너 + `다시 시도` |
| 예외 처리 | 별점만은 **숫자가 섞인 유니온**이라 문자열 허용 목록으로는 되돌릴 수 없다. 옵션 목록(별점 선택지의 단일 원천)에서 id 를 찾아 돌려주므로 허용 목록과 셀렉트가 그리는 값이 영원히 어긋나지 않는다 — `ReviewListPage.tsx:39-50` |
| 화면 변경 사항 | 목록영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

| 항목 | 내용 |

### 4.3 상세로 이동 · 삭제

| 항목 | 내용 |
|---|---|
| 기능 목적 | 답변을 달거나 리뷰를 지운다 |
| 실행 조건 | 삭제는 `remove` 권한 |
| 사용자 동작 | 행 클릭 · 연필 · 행 삭제 · 일괄 삭제 |
| 시스템 처리 | **행 클릭과 연필이 모두 상세로 간다** — 형제 목록 대부분(`/:id/edit`)과 다르다. 예전에는 그 사실이 이름 뒤에 숨어 표 캡션이 '수정 화면' 이라고 잘못 읽었다 — `ReviewListPage.tsx:215-217` |
| 데이터 처리 | 삭제는 리뷰를 목록에서 지운다 |
| Validation | 삭제 차단 규칙이 없다 |
| API 호출 여부 | Y — 삭제 N건 |
| 성공 처리 | 단건은 `MSG-COMMON-02`, 일괄은 `MSG-COMMON-03`([공통 문서](../../_common/index.md) §4.6) |
| 실패 처리 | 공통 삭제 확인 다이얼로그가 닫히지 않고 배너로 사유를 말한다 |
| 예외 처리 | 리뷰는 고객이 남긴 기록이지만 **삭제가 열려 있다**(클레임·주문과 다르다) |
| 화면 변경 사항 | 목록영역 · 선택 바 |
| 후속 동작 | 목록 재조회 |
| 로그 기록 여부 | 확인하지 못했다 |

**등록 액션은 존재하지 않는다** — 리뷰는 고객이 쓴다.

**⚠ 노출 토글에는 화면 쪽 권한 게이팅이 없다** — 공통 인라인 갱신 훅이 저장 시점에 다루며, 지금은 백엔드가 없어 실제로 막히지 않는다.

**권한이 없을 때 화면이 어떻게 되는가** — 조회 권한이 없으면 본문이 `접근 권한이 없어요` 로 바뀐다. `remove` 가 없으면 선택 체크박스와 일괄 삭제가 사라진다.
