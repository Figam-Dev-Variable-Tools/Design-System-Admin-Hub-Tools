# 상품

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 상품 |
| 화면 ID | `SCR-PRODUCTS` |
| 메뉴 경로 | 상품 관리 > 상품 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 등록된 상품을 카테고리·판매상태로 좁혀 보고, 전시 여부를 그 자리에서 바꾸고, 등록·수정·삭제로 들어간다.

**업무 배경** — 상품은 이 어드민이 **만들고 지우는** 몇 안 되는 자원이다. 그런데 금액은 이 화면이 정하지 않는다 — 결제 연동이 없으면 목록의 금액 칸이 전부 `가격문의` 가 되고 재고 열이 사라진다. 그 사실을 화면이 먼저 말하지 않으면 운영자는 값이 지워진 줄 알고 상품을 다시 저장한다.

**화면 설명** — 좌측은 필터 두 벌(카테고리 · 판매상태)이며 각각 건수 배지가 붙는다. 우측은 (결제가 없으면) 안내 배너 · 툴바(검색 + `상품 등록`) · 표다. 표에는 선택 체크박스와 행 액션(수정·삭제)이 있다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. '아우터' 카테고리를 눌러 12건을 본다 → 품절 배지가 붙은 행을 찾아 수정으로 들어간다.
2. 시즌이 끝난 상품의 전시 토글을 끈다 → 즉시 저장되고 성공 토스트가 뜬다.
3. 결제를 아직 연동하지 않았다 → 표 위에 안내 배너가 뜨고, 금액 칸이 전부 `가격문의` 이며, 재고 열 대신 **문의 건수** 열이 선다.

**선행 조건** — 로그인 · `/products` 리소스 조회 권한. 등록하려면 `create`, 수정·전시 토글은 `update`, 삭제는 `remove` 권한이 필요하다.

**후행 처리** — 등록·수정은 상품 폼(`SCR-PRODUCTS-FORM`)이 맡는다. 전시 토글은 이 화면에서 바로 저장된다. 삭제는 목록에서 사라진다.

**관련 화면** — `SCR-PRODUCTS-FORM`(등록·수정) · `SCR-PRODUCTS-CATEGORIES`(카테고리 배지가 가리키는 원장) · `SCR-PRODUCTS-INQUIRIES`(결제가 없을 때 문의가 쌓이는 곳) · `SCR-SETTINGS-PAYMENT`(금액 표시의 원인). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 사이드바 `상품 관리 > 상품`. 조건부 노출이 아니다.

**관련 기능** — **결제 설정이 이 화면의 세 가지를 바꾼다**: 금액 칸(가격문의 대체) · 재고 열(문의 열로 교체) · 안내 배너. 판정은 전부 결제 설정 층이 소유하고 화면이 다시 판단하지 않는다 — `apps/admin/src/pages/products/items/ProductListPage.tsx:188-198`.

**관련 API** — 상품 목록 조회 1건 · 카테고리 선택지 조회 1건 · 상품 갱신 1건(전시 토글) · 상품 삭제 N건. **백엔드가 없다** — `apps/admin/src/pages/products/items/data-source.ts:19,29` 의 `TODO(backend)`: `GET/POST /api/products` · `GET/PUT/DELETE /api/products/:id` · `GET /api/products/categories`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [상품 비기능 명세서](../../NFRS/products.md)

**구현 파일** — `apps/admin/src/pages/products/items/ProductListPage.tsx` (필터 `components/ProductFilterPanel.tsx` · 표시 헬퍼 `types.ts` · 도메인·픽스처 `apps/admin/src/pages/products/_shared/store.ts` · 어댑터 `data-source.ts` · 가격 표시 `apps/admin/src/shared/commerce/price-display.ts`)

## 4. 기능 명세

### 4.1 목록 조회 · 두 축 필터 · 검색

| 항목 | 내용 |
|---|---|
| 기능 목적 | 지금 손볼 상품을 찾는다 |
| 실행 조건 | 화면 진입 · 필터·검색 변경 |
| 사용자 동작 | 좌측 항목 클릭 · 검색어 타이핑 |
| 시스템 처리 | 상품 목록과 카테고리 선택지를 각각 조회하고, 카테고리 → 판매상태 → 검색어 순으로 좁힌다. 조회 조건은 주소 쿼리가 소유한다(`?category=` · `?status=` · 검색어) — `ProductListPage.tsx:140-148` |
| 데이터 처리 | 없음(읽기). **보고 있는 행 집합이 바뀌면 선택을 푼다** — 화면에 없는 행이 선택된 채 '선택 3건 삭제' 가 되지 않게 — `ProductListPage.tsx:167-169` |
| Validation | 주소의 모르는 판매상태 값은 조용히 '전체' 로 되돌린다 |
| API 호출 여부 | Y — 상품 목록 1 · 카테고리 선택지 1 |
| 성공 처리 | 표가 그려진다. 정렬은 상품명 가나다순(같은 이름은 id 안정 정렬) — `_shared/store.ts` 의 `sortProducts`. 토스트 없음 |
| 실패 처리 | 상품 조회 실패는 표 대신 배너. **카테고리 조회 실패는 좌측 필터 안에서만** 알리고 표는 그대로 둔다 |
| 예외 처리 | 두 '전체' 항목의 배지는 전체 상품 수를 쓴다. 건수를 못 셌으면 `—` |
| 화면 변경 사항 | 필터영역 · 목록영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**건수는 다른 축의 필터를 반영하지 않는다** — '아우터 12' 는 언제나 아우터 상품 전체 수다. 판매상태 필터를 걸었다고 이 숫자가 바뀌면 운영자는 '지금 무엇을 걸러냈는지' 를 읽을 기준을 잃는다 — `_shared/store.ts` 의 `countProductsByCategory` 머리말.

### 4.4 결제 연동에 따른 표시 파생 — **이 화면에서 가장 중요한 규칙**

| 항목 | 내용 |
|---|---|
| 기능 목적 | 금액을 노출할 수 있는지를 **사이트 전역 연동 상태**에서 파생시킨다 |
| 실행 조건 | 렌더할 때마다(설정을 바꾸면 다음 렌더가 곧바로 새 규칙을 쓴다) |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 결제 설정을 읽어 술어 하나(`resolvePriceDisplay`)를 부르고, 그 결과의 서로 다른 필드를 쓴다 — 목록은 표시 문구를, 폼은 입력 잠금을 — `apps/admin/src/shared/commerce/price-display.ts:21-25` |
| 데이터 처리 | **아무것도 저장하지 않는다.** 화면이 그리는 것은 이 함수의 결과이지 어딘가 저장된 '표시 상태' 가 아니다 |
| Validation | 해당 없음 |
| API 호출 여부 | N — 저장된 설정을 동기로 읽는다 |
| 성공 처리 | 아래 표대로 화면이 바뀐다. 토스트 없음 |
| 실패 처리 | 실패할 것이 없다(로컬 읽기). 다만 자격증명이 **실제로 유효한지는 프런트가 알 수 없다**(§7) |
| 예외 처리 | 문의 건수 조회기가 배선되지 않았으면 `—`(모름)이고 0 을 그리지 않는다 |
| 화면 변경 사항 | 잠금 안내영역 · 판매가 열 · 재고/문의 열 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**⚠ 상품마다 고르던 '가격 표시 방식' 라디오는 없다.** 예전에는 상품 폼에 '금액 노출 / 가격문의로 대체' 를 고르는 축이 있었고 그 값이 상품에 저장됐다. **운영 요구로 그 축을 없앴다** — "이렇게 선택하는 게 아니라, 결제 연동이 안 돼 있으면 전부 가격 문의로 대체". 즉 가격 표시는 **고르는 값이 아니라 연동 상태의 결과**다 — `price-display.ts:3-19`. 라디오를 찾는 사람은 그것을 찾을 수 없다.

**무엇이 판정을 정하는가** — 결제로 판매할 수 있는지는 세 가지를 모두 만족해야 한다 — `apps/admin/src/shared/commerce/payment-settings.ts:111-118`.

| 조건 | 만족하지 못하면 |
|---|---|
| ⓐ 운영자가 PG 를 쓰기로 설정했다 | 금액이 전부 `가격문의` |
| ⓑ 저장된 자격증명이 PG 에서 거절되지 않았다 | 〃 (**백엔드만 아는 사실**이다 — §7) |
| ⓒ 필수 공개 값이 다 차고 필수 비밀이 저장돼 있다 | 〃 |

**잠금 사유 세 갈래** — 화면의 배너 문구는 이 셋 중 하나다 — `payment-settings.ts:192-203`.

| 원인 | 문구 |
|---|---|
| PG 미사용 | `PG 결제를 쓰지 않도록 설정되어 있어 모든 금액이 가격문의로 대체돼요.` |
| 자격증명 거절 | `저장된 결제 자격증명이 PG 에서 거절되었어요. 값을 다시 확인해 주세요. 그동안 모든 금액은 가격문의로 대체돼요.` |
| 필수 값 미완 | `{PG 이름} 필수 자격증명이 아직 채워지지 않아({빠진 항목들}) 결제 연동이 성립하지 않아요. 그동안 모든 금액은 가격문의로 대체돼요.` |

**결과가 화면에서 어떻게 갈라지는가**

| 축 | 결제로 판매 가능 | 불가능 |
|---|---|---|
| 판매가 열 | 최종가(+ 할인 시 정가 취소선) | **전 행이 `가격문의`** |
| 재고 열 | 총 재고 + 품절/부족 배지 | **감춘다** — 차감 주체(주문)가 없어 0 을 그리면 품절로 읽힌다 |
| 그 자리 | — | **문의 건수 열**이 대신 선다 |
| 배너 | 없음 | info 배너 1장 + 결제 설정·문의 화면 링크 |
| 상품 등록·수정 | 열려 있다 | **그대로 열려 있다** — 카탈로그는 결제와 무관하게 자란다 |

**잠금은 값을 지우지 않는다** — 저장된 판매가·할인율·과세 구분은 그대로 남고, 연동을 마치면 되살아난다. 지우는 구현은 되돌릴 수 없다: 운영자는 결제를 잠시 끄는 것과 정책을 폐기하는 것을 구분해서 하고 있는데, 코드가 그 둘을 같은 것으로 만든다 — `apps/admin/src/shared/commerce/pg-lock.ts:7-10`.
