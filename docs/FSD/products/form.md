# 상품 등록 · 수정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 상품 등록 · 상품 수정 |
| 화면 ID | `SCR-PRODUCTS-FORM` |
| 메뉴 경로 | 상품 관리 > 상품 > `상품 등록` / (행 클릭) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 상품 한 건의 모든 축(기본정보 · 가격/할인 · 적립금 · 쿠폰 사용 · 옵션/재고 · 배송 · 이미지 · 상세설명)을 한 화면에서 채우고 저장한다.

**업무 배경** — 상품 등록은 채울 것이 여덟 구획이라 저장 버튼을 누른 뒤 오류가 두 화면 아래에 있을 수 있다. 그래서 좌측에 **구획 목차**를 두고 오류가 남은 구획에 붉은 점을 찍는다. 오른쪽에는 고객에게 보일 상품 카드 미리보기를 두어 '지금 이 값이 고객 화면에서 무엇이 되는가' 를 함께 본다.

**화면 설명** — 3단이다: 좌측 구획 목차 · 가운데 입력 카드 8장 · 우측 미리보기 카드. 좁은 화면에서는 1단으로 접힌다. `/products/new` 와 `/products/:id/edit` 는 **같은 화면**이며 id 유무로 등록/수정이 갈린다.

**주요 사용자** — 최상위 관리자 · 운영자(상품 등록·수정 권한 보유자).

**사용 시나리오**

1. `상품 등록` → 기본 정보와 가격을 채우고 옵션 두 축(색상·사이즈)을 넣으면 SKU 매트릭스가 자동으로 만들어진다 → 재고를 채우고 대표 이미지를 올려 등록한다.
2. 결제를 아직 연동하지 않았다 → **판매가·할인·과세 입력이 잠기고** 적립금·쿠폰·재고 수량·배송 구획에도 각각 잠금 안내가 붙는다. 상품명·옵션 구성·이미지·상세설명은 그대로 편집된다.
3. 저장을 눌렀는데 오류가 있다 → 좌측 목차의 그 구획에 붉은 점이 찍히고 첫 위반 필드로 포커스가 옮겨 간다.

**선행 조건** — 로그인 · `/products` 리소스의 `create`(등록) 또는 `update`(수정) 권한. 카테고리가 1개 이상 등록되어 있어야 저장할 수 있다.

**후행 처리** — 저장하면 목록으로 돌아간다. 카테고리 라벨은 저장 시점 값으로 상품에 복사된다(목록이 바로 쓴다).

**관련 화면** — `SCR-PRODUCTS`(목록) · `SCR-PRODUCTS-CATEGORIES`(카테고리 원장) · `SCR-PRODUCTS-COUPONS`(쿠폰 후보 목록) · `SCR-PRODUCTS-POINTS`(기본 적립률의 정본) · `SCR-PRODUCTS-SHIPPING`(전역 배송 정책) · `SCR-SETTINGS-PAYMENT`(잠금의 원인). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다.

**관련 기능** — 상품의 쿠폰 사용 설정과 쿠폰의 '발급 대상' 이 어긋나면 **상품이 이긴다** — 쿠폰은 캠페인이고 쿠폰 불가는 상품의 원가 사실이라, 둘이 어긋날 때 손해를 보는 쪽은 언제나 상품이다 — `apps/admin/src/pages/products/_shared/store.ts:157-170`.

**관련 API** — 상세 조회 1건(수정) · 카테고리 선택지 1건 · 쿠폰 후보 1건(결제가 열려 있을 때만) · 상품 생성/갱신 1건. **백엔드가 없다** — `apps/admin/src/pages/products/items/data-source.ts:19,29`.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [상품 비기능 명세서](../../NFRS/products.md)

**구현 파일** — `apps/admin/src/pages/products/items/ProductFormPage.tsx` (검증 `validation.ts` · 가격/적립/배송 카드 `components/ProductPricingCards.tsx` · 쿠폰 카드 `components/ProductCouponCard.tsx` · 옵션 매트릭스 `components/ProductOptionMatrix.tsx` · 미리보기 `components/ProductCardPreview.tsx` · 도메인 `../_shared/store.ts` · 잠금 `apps/admin/src/shared/commerce/pg-lock.ts`)

**⚠ 잠긴 칸은 검증하지 않는다 — 스키마가 결제 설정을 읽는다.** 결제 연동이 없으면 판매가·할인 입력이 잠기지만 **저장된 값은 지우지 않는다**. 그 값을 계속 검증하면 손댈 수 없는 칸 때문에 저장이 영원히 거절된다(상품명만 고치려 해도 막힌다). 그래서 스키마가 **화면과 같은 함수를 부른다** — 조건을 두 곳에 적으면 '입력은 잠겼는데 검증은 요구하는' 화면이 생긴다 — `validation.ts:6-16,37-45`.

**상세설명 길이는 마크업이 아니라 평문에 건다** — 값 길이로 재면 '굵게' 한 번에 17자가 붙어 사용자가 쓰지도 않은 글자수로 제출이 막힌다. 화면의 카운터와 **같은 함수**로 판정한다 — `validation.ts:134-137`.

**검증 실패는 첫 위반 필드로 포커스를 옮긴다** — 공통 폼 컨트롤러가 계약으로 고정한다.

| 네트워크 오류 | 위와 같다 | 자동 재시도 없음 | 재시도 |
| 권한 없음 | 저장 시점에 공통 폼이 막는다: `이 항목을 수정할 권한이 없어요. …` / `이 항목을 등록할 권한이 없어요. …` — [공통 문서](../_common/index.md) §7 | 검증도 요청도 시작하지 않고 폼 배너로 이유를 남긴다 | 관리자에게 권한 요청 |

**'업로드' 는 이미지 필드를 뜻한다** — 다만 **업로드 엔드포인트가 없어** 실제 전송은 일어나지 않는다(§7). 삭제는 이 화면에 없다(목록의 일이다).
