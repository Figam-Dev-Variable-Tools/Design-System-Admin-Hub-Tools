# 쿠폰 발급 현황

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 쿠폰 발급 현황 |
| 화면 ID | `SCR-PRODUCTS-COUPONS-ISSUANCE` |
| 메뉴 경로 | 상품 관리 > 쿠폰 > `발급 현황` — **사이드바에 자기 항목이 없다** |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 걸어 둔 쿠폰이 **얼마나 나갔고 얼마나 쓰였는지**, 그리고 어느 발급 기준으로 나갔는지 본다.

**업무 배경** — 쿠폰 목록이 답하는 질문은 '무엇을 걸어 뒀나' 이고, 여기가 답하는 질문은 '그래서 얼마나 나갔고 얼마나 쓰였나' 다. 두 질문을 한 표에 넣으면 열이 열두 개가 되고 어느 쪽도 읽히지 않는다 — `apps/admin/src/pages/products/coupons/CouponIssuanceListPage.tsx:3-6`.

**화면 설명** — 위에 **쿠폰별 요약 표**(쿠폰마다 한 줄), 아래에 **발급 이력 목록**(발급 한 건마다 한 줄)이 있다. 발급은 트리거가 만들고 운영자는 보기만 하므로 선택도 삭제도 없다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원.

**사용 시나리오**

1. 요약 표에서 사용률이 `0%` 인 쿠폰을 찾는다(경고 색) → 그 쿠폰을 눌러 설정을 손본다.
2. '고객 다운로드' 로 거른다 → 다운로드로 나간 발급만 남는다.
3. 고객이 자기 쿠폰 만료일을 묻는다 → 회원 이름으로 검색해 그 줄의 만료일을 읽는다.

**선행 조건** — 로그인 · **`/products/coupons` 리소스 조회 권한**(이 라우트는 자기 리소스 키가 없다 — §9).

**후행 처리** — 이 화면 자체는 아무것도 바꾸지 않는다. 행을 누르면 쿠폰 수정으로 간다.

**관련 화면** — `SCR-PRODUCTS-COUPONS`(부모 · 유일한 진입 경로) · `SCR-PRODUCTS-COUPONS-FORM`(행 클릭 목적지) · `SCR-USERS-MEMBERS-DETAIL`(같은 사건을 회원 쪽에서 본 보유 쿠폰). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — **사이드바에 잎이 없다.** 사이드바 잎은 `/products/coupons` 하나이고, 이 라우트는 그 아래의 **정적 보조 목록**이다. 라우트 순서상 `/products/coupons/:id/edit` 보다 먼저 잡힌다 — `apps/admin/src/App.tsx:345-347`.

**관련 기능** — 만료일은 쿠폰의 사용 기간 규칙이 정한다. **발급일 기준 쿠폰이면 회원마다 만료일이 다르고**, 캠페인 종료일이 먼저 오면 그날 만료된다.

**관련 API** — 발급 이력 조회 1건 · 쿠폰 목록 조회 1건(부모와 **같은 조회를 공유한다**). **백엔드가 없다** — `apps/admin/src/pages/products/coupons/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [쿠폰 목록](index.md) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [발급 현황 비기능 명세서](../../../NFRS/products/coupons-issuance.md)

**구현 파일** — `apps/admin/src/pages/products/coupons/CouponIssuanceListPage.tsx` (집계·만료 계산 `types.ts` · 어댑터 `data-source.ts`)

## 4. 기능 명세

### 4.1 두 조회와 요약 집계

| 항목 | 내용 |
|---|---|
| 기능 목적 | 발급 이력과 쿠폰 정의를 합쳐 쿠폰별 요약을 만든다 |
| 실행 조건 | 화면 진입 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 발급 이력을 조회하고, **쿠폰 목록은 부모와 같은 조회를 공유한다** — 같은 자료를 두 번 조회하지 않는다 — `CouponIssuanceListPage.tsx:113-119`. 이력을 쿠폰 id 로 묶어 집계한다 |
| 데이터 처리 | 없음(읽기) |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 발급 이력 1건(쿠폰 목록은 캐시를 쓴다) |
| 성공 처리 | 요약 표와 이력 표가 그려진다. 토스트 없음 |
| 실패 처리 | 이력 조회가 실패하면 아래 목록이 배너로 바뀐다. **요약 표는 쿠폰 목록만 있으면 그려지고, 그 안의 집계는 0 이 된다** |
| 예외 처리 | 발급이 0건인 쿠폰은 트리거별 배지 대신 `발급 없음` 을 그린다 |
| 화면 변경 사항 | 요약영역 · 목록영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**두 숫자가 무엇을 세는지 화면이 밝힌다** — 요약의 `발급`·`사용`·`사용률` 은 **이력 표본**에서 센 값이고, `소진율` 은 쿠폰이 들고 있는 **운영 누계**다. 픽스처 단계라 둘이 다르므로 캡션이 그 사실을 적어 둔다 — `CouponIssuanceListPage.tsx:8-10`.
