# 쿠폰

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 쿠폰 |
| 화면 ID | `SCR-PRODUCTS-COUPONS` |
| 메뉴 경로 | 상품 관리 > 쿠폰 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 걸어 둔 쿠폰을 발급유형·발급기준으로 좁혀 보고, 발급 상태를 그 자리에서 켜고 끄며, 등록·수정·삭제로 들어간다.

**업무 배경** — 쿠폰은 축이 둘이다: **언제 누구에게 나가는가**(발급 기준)와 **어디에 쓸 수 있는가**(사용 대상). 목록에서 가장 자주 확인되는 것은 앞의 축이라 '발급 기준' 열이 파라미터까지 실어 보인다('VIP 승급 시' 와 'VVIP 승급 시' 는 다른 정책이다).

**화면 설명** — (결제가 없으면) 잠금 배너 · 툴바(검색 + 유형 셀렉트 + 발급 기준 셀렉트 + `발급 현황` + `쿠폰 등록`) · 표. 표에는 선택 체크박스와 행 액션이 있다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원(조회만).

**사용 시나리오**

1. '고객 다운로드' 쿠폰만 걸러 소진율을 확인한다 → 절반이 넘었으면 발급 수량을 늘리러 수정으로 들어간다.
2. 상태 열에 `쿠폰 불가 상품 2건` 경고가 붙었다 → 그 쿠폰을 열어 대상에서 그 상품을 빼거나, 상품 쪽 설정을 푼다.
3. 결제를 아직 연동하지 않았다 → 배너가 뜨고 **등록 버튼과 발급 토글이 잠긴다**. 기존 쿠폰은 그대로 조회된다.

**선행 조건** — 로그인 · `/products/coupons` 리소스 조회 권한. 등록은 `create`, 삭제는 `remove` 권한.

**후행 처리** — 쿠폰의 사용 대상 설정은 상품 폼의 '쿠폰 사용 설정' 과 짝을 이루고, 어긋나면 **상품이 이긴다**.

**관련 화면** — `SCR-PRODUCTS-COUPONS-FORM`(등록·수정) · `SCR-PRODUCTS-COUPONS-ISSUANCE`(발급 현황) · `SCR-PRODUCTS-FORM`(쿠폰 사용 설정) · `SCR-SETTINGS-PAYMENT`(잠금의 원인) · `SCR-USERS-SETTINGS`(등급 정책). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `상품 관리 > 쿠폰`. 조건부 노출이 아니다. **발급 현황(`/products/coupons/issuance`)은 사이드바에 잎이 없고** 이 잎이 그 권한을 덮는다(§10).

**관련 기능** — 쿠폰이 지목한 상품이 쿠폰을 거부하면 목록의 상태 열에 경고 배지가 붙는다. 판정의 정본은 상품 도메인이다 — `apps/admin/src/pages/products/_shared/store.ts:157-170`.

**관련 API** — 쿠폰 목록 조회 1건 · 쿠폰 갱신 1건(발급 토글) · 삭제 N건. **백엔드가 없다** — `apps/admin/src/pages/products/coupons/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [쿠폰 비기능 명세서](../../../NFRS/products/coupons.md)

**구현 파일** — `apps/admin/src/pages/products/coupons/CouponListPage.tsx` (도메인·표시 헬퍼 `types.ts` · 어댑터 `data-source.ts` · 잠금 `apps/admin/src/shared/commerce/pg-lock.ts`)

### 4.5 쿠폰 ↔ 상품 충돌

| 항목 | 내용 |
|---|---|
| 기능 목적 | 운영자가 **모르는 채로 두는 것**을 막는다 |
| 실행 조건 | 쿠폰의 대상이 '특정 상품' 또는 '특정 카테고리' 일 때 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 그 쿠폰이 지목한 상품 중 **쿠폰을 거부하는** 것을 센다 — `types.ts:630-640` |
| 데이터 처리 | 없음 |
| Validation | 해당 없음 — **저장을 막지 않는다** |
| API 호출 여부 | N |
| 성공 처리 | 충돌이 있으면 상태 열에 warning 배지가 붙는다 |
| 실패 처리 | 해당 없음 |
| 예외 처리 | '전체 회원' 쿠폰 아래에 쿠폰 불가 상품이 섞여 있는 것은 **정상**이다 — 특가 한 건만 빼 두는 것이 그 설정의 목적이다. 운영자가 **콕 집어 만든 쿠폰**이 그 상품에서 거절당하는 것만이 실수다 |
| 화면 변경 사항 | 상태 열 |
| 후속 동작 | 쿠폰 폼에도 같은 경고가 뜬다 |
| 로그 기록 여부 | N |

**승자는 상품이다** — 쿠폰이 자기 대상으로 지목해도 상품이 '쿠폰 사용 불가' 면 그 쿠폰은 붙지 않는다. 쿠폰은 **캠페인**(기간이 끝나면 사라진다)이고 쿠폰 불가는 **상품의 원가 사실**(마진이 없다)이라, 둘이 어긋날 때 손해를 보는 쪽은 언제나 상품이다 — `_shared/store.ts:157-164`.
