# 쿠폰 등록 · 수정

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 쿠폰 등록 · 쿠폰 수정 |
| 화면 ID | `SCR-PRODUCTS-COUPONS-FORM` |
| 메뉴 경로 | 상품 관리 > 쿠폰 > `쿠폰 등록` / (행 클릭) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 쿠폰 한 건의 **정보 · 발급 기준 · 사용 조건**을 정하고 저장한다.

**업무 배경** — '누구에게 쓸 수 있나' 와 '언제 손에 들어오나' 는 다른 질문이다. VIP 승급 시 발급되는 쿠폰이 전 상품에 쓰일 수 있고, 전 회원이 받아 가는 다운로드 쿠폰이 특정 카테고리에만 쓰일 수 있다. 두 축을 한 필드로 묶으면 그 조합이 표현되지 않는다 — `apps/admin/src/pages/products/coupons/types.ts:8-10`.

**화면 설명** — 왼쪽에 입력 카드 3장(`쿠폰 정보` · `발급 기준` · `사용 조건`), 오른쪽에 실시간 쿠폰 카드 미리보기. `/products/coupons/new` 와 `/products/coupons/:id/edit` 는 **같은 화면**이다.

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. 신규 가입 쿠폰을 만든다 → 발급 시점 `회원 가입 시` → **조건 칸이 뜨지 않는다**(더 정할 것이 없다는 힌트가 그 자리를 대신한다).
2. VIP 승급 쿠폰 → 발급 시점 `회원 등급 승급 시` → **승급 대상 등급** 셀렉트가 붙는다.
3. 특정 상품 쿠폰 → 사용 대상 `특정 상품` → **1차 카테고리 → 2차 카테고리 → 상품** 세 셀렉트가 차례로 열리고, `추가` 로 대상을 쌓는다.
4. 1차 카테고리를 바꾼다 → 2차와 상품 선택이 **버려진다**.

**선행 조건** — 로그인 · `/products/coupons` 리소스의 `create`(등록) 또는 `update`(수정) 권한. 등록 버튼은 결제가 잠기면 사라진다.

**후행 처리** — 저장하면 목록으로 돌아간다. 사용 대상 설정은 상품의 '쿠폰 사용 설정' 과 짝을 이룬다.

**관련 화면** — `SCR-PRODUCTS-COUPONS`(목록) · `SCR-PRODUCTS-CATEGORIES`(연쇄 선택의 원천) · `SCR-PRODUCTS`(상품 선택지) · `SCR-USERS-SETTINGS`(회원 등급 원장) · `SCR-PRODUCTS-FORM`(쿠폰 사용 설정). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바에 자기 자리가 없다.

**관련 기능** — 회원 등급 목록의 정본은 고객 설정 화면이고, 이 화면은 **이음매로만** 읽는다. 배선이 없으면 빈 배열이 아니라 **`null`(모른다)** 이 온다 — `CouponFormPage.tsx:274-282`.

**관련 API** — 상세 조회 1건(수정) · 카테고리 선택지 1건(대상이 카테고리·상품일 때만) · 상품 선택지 1건(대상이 상품일 때만) · 쿠폰 생성/갱신 1건. **백엔드가 없다** — `apps/admin/src/pages/products/coupons/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [쿠폰 비기능 명세서](../../../NFRS/products/coupons.md)

**구현 파일** — `apps/admin/src/pages/products/coupons/CouponFormPage.tsx` (검증 `validation.ts` · 도메인·순수 규칙 `types.ts` · 연쇄 선택 `components/CouponScopePicker.tsx` · 등급 다중 선택 `components/CouponTargetPicker.tsx` · 미리보기 `components/CouponCardPreview.tsx`)

**왜 셋에는 조건이 없는가** — `types.ts:405-413`.

| 시점 | 이유 |
|---|---|
| 운영자 직접 발급 | 누구에게 줄지는 **발급을 실행하는 순간** 고른다 — 쿠폰 정의가 미리 담을 값이 없다 |
| 회원 가입 시 | 가입 이벤트에 파라미터가 없다(이 리포의 회원 모델에 가입 경로·추천인 축이 없다). 없는 데이터를 묻는 셀렉트는 저장은 되고 동작하지 않는다 |

**힌트가 조건 칸과 짝을 이룬다** — 조건을 더 묻는 시점의 힌트는 **아래에 무엇이 뜨는지**를 말하고, 묻지 않는 시점의 힌트는 **더 정할 것이 없다**고 못 박는다. 안 그러면 운영자는 조건 칸이 안 뜨는 것을 화면 고장으로 읽고 같은 자리를 계속 눌러 본다 — `types.ts:58-63`.

**승급 대상 등급은 기본 제공 3종뿐이다** — 발급 기준이 드는 등급은 닫힌 유니온이라 운영자가 추가한 등급은 아직 들 수 없다. **그 등급들을 조용히 빼지 않고 이름을 적는다**(§3 No.14) — 등급 정책 화면이 반대편에서 같은 사실을 말하고 있고, 두 화면의 말이 어긋나면 운영자는 어느 쪽도 믿지 못한다 — `CouponFormPage.tsx:312-322`.

**회원등급은 체크박스다** — 짧은 평면 목록이라 연쇄가 필요 없다. 필수 표시는 **개별 체크박스가 아니라 묶음**에 붙는다 — 필수인 것은 '어느 한 대상' 이 아니라 '고르는 행위' 이기 때문이다 — `components/CouponTargetPicker.tsx:6-9`.
