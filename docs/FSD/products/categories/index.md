# 카테고리 (상품)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 카테고리 |
| 화면 ID | `SCR-PRODUCTS-CATEGORIES` |
| 메뉴 경로 | 상품 관리 > 카테고리 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 상품 분류를 **2단계(대분류 → 중분류)** 로 만들고 고치고 지운다.

**업무 배경** — 카테고리는 상품이 참조하는 값이다. 쓰고 있는 카테고리를 지우면 그 상품들이 갈 곳을 잃는다. 그래서 이 화면의 핵심 제약은 **사용 중이면 삭제할 수 없다**이고, 정리하려는 운영자가 가장 먼저 하려는 일은 '지울 수 있는 것만 보기' 다 — 좌측 필터 축이 '사용 여부' 인 이유다.

**화면 설명** — 좌측은 사용 여부 필터(전체 · 사용 중 · 미사용). 우측은 건수 줄 + `카테고리 추가` + **2단 트리 목록**이다. 대분류 행은 접었다 펼 수 있고 그 아래에 중분류가 들여쓰기로 붙는다.

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. '미사용' 을 눌러 지울 수 있는 것만 본다 → 휴지통을 눌러 확인 후 지운다.
2. '아우터' 행의 `+` 를 눌러 하위 '코트' 를 만든다 → 상위가 '아우터' 로 미리 채워진 채 모달이 열린다.
3. 삭제 버튼이 잠긴 카테고리의 건수 배지를 누른다 → 그 카테고리로 걸러진 **상품 목록**이 열린다.

**선행 조건** — 로그인 · `/products/categories` 리소스 조회 권한. 추가는 `create`, 삭제는 `remove` 권한.

**후행 처리** — 카테고리 라벨을 고치면 그 카테고리를 쓰는 **상품의 비정규화 라벨도 함께 갱신된다**(백엔드가 붙으면 서버가 정합성을 맡는다) — `apps/admin/src/pages/products/_shared/store.ts` 의 `updateProductCategory`.

**관련 화면** — `SCR-PRODUCTS`(건수 배지의 목적지 · 카테고리 열) · `SCR-PRODUCTS-FORM`(2단 셀렉트) · `POP-PRODUCTS-CATEGORIES-CATEGORY-FORM`(추가·수정 모달) · `SCR-PRODUCTS-COUPONS-FORM`(쿠폰 대상의 연쇄 선택이 같은 목록을 읽는다). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `상품 관리 > 카테고리`. 조건부 노출이 아니다.

**관련 기능** — 이 목록은 상품 폼의 카테고리 셀렉트 2단과 **쿠폰 폼의 1차·2차 연쇄 선택**의 원천이다. 세 화면이 같은 조회를 읽는다.

**관련 API** — 카테고리 목록(사용 건수 포함) 조회 1건 · 생성/수정/삭제 각 1건. **백엔드가 없다** — `apps/admin/src/pages/products/categories/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [카테고리 비기능 명세서](../../../NFRS/products/categories.md)

**구현 파일** — `apps/admin/src/pages/products/categories/ProductCategoriesPage.tsx` (필터 `components/CategoryUsageFilter.tsx` · 모달 `components/ProductCategoryFormModal.tsx` · 표시 헬퍼 `types.ts` · 도메인 `../_shared/store.ts`)

### 4.4 2단계 제약과 삭제 차단

**계층은 최대 2단계다** — 3단계는 만들지 않는다. 저장소가 세 갈래로 막는다 — `_shared/store.ts` 의 `assertAssignableParent`·`updateProductCategory`·`removeProductCategory`.

| 규칙 | 언제 걸리는가 | 거부 사유 |
|---|---|---|
| 자기 자신을 상위로 | 수정에서 자기 id 를 상위로 지정 | `자기 자신을 상위 카테고리로 지정할 수 없어요.` |
| 없는 상위 | 상위 id 가 실재하지 않음 | `상위 카테고리를 찾을 수 없어요.` |
| 3단계 금지 | 상위로 지정한 것이 이미 2Depth | `카테고리는 2단계까지만 만들 수 있어요.` |
| 하위를 가진 것의 이동 | 하위가 있는 대분류를 다른 대분류 밑으로 | `하위 카테고리가 있는 카테고리는 다른 카테고리 밑으로 옮길 수 없어요.` |

**삭제 차단은 두 갈래다** — 화면은 버튼을 잠그고 그 사유를 이름·툴팁에 싣는다 — `ProductCategoriesPage.tsx:197-199`.

| 차단 사유 | 화면 표기 | 저장소의 거부 사유 |
|---|---|---|
| 그 카테고리를 쓰는 상품이 있다 | `{N}개 상품` | `사용 중인 카테고리는 삭제할 수 없어요.` |
| 하위 카테고리가 달려 있다 | `하위 카테고리 있음` | `하위 카테고리가 있는 카테고리는 삭제할 수 없어요.` |

**막힌 삭제의 다음 행동을 화면이 말한다** — 건수 배지가 곧 상품 목록 링크다(그 카테고리로 이미 걸러진 목록이 열린다). 거기서 카테고리를 바꾸거나 상품을 지우면 이 카테고리가 지워진다.

**⚠ 수정 버튼에는 권한 게이팅이 없다** — 화면이 `canCreate` 와 `canRemove` 만 읽고, 연필 버튼은 조회할 수 있으면 누구에게나 보인다. 저장은 서버가 막아야 한다(지금은 백엔드가 없어 실제로 막히지 않는다).

**권한이 없을 때 화면이 어떻게 되는가** — 조회 권한이 없으면 본문이 `접근 권한이 없어요` 로 바뀐다. `create` 가 없으면 추가·하위 추가 버튼이 사라지고, `remove` 가 없으면 삭제 버튼이 잠긴다(사유 없이 잠기는 것이 아니라 **버튼 자체가 조건에 걸려 비활성**이다).
