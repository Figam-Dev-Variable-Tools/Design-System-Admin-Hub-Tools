# 적립금 (정책)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 적립금 |
| 화면 ID | `SCR-PRODUCTS-POINTS` |
| 메뉴 경로 | 상품 관리 > 적립금 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 새 상품의 **기본 적립률**과, 상품에 속하지 않는 **적립금 사용·소멸 규칙**을 문서 1건으로 정한다.

**업무 배경** — 적립률은 상품마다 갈라졌다(상품 폼의 적립금 카드 — 정률/정액/미적용). 그런데 나머지 다섯 가지는 상품에 속하지 않는다: **회원가입 적립금**(가입 시점의 지급이라 상품이 없다), **최소 사용 포인트·사용 단위·1회 사용 한도**(주문 단위의 '사용' 규칙이지 적립이 아니다), **유효기간**(적립된 포인트의 소멸 규칙이라 어느 상품에서 왔는지와 무관하다). 그래서 이 화면이 남았다 — `apps/admin/src/pages/products/points/PointsPolicyPage.tsx:6-13`.

**화면 설명** — 카드 한 장이다. (결제가 없으면) 잠금 배너 + 적립 기준 셀렉트 + 숫자 6칸 + 저장 버튼.

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. 기본 적립률을 1% 에서 2% 로 올린다 → 저장 → **새로 만드는 상품**의 초기값이 2% 가 된다(이미 등록된 상품은 자기 값을 그대로 쓴다).
2. 적립금 유효기간을 12개월에서 24개월로 늘린다 → 저장.
3. 결제를 아직 연동하지 않았다 → **정책은 열리되 입력이 잠긴다**. 저장 버튼도 함께 잠긴다.

**선행 조건** — 로그인 · `/products/points` 리소스 조회 권한. 저장은 `update` 권한이 없으면 화면 전체가 접근 거부로 바뀐다(단일 문서 껍데기의 규약).

**후행 처리** — 기본 적립률은 상품 폼의 **초기값**이 된다. 나머지 다섯 규칙은 주문·회원 적립금 계산에 쓰인다.

**관련 화면** — `SCR-PRODUCTS-FORM`(상품별 적립 오버라이드) · `SCR-USERS-MEMBERS-DETAIL`(회원별 적립금 잔액·원장) · `SCR-SETTINGS-PAYMENT`(잠금의 원인) · `SCR-ORDERS-CLAIMS-DETAIL`(환불 시 적립금 복원). 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `상품 관리 > 적립금`. 조건부 노출이 아니다.

**관련 기능** — 이 화면과 상품 폼의 관계는 **배송 정책과 상품 배송 카드의 관계와 정확히 같다** — 정책이 기본값을 정하고 상품이 필요할 때만 덮어쓴다.

**관련 API** — 정책 문서 조회 1건 · 저장 1건. **백엔드가 없다** — `apps/admin/src/pages/products/points/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [적립금 비기능 명세서](../../../NFRS/products/points.md)

**구현 파일** — `apps/admin/src/pages/products/points/PointsPolicyPage.tsx` (검증 `validation.ts` · 기본값·선택지 `types.ts` · 잠금 `apps/admin/src/shared/commerce/pg-lock.ts`)

### 4.2 결제 연동에 따른 잠금 — **정책은 열되 입력은 잠근다**

| 항목 | 내용 |
|---|---|
| 기능 목적 | 결제가 없어 적립이 발생하지 않는 동안 **정책을 볼 수는 있게 하되 저장은 막는다** |
| 실행 조건 | 렌더할 때마다 |
| 사용자 동작 | 없음(자동) |
| 시스템 처리 | 잠금 표의 `points-policy` 섹션을 읽어 입력 전체를 잠근다. **입력이 잠기면 폼이 dirty 가 되지 않으므로 저장 버튼도 함께 잠긴다**(껍데기가 dirty 로 저장을 연다) — `PointsPolicyPage.tsx:99-103` |
| 데이터 처리 | **아무것도 저장하지 않는다.** 저장된 정책은 그대로 남는다 |
| Validation | 잠긴 상태에서는 제출 자체가 일어나지 않는다 |
| API 호출 여부 | N |
| 성공 처리 | info 배너 + 입력·저장 잠김. 조회는 그대로 되고 **값은 화면에 보인다** |
| 실패 처리 | 실패할 것이 없다 |
| 예외 처리 | 결제를 켜면 그 값이 곧바로 다시 쓰인다 |
| 화면 변경 사항 | 잠금 안내영역 · 입력영역 · 버튼영역 |
| 후속 동작 | 없음 |
| 로그 기록 여부 | N |

**두 정책 화면이 다르게 다뤄진다**

| 화면 | 결제가 없을 때 |
|---|---|
| `SCR-PRODUCTS-POINTS`(이 화면) | **입력이 잠긴다** — 잠금 표에 `points-policy` 섹션이 있다 |
| `SCR-PRODUCTS-SHIPPING`(배송 정책) | **잠기지 않는다** — 잠금 표의 `product-shipping` 은 **상품 폼의 배송 카드**에 걸리는 것이고 전역 배송 정책 화면에는 걸려 있지 않다 |

**잠금은 값을 지우지 않는다** — 저장된 정책은 그대로 남고, 연동을 마치면 그 값이 다시 쓰인다 — `pg-lock.ts:7-10`.
