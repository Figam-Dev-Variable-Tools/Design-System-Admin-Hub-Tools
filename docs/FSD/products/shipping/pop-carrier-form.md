# 택배사 추가 · 수정 (팝업)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 택배사 추가 · 택배사 수정 |
| 화면 ID | `POP-PRODUCTS-SHIPPING-CARRIER-FORM` |
| 메뉴 경로 | 상품 관리 > 배송 > `택배사 추가` / 연필 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 송장을 붙일 때 고를 택배사 한 곳의 **이름 · 연동 코드 · 추적 URL 템플릿 · 사용 여부**를 정한다.

**업무 배경** — 예전 정책 문서의 택배사는 텍스트 한 줄이었다. 그래서 '대한통운' 과 'CJ대한통운' 이 공존할 수 있었고 **추적 URL 을 만들 키가 없었다**. 목록에서만 고르게 하면 이름이 한 벌이 되고, 각 택배사가 자기 추적 주소를 들 수 있다 — `apps/admin/src/pages/products/shipping/components/CarrierSection.tsx:6-9`.

**화면 설명** — 이름 · 코드 · 추적 URL 템플릿 · 사용 여부 토글 + 버튼 둘. 하나의 모달이 등록과 수정을 겸한다.

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. `택배사 추가` → 이름 '가상택배', 코드 `VIRTUAL`, 추적 URL 에 `{{invoice}}` 자리를 포함한 주소를 넣고 `추가`.
2. 오타로 등록한 코드를 고친다 → **코드는 수정할 수 있다**. 이미 나간 배송 건은 id 를 가리키므로 기록이 끊기지 않는다.
3. 추적 주소를 모른다 → 템플릿을 비워 둔다. 그러면 목록의 송장번호가 링크가 되지 않고 번호만 보인다.

**선행 조건** — 배송 화면이 열려 있어야 한다. 추가는 `create`, 수정은 `update` 권한이 있어야 여는 버튼이 보인다.

**후행 처리** — 저장하면 택배사 표와 **기본 택배사 선택지**가 함께 갱신되고, 배송 처리 화면의 송장 선택지에도 반영된다.

**관련 화면** — `SCR-PRODUCTS-SHIPPING`(부모) · `SCR-ORDERS-SHIPMENTS` · `POP-ORDERS-SHIPMENTS-INVOICE-BULK`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 해당 없음 — 메뉴 항목이 아니다.

**관련 기능** — **식별자는 id 다.** 이름과 코드는 사람이 읽는 표기라 언제든 바뀔 수 있고, 바뀔 수 있는 값을 키로 쓰면 지난 배송 건이 어느 택배사였는지 영영 알 수 없다 — `apps/admin/src/shared/domain/shipment.ts:29-34`.

**관련 API** — 택배사 생성 또는 갱신 1건. **백엔드가 없다** — `apps/admin/src/pages/products/shipping/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [배송](index.md) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/products/shipping/components/CarrierFormModal.tsx` (검증 `../validation.ts` · 도메인 `apps/admin/src/shared/domain/shipment.ts` · 안내 문구 `../types.ts`)

## 4. 기능 명세

### 4.1 저장(추가 · 수정)

| 항목 | 내용 |
|---|---|
| 기능 목적 | 택배사 1곳을 만들거나 고친다 |
| 실행 조건 | 검증 통과 · 저장 중이 아님 |
| 사용자 동작 | `추가` 또는 `저장`(폼 제출) |
| 시스템 처리 | 이름·코드·템플릿의 **앞뒤 공백을 다듬어** 저장한다 — `' 가상택배 '` 가 그대로 들어가면 이름이 두 벌이 된다 — `CarrierFormModal.tsx:97-103` |
| 데이터 처리 | 택배사 1건 생성 또는 갱신 |
| Validation | §6 전부 |
| API 호출 여부 | Y — 생성 또는 갱신 1건 |
| 성공 처리 | 모달이 닫히고 부모가 토스트를 띄운다 — 수정이면 `'{이름}' 택배사를 저장했어요.`, 추가면 `'{이름}' 택배사를 추가했어요.`(부모 §4.3) |
| 실패 처리 | **모달이 닫히지 않는다.** 상단 danger 배너 `저장하지 못했어요. 잠시 후 다시 시도해 주세요.` |
| 예외 처리 | 검증 실패면 **이름 칸으로 포커스**를 옮긴다 — `CarrierFormModal.tsx:137` |
| 화면 변경 사항 | 모달 · 부모의 택배사 표 · 기본 택배사 선택지 |
| 후속 동작 | 배송 처리 화면의 송장 선택지에도 반영된다 |
| 로그 기록 여부 | 확인하지 못했다 |

| 추적 URL | 비우거나 `{{invoice}}` 를 포함 | 〃 | `추적 URL 에는 {{invoice}} 이 한 번 들어가야 해요 — 송장번호를 끼울 자리가 없으면 링크를 만들 수 없어요.` — `apps/admin/src/pages/products/shipping/validation.ts:87` | 인라인 오류 |
| 사용 여부 | 참/거짓뿐이라 검증 대상이 아니다 | — | — | — |

**소문자·한글을 막는 이유** — 코드는 사람이 읽는 이름이 아니라 **시스템이 대조하는 값**이다. `cj` 와 `CJ` 가 다른 택배사가 되는 순간 이름을 유한 집합으로 좁힌 이유가 사라진다 — `validation.ts:55-59`.

**⚠ 코드·이름의 중복을 막는 규칙이 없다** — 같은 코드를 두 번 등록할 수 있다.
