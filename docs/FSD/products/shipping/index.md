# 배송 (배송 정책 · 택배사)

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 배송 |
| 화면 ID | `SCR-PRODUCTS-SHIPPING` |
| 메뉴 경로 | 상품 관리 > 배송 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 스토어 전체의 배송비 규칙(문서 1건)을 정하고, 송장을 붙일 때 고를 **택배사 목록**을 관리한다.

**업무 배경** — 배송은 목록형이 아니라 **정책 설정형**이다: 권역·요금·묶음배송을 문서 한 건으로 관리한다. 여기에 택배사 표가 하나 붙는데, 별도 메뉴로 만들지 않은 이유는 행이 서넛인 참조 테이블이라 자기 메뉴 잎을 가질 무게가 아니기 때문이다 — 배송비를 정하러 온 운영자와 택배사를 등록하러 온 운영자는 **같은 사람**이다 — `apps/admin/src/pages/products/shipping/components/CarrierSection.tsx:1-5`.

**화면 설명** — 카드 두 장이 세로로 선다: `배송 정책`(단일 문서 폼 + 저장) · `택배사`(CRUD 표 + 추가 모달).

**주요 사용자** — 최상위 관리자 · 운영자.

**사용 시나리오**

1. 무료배송 기준을 5만원에서 7만원으로 올린다 → 저장 → 스토어 전체 배송비 계산에 반영된다.
2. 새 택배사를 등록한다 → 이름·코드·추적 URL 템플릿을 넣으면 **배송 처리 화면의 송장 선택지**에 곧바로 뜬다.
3. 계약이 끝난 택배사를 지우려 한다 → 그 택배사로 나간 배송 건이 있어 버튼이 잠긴다. 대신 사용 여부를 끄면 새 송장의 선택지에서만 빠진다.

**선행 조건** — 로그인 · `/products/shipping` 리소스 조회 권한. 정책 폼은 `update` 권한이 없으면 화면 전체가 접근 거부로 바뀐다.

**후행 처리** — 정책의 반품배송비는 **클레임 상세의 반품배송비 기본값**이 된다. 기본 택배사는 송장 입력의 기본 선택이 된다.

**관련 화면** — `SCR-ORDERS-SHIPMENTS`(택배사 목록을 소비한다) · `POP-ORDERS-SHIPMENTS-INVOICE-BULK` · `SCR-ORDERS-CLAIMS-DETAIL`(반품배송비) · `SCR-PRODUCTS-FORM`(상품별 배송 오버라이드) · `POP-PRODUCTS-SHIPPING-CARRIER-FORM`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md).

**관련 메뉴** — 사이드바 `상품 관리 > 배송`. 조건부 노출이 아니다.

**관련 기능** — 상품 폼의 배송 카드는 **이 정책과 별개의 상품별 오버라이드**다. 두 화면이 서로를 참조하지 않고, 택배사 모델·규칙은 공통 층(`shared/domain/shipment.ts`)이 갖는다.

**관련 API** — 정책 문서 조회·저장 각 1건 · 택배사 목록 조회 1건 · 택배사 생성/수정/삭제 각 1건. **백엔드가 없다** — `apps/admin/src/pages/products/shipping/data-source.ts` 의 `TODO(backend)`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md) · [배송 비기능 명세서](../../../NFRS/products/shipping.md)

**구현 파일** — `apps/admin/src/pages/products/shipping/ShippingPolicyPage.tsx` (택배사 섹션 `components/CarrierSection.tsx` · 모달 `components/CarrierFormModal.tsx` · 검증 `validation.ts` · 기본값·문구 `types.ts` · 도메인 `apps/admin/src/shared/domain/shipment.ts`)

### 4.4 택배사 삭제 차단(fail-closed)

**규칙** — `apps/admin/src/shared/domain/shipment.ts:144-163`.

| 순서 | 언제 막히는가 | 거부 사유(버튼 이름·툴팁에 그대로 실린다) |
|---|---|---|
| ① | 사용 건수를 **모른다**(조회기 미배선) | `배송 건을 확인하지 못해 삭제할 수 없어요. 잠시 후 다시 시도해 주세요.` |
| ② | 그 택배사로 나간 배송이 1건 이상 | `'{택배사명}' 으로 나간 배송 {N}건이 있어 삭제할 수 없어요. 사용 여부를 끄면 새 송장의 선택지에서만 빠져요.` |

**모르면 막는다** — 여기서 통과시키면 배송 건이 가리키던 택배사가 사라져 송장 열의 이름과 추적 링크가 동시에 죽는다. 그 배송 건은 다시는 어느 택배로 나갔는지 말할 수 없게 된다 — `shipment.ts:147-152`.

**대안을 문장이 말한다** — 삭제 대신 **사용 여부 끄기**가 답이다(새 송장의 선택지에서만 빠지고 과거 기록은 남는다).
