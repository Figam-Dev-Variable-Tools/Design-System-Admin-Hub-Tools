# 대시보드

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 대시보드 |
| 화면 ID | `SCR-DASHBOARD` |
| 메뉴 경로 | 일반 관리 > 대시보드 (가지가 아니라 최상위 잎이다) |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 로그인한 운영자가 **오늘 무엇부터 해야 하는지**를 한 화면에서 판단하게 한다. 업무 영역(상품 · 문의 · 영업)을 탭으로 고르고, 그 영역의 할일 건수와 최근 목록을 본 뒤, 아래 통계로 흐름을 확인한다.

**업무 배경** — 운영자는 아침에 열 개 남짓한 메뉴를 차례로 열어 '새로 들어온 것이 있나'를 확인하곤 한다. 그 순회를 한 화면으로 접는 것이 이 화면의 존재 이유다. 그래서 각 위젯의 숫자는 **그 숫자를 처리하는 화면으로 바로 이어진다**.

**화면 설명** — 위에서부터 업무 탭 → 탭 패널(오늘의 할일 + 리스트 카드 2장) → (결제를 쓰지 않을 때만) 문의 현황 → 통계(방문자 차트 · 기간별 분석) 순이다. **모든 위젯이 권한 키에 걸려 있어** 최상위 관리자가 권한 관리 화면에서 끄면 리로드 없이 즉시 사라진다.

**주요 사용자** — 최상위 관리자 · 운영자 · 상담원. 무엇이 보이는지는 역할이 정한다.

**사용 시나리오**

1. 로그인하면 이 화면이 열린다 → '상품' 탭의 신규주문 건수를 보고 눌러 주문 목록으로 간다.
2. 문의를 담당하는 운영자는 '문의' 탭만 켜져 있어 그 탭이 곧 첫 화면이 된다.
3. 방문자 차트의 기간 토글을 '월'로 바꿔 최근 흐름을 본다.

**선행 조건** — 로그인. 세션이 없으면 로그인 화면으로 밀려난다.

**후행 처리** — 이 화면은 아무것도 바꾸지 않는다. 읽고 이동할 뿐이다.

**관련 화면** — `SCR-LOGIN`(로그인 성공 시 기본 이동처) · 탭 3건은 별도 문서다: [상품](tab-products.md) · [문의](tab-inquiries.md) · [영업](tab-sales.md). 위젯이 보내는 곳: `SCR-ORDERS` · `SCR-ORDERS-CLAIMS` · `SCR-PRODUCTS-REVIEWS` · `SCR-SUPPORT-TICKETS` · `SCR-SALES-INQUIRIES` · `SCR-SALES-CONTRACTS` · `SCR-PRODUCTS-INQUIRIES` · `SCR-SETTINGS-PAYMENT`. 공통 규약은 [공통 확인 다이얼로그 · 토스트](../_common/index.md).

**관련 메뉴** — 사이드바 최상위 첫 항목. 조건부 노출이 아니지만, 메뉴 자체가 `menu.dashboard` 권한에 걸린다 — `apps/admin/src/shared/layout/nav-config.ts:124`.

**관련 기능** — 권한 관리(`SCR-USERS-ROLES`)가 위젯 7개를 개별로 켜고 끈다 — `apps/admin/src/shared/permissions/feature-registry.ts:67-75`. 결제 설정(`SCR-SETTINGS-PAYMENT`)이 이 화면의 구성(문의 현황 표시 여부 · 기간별 분석의 열 구성)을 바꾼다.

**관련 API** — 목적 둘. ① 활성 탭의 할일·리스트 데이터 조회 ② 선택한 기간의 통계 조회. 두 조회는 **독립**이라 한쪽이 실패해도 다른 쪽을 가리지 않는다. **백엔드가 없다** — `apps/admin/src/pages/dashboard/api.ts` 와 `stats-api.ts` 가 로컬 mock 이며, 실제 API 가 붙으면 그 내부만 교체한다.

**참고 문서** — [ID 규약](../id-convention.md) · [인벤토리](../inventory.md) · [품질 기준](../../reference/quality-bar.md)

**구현 파일** — `apps/admin/src/pages/dashboard/DashboardPage.tsx` (탭 해소 `useDashboardTabs.ts` · 탭 패널 `components/DashboardTabPanel.tsx` · 문의 현황 `components/InquiryPulseSection.tsx` · 통계 `components/StatsSection.tsx` · 조회 `queries.ts` · mock `api.ts`·`stats-api.ts`)

## 4. 기능 명세

### 4.1 업무 탭 해소 — 권한으로 걸러진다

| 항목 | 내용 |
|---|---|
| 기능 목적 | 이 역할이 볼 수 있는 업무 영역만 탭으로 세운다 |
| 실행 조건 | 화면이 그려질 때마다 |
| 사용자 동작 | 없음(자동). 탭 클릭은 §4.2 |
| 시스템 처리 | 정적 탭 목록 3개를 권한 키(`dashboard.tab.products` · `dashboard.tab.inquiries` · `dashboard.tab.sales`)로 거른다 — `useDashboardTabs.ts:17-21,35`. **꺼진 탭은 목록에서 사라진다** — 누를 수 없는 탭을 남기지 않는다 |
| 데이터 처리 | 없음 |
| Validation | 해당 없음 |
| API 호출 여부 | N — 권한은 앱이 이미 들고 있다 |
| 성공 처리 | 보이는 탭이 1개 이상이면 탭 줄을 그리고 활성 탭의 데이터를 조회한다 |
| 실패 처리 | 실패 경로가 없다(네트워크를 타지 않는다) |
| 예외 처리 | 보고 있던 탭의 권한이 **도중에 꺼지면** 남아 있는 첫 탭으로 흘러간다 — 죽은 탭에 갇히지 않는다 — `useDashboardTabs.ts:38-39`. 보이는 탭이 0개면 탭 줄 자체를 그리지 않는다 |
| 화면 변경 사항 | Navigation · 탭 패널 |
| 후속 동작 | 활성 탭의 데이터 조회 |
| 로그 기록 여부 | N |

**권한 변경은 리로드 없이 반영된다** — 권한 저장소가 바뀌면 이 훅을 쓰는 화면이 다시 그려지므로, 별도 코드 없이 탭이 사라지거나 나타난다.

### 4.2 탭 전환

| 항목 | 내용 |
|---|---|
| 기능 목적 | 다른 업무 영역의 할일·목록을 본다 |
| 실행 조건 | 보이는 탭이 2개 이상일 때 |
| 사용자 동작 | 탭 클릭 또는 키보드 이동 |
| 시스템 처리 | 받은 탭 id 를 **보이는 탭 목록에서 되찾아** 좁힌다 — 권한이 없는 탭 id 는 여기서 걸러진다 — `DashboardPage.tsx:49-55` |
| 데이터 처리 | 그 탭의 데이터를 조회한다. 이전 탭의 데이터는 유지되지 않는다(§8 참고) |
| Validation | 해당 없음 |
| API 호출 여부 | Y — 그 탭의 할일·리스트 조회 |
| 성공 처리 | 탭 패널이 새 탭의 내용으로 바뀐다. **토스트를 띄우지 않는다** |
| 실패 처리 | 탭 줄은 그대로 두고 패널 자리에 실패 배너를 세운다(§7) |

| 세션 만료 | 이 화면에 자기 문구가 없다. 401 이 관측되면 로그인 화면으로 보내고 그 화면이 알린다 | 이 화면은 **여러 조회가 동시에 뜨므로 401 도 동시에 여러 번 온다** — 첫 통지만 통과시켜 이동이 한 번만 일어난다 — `apps/admin/src/shared/auth/session-expiry.ts:19-25` | 재로그인하면 이 화면으로 돌아온다 |
| 데이터 충돌 | 해당 없음 — 이 화면은 아무것도 쓰지 않는다 | — | — |
| 중복 데이터 | 해당 없음 — 아무것도 만들지 않는다 | — | — |
| 파일 업로드 실패 | 해당 없음 — 이 화면에 파일 업로드가 없다 | — | — |

**⚠ 지금 확인되는 격차** — 탭 조회 실패 배너에는 **'다시 시도' 컨트롤이 없다.** 통계 카드도 마찬가지로 재시도 손잡이가 없고, 기간 토글을 움직여야 다시 조회된다. 품질 기준은 조회 실패에 명시적 재시도 컨트롤을 요구한다(`docs/reference/quality-bar.md` STATE-02). 화면을 새로 고치는 것 말고 다른 복구 수단이 이 화면에는 없다.

**권한이 없을 때 화면이 어떻게 되는가** — 세 층이 다르게 움직인다.

| 없는 것 | 결과 |
|---|---|
| 화면 읽기 권한 | 본문이 `접근 권한이 없어요` 화면으로 대체된다. 사이드바·헤더는 남는다 |

| 위젯 권한 전부 | 화면이 안내 한 줄로 대체된다 |

**플랜(엔타이틀먼트) 축** — 대시보드 경로에는 모듈 매핑이 없어 잠금 판정이 `granted` 로 열린다(매핑이 없으면 연다 — `apps/admin/src/shared/entitlements/plan.ts:27-28`). 또한 판매하지 않는 모듈(`absent`)의 화면은 **대시보드로 되돌려 보내진다** — 즉 이 화면은 그 되돌림의 착지점이기도 하다 — `apps/admin/src/shared/entitlements/RequireEntitlement.tsx:111`.
