# 연혁

> **이 문서는 기업 관리 목록형 화면의 정본이다.** [ESG](../esg/index.md) · [채용 공고](../careers/index.md)가
> 같은 공용 껍데기(`useCrudList` + `CrudListShell`)를 쓰므로 그 문서들은 **차이만** 적는다.
> [인증서/특허](../certificates/index.md)는 드래그 재정렬 때문에 껍데기를 쓰지 못하고 같은 모양을
> 손으로 조립한다 — 그 문서가 어디가 같고 어디가 다른지 밝힌다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 연혁 |
| 화면 ID | `SCR-COMPANY-HISTORY` |
| 메뉴 경로 | 기업 관리 > 연혁 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 회사 연혁을 연·월 단위로 쌓아 두고, 등록·수정·삭제한다.

**업무 배경** — 연혁은 홈페이지 회사 소개의 뼈대다. 항목이 늘 때마다 개발자를 부를 수 없고, 순서는 사람이 정하지 않는다 — **연·월이 곧 순서**다.

**화면 설명** — 필터도 검색도 없다. 오른쪽 끝의 '연혁 등록' 하나뿐인 툴바 아래에 요약 건수 · (선택이 있으면) 선택 바 · 표가 온다. 페이지네이션이 없다.

**주요 사용자** — 최상위 관리자 · 기업 정보를 다루는 운영자.

**사용 시나리오**

1. 새 수상 이력이 생겨 '연혁 등록'을 눌러 연·월·내용을 적는다.
2. 오래된 항목의 내용을 다듬는다 → 행의 연필을 눌러 폼으로 간다.
3. 잘못 넣은 항목 여러 건을 체크해 한 번에 지운다.

**선행 조건** — 로그인. 기업 관리 잎은 채용 공고를 뺀 여덟이 플랜 모듈 매핑에 없어 플랜 축으로 잠기지 않는다 — `apps/admin/src/shared/entitlements/module-resources.ts:64-76`.

**후행 처리** — 저장된 연혁은 홈페이지 회사 소개에 쓰인다. 홈페이지는 이 리포 밖이라 반영 경로를 코드에서 확인하지 못했다.

**관련 화면** — [연혁 등록·수정](form.md)(`SCR-COMPANY-HISTORY-FORM`) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md). **상세 화면이 없다.**

**관련 메뉴** — 기업 관리 가지의 세 번째 잎 — `apps/admin/src/shared/layout/nav-config.ts:157`.

**관련 기능** — 없음.

**관련 API** — 목록 조회 1건 · 단건 삭제 1건. **백엔드가 없다** — `apps/admin/src/pages/company/history/data-source.ts:19`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/company/history/HistoryListPage.tsx` (껍데기 `apps/admin/src/shared/crud/CrudListShell.tsx` · 컨트롤러 `useCrudList.tsx` · 데이터 `data-source.ts` · 규칙 `types.ts`)

---

**⚠ 지금 코드에서 확인되는 사실** — 기업 관리의 목록형 화면은 콘텐츠 관리와 달리 **쓰기가 실제로 반영된다**(픽스처 배열을 갱신한다) — `apps/admin/src/shared/crud/crud.ts:118-170`. 없는 항목에 대한 수정·삭제도 조용히 통과시키지 않고 충돌로 막는다(§7).
