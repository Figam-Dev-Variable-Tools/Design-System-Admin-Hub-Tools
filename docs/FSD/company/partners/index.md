# 파트너사

> **파트너사와 [고객사](../clients/index.md)는 같은 화면이다.** 두 라우트가 한 컴포넌트를 공유하고,
> 다른 것은 **자원 이름과 엔티티 라벨과 데이터 소스 셋뿐**이다 — `apps/admin/src/pages/company/partners/PartnersPage.tsx:8` · `apps/admin/src/pages/company/clients/ClientsPage.tsx:8`.
> 그래서 이 문서가 그 화면의 정본이고, [고객사](../clients/index.md)는 차이만 적는다.
>
> 등록·수정 팝업은 두 화면이 함께 쓰므로 [로고 등록·수정](../../_common/pop-logo-form.md)(`POP-COMMON-LOGO-FORM`)이 갖는다.

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 파트너사 |
| 화면 ID | `SCR-COMPANY-PARTNERS` |
| 메뉴 경로 | 기업 관리 > 파트너사 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 홈페이지의 파트너사 로고 띠에 걸릴 항목을 모아 두고, **보일 순서와 노출 여부를 목록에서 바로** 다루며, 등록·수정·삭제한다.

**업무 배경** — 로고 띠는 회사 소개의 신뢰 장치다. 로고가 늘고 줄 때마다 개발자를 부를 수 없고, **순서가 곧 의미**라 페이지를 나누지 않고 전부 한 화면에 그려 드래그로 옮긴다 — `apps/admin/src/pages/company/logo-list/LogoListPage.tsx:6-8`.

**화면 설명** — 툴바에 검색과 '`파트너사` 추가'. 그 아래 요약 · (선택이 있으면) 선택 바 · 표가 온다. 각 행에는 노출 토글이 있고, 검색어가 없을 때는 행을 끌어 순서를 바꿀 수 있다. 등록·수정은 목록 위에 뜨는 팝업이 맡는다.

**주요 사용자** — 최상위 관리자 · 기업 정보를 다루는 운영자.

**사용 시나리오**

1. 새 파트너사를 '파트너사 추가'로 등록한다 → 목록 맨 끝에 **노출 상태로** 붙는다.
2. 중요한 파트너사를 맨 앞으로 끌어 올린다 → 놓는 즉시 저장되고 토스트가 뜬다.
3. 계약이 끝난 파트너사의 토글을 꺼 홈페이지에서만 내린다(데이터는 남긴다).

**선행 조건** — 로그인. 기업 관리 잎은 채용 공고를 뺀 여덟이 플랜 모듈 매핑에 없어 플랜 축으로 잠기지 않는다 — `apps/admin/src/shared/entitlements/module-resources.ts:64-76`.

**후행 처리** — 저장된 목록이 홈페이지의 로고 띠에 쓰인다. 홈페이지는 이 리포 밖이라 반영 경로를 코드에서 확인하지 못했다.

**관련 화면** — [고객사](../clients/index.md)(`SCR-COMPANY-CLIENTS` — 같은 화면) · [로고 등록·수정](../../_common/pop-logo-form.md)(`POP-COMMON-LOGO-FORM`) · 공통 규약은 [공통 확인 다이얼로그 · 토스트](../../_common/index.md). **상세 화면이 없다.**

**관련 메뉴** — 기업 관리 가지의 여섯 번째 잎 — `apps/admin/src/shared/layout/nav-config.ts:160`. 고객사는 바로 다음 잎이다.

**관련 기능** — 없음. 두 화면은 **각자 자기 주소를 가지므로 검색어가 서로 섞이지 않는다** — `apps/admin/src/pages/company/logo-list/LogoListPage.tsx:9-10`.

**관련 API** — 목록 조회 1건 · 등록 1건 · 수정 1건 · 삭제 1건 · 순서 저장 1건 · 노출 토글 1건. **백엔드가 없다** — `apps/admin/src/pages/company/partners/data-source.ts:47`.

**참고 문서** — [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/company/partners/PartnersPage.tsx` → `apps/admin/src/pages/company/logo-list/LogoListPage.tsx` (표 `apps/admin/src/pages/company/logo-list/LogoListTable.tsx` · 팝업 `apps/admin/src/pages/company/logo-list/LogoFormModal.tsx` · 어댑터 `apps/admin/src/pages/company/logo-list/adapter.ts` · 순수 규칙 `apps/admin/src/pages/company/logo-list/types.ts`)

리소스 키는 경로에서 파생되므로 **파트너사와 고객사는 서로 다른 권한 키를 갖는다** — `apps/admin/src/shared/permissions/resources.ts:66-68`. 노출 토글과 순서 저장은 수정(`update`) 축이고, 업로드는 등록·수정 권한에 딸린다.

**권한이 없을 때 화면이 어떻게 되는가** — 읽기 권한이 없으면 `접근 권한이 없어요` 화면이 뜬다. ⚠ **쓰기 권한이 없어도 추가·연필·토글·드래그·삭제가 그대로 보이고 눌린다** — 이 화면은 공용 목록 껍데기를 쓰지 않고 권한 판정도 부르지 않는다. 실제 차단은 서버의 몫이며, 프런트 게이팅은 보안 경계가 아니라 UX 다 — `apps/admin/src/shared/permissions/RequirePermission.tsx:8-11`.
