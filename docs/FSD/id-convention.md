# 화면 ID 규약

기능 명세서 §1 의 **화면 ID** 칸과, 문서가 놓이는 **폴더·파일 이름**을 정한다.

---

## 1. 한 문장으로

**화면 ID 는 라우트 경로다.** 앞에 종류 접두를 붙이고, 대문자로 적는다.

```
/users/members            →  SCR-USERS-MEMBERS
/users/members/:id        →  SCR-USERS-MEMBERS-DETAIL
/company/history/new      →  SCR-COMPANY-HISTORY-FORM
   + /company/history/:id/edit  (같은 화면이다 — 아래 §3)
```

## 2. 왜 경로인가 — 번호를 쓰지 않는 이유

옛 체계는 `FS-041` 같은 일련번호였다. 세 가지가 아팠다:

| 아픈 자리 | 무슨 일이 일어났나 |
|---|---|
| 번호와 화면 사이에 **아무 관계도 없다** | `086` 이 채용 공고라는 것을 알려면 색인을 봐야 했다 |
| 화면이 사라지면 **결번 표**가 생긴다 | 번호를 재사용하지 않기로 하니, 왜 비었는지 설명하는 문서가 따로 필요해졌다 |
| 화면이 늘면 **번호대가 어긋난다** | 기업 관리인데 `086`, 시스템 설정인데 `078` — 섹션과 번호가 갈라진 것을 설명하는 문단이 색인에 있었다 |

경로를 쓰면 셋 다 성립하지 않는다. **화면이 늘거나 줄어도 다른 화면의 ID 는 하나도 바뀌지 않는다.**

경로는 이미 이 코드베이스가 **바꾸지 않기로 못박은 내부 식별자**다:

> `[경로는 라벨을 따라 바꾸지 않는다] 경로는 북마크와 권한 리소스 키(navPageResourceId)가 매달린
> 내부 식별자다 — 라벨을 고칠 때 같이 바꾸면 저장된 역할의 권한 키가 어긋나 화면이 통째로 안 보인다.`
> — `apps/admin/src/shared/layout/nav-config.ts:291-293`

메뉴 **라벨은 실제로 바뀐다**(같은 파일: `API Key 설정` ↔ `AI 모델 연동`). 경로는 바뀌지 않는다.
그래서 ID 는 라벨이 아니라 경로에서 나온다. 한국어 화면명은 명세 §1 의 **화면명** 칸이 갖는다.

## 3. 접두 넷

| 접두 | 무엇 | 예 |
|---|---|---|
| `SCR-` | 라우트를 가진 화면 (목록 · 상세 · 등록/수정 · 보조 목록) | `SCR-ORDERS-CLAIMS-DETAIL` |
| `POP-` | 팝업 — 자기 입력·검증을 가진 모달 | `POP-USERS-MEMBERS-CREATE-GROUP` |
| `TAB-` | 탭 — 같은 라우트에서 **다른 자료**를 보는 화면 | `TAB-SETTINGS-PLAN-BILLING` |
| `MSG-` | 안내 토스트 | `MSG-USERS-MEMBERS-01` |

### 3.1 꼬리 셋

| 꼬리 | 언제 붙는가 |
|---|---|
| `-DETAIL` | 읽기 전용 상세(`/:id`) |
| `-FORM` | 등록·수정. **`/new` 와 `/:id/edit` 가 같은 컴포넌트면 한 화면이다** — 문서도 하나다 |
| (없음) | 목록·단일 문서형 화면 |

`/new` 와 `/:id/edit` 를 한 문서로 접는 근거는 코드다: 같은 컴포넌트가 `:id` 유무로 등록/수정을 겸한다
(`apps/admin/src/App.tsx` 의 `element`). 문서를 둘로 가르면 같은 검증 규칙이 두 곳에 적히고, 반드시 한쪽만 갱신된다.
**두 라우트가 다른 컴포넌트면 두 문서다** — 판정 근거는 추측이 아니라 `App.tsx` 다.

### 3.2 라우트 파라미터 이름은 ID 에 넣지 않는다

`:id` · `:target` · `:provider` · `:providerId` 는 전부 `-DETAIL` 로 접는다.
파라미터 이름이 바뀌어도 화면은 같은 화면이다.

### 3.3 리다이렉트 라우트는 화면이 아니다

`/products/returns` → `/orders/claims` 같은 `<Navigate>` 6건은 ID 를 갖지 않는다.
도착 화면 명세 §1 의 **관련 화면** 칸에 '이 화면으로 오는 옛 경로' 로 적는다.

### 3.4 토스트 번호

토스트는 라우트도 컴포넌트도 아니라서 경로에서 ID 가 나오지 않는다.
**소속 화면 ID + 두 자리 일련번호**를 쓴다: `MSG-USERS-MEMBERS-01`.

- 번호는 **그 문서 안에서만** 산다. 다른 문서와 겹쳐도 상관없다(접두가 다르다).
- 토스트가 사라지면 그 행을 지우고 **번호는 다시 쓰지 않는다.** 문서 안의 빈 번호는 관리 대상이 아니다 —
  전역 번호가 아니므로 결번 표가 필요 없다.

## 4. 폴더와 파일

운영자 규칙: **폴더 > 페이지(목록) > 페이지(등록) > 팝업(토스트, 모달)**.
디렉터리가 그 계층을 그대로 따른다.

```
docs/FSD/
  users/                       ← 폴더 = 사이드바 가지 (경로 세그먼트)
    members/                   ← 잎 하나 = 디렉터리 하나
      index.md                 SCR-USERS-MEMBERS          (목록)
      detail.md                SCR-USERS-MEMBERS-DETAIL   (상세)
      pop-create-group.md      POP-USERS-MEMBERS-CREATE-GROUP
      pop-password-change.md   POP-USERS-MEMBERS-PASSWORD-CHANGE
    admins/
      index.md · detail.md · form.md · pop-create-group.md
  settings/
    plan/
      index.md · tab-usage.md · tab-billing.md · tab-history.md
  _common/                     ← 여러 화면이 공유하는 팝업·다이얼로그 (§5)
```

| 파일 | 화면 |
|---|---|
| `index.md` | 그 잎의 **목록·단일 문서형 화면** |
| `detail.md` | 상세 |
| `form.md` | 등록·수정 |
| `<정적 세그먼트>.md` | 잎에 딸린 보조 목록. 예: `products/coupons/issuance.md` |
| `tab-<탭 id>.md` | 탭 |
| `pop-<이름>.md` | 팝업 |

**폴더 이름은 경로 세그먼트다**(`users` · `content` · `settings`), 한국어 메뉴명이 아니다.
같은 이유다 — 메뉴명은 바뀌고 경로는 바뀌지 않는다. 한국어 이름은 [`inventory.md`](inventory.md) 가 표로 잇는다.

**폴더는 15개**다 — `nav-config.ts` 의 최상위 항목 15개(가지 14 + 잎 `대시보드` 1)와 1:1 이고,
셸 밖의 `login` 하나가 더 붙는다.

## 5. 무엇이 파일이고 무엇이 절인가 — 이 판정이 문서 수를 정한다

| 종류 | 판정 | 근거 |
|---|---|---|
| **팝업(모달)** — 입력·검색·선택을 담은 것 | **별도 파일** | 자기 컴포넌트·자기 검증·자기 이벤트·자기 예외를 갖는다. 양식 10절이 그대로 성립한다 |
| **확인 다이얼로그**(`ConfirmDialog`) | **부모 문서의 절** | 제목·본문·버튼 둘이 전부다. 부모의 §5 이벤트(행) · §7 예외 처리(행) · §10 화면 이동(팝업 여부 `Y`)에 적는다. 코드에 **51건**이 있고, 각각에 10절 문서를 만들면 아홉 절이 빈다 |
| **여러 화면이 공유하는 팝업·다이얼로그** | **`_common/` 에 한 번** | 부모마다 반복해 적으면 그것이 곧 누적 부담이다 — 이번 재작성이 없애려는 문제와 같은 모양이다. 부모는 §10 에서 참조만 한다 |
| **안내 토스트** | **부모 §4 기능 명세의 성공 처리·실패 처리 칸** | 토스트는 화면이 아니라 **한 기능의 결과 통지**다. 양식이 이미 그 칸을 갖고 있으므로 절을 새로 만들지 않는다. 대신 **빠짐없이** 적혔는지는 [`inventory.md`](inventory.md) 의 화면별 토스트 건수와 대조해 확인한다 |
| **탭** — 다른 자료를 보는 것 | **별도 파일** | 예: 플랜의 `이용현황`/`결제`/`결제내역`. 서로 다른 자료를 서로 다른 규칙으로 그린다 |
| **탭** — 같은 목록을 거르는 것 | **부모 §3 UI 컴포넌트** | 예: API Key 화면의 분류 필터 탭. 필터이지 화면이 아니다 |
| **패널·카드·구획** | **부모 §2 페이지(UI) 구성** | 화면의 영역이다 |

## 6. 상호 참조

다른 문서를 가리킬 때 쓸 수 있는 것은 둘뿐이다.

| 쓴다 | 형태 |
|---|---|
| **화면 ID** | `SCR-USERS-MEMBERS-DETAIL` — 표의 '이동 화면'·'관련 화면' 칸 |
| **상대 경로 링크** | `[회원 상세](../members/detail.md)` 형태 — 파일을 직접 가리킨다 |

**절 번호로 남의 문서를 가리키지 않는다.** `FSD-041 §3` 같은 참조는 절이 하나 늘면 조용히 틀린다.
같은 문서 **안에서** 절을 가리키는 것은 괜찮다 — 한 파일이라 함께 고쳐진다.

## 7. 문서가 자기 역사를 쓰지 않는다

명세는 **지금의 코드**를 기술한다. 무엇이 언제 바뀌었는지는 git 이 알고, 왜 그렇게 정했는지는 [`../adr/`](../adr/) 가 안다.

옛 `specs/**` 에서 실측된 누적 관용구다. **새 문서에 한 건도 없어야 한다.**

| 금지 | 옛 문서에서의 건수 | 대신 |
|---|---|---|
| 미결을 번호로 되짚는 역참조 (`§7 #4`) | 3,272 | 미결은 **그 칸 안에서** 한 문장으로 말한다. 다른 절을 가리키지 않는다 |
| ⚠ 를 이력 표식으로 쓰기 | 2,067 | ⚠ 는 **지금 위험한 사실**에만 |
| 날짜 도장 (`2026-07-nn`) | 896 | 지운다 |
| 취소선 (`~~…~~`) | 399 | 지운다 |
| `해소됨` · `해소했` · `해소된` | 233 | 지운다 |
| `범위 정정` · `이관 취소` | 111 | 지운다 |
| `결번` | 23 | 성립하지 않는다(§2) |

문서 버전은 §1 의 **버전** 칸 하나가 갖는다. 본문에 변경 이력을 적지 않는다.
