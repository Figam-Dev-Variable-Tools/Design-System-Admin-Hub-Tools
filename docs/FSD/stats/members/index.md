# 회원 통계

> **조회 조건 축과 껍데기는 [방문자 통계](../visitors/index.md) 와 같다.** 조회 조건 바(기간 프리셋 ·
> 직접 입력 · 비교 기준 · 세그먼트 · 내보내기) · 조회 상태 4분기 · KPI 줄 · 추이 카드 · 드릴다운 ·
> 빈 상태 · 권한 게이팅이 전부 한 벌에서 나온다.
> **이 문서는 그 한 벌을 다시 적지 않고 이 화면의 차이만 적는다.**

## 1. 화면 개요

| 항목 | 내용 |
|---|---|
| 화면명 | 회원 통계 |
| 화면 ID | `SCR-STATS-MEMBERS` |
| 메뉴 경로 | 분석 · 운영 > 통계 > 회원 통계 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |

**화면 목적** — 가입 경로별로 신규 가입과 탈퇴를 갈라 보고, 순증과 누적 회원 수의 흐름을 비교 기간과 견준다 — `apps/admin/src/pages/stats/members/MemberStatsPage.tsx:242`.

**업무 배경** — 신규 가입만 세면 탈퇴가 같은 수로 빠져나가도 화면은 '성장 중'으로 읽힌다. 회원이 실제로 늘었는지에 답하는 것은 순증뿐이라 파생 계산이 아니라 1급 지표로 세운다 — `apps/admin/src/pages/stats/members/types.ts:12-16`. 또한 국내 커머스의 가입은 소셜 로그인으로 크게 기울어, 경로별로 가르지 않으면 '카카오 가입이 막혔다' 같은 사고가 전체 평균에 묻혀 며칠씩 보이지 않는다 — `members/types.ts:18-20`.

**화면 설명** — 배치는 다른 통계 화면과 같다. 다른 것은 세그먼트가 `가입 경로`, 드릴다운 축이 `일자별`·`등급별` 이고, **등급별은 표가 아니라 구성비 막대**라는 점이다.

**주요 사용자** — 최상위 관리자 · 마케팅/CRM 담당 운영자.

**사용 시나리오**

1. 기간을 `최근 30일` 로 두고 KPI 의 `순증` 이 음수인 날이 있는지 표에서 확인한다.
2. 세그먼트를 `카카오` 로 좁혀 특정 경로의 가입이 마르지 않았는지 본다.
3. 드릴다운 축을 `등급별` 로 바꿔 회원이 어느 등급에 쏠려 있는지 막대 길이로 읽는다.
4. 조건 그대로 `엑셀 내보내기` 로 받는다(축에 따라 다른 파일이 나간다).

**선행 조건** — 로그인. 이 화면의 읽기 권한.

**후행 처리** — 아무것도 바꾸지 않는다.

**관련 화면** — 같은 껍데기: `SCR-STATS-VISITORS` · `SCR-STATS-REVENUE` · `SCR-STATS-ORDERS` · `SCR-STATS-TRAFFIC` · `SCR-STATS-KEYWORDS`. 회원 한 사람을 찾고 고치는 화면은 `SCR-USERS-MEMBERS` 이며, **이 화면에서 개별 회원으로 들어갈 수 없다**(열 이유가 없다) — `MemberStatsPage.tsx:3-8`. 등급 정책은 `SCR-USERS-SETTINGS` 가 갖는다.

**관련 메뉴** — 사이드바 `분석 · 운영 > 통계` 의 두 번째 항목 — `apps/admin/src/shared/layout/nav-config.ts:268`. **플랜으로 잠기지 않는다** — 심화 리포트 세 화면만 상위 플랜의 것이다.

**관련 기능** — 고객 설정(`SCR-USERS-SETTINGS`)이 등급을 만들고 조정하는 화면이다. ⚠ 그러나 이 화면의 등급 어휘는 그것과 다르다(§4.4).

**관련 API** — 목적 하나: 조회 기간과 비교 기간의 회원 집계(가입 경로별 일자별 · 등급 구성)를 한 번에 받는다. 쓰기 목적의 호출이 없다. 경계는 `apps/admin/src/pages/stats/members/data-source.ts` 의 `TODO(backend)` 다.

**참고 문서** — [방문자 통계](../visitors/index.md)(껍데기 정본) · [ID 규약](../../id-convention.md) · [인벤토리](../../inventory.md)

**구현 파일** — `apps/admin/src/pages/stats/members/MemberStatsPage.tsx` · `members/types.ts` · `members/data-source.ts` (공유 껍데기는 [방문자 통계](../visitors/index.md) 의 구현 파일 표)

## 4. 기능 명세

기능의 목록과 절차는 [방문자 통계](../visitors/index.md) 와 같다: 조회 · 기간 선택 · 비교 기준 · 세그먼트 · 추이 지표 · 드릴다운 · 정렬 · 페이지 · 내보내기 · 권한 판정. **성공·실패·예외의 처리 방식도 같다.** 아래는 이 화면에서만 달라지는 값이다.

### 4.4 ⚠ 등급 어휘가 도메인과 어긋난다 — 선재 결함

| 항목 | 내용 |
|---|---|
| 무엇이 어긋났는가 | 이 화면은 회원 등급을 **네 종류**로 들고 있다: `일반` · `실버` · `골드` · `VIP` — `apps/admin/src/pages/stats/members/types.ts:44-56`. 그런데 이 앱의 회원 등급 정본은 **세 종류**다: `일반회원` · `VIP` · `VVIP` — `apps/admin/src/shared/domain/member.ts:20-26` |
| 왜 결함인가 | 같은 '회원 등급'이라는 말이 화면마다 다른 목록을 가리킨다. 회원 관리(`SCR-USERS-MEMBERS`)의 등급 필터에는 `실버`·`골드` 가 없고, 이 화면에는 `VVIP` 가 없다. 운영자가 이 화면에서 본 `골드 320명` 을 회원 목록에서 찾으면 그 등급 자체가 없다 |
| 어떻게 드러나는가 | 등급별 구성비 막대의 항목 이름과 CSV 의 `등급` 열에 그대로 나간다 |
| 지금 코드의 상태 | 주문 통계는 같은 문제를 **이미 해결했다** — 주문 상태 어휘를 도메인 층으로 올리고 화면은 그것을 재수출한다(`apps/admin/src/pages/stats/orders/types.ts:20-26`). 회원 등급은 그 수렴이 되지 않은 채 로컬 사본으로 남아 있다 |
| 명세의 판단 | **이 문서는 코드에 있는 그대로 네 등급을 적는다.** 다만 그것이 도메인과 어긋난 상태임을 여기 표시한다 — 명세가 코드보다 앞서 세 등급으로 적으면 개발자와 QA 가 화면에 없는 값을 기준으로 삼게 된다 |

### 4.5 CSV 내보내기 — 축마다 다른 파일

| 항목 | 내용 |
|---|---|
| 기능 목적 | 지금 조건과 축에 걸린 전체를 파일로 받는다 |
| 실행 조건 | `export` 권한 + 내보낼 행 1건 이상 |
