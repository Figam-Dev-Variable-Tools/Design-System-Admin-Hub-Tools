# 비기능 명세서 (NFRS)

성능 · 사용성 · 보안 · 안정성 · 호환성 · 운영성 · 유지보수성 · 제약사항 **8개 절**로 기술한다.
절 이름과 순서는 [`_template.md`](_template.md) 가 정본이며 바꾸지 않는다.

---

## 1. 작성 단위 — 자원(사이드바 잎) 하나가 문서 하나

기능 명세서는 **화면마다** 쓴다(목록·상세·등록·팝업·탭이 각각).
비기능 명세서는 **자원마다** 쓴다 — 사이드바 잎 하나가 문서 하나다.

**왜 화면 단위가 아닌가**

| 사실 | 결과 |
|---|---|
| 목록·상세·등록은 **같은 데이터·같은 권한 축·같은 API 군**을 쓴다 | 성능 목표와 보안 요구가 셋에 똑같이 걸린다. 셋으로 가르면 같은 문장이 세 곳에 적히고 반드시 한 곳만 갱신된다 |
| 8절 중 ⑤호환성 · ⑥운영성 · ⑦유지보수성은 거의 전부 **리포 전역**이다 | 화면 126개로 쪼개면 '지원 브라우저' 문장이 126번 반복된다. 브라우저 지원 범위가 한 번 바뀌면 126곳을 고쳐야 한다 |

그래서 구조는 **공통 한 벌 + 자원별 차이**다.

```
docs/NFRS/
  README.md      ← 지금 이 문서
  _template.md   ← 8절 템플릿
  _common.md     ← 전역 기준 한 벌 (전 자원이 공통으로 지키는 것)
  <폴더>/<잎>.md  ← 그 자원의 수치·차이·제약
```

**규칙 N-1.** 자원 문서의 각 절은 **`공통 준수` 한 줄 + 이 자원만의 차이**를 적는다.
공통과 같은 내용을 다시 쓰지 않는다.

**규칙 N-2.** 차이가 없으면 `공통 준수 — 이 자원의 추가 요구 없음` 한 줄이면 된다.
빈 절을 남기지 않는다 — 빈 절은 '요구가 없다' 와 '아직 안 봤다' 를 구분하지 못한다.

**규칙 N-3.** 재지 않은 값을 적지 않는다. `측정하지 않음` 이라고 쓴다. **측정 불가는 통과가 아니다.**

## 2. 폴더와 파일

기능 명세서와 같은 계층을 쓰되, 잎 하나가 **파일 하나**다(잎 밑에 디렉터리를 만들지 않는다).

| 자원 | 파일 |
|---|---|
| 회원 관리 (`/users/members`) | `docs/NFRS/users/members.md` |
| 플랜·이용 현황 (`/settings/plan`) | `docs/NFRS/settings/plan.md` |
| 대시보드 (`/dashboard`) | `docs/NFRS/dashboard.md` |

**대상은 76건**이다 — 사이드바 잎 74 + 로그인 1 + 쿠폰 발급 현황 1(잎이 아닌 보조 목록이지만
자기 조회·자기 대량 데이터 특성을 갖는다). 전수 목록은 [`../FSD/inventory.md`](../FSD/inventory.md) 가 갖는다.

## 3. 품질 기준의 정본

수치와 판정 기준은 [`../reference/quality-bar.md`](../reference/quality-bar.md)(9차원 102요구사항)가 갖는다.
비기능 명세서는 그 기준을 **이 자원에 대고 판정한 결과**를 적는다. 기준 자체를 여기서 새로 정의하지 않는다.

## 4. 기능 명세서와의 경계

| 이것은 | 어디에 |
|---|---|
| 오류 메시지 문구 · 버튼 비활성 조건 · 화면 상태 전이 | **기능 명세서** (§6·§7·§8) |
| 응답 시간 목표 · 동시 사용자 수 · 대용량 처리 한계 · 지원 브라우저 · 로그 보존 | **비기능 명세서** |
| 권한별 조회/등록/수정/삭제 가능 여부 | **기능 명세서 §9** (표) |
| 인증 방식 · 개인정보 마스킹 규칙 · 입력값 검증 정책 | **비기능 명세서 ③보안** |

같은 사실을 두 문서에 적지 않는다. 겹치면 링크한다.
