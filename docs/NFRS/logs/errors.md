# 오류 로그 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 오류 로그 |
| 화면 ID | `SCR-LOGS-ERRORS` |
| 메뉴 경로 | 분석 · 운영 > 로그 관리 > 오류 로그 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [오류 로그](../../FSD/logs/errors/index.md) · 공유 팝업 [로그 원문 보기](../../FSD/_common/pop-log-payload.md) |

**이 자원의 성격** — 로그 4화면이 공유하는 껍데기 위의 화면이다. 성능·사용성·안정성·호환성·유지보수성의 값은 [관리자 로그](admin.md) 와 **같다** — 그 문서가 껍데기 한 벌을 갖는다. 이 문서는 **이 자원만의 차이**를 적는다. 차이의 뿌리는 하나다: **행위자가 없다** — 아무도 하지 않았는데 일어난 일을 담는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 전 항목 | [관리자 로그](admin.md) 준수 | 같다 — 같은 껍데기·같은 조회 엔진·같은 페이지 상한(100행, 기본 20) | 코드 확인 |
| 행 수와 사건 수의 분리 | — | 한 행이 여러 사건일 수 있다 — `발생 횟수` 가 그 수를 갖는다. **1회와 340회는 완전히 다른 사건인데, 이 숫자가 없으면 표에서 둘이 똑같이 한 줄로 보인다** — `apps/admin/src/pages/logs/errors/types.ts:47-50`. 즉 페이지 상한 100행은 사건 100건이 아니다 | 코드 확인 |
| 데이터 부피 | — | API 로그보다 작고 관리자 로그보다 크다 — 보존기간(180일)이 그 중간값의 근거로 기술되어 있다 — `errors/types.ts:98-101` | 코드 확인 |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 전 항목 | [관리자 로그](admin.md) 준수 | 같다 |
| 경고를 오류와 섞지 않기 | — | 심각도가 `치명` · `오류` · `경고` 셋으로 갈린다. **섞으면 진짜 사고가 잡음에 묻히고, 그것이 '오류 로그를 아무도 안 보는' 가장 흔한 이유다** — `errors/types.ts:17-19` |
| 정렬이 심각한 순서로 읽히기 | — | 심각도 정렬은 사전순이 아니라 `치명 → 오류 → 경고` 순위를 따른다. 사전순이면 '경고'가 맨 위로 온다 — `errors/types.ts:29-34` |
| 상태를 색으로만 전달하지 않기 | 공통 준수 | 심각도는 색 위에 **글자**가 있다 — 색을 못 봐도 '치명'은 치명으로 읽힌다 — `apps/admin/src/pages/logs/errors/ErrorLogPage.tsx:28-40` |
| 숫자 컬럼 표기 | — | `발생 횟수` 가 우측 정렬 + 등폭 숫자다 — `apps/admin/src/shared/ui/styles.ts:392-396` |
| 요약 경고의 우선순위 | — | 치명을 먼저, 없으면 오류를 말한다. **`경고` 만 있으면 요약 줄이 비어 있다** — 경고 300건보다 치명 1건이 급하다 — `ErrorLogPage.tsx:96-102` |
| 개발자에게 넘길 것 | — | 상세에 `추적 ID` 가 있고 **가리지 않는다.** 운영자가 이것을 복사해 개발자에게 준다 — 없으면 '오류 났어요'로 끝나 아무도 못 찾는다 — `errors/types.ts:52-56` |
| 빈 결과의 의미 | — | 이 화면에서 0건은 **좋은 소식**이다. 그럼에도 화면은 다른 화면과 같은 빈 상태 카드로 그린다(축하하지 않는다) |
| 나가는 길 | — | ⚠ 표에서 다른 화면으로 가는 링크가 **없다.** 고치는 일은 이 앱 밖(이슈 트래커)에서 하고, 그 연결이 코드에 배선되어 있지 않다 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 권한 | [관리자 로그](admin.md) 준수 | 리소스 키는 `/logs/errors` 이고 의미 있는 액션은 `read`·`export` 둘뿐이다 |
| 불변성 | [관리자 로그](admin.md) 준수 + 강화 | 쓰기 함수가 없는 것에 더해, **'해결됨' 토글도 없다.** 오류 추적 도구에는 있는 그 토글을 두지 않는 이유가 명시되어 있다: 일어난 일을 나중에 덧칠할 수 있다면 그것은 이미 불변 기록이 아니다 — `errors/types.ts:9-14` |
| 스택의 노출 | — | 스택·컨텍스트가 원문으로 표시되며 공통 마스킹 규칙을 거친다. **내부 구조가 담기는 화면**이므로 읽기 권한의 의미가 다른 화면보다 크다 |
| 오류 참조 코드와의 관계 | — | 화면이 사용자에게 보이는 짧은 참조 코드(공통 규칙)와 이 화면의 `추적 ID` 는 같은 목적의 값이다 — 서버 응답 원문·스택을 사용자 문장으로 노출하지 않되 대조는 가능하게 한다 |
| 입력값 검증 | [관리자 로그](admin.md) 준수 | 같다 |

## 6. 운영성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 보존기간 | 공통은 '측정하지 않음' | **180일.** 근거 `재발 추적에 필요한 기간. 보존기간이 지나면 자동 폐기돼요.` — `errors/types.ts:102-105`. API 로그(90일)보다 길고 관리자 로그(3년)보다 짧다 — 부피는 크지만 '작년에도 이랬나'(재발 판단)를 물을 일이 실제로 있고 반기면 대개 답이 된다 — `errors/types.ts:98-101` |
| ⚠ 보존기간과 조회 상한의 어긋남 | — | 보존은 180일인데 **한 번에 조회할 수 있는 구간은 90일**이다 — `apps/admin/src/pages/logs/types.ts:89`. '작년에도 이랬나'를 묻는 재발 판단은 **두 번 조회해야** 답이 나온다 |
| ⚠ 폐기의 주체 | — | [관리자 로그](admin.md) 와 같다 — 집행 코드가 리포에 없다 |
| 모니터링 | 공통 준수 | 없다. **이 화면이 사실상 유일한 오류 관측 수단**이며, 치명 오류 알림·임계 통지가 코드에 없다 — 운영자가 이 화면을 직접 열어야만 안다 |
| 배포 영향 | — | 껍데기를 고치면 4화면이 함께 바뀐다 |
| 재현 손잡이 | 공통 준수 | `?fail=list` · `?fail=export` · `?fail=all` · `?fail=logs-errors:list` — `apps/admin/src/pages/logs/adapter.ts:17-19` · `errors/data-source.ts:15` |
| 설정 관리 | — | 심각도 구분·발생 위치 목록이 **코드 상수**이며 어드민에서 바꿀 수 없다 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 전 항목 | [관리자 로그](admin.md) 준수 — 백엔드 없음 · 쓰기 API 없음 · 조회 기간 최대 90일 · 미래 조회 불가 · 페이지 100행 · CSV 단일 형식 · 상세는 링크로 공유 불가 |
| 보존기간 | 180일(§6). 조회 상한 90일과 어긋난다 |
| 상태 관리 | **'해결됨' 개념이 없다.** 이슈 트래커가 아니므로 처리 여부를 이 화면에서 추적할 수 없다 |
| 심각도 | `치명` · `오류` · `경고` 셋 고정 |
| 발생 위치 | 다섯 고정 목록(§4) |
| 프런트 예외 | 이 어드민 자신의 렌더 예외는 담기지 않는다(§4) |
| 이동 경로 | 표에서 나가는 길이 **없다**(§2) |
| 파일 용량 제한 · 업로드 확장자 | 해당 없음 — 파일 입력이 없다 |
