# 회원 활동 로그 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 회원 활동 로그 |
| 화면 ID | `SCR-LOGS-MEMBER-ACTIVITY` |
| 메뉴 경로 | 분석 · 운영 > 로그 관리 > 회원 활동 로그 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [회원 활동 로그](../../FSD/logs/member-activity/index.md) · 공유 팝업 [로그 원문 보기](../../FSD/_common/pop-log-payload.md) |

**이 자원의 성격** — 로그 4화면이 공유하는 껍데기 위의 화면이다. 성능·사용성·안정성·호환성·유지보수성의 값은 [관리자 로그](admin.md) 와 **같다** — 그 문서가 껍데기 한 벌을 갖는다. 이 문서는 **이 자원만의 차이**를 적는다. 차이의 뿌리는 하나다: **여기 담기는 것은 남의 개인정보다.**

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 전 항목 | [관리자 로그](admin.md) 준수 | 같다 — 같은 껍데기·같은 조회 엔진·같은 페이지 상한(100행, 기본 20)·같은 캐시 규칙 | 코드 확인 |
| 데이터 부피 | — | 관리자 로그보다 크고 API 로그보다 작다(회원 수 × 활동). 코드에 이 자원만의 상한이 따로 없다 — 페이지 상한과 90일 조회 상한이 그대로 적용된다 | 코드 확인 |

| 숫자 컬럼 | — | 없다 — 우측 정렬·등폭 규칙이 적용되는 컬럼이 이 화면에는 하나도 없다 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 권한 | [관리자 로그](admin.md) 준수 | 리소스 키는 `/logs/member-activity` 이고 의미 있는 액션은 `read`·`export` 둘뿐이다 |
| ⚠ 개인정보에 대한 추가 권한 | — | **없다.** 4화면 중 개인정보 밀도가 가장 높은데도 권한 축은 다른 로그 화면과 동일하며, 열람 사유 입력·건수 상한·2인 승인 같은 추가 통제가 코드에 없다. 확인된 유일한 완화 조치는 보존기간을 1년으로 짧게 잡은 것이다 |
| 개인정보·비밀 마스킹 | [관리자 로그](admin.md) 준수 | 원문 마스킹 규칙은 같다. **이 화면에서 그 규칙의 부담이 가장 크다** — 주문 원문에는 배송지와 결제수단이 들어 있다 — `apps/admin/src/pages/logs/member-activity/types.ts:8-10` |
| 표시되는 개인정보 | — | 회원 계정(이메일)·이름이 표에 뜬다. 타입 주석은 마스킹된 채로 내려온다는 전제를 적고 있으나(`member-activity/types.ts:40-43`), **가리는 주체가 서버인지 화면인지는 코드에서 확인하지 못했다** — 화면 쪽에 계정·이름을 가리는 함수가 없다 |
| ⚠ 반출 데이터 | — | CSV 에 회원 계정·이름이 그대로 담긴다. 원문 열은 빠지지만 **식별 가능한 개인정보가 파일로 나간다** — 파일 취급 정책이 코드에 없다 |
| 불변성 | [관리자 로그](admin.md) 준수 | 같다 — 쓰기 함수가 존재하지 않는다 |
| 입력값 검증 | [관리자 로그](admin.md) 준수 | 같다 |

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 보존기간 | 공통은 '측정하지 않음' | **1년.** 근거 `개인정보 최소 보관 원칙. 보존기간이 지나면 자동 폐기돼요.` — `member-activity/types.ts:92-95`. 관리자 로그(3년)보다 훨씬 짧은 이유가 명시되어 있다: 관리자 로그는 '우리 직원이 무엇을 했나'라 오래 들고 있는 것이 통제이지만, **이 로그는 남의 개인정보라 오래 들고 있는 것 자체가 위험이고 비용이다** — `member-activity/types.ts:85-91` |
