# 쿠폰 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 쿠폰 |
| 화면 ID | `SCR-PRODUCTS-COUPONS` · `SCR-PRODUCTS-COUPONS-FORM` |
| 메뉴 경로 | 상품 관리 > 쿠폰 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../../FSD/products/coupons/index.md) · [등록·수정](../../FSD/products/coupons/form.md) · [발급 현황](../../FSD/products/coupons/issuance.md) |

**이 자원의 성격** — **축이 둘인 폼**이다(발급 기준 · 사용 대상). 두 축이 각각 조건부 입력을 갖고, 사용 대상은 다른 세 자원(카테고리·상품·회원 등급)의 목록을 읽어야 한다. 그래서 비기능 요구가 **'선택지를 모른다' 는 상태를 정직하게 다루는 일**과 **연쇄 선택의 초기화 규칙**에 몰린다. 아래 각 절은 [공통 기준](../_common.md) 대비 이 자원만의 차이를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수 | 목록 1건. 폼은 상세 1 + **조건부** 카테고리 1 + **조건부** 상품 1 | 코드 확인 |
| 불필요한 조회 억제 | — | **쓰지 않을 목록을 부르지 않는다**: 카테고리는 사용 대상이 카테고리·상품일 때만, 상품은 사용 대상이 상품일 때만 조회한다 — `CouponFormPage.tsx:283-295` |
| 대용량 데이터 처리 | 가상화 없이 1,000행을 그리지 않는다 | ⚠ 목록에 **페이지네이션도 가상화도 없다**. 폼의 등급 다중 선택은 **선택 영역만 스크롤**시킨다(상품이 수백 건이 되면 카드가 세로로 무한히 늘어난다) — `components/CouponTargetPicker.tsx:33-35` |
| 연쇄 선택의 후보 계산 | — | 상품 후보는 좌표(1차·2차)에서 매 렌더 다시 걸러진다. **1차조차 고르지 않았으면 빈 목록**이라 전 상품을 셀렉트에 쏟지 않는다 |
| 충돌 판정 | — | 목록의 **행마다** '이 쿠폰이 지목한 상품 중 쿠폰을 거부하는 것' 을 센다. 쿠폰 수 × 상품 수에 비례한다 | 코드 확인 |
| 재조회 중 표시 | 공통 준수 | 스켈레톤은 최초 로드에서만 | 코드 확인 |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 공통 CRUD 껍데기. 폼의 연쇄 선택과 등급 다중 선택은 **이 화면 전용 부품**이다 |
| 한 줄에 입력 하나 | — | 카드 안의 칸은 전부 세로로 쌓는다. 가로로 두는 것은 **한 쌍이 곧 하나의 값**인 날짜 범위뿐이다 — 시작일과 종료일은 따로 읽을 수 없다 |
| 값의 자리를 읽는 순서로 정한다 | — | '최소 주문 금액' 이 '사용 조건' 이 아니라 '쿠폰 정보' 에 있는 이유: 할인값·최대 할인과 **한 문장으로 읽힌다**("3만원 이상 사면 15%, 최대 2만원") — `CouponFormPage.tsx:538-545` |
| 조건이 없는 시점을 지어내지 않는다 | — | 여섯 발급 시점 중 셋만 더 묻는다. 나머지에 억지로 셀렉트를 붙이면 **저장은 되고 아무 데도 쓰이지 않는 값**이 생긴다. 대신 힌트가 '더 정할 조건이 없다' 고 못 박는다 — 안 그러면 운영자는 조건 칸이 안 뜨는 것을 화면 고장으로 읽는다 — `types.ts:58-63,405-413` |
| 목록 열이 파라미터를 싣는다 | — | '발급 기준' 열이 `{등급} 승급 시` · `생일 {N}일 전` 처럼 값까지 보인다 — 'VIP 승급' 과 'VVIP 승급' 은 다른 정책인데 둘 다 '회원 등급 승급 시' 로만 보이면 목록에서 구분할 수 없다 |
| 모른다 ≠ 비었다 | 공통 규약 5 준수 | 선택지를 못 읽으면 **빈 셀렉트를 그리지 않는다** — 빈 셀렉트는 '고를 것이 없다' 는 **완결된 문장**이라 운영자가 카테고리를 다시 만들러 간다. 대신 '지금은 고를 수 없다' 고 말하고 이유를 적는다 — `components/CouponScopePicker.tsx:12-14` |
| 숨기지 않고 이름을 적는다 | — | 발급 기준이 들 수 없는 등급(운영자가 추가한 등급)을 조용히 빼지 않고 **이름을 나열한다** — 등급 정책 화면이 반대편에서 같은 사실을 말하고 있고, 두 화면의 말이 어긋나면 운영자는 어느 쪽도 믿지 못한다 |
| 잠긴 버튼은 이유를 단다 | — | 저장 차단 사유가 배너와 **버튼의 접근성 이름**에 함께 실린다 |
| 경고는 저장을 막지 않는다 | — | 쿠폰↔상품 충돌은 경고다. 막아야 할 것은 **운영자가 모르는 채로 두는 것**이지 저장 자체가 아니다 — 특가가 끝나 상품 설정이 풀리면 이 쿠폰은 그대로 살아난다 |
| 결과 통지 | 공통 준수 | 목록의 발급 토글 2문구 + 공통 저장·삭제 통지 |
| 접근성 — 다중 선택의 필수 | — | 필수는 **개별 체크박스가 아니라 묶음**에 붙는다 — 필수인 것은 '어느 한 대상' 이 아니라 '고르는 행위' 다. 개별 체크박스에 필수를 붙이는 것은 거짓말이다 — `components/CouponTargetPicker.tsx:6-9` |
| 접근성 — 선택 수 통지 | — | 선택 수와 오류가 같은 자리에 있고, 묶음이 그 줄을 설명으로 가리킨다 |
| 미저장 보호 | 공통 준수 | 폼 전체에 이탈 가드 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽 |
| 권한 | 공통 준수 | 리소스 키는 사이드바 잎 `/products/coupons`. **발급 현황 라우트도 이 키를 쓴다**(그 화면은 자기 잎이 없다) |
| 권한 실패 방향 | 공통 준수 | `create` 가 없으면 등록 버튼이, `remove` 가 없으면 선택·일괄 삭제가 **사라진다** |
| 개인정보 마스킹 | 공통 준수 | 이 자원(목록·폼)에는 개인정보가 없다. 발급 현황의 회원 표시명은 그 문서가 다룬다 |
| 입력값 검증 | 공통 준수 | 규칙의 정본은 zod 스키마. **폼은 평평하고 도메인은 판별 유니온**이라, 스키마는 고른 종류에 필요한 칸만 검사하고 유니온으로 올리는 일은 별도 함수 한 곳이 한다 |
| zod 밖의 한 겹 | — | zod 는 폼 값만 본다 — **'선택지 목록을 읽지 못했다' 는 폼 값이 아니다.** 그 상태에서 저장을 열어 두면 운영자는 고를 수 없었던 것을 고르지 않은 채로 저장한다. 그래서 술어를 한 벌 더 두고 **버튼의 비활성과 제출 거부가 둘 다 그것을 읽는다** — 하나만 읽으면 키보드 Enter 나 URL 로 걸어 들어온 제출이 그대로 통과한다 — `types.ts:433-439` |
| 저장 시 정규화 | — | 쿠폰 코드는 **대문자로** 저장된다 |
| 화면에 보이지 않던 값을 저장하지 않는다 | — | 사용 대상을 바꾸면 이전에 고른 id 를 함께 버린다. 그러지 않으면 화면에는 카테고리 선택지가 뜨는데 폼은 상품 id 를 들고 있고, **그대로 저장하면 화면에 보이지 않던 값이 저장된다** — `CouponFormPage.tsx:336-343` |
| 중복 제출 | 공통 준수 | 공통 폼의 동기 잠금 + 멱등키 |
| HTTPS | 공통 준수 | 측정하지 않음 |

## 8. 제약사항
