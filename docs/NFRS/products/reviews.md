# 리뷰 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 리뷰 |
| 화면 ID | `SCR-PRODUCTS-REVIEWS` · `SCR-PRODUCTS-REVIEWS-DETAIL` |
| 메뉴 경로 | 상품 관리 > 리뷰 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../../FSD/products/reviews/index.md) · [상세](../../FSD/products/reviews/detail.md) |

**이 자원의 성격** — 고객이 만들고 관리자는 **노출 여부와 답변만** 정하는 자원이다. 결제 잠금과 무관하고 조회 축도 단순하다. 대신 상세에 **권한 게이팅과 검증 스키마가 둘 다 없어**, 이 자원의 비기능 요구는 그 두 구멍에 몰린다. 아래 각 절은 [공통 기준](../_common.md) 대비 이 자원만의 차이를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수 | 목록 1건 · 상세 1건. 필터·검색은 받아 온 배열 위에서 좁힌다 | 코드 확인 |
| 대용량 데이터 처리 | 가상화 없이 1,000행을 그리지 않는다 | ⚠ **페이지네이션도 가상화도 없다.** 리뷰는 상품 수에 비례해 자라는 자원이라 이 상한 부재가 실제로 문제가 된다 | 코드 확인 |
| 이미지 | — | **목록에는 이미지 열이 없다**(전 섹션 규칙). 포토 리뷰의 이미지는 상세의 미리보기에서만 그려진다 | 코드 확인 |
| 재조회 중 표시 | 공통 준수 | 스켈레톤은 최초 로드에서만 | 코드 확인 |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 공통 CRUD 껍데기. 상세는 카드 2단 + 전용 미리보기 부품이다 |
| 업무가 루프다 | — | 이 화면의 업무는 **낮은 별점 훑기 → 답변 → 다음**이라 조회 조건이 주소에 남는 것이 특히 중요하다. 예전에는 한 건을 처리하고 뒤로 오면 필터가 풀린 전체 목록에 착지해 방금 처리하던 줄을 처음부터 다시 찾아야 했다 — `ReviewListPage.tsx:6-11` |
| 행 클릭의 목적지 | — | 형제 목록 대부분이 수정 폼으로 가는 것과 달리 **상세**로 간다. 예전에는 그 사실이 이름 뒤에 숨어 **표 캡션이 '수정 화면' 이라고 잘못 읽었다** — 지금은 목적지를 명시적으로 준다 |
| 실시간 미리보기 | — | 상세의 미리보기는 **편집 중인 값**(답변·노출)과 저장된 값을 함께 그린다 — 저장 전에 고객 화면을 볼 수 있다 |
| 색만으로 말하지 않기 | — | 별점은 별 문자열이 장식이고 **뜻은 접근성 이름**(`5점 만점에 N점`)이 전한다 |
| 빈 사유를 빈칸으로 두지 않는다 | — | 신고 사유가 비면 `사유 미기재` 라고 적는다 |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 이 자원에는 `null` 로 오는 값이 없다. 답변이 비어 있는 것은 **'답변하지 않았다' 는 사실**이라 표시 배지가 붙지 않는 것으로 표현된다 |
| 결과 통지 | 공통 준수 | 목록의 노출 토글 2문구 · 상세의 저장·삭제 2문구 |
| 되돌릴 수 있는 일에는 확인을 붙이지 않는다 | — | 노출 토글과 답변 저장은 확인 없이 저장된다. 확인을 세우는 것은 삭제뿐이다 |
| 미저장 보호 | 공통 준수 | 상세에 이탈 가드 |
| ⚠ 답변 오류를 말할 자리가 없다 | — | 상세에 **검증 문구가 하나도 없다** — 입력 상한(500자)만 걸고 그 밖의 판정을 하지 않는다(§3) |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽 |
| 권한 | 공통 준수 | 리소스 키는 사이드바 잎 `/products/reviews`. 목록은 `remove` 를 읽어 선택·일괄 삭제를 감춘다 |
| ⚠ 상세에 쓰기 게이팅이 없다 | — | **상세 화면이 권한을 읽지 않는다** — 저장·삭제 버튼이 조회할 수 있는 누구에게나 보인다. 목록에서는 삭제가 `remove` 에 묶여 있는 것과 대비된다. 막는 층은 서버뿐이며 지금은 백엔드가 없어 실제로 막히지 않는다 |
| ⚠ 노출 토글에도 게이팅이 없다 | — | 목록의 토글은 `update` 권한을 보지 않는다(공통 인라인 갱신 훅이 저장 시점에 다룬다) |
| 개인정보 마스킹 | 공통 준수 | 작성자는 픽스처에 **이미 마스킹된 닉네임**으로 들어 있다. **가리는 주체가 화면인지 서버인지는 확인하지 못했다** — 화면 쪽에 마스킹 함수가 없다 |
| 입력값 검증 | 공통 준수 | ⚠ **이 자원에는 zod 스키마가 없다.** 리뷰 모듈에 `validation.ts` 가 없고 답변은 입력 상한(500자)만으로 다뤄진다 — 프로그램적으로 채워진 값을 거르는 것은 서버의 몫이다 |
| 고객이 쓴 내용 | — | 리뷰 본문은 **평문**으로 그려진다(서식 없는 텍스트). 서식 있는 본문을 받는 자리가 아니다 |
| 중복 제출 | 공통 준수 | 저장·삭제 중에는 버튼이 잠긴다 |
| HTTPS | 공통 준수 | 측정하지 않음 |
