# 배송 (배송 정책 · 택배사) — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 배송 정책 · 택배사 |
| 화면 ID | `SCR-PRODUCTS-SHIPPING` · `POP-PRODUCTS-SHIPPING-CARRIER-FORM` |
| 메뉴 경로 | 상품 관리 > 배송 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [배송](../../FSD/products/shipping/index.md) · [택배사 추가·수정 팝업](../../FSD/products/shipping/pop-carrier-form.md) |

**이 자원의 성격** — 한 화면에 성격이 다른 둘이 산다: **정책 문서 1건**(단일 문서형)과 **택배사 CRUD 표**(목록형). 택배사 목록은 배송 처리 화면이 소비하는 참조 데이터라, 이 자원의 비기능 요구는 **삭제 차단(fail-closed)** 과 **두 화면의 이음매**에 몰린다. 아래 각 절은 [공통 기준](../_common.md) 대비 이 자원만의 차이를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수 | 조회 2건(정책 문서 · 택배사 목록). **기본 택배사 선택지는 표와 같은 조회를 읽는다** — 별도 조회를 만들면 한쪽만 새로고침되어 '표에는 있는데 고를 수 없는' 택배사가 생긴다 | `ShippingPolicyPage.tsx:90-93` |
| 대용량 데이터 처리 | 가상화 없이 1,000행을 그리지 않는다 | 택배사는 본질적으로 작다(행이 서넛). 페이지네이션은 없다 | 코드 확인 |
| 사용 건수 조회 | — | 행마다 **배송 원장에 사용 건수를 묻는다**(조회기 호출). 택배사 수만큼 돈다 | `components/CarrierSection.tsx:166-168` |
| 재조회 중 표시 | 공통 준수 | 표는 이전 행을 유지하고 건수 줄에 `· 새로고침 중…` 만 덧붙인다 | 코드 확인 |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 정책은 단일 문서 껍데기, 택배사는 디자인 시스템 표 + 액션 열이다. 모달·확인 다이얼로그도 공통이다 |
| 두 카드의 분리 | — | 택배사 표는 정책 폼 **밖**에 선다 — 안에 두면 폼 안에 모달의 폼이 겹친다. 저장소도 서로 달라 저장 버튼이 서로를 건드리지 않는다 |
| 자유 입력을 없앤 이유 | — | 예전 정책의 택배사는 텍스트 한 줄이었다. '대한통운' 과 'CJ대한통운' 이 공존할 수 있었고 **추적 URL 을 만들 키가 없었다** — `components/CarrierSection.tsx:6-9` |
| 현재 값을 선택지에서 빼지 않는다 | — | 끈 택배사도 **지금 정책의 값이면** 선택지에 남긴다(`{이름} (미사용)`). 빼 버리면 셀렉트가 조용히 첫 항목으로 튀어 **운영자가 건드리지도 않은 정책이 저장 시 바뀐다** |
| 빈 선택지는 침묵하지 않는다 | — | 택배사가 0곳이면 선택지 자체가 `등록된 택배사가 없어요` 를 말한다 |
| 막힌 버튼은 이유를 말한다 | — | 삭제가 잠기면 사유가 **버튼 이름과 툴팁**에 함께 실린다 |
| 대안을 문장이 말한다 | — | 삭제 대신 **사용 여부 끄기**가 답이라는 것을 차단 문구와 안내 줄이 모두 말한다 |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 사용 현황이 `null` 이면 `사용 현황 확인 중` 이고 **그 상태에서는 삭제가 열리지 않는다**(0 으로 뭉개면 조회 실패 순간 삭제가 열린다) |
| 없는 링크를 그리지 않는다 | — | 추적 URL 템플릿이 비면 표에 `추적 링크 없음` 이라 적고, 배송 처리 화면에서도 링크를 만들지 않는다 |
| 결과 통지 | 공통 준수 | 정책 저장 1 · 택배사 추가/저장 2 · 택배사 삭제 1 |
| 접근성 — 액션 열 | — | 액션 열의 머리에 **보이지 않는 이름**(`행 액션`)이 붙는다 |
| 접근성 — 검증 실패 | 공통 준수 | 모달의 검증 실패는 이름 칸으로 포커스를 옮긴다 |
| 미저장 보호 | 공통 준수 | 정책 폼에 이탈 가드, 모달에 네 경로 dirty 가드 |

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|

| 개인정보 마스킹 | 공통 준수 | 해당 없음 — 이 자원에는 개인정보가 없다 |
| 입력값 검증 | 공통 준수 | 규칙의 정본은 zod 스키마 둘(정책 · 택배사) |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 셸 안쪽 경계 |
| 장애 대응 | 공통 준수 | **택배사 조회가 실패해도 정책 폼은 남는다** — 표만 배너로 바뀌고, 정책 쪽에는 '기본 택배사를 고를 수 없다' 는 힌트가 붙어 어디서 복구하는지 말한다 |
| 재시도 | 공통 준수 | 정책·택배사 각각 `다시 시도`. 삭제 실패는 다이얼로그가 닫히지 않는다 |
| 삭제 차단(fail-closed) | — | 사용 건수를 **모르면 막는다**. 통과시키면 배송 건이 가리키던 택배사가 사라져 송장 열의 이름과 추적 링크가 동시에 죽고, 그 배송 건은 다시는 어느 택배로 나갔는지 말할 수 없다 — `apps/admin/src/shared/domain/shipment.ts:147-152` |
| 식별자와 표기의 분리 | — | 배송 건은 택배사 **id** 를 가리킨다. 이름·코드가 바뀌어도 과거 기록이 끊기지 않으므로 **코드 수정을 열어 둘 수 있다** — 고칠 길이 없으면 운영자는 택배사를 하나 더 만든다 |
| 저장 전 다듬기 | — | 이름·코드·템플릿의 앞뒤 공백을 뗀다 — `' 가상택배 '` 가 그대로 들어가면 이름이 두 벌이 된다 |
| 실시간 추적을 흉내 내지 않는다 | — | 화면이 하는 일은 주소를 만드는 것뿐이고 그 뒤는 택배사 사이트가 답한다. **상태를 지어내면 거짓말이 된다** — `shipment.ts:105-110` |
| 로그 기록 | 공통 준수 | **확인하지 못했다** |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 택배사 **추적 사이트**로 나가는 링크가 전부다. 접수 API 연동은 없다 |
| API 제약 | **백엔드가 없다.** 연동 지점은 `apps/admin/src/pages/products/shipping/data-source.ts` 의 `TODO(backend)` 다 |
| 택배사 코드 | 영문 **대문자**·숫자·하이픈, 20자 |
| 택배사 이름 | 40자 |
| 추적 URL | `http(s)://` 로 시작하고 `{{invoice}}` 토큰을 포함해야 한다. 비우면 링크를 만들지 않는다 |
| 중복 | ⚠ **이름·코드 중복을 막지 않는다** |
| 삭제 | 그 택배사로 나간 배송 건이 있거나 **건수를 모르면** 불가(§4) |
| ⚠ '기본 배송비' 검증 | 화면에서는 배송비 정책이 무료일 때 그 칸이 사라지지만 **스키마는 값을 계속 요구한다**. 폼이 기본값을 들고 있어 실무에서 드러나지 않을 뿐, 값을 비운 채 무료로 바꾸면 보이지 않는 필드 때문에 저장이 막힌다 |
| 조회 전용 접근 | ⚠ **없다** — `update` 권한이 없으면 화면 전체가 접근 거부다(§3) |
| 권역 | 제주·도서산간 두 가지 고정이다 — 권역을 더할 UI 가 없다 |
| 페이지 크기 상한 | 없다(§1) |
| 내보내기 | CSV 내보내기가 없다 |
| 파일 용량 제한 | 해당 없음 — 파일 입력이 없다 |
| 업로드 가능 확장자 | 해당 없음 — 같은 이유 |
