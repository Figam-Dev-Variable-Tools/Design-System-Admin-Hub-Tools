# 문의 (상품 · 조건부 노출) — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 상품 문의 |
| 화면 ID | `SCR-PRODUCTS-INQUIRIES` · `SCR-PRODUCTS-INQUIRIES-DETAIL` |
| 메뉴 경로 | 상품 관리 > 문의 — **조건부 노출** |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../../FSD/products/inquiries/index.md) · [상세](../../FSD/products/inquiries/detail.md) |

**이 자원의 성격** — **결제를 쓰지 않는 동안에만 자라는 자원**이다. 그래서 사이드바 노출 자체가 조건부이고(세 상태), 그 판정이 이 자원의 가장 중요한 비기능 요구다 — 잘못 판정하면 **과거 문의가 통째로 접근 불가**가 된다. 그다음이 견적 퍼널로의 이음매다. 아래 각 절은 [공통 기준](../_common.md) 대비 이 자원만의 차이를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수 | 목록 1건 · 상세 1건. 필터·검색은 받아 온 배열 위에서 좁힌다 | 코드 확인 |

| 견적 발행 요청 수 | — | 담은 건수만큼 문의 갱신이 **건별로** 나간다(묶음 엔드포인트가 없다). 합쳐진 문의는 모두 같은 견적 id 를 갖는데, **하나라도 빠지면 그 문의는 견적 없이 남는다** — `ProductInquiryListPage.tsx:213-219` |
| 재조회 중 표시 | 공통 준수 | 스켈레톤은 최초 로드에서만 | 코드 확인 |
| 건수 집계 | — | 좌측 5갈래 건수와 미답변 수는 **필터 이전 전체 집합**에서 매 렌더 다시 센다. 못 셌으면 `null` 이다 | 코드 확인 |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 **읽기 전용 껍데기 + 담기 열 하나**를 얹었다. 그 껍데기의 선택 체크박스는 일괄 삭제용이라 어떤 역할에게도 없고, 여기 필요한 것은 지우는 선택이 아니라 **담는 선택**이라 열로 따로 만들었다 |
| 중심 열이 경과다 | — | 운영자가 여기서 하는 판단은 '무엇부터 답할까' 하나뿐이라, 상품도 제목도 아니고 **얼마나 오래 답을 못 받았는가**가 중심이다 |
| 색만으로 지연을 말하지 않는다 | — | 경과 배지 안에 `3일째 미답변` 이라는 **문구가 함께 실린다** |
| 미답변 두 상태를 같은 색으로 묶지 않는다 | — | '답변 중' 은 이미 사람이 붙은 상태라 방치된 '접수' 와 같은 색이면 운영자가 무엇을 먼저 집어야 할지 알 수 없다 — `types.ts:20-25` |
| 견적 발행을 '답변 완료' 로 뭉개지 않는다 | — | 견적만 나가고 답변 글은 아직인 건은 경과 문구가 `견적 발행` 이다 — '답변 완료' 도 '미답변' 도 사실이 아니다 |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 좌측 건수는 못 셌으면 `—`, 미답변 안내는 `미답변 건수를 세는 중이에요.`(0건과 다른 문장). **경과를 계산할 수 없으면 `—` 이고 0일로 위장하지 않는다** |
| 죽은 컨트롤 금지 | — | 바구니 막대는 **담은 것이 있을 때만** 나타난다 — 아무것도 담기지 않은 화면에 죽은 버튼을 두지 않는다. 상세의 상태 전환 버튼도 조건에 맞을 때만 렌더된다 |
| 상태는 버튼이 아니라 규칙이 정한다 | — | '답변 완료' 를 직접 고르는 셀렉트가 없다. 있으면 **'답변 없이 답변 완료'** 가 만들어지고, 그 순간 목록의 미답변 집계와 경과 문구가 전부 거짓말이 된다 — `ProductInquiryDetailPage.tsx:6-9` |
| 고객이 쓴 글을 뭉개지 않는다 | — | 문의 내용은 **줄바꿈이 의미**라 문단을 유지해 그린다 |
| 결과 통지 | 공통 준수 | 목록 2문구(성공·실패) · 상세 5문구(전부 성공) |
| 접근성 — 담기 상태 | — | 담긴 버튼은 눌린 상태로도 전달된다(`aria-pressed`) |
| 미저장 보호 | 공통 준수 | 상세의 답변 입력에 이탈 가드 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽 |
| 권한 | 공통 준수 | 리소스 키는 사이드바 잎 `/products/inquiries`. 의미 있는 액션은 `read` 와 `update` — 등록·삭제 컨트롤이 없다 |
| 메뉴 노출 ≠ 권한 | — | **조건부 노출은 메뉴 규칙이지 권한 규칙이 아니다.** 사이드바에서 사라져도 리소스 키와 판정은 살아 있고 라우트도 열린다 |
| 권한 실패 방향 | 공통 준수 | `update` 가 없으면 목록의 **담기 열과 바구니 막대가**, 상세의 **버튼 셋이** 사라진다 |
| 개인정보 마스킹 | 공통 준수 | ⚠ **문의자 이름과 연락처가 평문**이다(상세의 문의자 정보 카드). 픽스처가 가상 값이라 지금은 드러나지 않지만 실데이터에서는 서버가 정책을 정해야 한다 — 화면 쪽에 마스킹 함수가 없다 |
| 입력값 검증 | 공통 준수 | 답변의 규칙은 zod 스키마가 갖고, **저장소도 같은 규칙으로 한 번 더 막는다** — 빈 답변이 '답변 완료' 로 상태를 넘기면 고객은 아무것도 받지 못한 채 처리된 것으로 집계된다 — `validation.ts:4-6` |
| 중복 제출 | 공통 준수 | 저장 중에는 입력과 버튼이 잠긴다. **견적 발행은 도메인 층에서 멱등**이다(§4) |
| HTTPS | 공통 준수 | 측정하지 않음 |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 셸 안쪽 경계 |
| 장애 대응 | 공통 준수 | 상세의 조회 실패는 **없는 문의와 고장난 문의를 가른다** — 없는 문의에 '다시 시도' 를 권하지 않는다 |
| 재시도 | 공통 준수 | 목록·상세 각각 `다시 시도`(없는 대상에는 붙이지 않는다) |
| 메뉴 판정의 fail-open | — | 잔여 문의 조회기가 배선되지 않으면 `null`(모른다)이고, 메뉴 규칙은 그것을 **'남긴다'** 로 읽는다. 0 으로 읽어 메뉴를 지우면 **과거 문의로 가는 길이 통째로 끊기고**, 원인은 데이터가 없어서가 아니라 배선을 잊어서다 — **지우는 쪽으로 실패하면 복구가 없다** — `apps/admin/src/shared/commerce/inquiry-backlog.ts:10-13` |
| 잔여의 정의 | — | 잔여는 **종결 포함 전체 건수**다. 미종결만 세면 답변을 끝낸 문의(고객이 문의번호를 대며 다시 물을 기록 · 감사 대상)를 0으로 읽고 메뉴를 지운다 — `inquiry-backlog.ts:19-25` |
| 라우트는 사라지지 않는다 | — | 메뉴 해소 함수는 **라우트 생성 경로를 지나지 않는다** — 감추는 것과 없애는 것은 다른 결정이다 — `apps/admin/src/shared/layout/nav-config.ts:320-325` |
| 상태와 값이 갈라지지 않는다 | — | 답변·답변 시각·상태가 **한 함수에서 함께** 옮겨 간다. 견적 id 와 상태도 마찬가지다 — 둘이 갈라지면 '상태는 견적 발행인데 견적이 없는' 문의가 생긴다 |
| 응대 속도를 조작할 수 없다 | — | **최초 답변 시각은 재수정에서 바뀌지 않는다** — 오탈자를 고칠 때마다 갱신되면 응대 속도가 사후에 조작된다 — `_shared/store.ts:120-126` |
| 견적 발행의 멱등 | — | 견적 id 가 멱등키다. **두 번 눌러도 견적은 하나**이고, 저장소가 문의 id 로 한 번 더 교차 확인한다(이중 방어) |
| 부분 실패 | — | 목록의 묶음 발행이 중간에 실패하면 **성공한 문의만 견적을 가리킨다**. 바구니는 비우지 않아 다시 시도할 수 있다 |
| 기준일 고정 | — | 경과의 기준일이 화면 상수다 — 픽스처의 경과가 실행하는 날마다 달라지면 회귀 비교가 매일 깨진다. **백엔드가 붙으면 서버가 내려주는 기준 시각으로 바뀐다** — `ProductInquiryListPage.tsx:74-81` |
| 로그 기록 | 공통 준수 | **확인하지 못했다** |
