## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 읽기 전용 껍데기 + 좌측 필터 레일. 상세는 껍데기가 없어 카드를 직접 배치한다 |
| 접근성 | 공통 준수 | 행 클릭(마우스)의 짝으로 **제목 링크**가 있어 Tab 으로도 상세에 닿는다. 바구니 담기 버튼이 `aria-pressed` 로 상태를 말한다 — `apps/admin/src/pages/programs/inquiries/ProgramInquiryListPage.tsx:310` |
| 색만으로 말하지 않기 | 공통 준수 | 경과·상태·유형이 전부 배지 안에 문구를 함께 싣는다 |
| 상태를 뭉개지 않기 | 다른 사실은 다른 색으로 | **미답변 두 상태를 같은 색으로 묶지 않는다** — '답변 중'은 이미 사람이 붙은 상태라 방치된 '접수'와 같은 색이면 무엇을 먼저 집을지 알 수 없다 — `apps/admin/src/pages/programs/inquiries/types.ts:25-29`. **견적만 나가고 답변 글은 아직인 상태**도 '답변 완료'로 뭉개지 않고 따로 표기한다 |
| 사용자 피드백 | 공통 준수 | 목록의 토스트는 **성공 1 · 실패 1**, 상세는 **성공 1**([인벤토리](../../FSD/inventory.md) §9.3 과 일치). 상세의 네 동작(답변·착수·종결·발행)이 한 저장 경로를 지나며 그 하나의 성공 토스트에 서로 다른 문구를 싣는다 |
| 죽은 컨트롤을 두지 않기 | 누를 수 없는 것을 보여 주지 않는다 | 바구니 막대는 **담은 것이 있을 때만** 나타나고, 상세의 액션 넷은 각각 전이 규칙이 열어 준 것만 존재한다 |
| 권한 예고 | 상세에 들어가서야 알게 하지 않는다 | 목록 좌측이 `답변 권한이 없어 조회만 가능해요.` 를 미리 띄운다 — `apps/admin/src/pages/programs/inquiries/ProgramInquiryListPage.tsx:386` |

| 공통 컴포넌트 사용 여부 | 페이지가 UI 를 손수 조립하지 않는다 | 목록은 읽기 전용 껍데기와 공통 필터 레일을 쓴다. 상세는 껍데기가 없어 카드·타임라인을 직접 배치한다 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 문의를 만드는 것은 **고객 채널**(PG 를 끈 프로그램 페이지의 '문의하기' 버튼 등)이고 이 리포 밖이다. 관리자 화면에는 생성 경로가 없다 |
| API 제약 | **백엔드가 없다.** 서버 계약은 §6 이 갖는다 |
| 생성·삭제 | **둘 다 없다.** 관리자는 답변하고 종결할 뿐이다 |
| 조건부 노출 | 이 잎은 결제를 쓰지 않는 동안에만 메뉴에 보인다(§6). **라우트는 언제나 살아 있다** |
| 종결의 성격 | **종점이다** — 되돌리는 전이가 없고 답변도 고칠 수 없다. ⚠ 그런데 확인 다이얼로그가 없다(§4) |
