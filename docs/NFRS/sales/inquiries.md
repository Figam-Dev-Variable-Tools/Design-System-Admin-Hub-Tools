## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | — |
| 권한 | 공통 준수 | 리소스 키는 `sales-inquiries`. **`create`·`remove` 는 이 자원에서 아무것도 열지 않는다** — 관리자 화면에 생성·삭제 경로가 없다 |
| ⚠ 쓰기 게이팅의 예외 | 쓰기 컨트롤은 권한이 없으면 사라진다(공통) | **이 자원의 상세는 그 규약을 따르지 않는다.** 담당·상태·작성칸·저장 버튼이 `canUpdate` 판정 없이 언제나 그려진다(형제 화면인 견적·청구 상세는 감춘다). 막는 것은 서버뿐이다 — `InquiryDetailPage.tsx` 에 `useRouteWritePermissions` 호출이 없다 |
| 플랜(엔타이틀먼트) | 공통 준수(fail-open) | `sales.pipeline` 모듈 |
| 개인정보 마스킹 | 필드 이름으로 적는다 | **마스킹 규칙이 없다.** `customerName`(고객명) · `contact`(연락처 — 이메일 또는 휴대전화) · `company`(회사명)가 목록과 상세에 평문으로 그려진다. 연락처는 개인정보이며 로그 화면의 마스킹 규칙(`apps/admin/src/pages/logs/masking.ts`)은 이 화면에 걸리지 않는다 |
| 입력값 검증 | 공통 준수 | **이 자원에는 zod 스키마가 없다** — 저장 대상이 세 값뿐이라 폼 라이브러리를 얹지 않았다. 상태값만 타입가드로 좁힌다 — `apps/admin/src/pages/sales/inquiries/types.ts:177-180` |
| 중복 제출 | 금액·생성·발송에서 두 번째 요청을 만들지 않는다 | 저장 버튼이 저장 중 잠긴다. **견적 중복 발행은 두 겹으로 막힌다**: 어댑터가 문의의 `quoteId` 를 보고, 견적 저장소가 원본 문의 id 로 교차 확인한다 |
| HTTPS | 공통 준수 | **측정하지 않음** |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | ⚠ **상세의 조회 실패가 '없음'과 '오류'를 가르지 않는다** — 둘 다 `문의를 불러오지 못했어요.` 로 말하고 `목록으로` 만 준다. 형제 화면(견적·계약·청구·거래처 상세)은 전부 가르고 오류일 때만 재시도를 준다 — `apps/admin/src/pages/sales/inquiries/InquiryDetailPage.tsx:208-219` |
| 부수효과의 원자성 | 두 일이 하나만 성공하는 창을 만들지 않는다 | **견적 발행은 문의 갱신의 부수효과로 어댑터 안에서 일어난다.** 화면이 '문의 저장'과 '견적 생성'을 두 번 호출하면 하나만 성공하는 창이 생기고, 그 창에서 새로고침하면 상태는 '견적 발행'인데 견적이 없는 문의가 남는다 — `apps/admin/src/pages/sales/inquiries/data-source.ts:6-8` |
| ⚠ 동시 저장 | 공통 준수(last-write-wins) | **충돌 다이얼로그가 없다.** 이 저장은 타임라인을 포함한 문서 한 벌을 통째로 보내므로, 내가 화면을 연 뒤 다른 관리자가 남긴 이력이 사라질 수 있다. 청구 자원은 같은 위험에 원장 소실 가드를 세웠지만 이 자원에는 없다 |
| 재시도 | 공통 준수 | 저장 실패는 배너로 남고 버튼 재클릭이 재시도다 |
| 취소는 실패가 아니다 | 공통 준수 | 화면을 떠나 요청이 중단되면 성공·실패 어느 표시도 하지 않는다 — `apps/admin/src/pages/sales/inquiries/InquiryDetailPage.tsx:190,201` |
| 로그 기록 | 공통 준수 | **확인하지 못했다** — 상태 전이·견적 발행을 관리자 로그에 남기는 배선을 찾지 못했다. 다만 **문의 자신의 타임라인**에는 상태 변경과 견적 자동 생성이 남는다(append-only) |

| 백엔드 없음의 이음매 | 서버가 맡을 일이 코드에 표시되어 있다 | `TODO(backend): GET /api/sales/inquiries · GET/PUT /api/sales/inquiries/:id … '견적 발행' 전이는 견적 생성을 동반한다 — 서버는 문의 갱신 + 견적 생성 + 양방향 링크를 한 트랜잭션으로 처리하고, 이미 발행된 문의의 재발행은 기존 견적을 돌려준다(멱등)` — `apps/admin/src/pages/sales/inquiries/data-source.ts:160-162` |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 문의를 만드는 것은 **고객 채널**(홈페이지·전화·이메일·방문)이고 이 리포 밖이다. 관리자 화면에는 생성 경로가 없다 |
| API 제약 | **백엔드가 없다.** 서버 계약은 §6 의 `TODO(backend)` 두 줄이 갖는다 |
| 생성·삭제 | **둘 다 없다.** 관리자는 처리·답변만 한다 |
| 견적 발행 | 그 문의로 이미 발행된 견적이 있으면 **다시 만들지 않는다**(멱등). 영업 문의는 품목명을 갖지 않아 **품목 없는 견적**으로 태어난다 — 무엇을 파는지 문의가 모르기 때문이다 — `apps/admin/src/pages/sales/inquiries/data-source.ts:136` |
| 거래처 연결 | 발행된 견적의 `accountId` 는 **미등록('')으로 시작한다** — 문의는 고객이 적은 회사명 문자열만 갖고 있어 마스터의 어느 행인지 모른다. 이름을 맞춰 추측하면 동명이인 거래처에 남의 견적이 붙는다 |
| 문자 수 상한 | 답변·메모 1,000자 — `apps/admin/src/pages/sales/inquiries/types.ts:76` |
| 타임라인 | append-only. 수정·삭제 경로가 없다 |
| 파일 업로드 | 해당 없음 — 이 자원에 파일 입력이 없다 |
