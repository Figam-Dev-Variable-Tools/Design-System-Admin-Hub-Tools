## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 공통 껍데기, 폼은 공통 컨트롤러. 상세의 역방향 카드는 이 자원 고유 컴포넌트(`RelatedRecordsCard`)이고 표 골격만 디자인 시스템 Table 에서 가져온다 |
| 접근성 | 공통 준수 | 행 클릭의 짝으로 **사업자명 링크**가 있어 Tab 으로 상세에 닿는다. 상태 토글에는 `{거래처명} 거래 여부` 맥락 라벨이 붙는다 |
| 입력 편의성 | 공통 준수 | 사업자번호는 입력 중 하이픈이 자동으로 붙고, **검색은 하이픈을 무시한다**(숫자만 뽑아 비교) — `apps/admin/src/pages/sales/accounts/types.ts:134-144` |
| 사용자 피드백 | 공통 준수 | 이 자원에는 **자기 토스트가 없다**([인벤토리](../../FSD/inventory.md) §9.3 에 없다). 상태 토글은 공통 인라인 갱신 토스트에 이 화면 문구를 싣는다: `'{이름}'을(를) 거래중으로 바꿨어요.` / `'{이름}' 거래를 중지했어요.` — `apps/admin/src/pages/sales/accounts/AccountListPage.tsx:171-174` |
| 세 상태를 뭉개지 않기 | 실패·미도착·0건이 서로 다른 화면이다 | 역방향 카드가 그 규약의 본보기다: 못 불러왔다(배너 + 다시 시도) / 아직 못 읽었다(스켈레톤 2행) / 정말 없다(안내 문구). 셋을 합치면 **조회가 실패한 거래처가 '거래 이력 없는 거래처'로 보인다** — `apps/admin/src/pages/sales/accounts/components/RelatedRecordsCard.tsx:8-10` |
| 빈 값 표기 | 빈 칸을 그대로 두지 않는다 | 상세는 빈 문자열을 `—` 로 그린다 — 빈 칸으로 두면 로딩 중인지 없는 것인지 구분되지 않는다 — `apps/admin/src/pages/sales/accounts/AccountDetailPage.tsx:165-166` |
| 읽기와 고치기의 분리 | 보기만 하려는 사람에게 편집 화면을 열지 않는다 | 목록의 **행 클릭은 상세로, 연필은 수정 폼으로** 간다. 예전에는 둘 다 수정 폼이었다 — `apps/admin/src/pages/sales/accounts/AccountListPage.tsx:238-240` |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | — |
| 권한 | 공통 준수 | 리소스 키는 `sales-accounts`. **영업 관리에서 `create` 가 실제로 열리는 유일한 자원이다** — 나머지 네 칸은 앞 칸에서만 생긴다 |
| 플랜(엔타이틀먼트) | 공통 준수(fail-open) | `sales.pipeline` 모듈 |

| 입력값 검증 | 공통 준수(스키마 하나가 정본) | `validation.ts`. **사업자등록번호를 국세청 체크섬으로 실검증한다** — 이 앱에서 그 검증을 하는 유일한 자리다(견적 폼은 표기만 맞춘다) — `apps/admin/src/pages/sales/_shared/business.ts:25-42` |
| ⚠ 중복 방지 | 같은 실체가 두 행이 되지 않아야 한다 | **막지 않는다.** 같은 사업자등록번호로 두 행을 만들 수 있다. 이 마스터가 생긴 이유가 '오타 하나가 한 거래처를 둘로 쪼갰다'였는데, **정확히 같은 번호의 두 행은 여전히 만들어진다** — 검증에도 저장소에도 그 규칙이 없다 |
| 중복 제출 | 공통 준수 | 폼은 동기 잠금 + 멱등키 |
| HTTPS | 공통 준수 | **측정하지 않음** |

## 7. 유지보수성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 코드 확장성 | 공통 준수 | 거래유형·과세유형·신용등급·결제조건이 전부 옵션 배열이다 |
| 참조 규약의 단일 정본 | 같은 규칙을 네 도메인이 각자 적지 않는다 | `AccountRef`(id + 표시 라벨)와 그 헬퍼(`currentAccountName` · `filterByAccount` · `indexById`)가 한 모듈에 모여 있다. **`list.find(...) ?? list[0]` 같은 폴백을 쓰지 않는다** — 못 찾았을 때 엉뚱한 거래처를 반환해 삭제된 거래처가 첫 거래처로 둔갑한다 — `apps/admin/src/pages/sales/_shared/account-reference.ts:44-57` |
| 미등록을 허용한 이유 | 규칙이 경로를 막지 않게 한다 | 문의 → 견적 발행은 회사명 문자열만 승계하므로 `accountId` 를 강제하면 그 경로가 통째로 막힌다. 그래서 ''를 허용하되 **화면이 그 대가를 반드시 드러낸다** — `apps/admin/src/pages/sales/_shared/account-reference.ts:15-18` |
| 화면 확장성 | 열·필터·상태값이 늘 때 견디는가 | 조회 조건은 주소가 소유한다. **역방향 카드는 한 벌(`RelatedRecordsCard`)을 셋이 공유하므로 축을 하나 더 붙이기 쉽다**(예: 청구 이력 — 지금은 없다) |
| 공통 컴포넌트 사용 여부 | 페이지가 UI 를 손수 조립하지 않는다 | 목록·폼은 공통 껍데기를 쓴다. 상세는 껍데기가 없어 카드와 정의목록을 직접 배치한다 |
| 껍데기를 재사용하지 않은 근거 | 비슷하다고 같은 것이 아니다 | 역방향 카드가 `CrudReadListShell` 을 쓰지 않는 이유: 저 껍데기는 **화면 하나가 통째로 목록일 때**의 것이다(툴바·빈 상태 복구 액션·행 클릭 목적지·URL 조회 상태). 여기는 상세 안의 세 구획이라 필터도 검색도 없고, 비었을 때 권할 것도 '검색을 지우세요'가 아니다 — `apps/admin/src/pages/sales/accounts/components/RelatedRecordsCard.tsx:4-6` |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | **없다.** 사업자등록 상태 조회(국세청 API) 연동이 없다 — 체크섬만 로컬에서 계산한다 |
| API 제약 | **백엔드가 없다.** 서버 계약은 §6 이 갖는다 |
| 등록·삭제 | **영업 관리에서 유일하게 손으로 만들고 지우는 자원이다** |
| 중복 사업자번호 | **막지 않는다**(§3) |
| 담당자 수 상한 | 8명 — `apps/admin/src/pages/sales/accounts/types.ts:62`. 최소 1명은 필수다 |
| 문자 수 상한 | 상호 60자 · 대표자명 40자 · 비고 500자 — `apps/admin/src/pages/sales/accounts/types.ts:60-61` |
| 여신한도 | **0 은 '한도 없음'이 아니라 '아직 정하지 않음'이다** — 상세가 `미설정` 으로 그린다 |
| 주소 입력 | 자유 입력이다. 오시는 길 화면의 주소 검색 팝업(`POP-COMPANY-DIRECTIONS-ADDRESS-SEARCH`)을 쓰지 않는다 |
| 전화번호 | 형식을 검증하지 않는다 |
| 파일 업로드 | 해당 없음 — 이 자원에 파일 입력이 없다(사업자등록증 사본을 보관하지 않는다) |
