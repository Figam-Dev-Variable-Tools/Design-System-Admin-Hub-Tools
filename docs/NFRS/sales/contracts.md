## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 공통 껍데기, 폼은 공통 컨트롤러를 쓴다. 미리보기(`ContractSummaryPreview`)만 이 자원 고유 컴포넌트다 |
| 접근성 | 공통 준수 | 원 견적 링크에 `{계약명} 원 견적 {견적번호}` 맥락 aria-label 이 붙는다 — `apps/admin/src/pages/sales/contracts/ContractListPage.tsx:136` |
| 입력 편의성 | 공통 준수 | 금액·통지기한은 문자열로 받아 원값을 보존하고 저장 직전에 숫자만 남긴다 — `1,200,000` 을 붙여 넣어도 값이 살아남는다 |
| 사용자 피드백 | 공통 준수 | 이 자원에는 **자기 토스트가 없다**([인벤토리](../../FSD/inventory.md) §9.3 에 없다). 저장은 공통 폼 토스트(`MSG-COMMON-01`), 삭제는 공통 삭제 토스트(`MSG-COMMON-02`/`03`)를 쓴다 |
| 침묵하지 않기 | 버튼이 없는 자리에 이유를 둔다 | `프로젝트 만들기` 가 없을 때 그 자리에 **왜 없는지**가 배너로 온다(체결 전이면 info, 이미 있으면 success + 링크) — `apps/admin/src/pages/sales/contracts/ContractFormPage.tsx:372-388` |
| 옛 데이터 고지 | 규칙 이전 데이터를 정상인 척하지 않는다 | 견적 없이 등록된 계약에는 warning 배너가 뜬다. **지우지도 않는다** — 실제로 맺어진 계약이고, 소급해서 견적을 지어내면 없던 거래가 하나 생긴다 — `apps/admin/src/pages/sales/contracts/ContractFormPage.tsx:362-369` |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | — |
| 권한 | 공통 준수 | 리소스 키는 `sales-contracts`. **`create` 는 이 자원에서 아무것도 열지 않는다** — `/new` 가 차단 화면이다 |
| ⚠ 프로젝트 생성 버튼의 게이팅 | 쓰기 컨트롤은 권한 판정을 거친다(공통) | **이 폼의 `프로젝트 만들기` 에는 권한 판정이 없다.** 버튼의 존재를 정하는 것은 전이 가드(`projectDraftBlock`)뿐이라 **읽기 권한만 있어도 보인다.** 견적 상세가 계약·청구 버튼을 `canUpdate` 로 함께 가리는 것과 다르다 — `apps/admin/src/pages/sales/contracts/ContractFormPage.tsx:343` |
| 플랜(엔타이틀먼트) | 공통 준수(fail-open) | `sales.pipeline` 모듈 |
| 개인정보 마스킹 | 필드 이름으로 적는다 | **마스킹 규칙이 없다.** `ownerName`(담당자)이 평문이고, **계약서 스캔 첨부**(`attachments`)에는 도장·서명·개인정보가 담길 수 있으나 접근 제어가 화면 권한 하나뿐이다 |
| 입력값 검증 | 공통 준수(스키마 하나가 정본) | `validation.ts` 의 zod 스키마. 금액 0 초과·기간 역전 금지·자동갱신 시 통지기한 숫자 |
| 중복 제출 | 공통 준수 | 폼은 동기 잠금 + 멱등키. **프로젝트 생성은 계약 id 를 멱등키로 저장소가 한 번 더 막는다** |
| HTTPS | 공통 준수 | **측정하지 않음** |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 폼의 조회 실패를 **없음과 오류로 가른다** — `apps/admin/src/pages/sales/contracts/ContractFormPage.tsx:300-322` |
| 되돌려 쓰기가 없는 이유 | 파생값을 두 곳에 적지 않는다 | 계약 → 프로젝트에는 **반쪽 저장이라는 상태가 성립하지 않는다.** 계약에는 '프로젝트가 생겼다'를 적을 칸이 없고 만들지도 않는다 — 그 사실의 정본은 프로젝트의 `contractId` 하나다 — `apps/admin/src/pages/sales/projects/data-source.ts:149-150` |
| 판정의 근거 | 화면 값이 아니라 저장된 값을 본다 | 프로젝트 생성 가능 판정은 **저장된 원본**(`loaded`)을 읽는다. 폼에서 상태만 '진행중'으로 바꿔 놓고 저장하지 않은 채 프로젝트를 만들 수 없다 — `apps/admin/src/pages/sales/contracts/ContractFormPage.tsx:260-266` |
| 장애 대응 | 공통 준수 | 목록 조회 실패는 배너 + `다시 시도` |
| 재시도 | 공통 준수(자동 재시도 없음) | 저장 실패는 폼 배너 + 참조 코드 |
| 삭제의 파급 | 지운 뒤 남는 참조를 감춰서는 안 된다 | ⚠ **계약을 지워도 앞뒤가 정리되지 않는다.** 그 견적의 상태는 `수주(계약 진행)` 로 남고(되돌리는 전이가 없다) 견적 상세는 그것을 종점으로 보아 아무 경고도 띄우지 않는다. 그 계약에서 만든 프로젝트의 `contractId` 도 남아 '원 계약 보기'가 죽은 목적지를 가리킨다. **다만 견적 쪽의 `계약 만들기` 버튼은 되살아난다** — 존재 여부를 렌더 시점에 다시 조회하기 때문이다 |
| 로그 기록 | 공통 준수 | **확인하지 못했다** |

## 6. 운영성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 로그 관리 | 공통 준수 | **확인하지 못했다** |
| 모니터링 | 공통 준수 | **없다** |
| 배포 영향 | 이 자원을 고치면 어느 화면이 함께 영향을 받는가 | `isConcludedContract`(진행중 + 서명완료)는 **프로젝트 생성의 유일한 관문**이다 — 이 판정을 바꾸면 프로젝트가 생기는 조건이 통째로 바뀐다. 계약금액은 프로젝트 예상매출의 승계값이고, 계약의 존재 여부가 견적 상세의 버튼과 경고를 정한다 |
| 설정 관리 | 이 자원의 동작을 바꾸는 설정값이 어디 있는가 | 없다. 플랜의 `sales.pipeline` 이 이 자원 전체를 잠근다 |
| '계약 완료' 상태를 만들지 않은 근거 | 없는 상태를 함부로 만들지 않는다 | 상태 유니온에 '완료'가 없다. 상태를 하나 늘리면 목록 필터·배지·시드가 전부 그 칸을 다시 정의해야 하고, 운영자가 말한 '계약 완료'는 애초에 **이미 있는 두 사실의 교집합**이다(서명완료 + 진행중) — `apps/admin/src/pages/sales/contracts/types.ts:220-233` |
| 백엔드 없음의 이음매 | 서버가 맡을 일이 코드에 표시되어 있다 | `TODO(backend): GET/POST /api/sales/contracts · GET/PUT/DELETE /api/sales/contracts/:id`(`apps/admin/src/pages/sales/contracts/data-source.ts:182`) · `TODO(backend): POST /api/sales/quotes/:id/contract — 서버가 계약 생성 + 견적 상태 전이를 한 트랜잭션으로 처리하고, 이미 계약이 있으면 기존 계약을 돌려준다. 그때 quoteWriteBack 은 사라진다`(`apps/admin/src/pages/sales/contracts/data-source.ts:166-167`) |

## 7. 유지보수성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 코드 확장성 | 공통 준수 | 계약유형·상태·서명 상태가 전부 옵션 배열이다. 프로젝트 생성 판정은 순수 술어(`projectDraftBlock`) 하나 |
| 상태 전이 규칙의 부재 | 규칙이 없으면 그 사실을 적는다 | **계약 상태에는 전이 제약이 없다** — 어느 상태로든 셀렉트로 갈 수 있다. **그것이 막다른 골목을 없애는 장치이기도 하다**: 만료·해지로 넘어간 뒤 프로젝트가 필요해지면 '진행중'으로 되돌릴 수 있다 — `apps/admin/src/pages/sales/contracts/types.ts:231-233` |
| 화면 확장성 | 열·필터·상태값이 늘 때 견디는가 | 조회 조건은 주소가 소유한다. 상태 필터 값은 옵션 배열에서 파생된다 |
| 공통 컴포넌트 사용 여부 | 페이지가 UI 를 손수 조립하지 않는다 | 목록은 `CrudListShell`, 폼은 `useCrudForm`. **다만 폼은 다중 카드 + 미리보기 2단이라 껍데기(`FormPageShell`) 대신 직접 배치한다** |
| 결합 회피 | 페이지가 다른 페이지의 모듈을 알지 않는다 | 계약 → 프로젝트는 **같은 페이지 트리(`pages/sales`) 안**이라 저장소를 직접 부른다. 페이지 경계를 넘는 문의 → 견적만 공통 이음매를 지난다 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 없음. **전자서명 연동이 없다** — `signStatus`(미발송·서명대기·일부서명·서명완료)는 운영자가 손으로 적는 값이다 |
| API 제약 | **백엔드가 없다.** 서버 계약은 §6 이 갖는다 |
| 등록 경로 | **없다.** `/sales/contracts/new` 는 차단 안내 화면이며 어떤 권한으로도 폼이 열리지 않는다 |
| 문자 수 상한 | 계약명 80자 · 조항 요약 1,000자 · 비고 500자 — `apps/admin/src/pages/sales/contracts/types.ts:59-60` |
| 첨부 | 최대 5개 — `apps/admin/src/pages/sales/contracts/types.ts:61` · 이미지 · 최대 5MB. ⚠ **업로드 엔드포인트가 없다** — 파일이 서버로 가지 않고 브라우저 안 미리보기만 만들어져, 폼을 떠나면 썸네일이 깨진다. 고칠 곳은 업로드 이음매다 — `apps/admin/src/shared/crud/validation.ts:41-62` |
| 부가세 표기 | 견적에서 만든 계약의 `부가세 포함` 은 **과세 견적일 때만 참**이다 — 면세·영세율 합계에는 세액이 없어 '포함'이라고 적으면 거짓이 된다 — `apps/admin/src/pages/sales/contracts/types.ts:238-244` |
| 기본 계약기간 | 오늘부터 1년(국내 관례). 초안의 출발값일 뿐 폼에서 고친다 — `apps/admin/src/pages/sales/contracts/types.ts:195-196` |
| 거래처 연결 | `accountId` 는 빈 값을 허용한다 — 아직 등록되지 않은 거래처와의 첫 계약을 막지 않기 위해서다. 대가는 폼이 warning 으로 드러낸다 |
| 파일 용량 제한 | 이미지 최대 5MB — `packages/ui/src/molecules/ImageUploadField/ImageUploadField.tsx:136` |
| 업로드 가능 확장자 | 이미지 파일만. 화면 표기는 `PNG · JPG · GIF · 최대 5MB` |
