# 고객 설정 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 고객 설정 |
| 화면 ID | `SCR-USERS-SETTINGS` · `POP-USERS-SETTINGS-TIER-FORM` |
| 메뉴 경로 | 일반 관리 > 사용자 관리 > 고객 설정 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [고객 설정](../../FSD/users/settings/index.md) · [등급 추가·이름 수정](../../FSD/users/settings/pop-tier-form.md) |

**이 자원의 성격** — **단일 문서형 설정 화면**이다. 목록도 페이지네이션도 없고, 화면 전체가 하나의 편집 초안이며 하단의 저장 한 번이 그것을 확정한다. 비기능 관점의 특징은 셋이다: ① **정책 변경이 전 회원의 등급과 할인율을 움직인다** — 그래서 저장 전 미리보기와 확인 한 겹이 있다 ② **검증이 화면이 아니라 모델에 있다** — 입력을 막지 않고 저장을 거부한다 ③ **삭제 가드가 다른 자원(회원 명부)에 되묻는다**.

아래 각 절은 [공통 기준](../_common.md) 대비 **이 자원만의 차이**를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다. 지금 기다리는 것은 픽스처 어댑터의 인위적 지연(기본 400ms)이다 — `apps/admin/src/pages/customer-settings/data-source.ts:15` | 코드 상수 확인 |

| 재조회 중 표시 | 공통 준수 | **공통 규칙을 따른다** — 스켈레톤 조건은 '데이터가 아직 없을 때' 하나뿐이다. 그대로 두면 **저장이 정책을 무효화할 때마다 편집 중이던 폼 전체가 스켈레톤으로 교체**되어 방금 저장한 사람이 자기 화면을 잃는다 — `apps/admin/src/pages/customer-settings/CustomerSettingsPage.tsx:158-164` | 코드 확인 |

| 대용량 데이터 처리 | 공통 준수 | 해당 없음 — 목록·페이지네이션·내보내기·일괄 처리가 없다. **행 수의 상한은 등급 수**이며 그것을 제한하는 규칙은 없다(운영자가 계속 추가할 수 있다) | — |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 없다 |
| API 제약 | **백엔드가 없다.** 연동 지점은 `TODO(backend): GET /api/member-tiers` · `PUT /api/member-tiers` — `data-source.ts:55,71`. 저장 계약은 **목록이 곧 등급의 전부**라는 규약이다: 빠진 등급은 삭제, 새 id 는 추가, 바뀐 라벨은 개명. id 는 프런트가 만들며, 서버가 다른 id 를 발급한다면 응답의 목록으로 갈아탄다 — `data-source.ts:72-77` |
