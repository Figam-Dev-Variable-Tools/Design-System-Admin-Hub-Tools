**이 자원의 성격** — 시스템 설정 섹션에서 **유일하게 완전한 읽기 전용** 화면이다. 저장·충돌·중복 제출 축이 성립하지 않는 대신 셋이 이 자원의 전부다: ① **모름을 세 층으로 가르는 표시 규율** ② **실패 방향이 권한 축과 정반대**(fail-open) ③ **탭 상태를 주소에 싣는 규약**. 아래 각 절은 [공통 기준](../_common.md) 대비 **이 자원만의 차이**를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | **측정할 대상이 없다** — 플랜은 브라우저 저장소에서 동기로 읽고, 구독 결제 두 조회기는 상수를 돌려준다 — `shared/entitlements/entitlement-store.ts:44-56` · `subscription.ts:152-175` | 코드 확인 |
| 조회 성능 | 공통 준수 | **네트워크 조회가 0건이다.** 그래서 공통의 신선도·재시도·창 포커스 규칙이 이 자원에는 아직 적용되지 않는다 | 코드 확인 |
| 로딩 상태 | 공통 준수 | **스켈레톤이 없다** — 첫 렌더에 완성된 화면이 나온다. 서버가 붙으면 로딩 상태가 새로 필요해진다 | 코드 확인 |
| 재렌더 억제 | — | 플랜을 **스토어 조각으로 구독**해 플랜이 바뀌지 않으면 다시 그리지 않는다 — `shared/entitlements/RequireEntitlement.tsx:34-36` | 코드 확인 |
| 파생값 저장 금지 | — | 두 조회기를 렌더마다 부른다. **값이 오는 자리를 코드에 남겨 두기 위해서**이고, 서버가 붙으면 그 두 함수의 본문만 바뀐다 — `PlanPage.tsx:108-114` | 코드 확인 |
| 대용량 데이터 처리 | 공통 준수 | 결제내역 표에 **페이지네이션·가상화가 없다.** 지금은 목록이 늘 비어 있어 문제가 없지만, 서버가 붙어 청구가 수십 건 쌓이면 상한 규칙이 필요해진다 — 지금 코드에는 그 상한이 없다 | 코드 확인 |
| 동시 사용자 | 공통 준수 | 측정하지 않음 — 쓰기가 없어 경합 축이 성립하지 않는다 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 탭·카드·알림·배지·표·빈 상태가 전부 디자인 시스템 것이다. 손수 조립한 것이 **없다** — 이 자원은 값을 읽어 그리기만 한다 |
| **모름의 낱말을 고정한다** | — | `—` · `없음` · `미확인` 이 섞이면 운영자는 그중 어느 것이 '0'이고 어느 것이 '모름'인지 구분할 수 없다. **이 화면에서 그 구분이 전부이므로** 낱말을 `확인할 수 없음` 하나로 못박았다 — `subscription.ts:177-190` |
| 결과가 아니라 효과를 말한다 | — | 결제 탭은 상태 배지만 찍고 끝내지 않고 **그 상태가 지금 앱에 무엇을 하고 있는지**를 함께 말한다 — 운영자가 '왜 저장 버튼이 잠겼지'를 여기서 끝낼 수 있어야 그 연결이 산 것이다 — `components/BillingPanel.tsx:11-20` |
| 두 질문을 나눈다 | — | 배너는 '무엇을 해야 하는가'(사내 홈페이지에서 확인하세요), 카드는 '무엇이 잠겼는가'. 같은 문구를 두 번 말하는 것이 아니다 — `BillingPanel.tsx:54-58` |
| 배너의 자리 | — | 청구 경고와 변경 예고는 **특정 탭의 사실이 아니라 이 계정의 사실**이라 탭 위에 있다. 결제 탭에만 두면 이용현황을 보러 온 운영자는 자기 앱이 조회 전용이라는 것을 모른 채 저장 버튼이 잠긴 화면을 만난다 — `PlanPage.tsx:22-24` |
| 티저를 만들지 않는다 | — | 판매하지 않는 모듈은 **표에 행 자체가 없다.** 살 수 없는 것을 표에 올리면 그것이 곧 티저이고, 운영자는 결제 페이지를 뒤지다 아무것도 찾지 못한다 — `components/UsagePanel.tsx:7-8` |
| 낱말을 뭉치지 않는다 | — | 값의 형태 셋을 한 낱말('포함')로 뭉치면 '포함인데 왜 등록이 안 되죠?'(한도 소진)를 화면이 설명하지 못한다. 그래서 **'종류' 열**을 세운다 — `UsagePanel.tsx:10-17` |
| 앱이 데이터를 판결하지 않는다 | — | 한도를 넘긴 상태(`201/200`)에서 **어느 항목이 초과분인지 앱이 정하지 않는다** — 숫자로만 말한다. 정하면 그 순간 앱이 데이터를 판결한다 — `UsagePanel.tsx:73-78` |
| 색·아이콘만으로 말하지 않기 | 공통 준수 | 상태 배지의 문구가 의미를 싣고 색은 보조다 — `BillingPanel.tsx:47` · `components/InvoiceHistoryPanel.tsx:46` |
| 죽은 버튼을 만들지 않는다 | — | 결제내역이 0건일 때 빈 상태의 **복구 콜백을 넘기지 않는다** — 검색·필터가 없는 표라 누를 것이 없는 버튼을 그리면 그것이 곧 죽은 버튼이다 — `InvoiceHistoryPanel.tsx:157-159` |
| 가짜 링크를 만들지 않는다 | — | 영수증이 없는 청구가 정상적으로 있다(취소·환불). 그때 `—` 를 그리고 없는 링크를 만들지 않는다 — `InvoiceHistoryPanel.tsx:120-122` |
| 숫자 비교 | — | 금액 열은 오른쪽 정렬 + 고정폭 숫자다 — **자릿수를 눈으로 비교하는 유일한 열**이다. 이용현황의 사용량 칸도 고정폭이라 행마다 흔들리지 않는다 — `InvoiceHistoryPanel.tsx:60-66` · `UsagePanel.tsx:59-60` |
| 사용자 피드백 | 공통 준수 | **이 자원에는 결과 통지 토스트가 없다** — 읽기 전용이라 통지할 결과가 없다 |
| 조회 실패 표시 | 공통 준수(인라인 배너 + 명시적 다시 시도) | ⚠ **이 자원은 그 축을 갖고 있지 않다.** 조회 실패 표시도 `다시 시도` 컨트롤도 없다 — 지금은 실패 경로가 성립하지 않기 때문이고, 서버가 붙으면 '못 받았다'와 '조회가 실패했다'를 가르는 표시가 새로 필요해진다 |
| 링크의 목적 | 공통 준수 | 바깥으로 나가는 링크만 있다(사내 요금제 안내 · 영수증). 전부 새 창으로 연다 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽이라 세션이 없으면 렌더되지 않는다 |
| 권한 | 공통 준수 | 리소스 키는 잎 `/settings/plan` 하나다. **쓰기 액션이 존재하지 않으므로 읽기 권한만 의미가 있다** |
| **실패 방향이 권한과 정반대다** | — | 권한은 fail-**closed**(모르면 막는다), 엔타이틀먼트는 fail-**open**(모르면 연다). 저장값 없음·파손·키 없음·매핑 없음이 전부 열린 상태로 수렴한다 — **고객이 돈을 낸 기능이 우리 조회 실패로 멈추는 것은 어떤 과금 실수보다 나쁘다** — `shared/entitlements/plan.ts:19-28` |
| 그 반대를 복사하지 않는다 | — | 이 리포의 다른 이음매는 전부 fail-closed 관성이라, 그 모양을 그대로 복사하면 **'엔타이틀먼트 조회 실패 = 전 기능 정지'** 가 된다. 스토어가 그 함정을 명시적으로 피한다 — `entitlement-store.ts:11-16` |
| 판정 순서 | 공통 준수 | 인증 → **플랜** → 권한. 사지 않은 기능에 '권한이 없습니다'라고 말하면 운영자는 켜 줄 수 없는 관리자에게 권한을 요청하고 그것이 지원 티켓이 된다 — `plan.ts:10-17` |
| 프런트 게이팅의 지위 | 공통 준수 | **UX 층이지 보안이 아니다.** 위조된 브라우저 저장값으로 이 가드는 우회되며, 실제로 막는 것은 서버 응답이다 — `RequireEntitlement.tsx:14-18` |
| 저장값 방어 | — | 티어를 못 읽으면 기본 상태(전 기능 가용)로 돌아가고, 개별 키가 깨졌으면 **그 키만 없는 것으로** 두어 열린 상태로 읽히게 한다. 모르는 청구 상태는 **정상**으로 읽는다 — 오탈자 하나가 전 고객을 읽기 전용으로 만들지 않게 — `plan.ts:560-604` |
| 응답에 없는 키 | — | **티어 정의로 채우지 않는다.** 채우면 앱이 서버 카탈로그보다 앞선 순간 새 모듈이 전 고객에게서 잠긴다 — `plan.ts:593-599` |
| 토큰에 싣지 않는다 | — | ⚠ 엔타이틀먼트를 액세스 토큰 클레임에 싣지 않기로 했다 — 플랜 변경이 토큰 만료까지 반영되지 않기 때문이다 — `plan.ts:33` |
| 개인정보 마스킹 | 공통 준수 | **카드번호·계좌번호 전체가 이 어드민에 오지 않는다** — 올 이유가 없다. 오는 것은 사람이 자기 수단을 알아보는 최소 조각(`신한 ····1234`)뿐이고, 그 값도 **서버가 그렇게 만들어 보내는 것**이지 화면이 가리는 것이 아니다 — `subscription.ts:49-53` |
| 입력값 검증 | 공통 준수 | **해당 없음** — 입력이 없다. 주소의 탭 값은 목록에서 되찾아 좁히므로 목록 밖의 값이 조용히 버려진다 — `tabs.ts:40-42` |
| 쓰기 표면 | — | **없다.** 조회 엔드포인트 셋 다 조회 전용이고, 여기에 쓰기를 만들면 실제 계약과 어긋나는 두 번째 정본이 생긴다 — `subscription.ts:31-32` |
| HTTPS | 공통 준수 | 측정하지 않음 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | **구독·결제·계약은 사내 홈페이지 소관이다.** 이 앱은 값을 받아 반영만 하고, 바깥으로 나가는 링크 하나가 유일한 행동이다 — `plan.ts:500-508` |
| API 제약 | **백엔드가 없다.** 플랜은 브라우저 저장소 픽스처, 구독 결제 두 조회기는 상수 `null` 이다. 계약은 `plan.ts:30-34` 와 `subscription.ts:29-32` 의 `TODO(backend)` 가 갖는다. **세 엔드포인트 모두 조회 전용이다** |
| **결제·결제내역에 데이터가 없다** | ⚠ 두 조회기가 상수 `null` 을 돌려주므로 **다섯 칸이 채워지는 경로도, 표가 그려지는 경로도 지금 실행에서 볼 수 없다.** 화면은 그 사실을 감추지 않고 배너로 말한다 |
| **청구 상태의 효과가 배선되지 않았다** | ⚠ 판정 함수(`planReadOnly` · `usePlanWriteBlock`)는 있으나 **그것을 읽어 쓰기 컨트롤을 잠그는 호출부가 앱 코드에 0건이다.** 결제 탭의 '조회 전용이에요' 문장은 설계 의도이지 지금 관측되는 동작이 아니다 — `RequireEntitlement.tsx:82-92` |
| **한도 초과의 생성 차단이 배선되지 않았다** | ⚠ 같은 이유로 `useCreateBlock` · `planQuotaCreateBlock` 의 호출부도 0건이다. 이용현황 표는 `201/200` 을 정확히 보여 주지만 그 상태에서 등록이 실제로 막히지는 않는다 — `RequireEntitlement.tsx:66-77` |
| 사용량 조회기의 범위 | 지금 실제 건수를 아는 키는 **상품 관리 하나**다. 나머지 키는 조회기가 `null` 을 돌려주고 저장값의 사용량으로 되돌아간다(0 으로 떨어뜨리지 않는다) — `apps/admin/src/wiring.ts:225-227` |
| 결제내역의 정렬 | 표 설명이 '최근 건이 위에 와요'라고 말하므로 **정렬은 서버가 보장해야 한다** — 화면이 다시 정렬하지 않는다 |
| 결제내역의 규모 | 페이지네이션·가상화·필터·검색이 **없다.** 청구가 많이 쌓이는 계약에서는 상한 규칙이 새로 필요해진다 |
| 파일 용량 제한 | 해당 없음 — 파일 입력이 없다 |
| 업로드 가능 확장자 | 해당 없음 — 같은 이유 |
| 다국어 | 공통 준수 — 측정하지 않음. 금액만은 통화와 한 벌로 다뤄 해외 계약이 들어와도 화면이 틀린 기호를 붙이지 않는다 |
