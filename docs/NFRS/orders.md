# 주문 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 주문 |
| 화면 ID | `SCR-ORDERS` · `SCR-ORDERS-DETAIL` · `POP-ORDERS-DETAIL-CANCEL` |
| 메뉴 경로 | 주문 관리 > 주문 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../FSD/orders/index.md) · [상세](../FSD/orders/detail.md) · [주문 취소 팝업](../FSD/orders/pop-cancel.md) |

**이 자원의 성격** — 관리자가 **만들지도 지우지도 않는** 자원이다. 등록·삭제 축이 통째로 없고, 남는 것은 조회와 **되돌릴 수 없는 상태 전이**다. 그래서 이 자원의 비기능 요구는 두 곳에 몰린다: ① 일괄 처리의 부분 실패를 정직하게 다루는 일 ② 상태·재고·이력이 **갈라지지 않게** 하는 일. 아래 각 절은 [공통 기준](_common.md) 대비 **이 자원만의 차이**를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다. 기다리는 것은 픽스처 어댑터의 인위적 지연이다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수(신선도 30초 · 자동 재시도 없음 · 창 포커스 재조회 없음) | 목록은 **한 번만** 조회하고 상태 필터·검색은 받아 온 배열 위에서 좁힌다 — 필터를 눌러도 요청이 늘지 않는다. 상세는 별도 조회다 | 코드 확인 |
| 조회 건수 | — | 목록 1 · 상세 1. 쓰기 성공 직후 그 항목을 다시 조회한다(자기 변경은 즉시 보인다) | 코드 확인 |
| 대용량 데이터 처리 | 가상화 없이 1,000행을 그리지 않는다 | ⚠ **페이지네이션이 없다.** 이 목록은 받아 온 주문 전부를 한 표에 그린다 — 페이지 크기 상한도, 가상화도, CSV 내보내기도 없다. 지금 픽스처가 10건이라 드러나지 않을 뿐, 실제 주문량에서는 이 자원이 공통 상한(100행)의 바깥에 있다 | 코드 확인 — `apps/admin/src/pages/orders/OrderListPage.tsx` 에 페이지 상태가 없다 |
| 일괄 처리 건수 상한 | — | **상한이 없다.** 선택한 건수만큼 갱신 요청이 **건별로** 나간다(한 번에 묶어 보내는 엔드포인트가 없다). 30건을 고르면 요청 30건이다 | `OrderListPage.tsx:186-192` |
| 일괄 처리 동시성 | — | 전부 동시에 시작하고 **모두 정착할 때까지 기다린다**(하나가 실패해도 나머지는 계속된다) — `apps/admin/src/shared/bulk` 의 규약 | 코드 확인 |
| 목록 재조회 중 표시 | 공통 준수 | 스켈레톤은 **최초 로드에서만**. 재조회 중에는 이전 행을 유지하고 요약줄에 `· 새로고침 중…` 만 덧붙인다 — `OrderListPage.tsx:133,310` | 코드 확인 |
| 건수 집계 | — | 좌측 필터의 건수는 **필터 이전 전체 집합**에서 매 렌더 다시 센다(서버가 세지 않는다). 취소 건은 상태 항목에서 빠지고 자기 항목에서만 세어진다 | `apps/admin/src/pages/orders/types.ts:71-90` |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | ⚠ **목록 표를 화면이 직접 조립한다.** 공통 목록 껍데기 둘 중 어느 것도 이 화면의 모양을 표현하지 못하기 때문이다 — 읽기 전용 껍데기에는 선택 체크박스가 없고, 쓰기 껍데기의 체크박스는 **삭제 권한**에 묶여 있는데 주문은 지우지 않는다. 손으로 표를 그리는 것이 아니라 같은 프리미티브(DS Table · 선택 셀 · 순번 셀)를 쓴다 — `apps/admin/src/pages/orders/components/OrderTable.tsx:4-19` |
| 선택의 권한 축 | — | 체크박스는 `update` 에 묶인다(`remove` 가 아니다). 이 화면에서 선택이 하는 일은 상태 전이, 곧 수정이다 — 수정 권한이 없는 역할에게 체크박스를 남기면 **아무것도 일어나지 않는 UI** 를 고르게 된다 |
| 결과 통지 | 공통 준수 | 목록 1문구 · 상세 4문구. **실패는 토스트로 가지 않는다** — 확인 다이얼로그의 배너와 카드 배너가 받는다(다이얼로그가 닫히지 않아야 재클릭이 곧 재시도가 된다) |
| 죽은 컨트롤 금지 | — | 세 갈래로 지킨다: ① 갈 수 없는 단계는 **버튼이 아예 없다**(비활성이 아니다) ② 일괄 버튼은 **처리 가능 건수**를 라벨에 싣고 0이면 잠긴다 ③ 취소가 막히면 비활성 버튼 대신 **사유 문장**이 그 자리에 온다 — `OrderDetailPage.tsx:426-428` |
| 버튼과 저장이 같은 규칙을 읽는다 | — | 전이 가능 여부의 정본은 도메인 술어 하나이며 버튼·일괄 건수·저장이 모두 그것을 부른다. 규칙이 두 벌이면 '눌리는데 거부당하는 버튼' 또는 '눌리지 않는데 서버는 허용하는 동작' 이 생긴다 — `apps/admin/src/shared/domain/order.ts:272-277` |
| 되돌릴 수 있는 일에는 확인을 붙이지 않는다 | — | 상태 전이·입금 확인·취소는 확인을 거치고, **처리 메모 저장은 곧바로 저장한다**. 되돌릴 수 있는 일에까지 확인을 붙이면 정작 중요한 확인이 무시된다 — `OrderDetailPage.tsx:10-12` |
| 부분 실패 통지 | — | 일괄 처리의 실패는 **건수**로 말한다(`{N}건을 처리하지 못했어요.`). 성공한 건은 되돌리지 않는다 |
| 구조적 0 을 그리지 않기 | — | 결제를 쓸 수 없으면 빈 목록에 '오늘 0건' 이 아니라 **왜 주문이 들어올 수 없는지**와 지금 봐야 할 곳(상품 문의)을 함께 말한다 — `OrderListPage.tsx:216-232` |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 좌측 건수는 아직 못 셌으면 `—`(0 이 아니다). 상세에서는 입금 시각이 비면 `미확인`, 차감 시각이 비면 `아직 차감되지 않았어요`, 복원 시각이 비면 **그 줄을 그리지 않는다** |
| 접근성 — 라이브 영역 | 공통 준수 | 목록에 **항상 마운트된** polite 라이브 영역이 있어 필터·검색으로 0행이 되는 전환이 소리로 들린다 — `OrderListPage.tsx:283-291` |
| 접근성 — 색만으로 말하지 않기 | — | 취소 행에 위험 색조를 얹되 **뜻은 '취소' 배지가 전한다**. 부분배송도 색이 아니라 `부분배송 1/3` 이라는 글자가 말한다 |
| 접근성 — 행 클릭과 키보드 | — | 행 클릭은 마우스 보조 수단이고, 키보드 사용자는 **주문번호 링크**로 상세에 닿는다. 표 캡션이 권한에 따라 달라져 무엇을 할 수 있는지 먼저 읽힌다 |
| 입력 편의성 — IME | 공통 준수 | 검색은 조합 중 커밋하지 않고 Enter 제출도 막는다 |
| 미저장 보호 | 공통 준수 | 상세의 **처리 메모**에만 걸린다. ⚠ **주문 취소 팝업에는 미저장 가드가 없다** — 사유를 반쯤 적은 채 닫으면 확인 없이 사라진다(다른 입력 팝업 셋은 전부 그 가드를 쓴다) |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽이라 세션 없이 렌더되지 않는다 |
| 권한 | 공통 준수 | 리소스 키는 사이드바 잎 `/orders` 에서 파생되고 **상세도 같은 키**를 쓴다(가장 구체적인 잎이 그것이다). 의미 있는 액션은 `read` 와 `update` 둘뿐 — `create`·`remove` 컨트롤이 화면에 존재하지 않는다 |
| 권한 실패 방향 | 공통 준수(fail-closed) | 수정 권한이 없으면 체크박스·일괄 바·액션 버튼이 **사라진다**(비활성이 아니다) |
| 프런트 게이팅의 한계 | 공통 준수 | 실제 차단은 서버의 몫이다. 지금은 백엔드가 없어 **어느 층도 실제로 막지 않는다** |
| 개인정보 마스킹 | 공통 준수 | ⚠ 이 화면은 **가리지 않는다.** 목록의 `주문자`, 상세의 `주문자 이름·연락처·이메일·수령인 이름·연락처·배송지 전체` 가 평문으로 그려진다. 픽스처가 가상 인물이라 지금은 드러나지 않지만, 실데이터에서는 **서버가 마스킹 정책을 정해 보내야 한다** — 화면 쪽에 마스킹 함수가 없다 |
| 입력값 검증 | 공통 준수 | 사람이 타이핑하는 칸은 둘뿐이다(처리 메모 500자 · 취소 사유 필수·200자). 규칙의 정본은 zod 스키마이며 화면이 자기 조건문으로 다시 판단하지 않는다 — `apps/admin/src/pages/orders/validation.ts:34-38` |
| 상태 전이 권한 | — | 전이 가능 여부는 권한이 아니라 **도메인 규칙**이다. 권한이 있어도 규칙에 막히면 버튼이 없다 |
| 중복 제출 | 공통 준수 | 확인 다이얼로그의 확인 버튼이 `busy` 동안 잠긴다. **입금 확인·재고 차감·재고 복원은 도메인 층에서 각각 멱등**이라 두 번째 저장이 값을 덮지 않는다 — `shared/domain/order.ts:367,504,529` |
| HTTPS | 공통 준수 | 측정하지 않음 |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 셸 안쪽 경계가 덮는다 |
| 장애 대응 | 공통 준수 | 목록 조회 실패는 표 자리만 배너로 바뀌고 좌측 필터·검색은 남는다. 상세 조회 실패는 **없는 주문과 고장난 주문을 가른다** — 없는 주문에 '다시 시도' 를 권하지 않는다 |

| 금액을 두 곳에서 더하지 않는다 | — | 최종 결제금액은 저장하지 않고 품목·결제에서 **매번 계산한다**. 합계까지 저장하면 배송비를 정정한 순간 총액과 항목이 어긋난 주문서가 남고, 그때 어느 쪽이 사실인지 아무도 답할 수 없다 — `order.ts:159-168` |
| 스냅숏 | — | 품목은 상품 id 만 가리키지 않고 **주문 시점의 상품명·옵션·단가·적립률을 복사해** 든다. 참조만 들면 상품 가격을 고친 날 3개월 전 주문의 결제금액이 함께 바뀐다 — `order.ts:24-29` |
| 재고 부수효과의 원자성 | — | 주문 갱신과 재고 증감을 화면이 두 번 호출하지 않는다 — 어댑터가 한 덩이로 묶는다. 백엔드가 붙으면 이 덩이가 **트랜잭션 엔드포인트 한 방**이 된다 — `apps/admin/src/pages/orders/data-source.ts:7-9` |
| 배선되지 않았으면 못 박지 않는다 | — | 재고 적용기가 '적용할 곳을 모른다' 를 주면 **차감 시각을 찍지 않는다**. 찍으면 재고는 그대로인데 주문만 '차감 완료' 라고 말하는 상태가 영구히 남고 다시는 차감되지 않는다 — `data-source.ts:39-42` |

| 부분 실패 | — | 일괄 처리는 **성공한 건을 되돌리지 않는다**. 실패 건수만 알리고 다이얼로그를 열어 둔다 |
| 로그 기록 | 공통 준수 | **확인하지 못했다** — 상태 전이·취소를 관리자 로그에 남기는 배선을 찾지 못했다. 다만 주문 **자기 처리 이력**은 추가만 되는 원장으로 남고 상세에서 보인다 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 없다. 결제 승인·웹훅은 **서버의 몫**이며 이 화면은 결과만 읽는다 |
| API 제약 | **백엔드가 없다.** 연동 지점은 `apps/admin/src/pages/orders/data-source.ts:61-64` 한 곳이다: `GET /api/orders` · `GET/PUT /api/orders/:id`. **`POST` 는 열지 않는다** — 주문을 만드는 것은 관리자가 아니라 고객의 결제다. `DELETE` 도 없다(거래 기록이다). 서버는 주문 갱신 + SKU 재고 증감을 한 트랜잭션으로 처리하고 재고가 모자라면 422 로 거절해야 한다. 멱등키는 `stockAppliedAt`·`stockRestoredAt` |
| 일괄 처리 API | **묶음 엔드포인트가 없다** — 지금은 건별 `PUT` N회다. 서버가 붙으면 부분 실패 응답 형식(성공/실패 건과 사유)을 계약으로 정해야 한다 |
| 페이지 크기 상한 | **없다**(§1). 실데이터 전에 페이지네이션 또는 가상화가 필요하다 |
| 정렬·기간 필터 | 정렬은 주문일시 내림차순 고정이고 **기간 필터가 없다**. '지난달 주문' 을 좁힐 수단이 이 화면에 없다 |
| 내보내기 | **CSV 내보내기가 없다** — 이 자원에는 `export` 액션 컨트롤이 없다 |
| 재고 차감 시점 설정 | 값은 있으나 **그것을 바꾸는 화면이 없다**(§6) |
| 관리자 로그 | 상태 전이·취소가 관리자 로그에 남는 배선을 확인하지 못했다(§4) |
| 개인정보 | 주문자·수령인 정보가 **가려지지 않고** 그려진다(§3) |
| 파일 용량 제한 | 해당 없음 — 이 자원에 파일 입력이 없다 |
| 업로드 가능 확장자 | 해당 없음 — 같은 이유 |
