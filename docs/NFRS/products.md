# 상품 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 상품 |
| 화면 ID | `SCR-PRODUCTS` · `SCR-PRODUCTS-FORM` |
| 메뉴 경로 | 상품 관리 > 상품 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../FSD/products/index.md) · [등록·수정](../FSD/products/form.md) |

**이 자원의 성격** — 이 어드민이 **만들고 지우는** 몇 안 되는 자원이고, 폼이 여덟 구획으로 가장 크다. 그리고 **결제 연동 상태가 표시와 입력을 파생시킨다** — 상품마다 고르던 '가격 표시 방식' 축이 제거되고 사이트 전역 술어 하나가 그 자리를 대신한다. 아래 각 절은 [공통 기준](_common.md) 대비 이 자원만의 차이를 적는다.

## 1. 성능

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 공통 CRUD 껍데기 그대로. 폼은 **다중 섹션 + 미리보기 2단**이라 단일 카드 껍데기를 쓰지 않고 이 화면이 배치한다 |
| 폼의 길이를 다루는 방법 | — | 좌측 **구획 목차**가 여덟 구획으로 점프하고, 오류가 남은 구획에 붉은 점을 찍는다. 상태 표기는 `aria-pressed` 가 아니라 **`aria-current`** 다 — 선택이 아니라 위치이기 때문이다 |
| 방향을 적지 않는다 | — | 안내 문구가 '왼쪽·오른쪽' 을 말하지 않는다 — 좁은 화면에서는 목차가 폼 위로 접혀 올라간다 |
| 실시간 미리보기 | — | 폼은 **타이핑 중인 값**으로 고객 카드를 그린다(검증을 통과하기 전 값도 숫자만 건져내 그린다). 저장 경로는 그 관대함을 쓰지 않는다 |
| 파생 표시의 일관 | — | 목록·폼·미리보기가 **같은 술어 하나**를 부른다. 목록은 그 결과의 표시 문구를, 폼은 입력 잠금을 읽는다 — 둘이 갈리면 '금액은 입력되는데 표시는 문의' 이거나 그 반대가 생긴다 — `apps/admin/src/shared/commerce/price-display.ts:21-25` |
| 잠금 안내의 모양 | — | 잠긴 구획의 배너는 **한 벌의 같은 모습**이다(사유 + 결제 설정 링크 + 값 보존 약속). 화면마다 배너를 그리면 운영자가 화면마다 새로 읽어야 한다 |
| 배너 수 | — | 목록의 잠금 배너는 **한 장**이다 — 금액 대체와 재고 열 교체는 같은 사실의 두 결과라, 두 장을 세우면 운영자가 같은 이야기를 두 번 읽고 원인이 둘이라고 오해한다 |
| 구조적 0 을 그리지 않기 | — | 결제가 없으면 **재고 열을 감춘다**(0 을 그리면 품절로 읽힌다). 그 자리에 실제로 쌓이는 것(문의 건수)을 세운다 |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 좌측 건수는 못 셌으면 `—`, 문의 건수는 배선 전이면 `—`, 0건이면 `0건`(링크가 아니다) |
| 결과 통지 | 공통 준수 | 목록의 전시 토글 2문구 + 공통 삭제·저장 통지. 폼은 공통 통지만 쓴다 |
| 죽은 컨트롤 금지 | — | 등록 버튼은 `create` 가 있을 때만 **존재한다**. 중분류가 없는 대분류에서는 중분류 셀렉트가 잠기고 힌트가 그 이유를 말한다 |
| 되돌릴 수 있는 일에는 확인을 붙이지 않는다 | — | 전시 토글은 확인 없이 즉시 저장된다 |
| 접근성 — 검증 실패 | 공통 준수 | 첫 위반 필드로 포커스가 옮겨 가고, 좌측 목차의 해당 구획에 오류 표시가 붙는다 |
| 접근성 — 필수 표시 | 공통 준수 | 별표는 장식이고 필수 여부가 별도로 보조기술에 전달된다. **쿠폰 다중 선택은 묶음에 필수를 싣는다** — 필수인 것은 '어느 한 쿠폰' 이 아니라 '고르는 행위' 다 |
| 입력 편의성 — IME | 공통 준수 | 검색은 조합 중 커밋하지 않는다 |
| 미저장 보호 | 공통 준수 | 폼 전체에 이탈 가드. **미리보기의 결제 설정 링크는 앵커**라 가드가 가로챈다(프로그램 이동이면 그냥 지나친다) |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 없다. 고객이 보는 상품 페이지는 이 리포 밖이다 |
| API 제약 | **백엔드가 없다.** 연동 지점은 `apps/admin/src/pages/products/items/data-source.ts:19,29` 다: `GET/POST /api/products` · `GET/PUT/DELETE /api/products/:id` · `GET /api/products/categories` |
| 파일 업로드 | ⚠ **업로드 엔드포인트가 없다.** 이미지 필드는 파일을 서버로 보내지 않고 브라우저 안 미리보기만 만든다 — **폼을 떠나면 썸네일이 깨진다.** 검증을 조여도 해결되지 않으며(조이면 저장 자체가 불가능해진다) 고칠 곳은 업로드 이음매다 — `apps/admin/src/shared/crud/validation.ts:41-62` |
| 파일 용량 제한 | 이미지 **최대 5MB**(공통 기본값) |
| 업로드 가능 확장자 | 이미지 파일만. 화면 표기는 `PNG · JPG · GIF · 최대 5MB` |
| 이미지 수 | 대표 1장(필수) + 상세 최대 10장 |
| 옵션 | 그룹 **최대 3개**. 각 그룹의 값 개수에는 상한이 없다 |
| 재고 상한 | 0~999,999 |
| 판매가 상한 | 1억 원 |
| 상세설명 | 평문 기준 2,000자 |
| 검색 태그 | 최대 20개 |
| 가격 표시 축 | ⚠ **상품마다 고르던 라디오가 제거됐다.** 표시는 사이트 전역 연동 상태의 파생값이며 저장 필드가 없다. 이 판단은 `docs/adr/0014` 와 어긋나고, ADR 개정은 문서 소유자의 몫으로 남아 있다 — `price-display.ts:13-18` |
| 상품코드 중복 | **막지 않는다** — SKU 조합 안의 중복만 막는다 |
| 페이지 크기 상한 | **없다**(§1) |
| 정렬·내보내기 | 정렬 변경도 CSV 내보내기도 없다 |
