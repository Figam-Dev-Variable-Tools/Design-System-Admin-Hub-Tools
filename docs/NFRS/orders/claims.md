# 취소/교환/반품 (클레임) — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 취소/교환/반품(클레임) |
| 화면 ID | `SCR-ORDERS-CLAIMS` · `SCR-ORDERS-CLAIMS-DETAIL` |
| 메뉴 경로 | 주문 관리 > 취소/교환/반품 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../../FSD/orders/claims/index.md) · [상세](../../FSD/orders/claims/detail.md) |

**이 자원의 성격** — **축이 둘인 처리 워크플로**다(처리 상태 · 환불). 목록은 완전한 읽기 전용이고, 쓰기는 상세 한 화면에 몰려 있다. 그 쓰기가 건드리는 것은 클레임만이 아니다 — **재고**와 **적립금 원장**이 함께 움직이며 둘 다 되돌릴 수 없다. 그래서 이 자원의 비기능 요구는 ① 멱등성 ② 다중 위반을 하나도 버리지 않는 실패 표면 ③ 다른 원장과의 경계에 몰린다. 아래 각 절은 [공통 기준](../_common.md) 대비 이 자원만의 차이를 적는다.

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 목록은 **읽기 전용 공통 껍데기**를 그대로 쓴다(선택·삭제·액션 열이 어떤 역할에게도 없다). 상세는 카드 조립이며 환불 카드·교환 옵션·재고 이력 표는 이 화면 전용 부품이다 |
| 두 축을 섞지 않는다 | — | 클레임 처리와 환불은 **별개 카드**다. 한 카드에 섞으면 '완료' 버튼이 무엇을 완료하는지 화면에서 읽히지 않는다 |
| 죽은 컨트롤 금지 | — | 상태 선택지는 **지금 갈 수 있는 곳 + 현재 상태**만 연다. 갈 수 없는 상태를 열어 두고 저장에서 막으면 운영자는 이미 마음을 정한 뒤에 거절당한다 — `ClaimDetailPage.tsx:232-237`. 환불 버튼도 가드 사유가 있으면 잠기고 **그 사유가 버튼 옆에 뜬다** |
| 눌리는데 거부당하지 않는다 | — | 환불 버튼은 **저장이 실제로 보낼 값**으로 판정한다. 상태 select 만 '완료' 로 바꿔 둔 화면에서 버튼이 열리면 어댑터가 저장된 상태를 보고 거절한다 — 대신 `저장하지 않은 처리 내용이 있어요. 처리 저장을 먼저 한 뒤 환불을 진행하세요.` 로 **무엇을 먼저 해야 하는지** 말한다 — `refund.ts:220-238` |
| 되돌릴 수 있는 일에는 확인을 붙이지 않는다 | — | 진행·반려·메모 수정은 곧바로 저장한다. 확인을 세우는 것은 **재고를 실제로 움직이는 저장**과 **환불 완료** 둘뿐이다 |
| 실패의 표면은 셋이다 | — | 문구 · 자리 · **포커스**. 배너가 떠도 포커스가 방금 누른 버튼에 남아 있으면 스크린리더 사용자는 저장이 실패한 사실 자체를 모른 채 서 있게 된다 — `ClaimDetailPage.tsx:15-18,266-287` |
| 위반을 하나도 버리지 않는다 | — | 한 응답에 위반이 여럿이면 **전부** 배치한다(인라인 자리가 있는 것은 그 입력으로, 나머지는 전부 배너로). 예전에는 첫 위반만 읽어 나머지가 조용히 사라졌고, 하나를 고쳐 다시 저장하면 또 실패하는 경험이 됐다 — `types.ts:480-511` |
| 침묵하는 실패를 만들지 않는다 | — | 필드를 하나도 지목하지 않은 거절도 반드시 배너에 뜬다. 예전에는 취소·반품에서 전이·환불·재고 위반이 나면 **배너도 토스트도 인라인도 없이 버튼만 조용히 다시 활성화됐다**(인라인 자리가 교환 클레임에만 있기 때문) — `types.ts:432-441` |
| 결과 통지 | 공통 준수 | 성공 4문구(처리 저장 2 · 환불 접수 · 환불 완료). 실패는 배너·인라인이 받는다 |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 원 주문을 못 풀면 `주문 정보를 확인할 수 없어요.` 라고 **그대로 말하고**, 취소 유형이면 처리 자체를 막는다(fail-closed). 반품배송비 정책을 모르면 **0 으로 채우지 않고** 빈 칸을 두고 이유를 밝힌다 |
| 없는 일과 안 한 일을 가른다 | — | 목록의 환불 열에서 **교환은 `해당 없음`** 이다. '환불 없음' 을 달면 '아직 환불하지 않았다' 로 읽혀 할 일처럼 보인다 — `refund.ts:61-66` |
| 접근성 — 인라인 오류 | 공통 준수 | 교환 옵션 셀렉트가 `aria-invalid` 와 그 이유를 함께 내보낸다 |
| 접근성 — 재고 미리보기 | — | 옵션을 고르면 재고 이동 예고가 **polite 라이브 영역**으로 읽힌다 — `components/ExchangeOptionField.tsx:118` |
| 접근성 — 모달 안 실패 | — | 확인 다이얼로그가 떠 있는 동안의 실패는 **포커스를 옮기지 않는다** — 그 실패는 다이얼로그 안 배너가 즉시 읽어 주고, 모달 밖으로 포커스를 끌어내면 트랩이 깨진다 |
| 미저장 보호 | 공통 준수 | 상세의 편집(상태·메모·교환 옵션·차감·쿠폰 복원) 중 하나라도 남으면 이탈 가드가 뜬다 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽 |
| 권한 | 공통 준수 | 리소스 키는 사이드바 잎 `/orders/claims` 이고 상세도 같은 키다. 의미 있는 액션은 `read` 와 `update` — 등록·삭제 컨트롤이 없다. **클레임의 '반려' 는 권한 액션이 아니라 상태 전이**이므로 `update` 하나로 열린다 |
| 권한 실패 방향 | 공통 준수 | 수정 권한이 없으면 입력이 잠기고 저장·환불 버튼이 **사라진다** |
| 개인정보 마스킹 | 공통 준수 | 신청자 이름은 픽스처에 **이미 가려진 형태**(`김**`)로 들어 있다. 다만 **가리는 주체가 화면인지 서버인지는 확인하지 못했다** — 화면 쪽에 마스킹 함수가 없다. 백엔드 연동 시 서버가 가려 보내야 한다 |
| 입력값 검증 — 화면 | — | 반품배송비는 **원 단위 정수만** 받는다. 소수점 환불은 실무에 없고, `Number('3,000')` 이 `NaN` 이 되어 **조용히 0원 차감으로 흐르는 길**을 애초에 막는다 — `refund.ts:166-173` |
| 입력값 검증 — 서버 | 공통 준수 | 화면의 입력 상한·선택지·숫자 파서는 **브라우저 입력만** 막는다. 프로그램적으로 채워진 값은 그대로 나가고 그것을 거르는 것은 서버의 일이다 — 지금은 픽스처가 그 자리에서 400(형식)과 422(규칙)를 낸다 — `data-source.ts:306-317` |
| 400 과 422 를 가른다 | — | 형식 위반이 규칙 위반보다 **먼저** 판정된다. 값이 형식조차 맞지 않으면 전이 판정이 무엇을 근거로 했는지 말할 수 없다 — `data-source.ts:454-458` |
| 오류 노출 | 공통 준수 | 서버 원문·스택·상태 코드를 문장으로 노출하지 않고 **짧은 참조 코드**만 남긴다 |
| 중복 제출 | 공통 준수 | 확인 다이얼로그의 버튼이 진행 중 잠긴다. 더 중요한 것은 **도메인 멱등성**이다(§4) |
| HTTPS | 공통 준수 | 측정하지 않음 |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 셸 안쪽 경계 |
| 장애 대응 | 공통 준수 | 상세 조회 실패는 **없는 클레임과 고장난 클레임을 가른다**. 옵션 조회 실패는 그 카드 안에서만 알리고 다른 카드를 가리지 않는다 |
| 재시도 | 공통 준수 | 목록·상세·옵션 각각 `다시 시도` 를 갖는다(없는 대상에는 붙이지 않는다) |
| 멱등성 — 재고 | — | 재고 반영은 **`stockAppliedAt` 이 멱등 키**다. 이미 반영된 클레임을 다시 저장해도 재고가 두 번 움직이지 않는다 — `data-source.ts:395` |
| 멱등성 — 환불 | — | 환불 완료는 **`refund.completedAt` 이 멱등 키**다. 적립금 원장은 **추가만 되는 장부**라 두 번 적힌 지급은 되돌릴 수도 없다 — 가장 고치기 어려운 종류의 사고다 — `refund.ts:255-259` · `data-source.ts:427` |
| 배선되지 않았으면 못 박지 않는다 | — | 재고 적용기가 배선되지 않았으면 **멱등 키를 찍지 않는다**. 원장이 배선되지 않았으면 **환불 완료를 거절한다**(500) — 여기서 통과시키면 키가 찍히고, 적립금은 돌아오지 않은 채 화면만 '복원 완료' 라고 말하는 상태가 영구히 남는다 — `data-source.ts:391-392,438-443` |
| 부수효과의 원자성 | — | 클레임 갱신·재고 증감·원장 기입을 화면이 따로 호출하지 않는다 — 어댑터가 한 덩이로 묶는다. 백엔드가 붙으면 **트랜잭션 엔드포인트 한 방**이 된다 — `data-source.ts:7-9` |
| 두 원장이 같은 수량을 두 번 움직이지 않는다 | — | **취소 클레임은 재고를 움직이지 않는다.** 복원의 주체는 주문 취소다(그쪽에 자기 멱등 키가 있다). 클레임이 여기서 또 되돌리면 같은 수량이 두 번 들어오고, 두 원장 중 어느 쪽이 거짓인지 나중에는 아무도 가리지 못한다 — `types.ts:336-343` |
| 스냅숏 | — | 결제액·적립금·쿠폰은 **접수 시점의 스냅숏**이다. 주문을 참조만 하면 나중에 주문서가 정정될 때 지난 환불의 근거 금액이 함께 바뀐다 — 이미 돈이 오간 거래가 사후에 다른 금액이 되는 것이다 — `refund.ts:77-83` |
| 완료된 환불은 사후에 고칠 수 없다 | — | 차감·쿠폰 복원 여부·결제액을 바꾸려 하면 422 로 막는다: `환불이 완료되어 차감 내역을 바꿀 수 없어요.` — 이미 나간 돈의 근거를 사후에 고치는 일이다 — `data-source.ts:353-365` |
| 금액을 두 곳에서 빼지 않는다 | — | 실제 환불액은 저장하지 않고 매번 계산한다. 차감을 정정한 순간 총액과 항목이 어긋난 환불서가 남고 어느 쪽이 사실인지 답할 수 없게 된다 — `refund.ts:128-134` |
| fail-closed | — | 원 주문을 모르면 취소 처리를 **막는다**. 모르는 채로 취소를 완료하면 이미 나간 물건을 '출고 전' 으로 처리해 재고와 정산이 함께 어긋난다 |
| 접수 사실을 지우지 않는다 | — | 철회(withdrawn)가 상태로 남는 이유다. 지우면 고객이 접수했다는 사실 자체가 사라지고, 같은 건이 다시 들어왔을 때 재접수인지 최초인지 아무도 모른다 — `types.ts:74-79` |
| 로그 기록 | 공통 준수 | **확인하지 못했다** — 관리자 로그 배선을 찾지 못했다. 재고 이동은 클레임 자신의 이력 표에 남는다 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 없다. 실제 환불 송금은 이 앱 밖의 일이며, 이 화면은 **'환불 완료로 기록' 하고 적립금·쿠폰을 되돌릴 뿐**이다 |
| API 제약 | **백엔드가 없다.** 연동 지점은 `apps/admin/src/pages/orders/claims/data-source.ts:467-471`(`GET /api/claims` · `GET/PUT /api/claims/:id`)과 `:484-485`(`GET /api/products/:id/variants`)다. **`POST` 는 열지 않는다** — 클레임을 만드는 것은 관리자가 아니라 고객의 접수다. `DELETE` 도 없다. 서버는 클레임 갱신 + SKU 재고 증감 + 원장 append 를 한 트랜잭션으로 처리하고 재고가 모자라면 422 로 거절해야 한다 |
| 쿠폰 복원 | ⚠ **아직 발급을 만들지 않는다.** 지금은 복원 여부만 클레임에 기록하고, 재발급은 서버가 `POST /api/members/:id/coupons` 로 처리한다. 복원은 **발급 1건을 새로 만드는 일**이지 사용 표시를 지우는 것이 아니다 — '언제 썼다' 는 사실은 남아야 한다 — `data-source.ts:445-447` |
| 적립금 원장 | 복원은 **양수 1건 추가**다(수정이 아니다). 원장이 배선되지 않으면 환불 완료가 거절된다 |
| 부분 환불 | **없다.** 환불액은 결제액에서 차감을 뺀 한 값이고, 품목 일부만 환불하는 축이 모델에 없다 |
| 클레임 접수 | 관리자가 접수할 수 없다(고객 채널이 만든다). 그래서 등록 화면이 없다 |
| 일괄 처리 | 없다(§1) |
| 페이지 크기 상한 | **없다**(§1) |
| 기간 필터·정렬·내보내기 | 전부 없다 |
| 발급 기준 등급의 한계 | 해당 없음 — 이 자원의 축이 아니다 |
| 파일 용량 제한 | 해당 없음 — 이 자원에 파일 입력이 없다 |
