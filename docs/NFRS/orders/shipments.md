# 배송 처리 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 배송 처리 |
| 화면 ID | `SCR-ORDERS-SHIPMENTS` · `POP-ORDERS-SHIPMENTS-INVOICE-BULK` |
| 메뉴 경로 | 주문 관리 > 배송 처리 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../../FSD/orders/shipments/index.md) · [송장 일괄 입력 팝업](../../FSD/orders/shipments/pop-invoice-bulk.md) |

**이 자원의 성격** — **두 원장이 만나는 화면**이다: 주문(행)과 배송 건(그 행에 딸린 송장). 그래서 조회도 둘, 쓰기도 둘(배송 건 생성·갱신 + 주문 갱신)이고, 한 번의 일괄 처리가 두 원장을 함께 바꾼다. 비기능 요구는 그 **경계의 정합**에 몰린다. 아래 각 절은 [공통 기준](../_common.md) 대비 이 자원만의 차이를 적는다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | 측정하지 않음 — 백엔드가 없다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수 | **조회가 둘이다**(주문 목록 · 배송 건 목록). 둘 다 도착해야 행이 성립하므로 하나만 실패해도 화면 전체가 실패로 간다 — `apps/admin/src/pages/orders/shipments/ShipmentListPage.tsx:145-153`. 필터·검색은 받아 온 배열 위에서 좁힌다(요청이 늘지 않는다) | 코드 확인 |
| 행 합성 비용 | — | 행마다 **SKU 단위 커버리지**를 계산한다(품목 × 배송 건). 순수 함수이고 메모이제이션되지만 계산량은 주문 수 × 품목 수 × 배송 건 수에 비례한다 | `apps/admin/src/shared/domain/shipment.ts:299-361` |
| 대용량 데이터 처리 | 가상화 없이 1,000행을 그리지 않는다 | ⚠ **페이지네이션도 가상화도 없다.** 취소·입금전 주문이 빠질 뿐 나머지 전부를 한 표에 그린다 | 코드 확인 |
| 일괄 처리 건수 상한 | — | **상한이 없다.** 세 일괄 모두 선택 건수만큼 요청이 나간다. 특히 **발송처리는 행 하나가 요청 여럿**을 만든다(그 행의 발송 가능한 배송 건마다 갱신 1 + 주문 갱신 1) — `ShipmentListPage.tsx:305-345` |
| 송장 등록 요청 수 | — | 줄 하나당 배송 건 생성 1 + (커버리지가 완성되면) 주문 갱신 1. 10줄이면 최대 20요청이다 |
| 재조회 중 표시 | 공통 준수 | 스켈레톤은 최초 로드에서만. 두 조회 중 하나라도 처음이면 스켈레톤이다 |
| 건수 집계 | — | 좌측 4갈래 건수는 **필터 이전 전체 행**에서 매 렌더 다시 센다. 아직 못 셌으면 `null` 이고 화면은 `—` 를 그린다 |
| 동시 사용자 | 공통 준수 | 측정하지 않음 | — |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | 주문 목록과 **같은 이유로** 표를 직접 조립한다(선택은 있으나 삭제가 없다). 골격은 여전히 디자인 시스템 표다 |
| 행의 단위 | — | 행은 **배송 건이 아니라 주문**이다. 배송 건을 행으로 삼으면 송장이 아직 없는 주문이 목록에서 아예 사라진다 — 정작 할 일이 가장 많은 주문이 — `types.ts:8-11` |
| 죽은 컨트롤 금지 | — | 세 일괄 버튼 모두 **처리 가능 건수**를 라벨에 싣고 0이면 잠긴다. 30건을 골라 눌렀는데 28건이 조용히 거절당하는 일을 만들지 않는다 |
| 어휘의 일관 | — | 배송 건의 상태는 주문 상태의 배송 구간과 **같은 낱말**을 쓴다(배송대기·배송중·배송완료). 같은 사건에 두 어휘가 생기면 운영자는 둘이 같은 것인지 확인할 방법이 없다 — `shipment.ts:166-171` |
| 부분 발송의 표기 | — | 부분 발송은 상태가 아니라 **배지**다(`부분발송 1/3`). 작업 상태는 `발송대기` 로 남는다 — 이 목록의 건수는 곧 오늘 처리할 양이어야 하고, 반쯤 끝난 일을 완료 쪽에 놓으면 그 숫자가 거짓말이 된다 |
| 결과 통지 | 공통 준수 | 성공 3문구. 실패는 확인 다이얼로그·팝업의 배너가 받는다(닫히지 않아야 재클릭이 재시도가 된다) |
| 저장 전 전량 검증 | — | 송장 팝업은 **한 줄이라도 틀리면 아무것도 저장하지 않는다**. 절반만 들어가면 어느 주문에 송장이 붙었는지 다시 세어 봐야 하고, 그 확인은 팝업을 다시 여는 것으로만 된다 — `components/InvoiceBulkDialog.tsx:8-9` |
| 치는 도중 붉게 물들이지 않기 | — | 팝업의 줄 오류는 **제출을 눌렀을 때만** 채워진다 — `InvoiceBulkDialog.tsx:132-133` |
| 값이 없는 것과 모르는 것 | 공통 규약 5 준수 | 택배사 목록이 **`null`(모른다)** 이면 warning 배너를 띄우고 입력을 잠근다. 빈 목록('없다')과 섞지 않는다 — 빈 목록은 '먼저 등록하라' 는 완결된 지시가 되어 이미 있는 것을 다시 만들게 한다. 표의 택배사·송장 열도 배송 건이 없으면 `—` 다 |
| 조용히 고치지 않기 | — | 송장번호의 **가운데 공백은 오류이지 정리 대상이 아니다**. 앞뒤 공백만 뗀다 — `shipment.ts:257-260` |
| 접근성 — 라이브 영역 | 공통 준수 | 항상 마운트된 polite 영역이 결과 건수를 알린다 |
| 접근성 — 인라인 오류 | 공통 준수 | 팝업의 송장번호 입력은 `aria-invalid` 를 **언제나 그 이유와 함께** 내보낸다(`aria-describedby` 가 오류 문장을 가리킨다) — `InvoiceBulkDialog.tsx:286-288` |
| 접근성 — 초기 포커스 | — | 팝업이 열리면 **첫 송장번호 칸**에 포커스가 간다 |
| 새 창 링크 | — | 송장번호는 추적 URL 을 만들 수 있을 때만 링크가 되고, **`noreferrer noopener`** 로 새 창에 연다. 링크 이름에 `(새 창)` 이 붙는다 — `components/ShipmentTable.tsx:81-91` |
| 미저장 보호 | 공통 준수 | 팝업은 네 경로(Esc·딤·×·취소)를 한 요청으로 모아 `DLG-COMMON-LEAVE-MODAL` 을 세운다. 확인 다이얼로그는 **모달 밖에** 렌더된다 — 안에 두면 포커스 트랩이 그것을 가둔다 |

## 3. 보안

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 인증 | 공통 준수 | 셸 안쪽 |
| 권한 | 공통 준수 | 리소스 키는 사이드바 잎 `/orders/shipments` 다. 의미 있는 액션은 `read` 와 `update` 뿐 — **배송 건 생성도 `update` 에 묶여 있다**(배송 처리는 주문의 수정이라는 판단) |
| 권한 실패 방향 | 공통 준수 | 수정 권한이 없으면 체크박스와 세 일괄 버튼이 **사라진다** |
| 개인정보 마스킹 | 공통 준수 | ⚠ **수령인 이름이 평문**이다. 배송지·연락처는 이 표에 없지만(주문 상세에 있다) 수령인은 검색 대상이자 표시 값이다. 실데이터에서는 서버가 정책을 정해야 한다 |
| 입력값 검증 | 공통 준수 | 송장번호는 **숫자와 하이픈만, 양 끝은 숫자, 30자 이하**로 좁힌다. 국내 송장번호는 전부 숫자이고 운영자는 엑셀에서 복사해 붙여 넣는데, 한글·공백·따옴표가 섞이면 택배사 접수 파일이 그 행에서 깨진다 — 깨진 것을 아는 시점은 이미 물건이 나간 뒤다 — `shipment.ts:243-249` |
| 중복 방지 | — | **같은 택배사 + 같은 송장번호**를 두 번 붙일 수 없다. 택배사를 함께 보는 이유는 번호 체계가 택배사마다 따로라 다른 택배사에서 같은 숫자가 나오는 것이 정상이기 때문이다. 대조군은 **저장된 배송 건 전체 + 이 팝업 안의 다른 줄** 둘 다다 |
| 검증의 층 | 공통 준수 | 화면 검증은 UX 다. 서버가 같은 값을 다시 검증해야 하며, 특히 **중복 판정은 서버가 최종 권한**을 가져야 한다(팝업이 보는 것은 조회 시점의 목록이다) |
| 중복 제출 | 공통 준수 | 다이얼로그·팝업의 실행 버튼이 진행 중 잠긴다 |
| HTTPS | 공통 준수 | 측정하지 않음 |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 셸 안쪽 경계가 덮는다 |
| 장애 대응 | 공통 준수 | 두 조회 중 하나만 실패해도 표 전체가 배너로 바뀐다(합쳐야 행이 성립한다). `다시 시도` 는 **둘 다** 다시 부른다 |
| 재시도 | 공통 준수 | 명시적 `다시 시도` 컨트롤이 있다. 쓰기 실패는 다이얼로그·팝업이 닫히지 않아 재클릭이 곧 재시도다 |
| 취소는 실패가 아니다 | 공통 준수 | 취소·Esc·딤은 진행 중 요청을 중단하고 배너·토스트를 만들지 않는다 |
| 규칙이 두 벌이 되지 않는다 | — | 주문을 미는 판정은 언제나 **주문 축을 먼저 묻고**(취소·미입금·역방향) 그 위에 배송 축의 조건 하나(전 품목이 덮였는가)만 얹는다. 규칙이 갈라지면 주문 목록의 버튼과 이 화면의 버튼이 같은 주문을 두고 다른 답을 한다 — `shipment.ts:15-18,407-412` |
| 커버리지 계산의 안전 | — | 품목 수량을 **넘겨 배정하지 않는다**. 한 품목에 과다 입력된 수량이 다른 품목까지 '발송됨' 으로 만들면 실제로는 창고에 남아 있는 물건이 화면에서 사라진다 — `shipment.ts:305-311` |
| 품목이 없는 주문 | — | 품목 0건인 주문은 '전 품목 발송' 이라고 말할 대상이 없다 — **완료로 통과시키지 않는다** — `shipment.ts:354-360` |
| 출고 수량은 계산의 결과다 | — | 주문 품목의 출고 수량을 화면이 더해 넣지 않는다. 목록의 배지와 이 화면의 판정이 다른 셈에서 나오면 같은 주문을 두 화면이 다르게 설명하는 순간이 반드시 온다 — `shipment.ts:388-392` |
| 시각과 상태가 갈라지지 않는다 | — | 배송 건의 상태를 옮길 때 **발송 시각이 함께 찍힌다**. 둘이 갈라지면 '배송중인데 언제 나갔는지 모르는' 건이 생긴다. 이미 찍힌 시각은 덮지 않는다 — `shipment.ts:475-494` |
| 삭제된 택배사 | — | 배송 건은 택배사 **id** 를 가리키므로 이름이 바뀌어도 과거 기록이 끊기지 않는다. 그럼에도 못 찾으면 표는 `알 수 없는 택배사` 라고 말한다(빈칸으로 두지 않는다) |
| 부분 실패 | — | 세 일괄 모두 성공한 건을 되돌리지 않고 실패 건수만 알린다 |
| 로그 기록 | 공통 준수 | 주문 상태가 움직인 건은 그 주문의 처리 이력에 남는다. 배송 건 자체의 이력 화면은 없고, 관리자 로그 배선은 **확인하지 못했다** |

| 코드 확장성 | — | 배송 건 상태를 더하면 순서 배열 한 곳만 고친다(전이 판정이 인덱스로 이뤄진다). 작업 상태 4갈래는 이 화면만의 축이라 화면 쪽 표가 갖는다 |

## 8. 제약사항

| 항목 | 내용 |
|---|---|
| 외부 시스템 연계 | 택배사 **추적 사이트**로 나가는 링크가 유일하다. 접수 API 연동은 없다 — 송장번호는 사람이 입력한다 |
| API 제약 | **백엔드가 없다.** 연동 지점은 배송 건 어댑터(`apps/admin/src/pages/orders/shipments/data-source.ts`)와 주문 어댑터(`apps/admin/src/pages/orders/data-source.ts:61-64`) 둘이다. 서버가 붙으면 '배송 건 생성 + 주문 갱신' 을 한 트랜잭션으로 묶어야 한다 |
| 일괄 처리 API | 묶음 엔드포인트가 없다 — 건별 요청 N회다(§1) |
| 엑셀 업로드 | **없다.** 이 회차는 화면 안 표 입력까지만 한다 — 파일 규격·인코딩·매핑 실패라는 세 겹의 예외를 백엔드 없이 흉내 내면 '되는 것처럼 보이지만 아무 데도 닿지 않는' 화면이 된다 |
| 부분 송장 | 한 주문에 송장을 나눠 붙일 수 없다(§7) |
| 페이지 크기 상한 | **없다**(§1) |
| 기간 필터·정렬 | 없다(§7) |
| 내보내기 | CSV 내보내기가 없다 |
| 개인정보 | 수령인 이름이 가려지지 않는다(§3) |
| 파일 용량 제한 | 해당 없음 — 파일 입력이 없다 |
| 업로드 가능 확장자 | 해당 없음 — 같은 이유 |
