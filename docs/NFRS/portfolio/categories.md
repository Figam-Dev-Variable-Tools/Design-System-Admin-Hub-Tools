# 카테고리 — 비기능 명세서

| 항목 | 내용 |
|---|---|
| 자원명 | 카테고리(포트폴리오 분류) |
| 화면 ID | `SCR-PORTFOLIO-CATEGORIES` · `POP-PORTFOLIO-CATEGORIES-CATEGORY-FORM` |
| 메뉴 경로 | 비즈니스 > 포트폴리오 관리 > 카테고리 |
| 작성자 | Admin Hub 문서화 |
| 작성일 | 2026-07-22 |
| 버전 | 1.0 |
| 관련 기능 명세서 | [목록](../../FSD/portfolio/categories/index.md) · [추가·수정 팝업](../../FSD/portfolio/categories/pop-category-form.md) |

**이 자원의 성격** — 포트폴리오 묶음에서 **유일하게 공용 목록 껍데기를 쓰지 않는 화면**이다. 표가 아니라 카드 안의 줄 목록이고, 등록·수정은 라우트가 아니라 팝업이다. 이 자원의 중심 규칙은 목록이 아니라 **'사용 중이면 못 지운다'** 다.

## 1. 성능

| 항목 | 요구 | 이 자원의 값 | 측정 방법 |
|---|---|---|---|
| 응답 시간 | 공통 준수 | **측정하지 않음** — 백엔드가 없다 | 백엔드 연동 후 재작성 |
| 조회 성능 | 공통 준수(자동 재시도 없음) | 요청 1건. **사용 건수를 별도로 세지 않는다** — 목록 항목이 '분류 + 사용 중 항목 수'로 온다 — `apps/admin/src/pages/portfolio/categories/data-source.ts:3-4` | 코드 확인 |
| 동시 사용자 | 공통 준수 | **측정하지 않음** | — |
| 대용량 데이터 처리 | 가상화 없이 대량 행을 그리지 않는다 | 페이지네이션도 가상화도 없다. **다만 분류는 십수 개 규모의 자원**이라 실질 위험이 이 앱에서 가장 낮다 | 코드 확인 |
| 재조회 중 화면 | 데이터가 눈앞에서 사라지지 않는다 | **스켈레톤은 최초 로드에만 뜬다.** 예전에는 무효화가 걸릴 때마다 채워져 있던 줄이 스켈레톤으로 지워져 목록을 훑던 운영자 밑에서 데이터가 사라졌다 — `apps/admin/src/pages/portfolio/categories/PortfolioCategoriesPage.tsx:180-189` | 코드 확인 |

## 2. 사용성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| UI 일관성 | 공통 준수 | ⚠ **공용 목록 껍데기를 쓰지 않는다** — 항목이 이름·건수 둘뿐이라 열이 둘인 표가 표의 값을 못 한다. 대신 저수준 조회·삭제 훅 위에 카드+줄 목록을 직접 조립했다 — `PortfolioCategoriesPage.tsx:1-5` |
| 접근성 | 공통 준수 | 아이콘 버튼에 이름을 준다(`{이름} 수정`). **잠긴 삭제 버튼의 이름이 사유를 말한다** — 잠긴 버튼은 눌러도 아무 일이 없으므로 이름이 곧 설명이어야 한다 — `PortfolioCategoriesPage.tsx:154-156` |
| ⚠ 접근성 — 조사 | 조립한 문장이 한국어로 읽힌다 | 잠긴 삭제 버튼의 이름이 `{N}개 사용 중라 삭제할 수 없어요` 로 조립된다 — **`사용 중이라` 가 되지 않는다.** 조사를 리터럴로 붙였다 — `PortfolioCategoriesPage.tsx:155` |
| 막힌 곳에서 나갈 길 | 금지만 알리고 끝내지 않는다 | 사용 건수 배지가 **그 분류로 걸러진 포트폴리오 목록**으로 가는 링크다. 삭제를 막는 근거와 그것을 해소할 자리가 같은 줄에 있다 — `PortfolioCategoriesPage.tsx:126-138`. **0건은 링크로 만들지 않는다**(열어 봐야 빈 목록이다) |
| 규칙을 미리 알리기 | 막히고 나서 이유를 찾게 하지 않는다 | 카드 바닥에 규칙과 다음 행동이 **상주한다** — `PortfolioCategoriesPage.tsx:274-277` |
| 왕복을 줄이기 | — | 입력이 이름 한 칸이라 **팝업으로 처리한다.** 대신 입력이 있는 채로 닫으려 하면 확인을 세운다 — ESC·딤·×·취소 **네 경로를 한 번에** 덮는다 — `apps/admin/src/pages/portfolio/categories/components/PortfolioCategoryFormModal.tsx:62-66` |
| 사용자 피드백 | 공통 준수 | 이 자원의 토스트는 **호출 지점 2곳**([인벤토리](../../FSD/inventory.md) §9.3 과 일치)이며 문구는 셋이다(추가·저장·삭제 성공). 전부 **여는 화면이** 띄운다 — 팝업은 성공과 동시에 사라지므로 그 안에서 띄운 통지는 함께 사라진다 — `PortfolioCategoriesPage.tsx:217,229` |
| 입력 편의성 | — | 팝업이 열리면 **커서가 입력 칸에 가 있다** — `PortfolioCategoryFormModal.tsx:111`. Enter 로 제출된다 |

## 4. 안정성

| 항목 | 요구 | 이 자원의 값 |
|---|---|---|
| 예외 처리 | 공통 준수 | 조회 실패는 배너, 삭제 실패는 **다이얼로그 안**, 저장 실패는 **팝업 안**에서 알린다 — 실패한 자리에서 알린다 |
| 실패 시 닫지 않기 | 무엇이 실패했는지 잃지 않는다 | 삭제 실패 시 확인 다이얼로그가 닫히지 않고, 저장 실패 시 팝업이 닫히지 않는다 — `PortfolioCategoriesPage.tsx:221` · `PortfolioCategoryFormModal.tsx:79-82` |
| 취소된 요청 | 취소를 실패로 알리지 않는다 | 다이얼로그를 닫으면 **진행 중 삭제 요청을 취소하고**, 취소로 끝난 것은 오류로 표시하지 않는다 — `PortfolioCategoriesPage.tsx:196-202,220`. 팝업이 사라질 때도 진행 중 요청을 취소한다 — `PortfolioCategoryFormModal.tsx:71` |
| `null` 과 `[]` 의 구분 | 모름과 없음을 섞지 않는다 | **구분된다** — 조회 실패는 배너로, 정말 0건인 것은 카드 안 안내로 다르게 보인다. [포트폴리오](items.md) 의 분류 선택지가 두 경우를 구분하지 못하는 것과 대비된다 |
| 재시도 | 자동 재시도 여부 | **자동 재시도 없음**(공통). 사용자 재시도는 배너의 `다시 시도` |
| ⚠ 덮어쓰기 방지 | — | **충돌 처리가 없다** — 다른 관리자가 같은 분류의 이름을 먼저 고쳤어도 그대로 덮어쓴다. 공용 폼 컨트롤러를 쓰지 않아 공통 충돌 다이얼로그가 붙지 않았다 |
| 로그 기록 | — | **확인하지 못했다** |
