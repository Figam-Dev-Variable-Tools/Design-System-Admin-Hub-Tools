# BE-079. 플랜·이용 현황 백엔드 기능 명세

## 1. 개요

| 항목 | 내용 |
|---|---|
| 대상 화면 | FS-079 플랜·이용 현황 (`/settings/plan`) — 단일 라우트, 하위 라우트 없음 |
| 범위 | **테넌트 엔타이틀먼트 문서 1건의 조회.** 그리고 그 값이 앱 안에서 만드는 **판정 계약**(3상태 · 모듈↔리소스 매핑 · fail-open 방향 · 저장값 방어)을 서버가 어떻게 강제해야 하는가. **웹훅 `entitlements.updated` 의 수신 계약**(심 `plan.ts:31`) |

## 2. 공통 (상속)

- **에러 봉투**: BE-003 §2. 공통 에러코드 동일. **다만 이 축은 새 코드 하나를 요구한다** — 플랜 계열의 거절은 권한(403)이 아니라 **402 `PAYMENT_REQUIRED`** 계열이어야 한다(§7.3).
- **권한**: `admin`, `operator` **모두 조회 가능**. 플랜 현황은 '이 계정이 무엇을 샀는가'이며 운영자가 잠금을 만났을 때 스스로 확인해야 하는 값이다 — 숨기면 그 운영자는 관리자에게 물어야 하고, 그것이 ADR-0013 이 없애려던 지원 티켓이다. **쓰기 권한 개념이 없다**(엔드포인트가 없다). 리소스는 `page:/settings/plan`.
- **CSRF**: 해당 없음 — **쓰기 엔드포인트가 없다**(§7.1).
- **타임아웃**: 조회 5초 → 504. **⚠ 타임아웃도 fail-open 이어야 한다** — 실패가 전 기능 정지가 되면 안 된다(§7.2 · §7.9 #2).
- **엔타이틀먼트**: **이 라우트 자신은 게이팅 대상이 아니다.** `MODULE_RESOURCES`(`module-resources.ts:29-91`)에 `/settings/**` 가 없어 언제나 `granted` 다(`route-entitlement.ts:27-28`, 회귀 `route-entitlement.test.ts:56-57`). **잠금을 설명하는 화면이 잠기면 설명을 볼 수 없다** — 이것은 우연이 아니라 규칙이다(`module-resources.ts:15-17`).
- **프론트 게이팅은 보안 경계가 아니다**: 세 파일이 같은 말을 한다(`plan.ts:34` · `RequireEntitlement.tsx:14-17` · `entitlement-store.ts:16`) — '**위조된 localStorage 로 이 가드는 우회되며, 그때 실제로 막는 것은 API 응답의 402/403 이다. 둘은 대체재가 아니라 층이다.**' §7.3 이 그 층을 계약으로 만든다.

### 6.1 어댑터 본문에 요구되는 사항 (시그니처 불변)

1. **`loadState()` 를 `GET /api/tenant/entitlements` 로 교체**한다. 반환 타입 `PlanState` 는 그대로이며 **반드시 `normalizePlanState` 를 통과시킨다** — 서버 응답도 저장값과 같은 방어를 받는다(§7.7).
2. **동기 → 비동기 전환이 화면 계약을 바꾼다.** `usePlan()` 이 지금 동기라 이 화면에 **로딩·조회 실패·재시도 세 상태가 없다**(FS-079-EL-025 ②). 그 셋을 신설하되 **판정은 그동안에도 `granted` 여야 한다** — '실패 배너를 띄우면서도 기능은 열어 두는' 조합이며 이 앱에 선례가 없다(§7.9 #2).
3. **`saveState` 는 로컬 캐시로 남긴다** — 서버로 쓰지 않는다. 오프라인·조회 실패 시 마지막으로 알던 값을 쓰는 자리다.
4. **`receivePlan` 을 조회 응답의 착지점으로 배선**한다 — 이미 `normalizePlanState` 를 통과시킨다(`entitlement-store.ts:112`).
5. **DEV 패널 두 액션(`devSetTier`·`devSetBillingState`)은 그대로 둔다** — 서버가 붙어도 개발 재현 수단은 필요하다. 다만 **운영 빌드 배제 조건(`import.meta.env.DEV`)이 유지되는지 배포 검증에 포함한다**(§7.1).
6. **쿼터 사용량 조회기(`registerEntitlementUsageLookup`)를 서버 값으로 대체할지 결정한다** — §7.5.

## 7. 핵심 판정

### 7.1 이 앱에서 플랜은 **읽기 전용**이다 — DEV 패널은 운영 빌드에 존재하지 않는다 【보안 판정】

**코드로 가른 사실.**

| 통로 | 성격 | 운영 빌드에 존재하는가 | 근거 |
|---|---|---|---|
| `receivePlan(next)` | 사내 어드민·웹훅이 준 값을 받는 자리 | **코드는 있으나 호출부 0건** | `entitlement-store.ts:84-88,111-113` |
| `devSetTier(tier)` · `devSetBillingState(state)` | **개발용 재현 스위치** | **아니다 — 코드째 제거된다** | `PlanPage.tsx:232-234` `{import.meta.env.DEV && <PlanDevPanel />}` · `PlanDevPanel.tsx:8-10` |
| 화면의 플랜 변경 UI | — | **없다** | `PlanPage.tsx:1-9,179-190`(`이 화면에서는 바꿀 수 없습니다.`) · 나가는 길은 `PLAN_PORTAL_URL` 하나(`plan.ts:507`) |

**판정 1 — `PlanDevPanel` 은 실제 저장이며, 그래서 운영 빌드 배제가 보안 요구다.** 이 패널은 흉내가 아니다: `devSetTier` 는 `planStateForTier(tier)` 로 스토어를 갈아끼우고 `mutate` 가 그것을 **localStorage 에 영속시킨다**(`entitlement-store.ts:100-106,115-123`). 즉 **클라이언트가 자기 플랜을 엔터프라이즈로 올릴 수 있다.** 그것이 허용되는 이유는 단 하나 — **vite 가 운영 빌드에서 `import.meta.env.DEV` 분기를 상수 `false` 로 접어 패널 코드까지 통째로 제거하기 때문**이다(`PlanPage.tsx:232-233`).

**따라서 계약으로 못박는다**: 이 조건부가 사라지거나 번들러 설정이 바뀌어 패널이 운영 번들에 남으면 **결제 없이 상위 모듈이 열리는 UI 가 제품에 실린다.** 배포 검증이 그것을 확인해야 한다(§7.9 #3). ⚠ **DEV 빌드에서는 조회 전용 역할도 플랜을 바꿀 수 있다**(FS-079 §7 #13) — 개발·스테이징을 실제 운영자에게 보여 주는 순간 그것이 '플랜 변경 UI'로 보인다.

**판정 2 — 그러나 이 모든 것이 보안의 본체는 아니다.** 패널을 지워도, `import.meta.env.DEV` 를 완벽히 접어도, **누구든 `localStorage['tds-admin.plan']` 을 손으로 고쳐 엔터프라이즈가 될 수 있다.** 세 파일이 같은 문장을 적어 둔 이유가 그것이다(§2). **프론트 게이팅은 UX 이고, 실제로 막는 것은 §7.3 이다.**

### 7.2 이 축은 **fail-open** 이다 — 모든 미지가 `granted` 로 수렴한다 【정합 판정】

`plan.ts:19-28` 이 이것을 **'이 파일에서 가장 중요한 문장'** 이라고 선언한다:

> 권한은 fail-**closed**: 모르면 막는다. 엔타이틀먼트는 fail-**open**: 모르면 연다(가용성 우선). … **고객이 돈을 낸 기능이 우리 조회 실패로 멈추는 것은 어떤 과금 실수보다 나쁘다.**

**수렴 경로를 전수로 확인했다.**

| 미지의 경우 | 결과 | 근거 |
|---|---|---|
| 저장값 없음 / JSON 파손 / 저장소 접근 불가 | `DEFAULT_PLAN_STATE`(**전 기능 가용 · `tier: 'enterprise'`**) | `entitlement-store.ts:43,45-48` · `plan.ts:316-332` |
| `version` 불일치 · `tier` 를 못 읽음 | `DEFAULT_PLAN_STATE` | `plan.ts:578-579` |
| 개별 엔타이틀먼트의 형태가 깨짐 | **그 키만 없는 것으로** → `granted` | `plan.ts:551-555` |
| 응답에 키가 없음 | `granted` | `plan.ts:383,592-598` |
| 카탈로그에 없는 키가 꺼져 있음 | `granted` | `plan.ts:387-388` |
| **리소스에 모듈 매핑이 없음** | `granted` | `route-entitlement.ts:27-28` |
| **경로를 감싸는 잎이 없음** | `granted` | `route-entitlement.ts:38-42`(회귀 `route-entitlement.test.ts:95-98` — `/전혀-모르는-경로`) |
| 모르는 청구 상태 | `'active'`(배너 없음) | `plan.ts:563-566` |

`route-entitlement.ts:8-12` 가 그 이유를 한 줄로 적는다 — **'이 축은 모든 미지의 경우가 granted 로 수렴한다 — 판정 실패가 기능 정지가 되면 안 된다.'**

**판정 — 서버도 이 방향을 지킨다.**

1. **EP-01 이 실패해도(5xx·504·429) 프론트 게이팅은 열려 있어야 한다.** 조회 실패를 잠금으로 읽는 구현을 **금지한다.**
2. **미설정 테넌트에 404 를 주지 않는다** — 기본 상태를 200 으로 준다(§5). 404 를 주면 프론트가 그것을 어떻게 읽든 '없음'이 상태가 되고, 그 상태의 안전한 해석은 이 축에서 **granted** 뿐이다.
3. **부분 응답을 허용한다** — `entitlements` 에 일부 키만 실려도 유효한 응답이다(§4 EP-01).

**⚠ 그러나 이 관대함은 UX 층에만 적용된다.** 서버의 도메인 API 는 **fail-open 이 아니다** — §7.3 이 그 선을 긋는다.

**세 축의 방향 — quality-bar EXC-22 의 근거**

| 축 | 방향 | 미지의 경우 | 정본 |
|---|---|---|---|
| 권한(RBAC) | **fail-closed** | 모르는 리소스·파손된 저장값·조회 실패 → **막는다** | `shared/permissions/**` |
| **엔타이틀먼트(플랜)** | **fail-open** | 위 표 전부 → **연다** | 이 문서 · `plan.ts:19-28` |
| 결제 설정(축 A) | **fail-closed** | `usePg` 를 켜 두고 MID 가 공란 → **판매 불가로 수렴** | **BE-078 §7.1** |

**한 화면이 두 계열을 동시에 말하지 않는다**(EXC-21) — 판정 순서가 **인증 → 플랜 → 권한 → 설정**이고(`plan.ts:10-17`) `RequireEntitlement` 가 `RequirePermission` **바깥**에 중첩된다(`AppShell.tsx:407-411`). 뒤집으면 **사지 않은 기능에 '권한이 없습니다'** 라고 말하게 되고, 운영자는 켜 줄 수 없는 관리자에게 권한을 요청해 지원 티켓이 된다.

### 7.3 엔타이틀먼트도 **서버가 정본이다** — 그러나 응답 코드는 권한이 아니다 【보안 판정】

**BE-044 §2 가 권한 축에 세운 문장('프론트 권한 게이팅은 보안 경계가 아니다')을 이 축에 그대로 적용한다.** 근거가 코드에 세 벌 있다(`plan.ts:34` · `RequireEntitlement.tsx:14-17` · `entitlement-store.ts:16`):

> **위조된 localStorage 로 이 가드는 우회되며, 그때 실제로 막는 것은 API 응답의 402/403 이다. 둘은 대체재가 아니라 층이다.**

**판정 1 — 서버가 모든 도메인 API 에서 엔타이틀먼트를 독립적으로 재판정한다.** `RequireEntitlement` 가 막는 것은 **화면 렌더**뿐이다. `GET /api/quotes` · `POST /api/products` 를 직접 호출하면 그 가드를 한 줄도 거치지 않는다. 서버는 **§3 의 매핑표와 판정 규칙을 자기 쪽에서 구현**하고, 요청이 속한 모듈이 이 테넌트에 없으면 거절한다.

**판정 2 — 그 거절은 권한 계열이 아니라 플랜 계열이다.** quality-bar EXC-21 이 요구하는 4계열 분리를 이 계약이 지킨다:

| 계열 | 응답 | 이 앱의 화면 | 왜 섞지 않는가 |
|---|---|---|---|
| ① 권한 없음 | **403 `FORBIDDEN`** | `ForbiddenScreen`('접근 권한이 없습니다') | 관리자가 켜 줄 수 있다 |
| ② **플랜(엔타이틀먼트) 없음** | **402 `PAYMENT_REQUIRED`**(또는 그와 구분되는 전용 코드) + `{ upgradeTo, reason }` | `UpgradeScreen`('<티어> 플랜에서 사용할 수 있습니다') | **관리자도 켤 수 없다.** 결제하는 사람이 다르다 |
| ③ 인증 없음 | 401 | `/login` | — |
| ④ 도메인 설정 off | 200 + 화면 안 문맥 전환 | PG 스위치(BE-078) | 설정 화면에서 스스로 켤 수 있다 |

`ErrorScreens.tsx:91-98` 이 그 분리의 근거를 적는다 — **'「권한 요청」과 「결제 필요」는 다른 사람이 다른 행동을 해야 한다.'** 그래서 잠금 화면에는 **참조 코드도 없다**(`:100`) — 신고할 고장이 아니라 계약 상태이기 때문이다.

**판정 3 — 응답에 `upgradeTo` 를 함께 싣는다.** 프론트가 사유 문구를 **각자 짓지 않는 것**이 이 축의 규율이다(`plan.ts:100-107` — 상태·사유·`upgradeTo` 를 한 벌로 낸다). 서버의 거절도 같은 모양이어야 화면이 `UpgradeScreen` 에 그대로 꽂을 수 있다.

**판정 4 — `absent` 는 응답에서 `locked` 와 구분돼야 한다.** 살 수 없는 모듈(`minTier === null`)의 거절에 업그레이드 안내를 붙이면 **운영자는 결제 페이지를 뒤지다 아무것도 찾지 못한다**(`plan.ts:148-150`). 프론트는 그때 화면 대신 `/dashboard` 로 리다이렉트한다(`RequireEntitlement.tsx:102-111`).

**판정 5 — 엔타이틀먼트를 액세스 토큰 클레임에 싣지 않는다**(심 `plan.ts:33`). 실으면 다운그레이드가 **토큰 만료까지 반영되지 않는다** — 결제가 끊긴 계정이 최대 토큰 수명만큼 상위 기능을 계속 쓴다.

### 7.4 fail-open 의 대가 — **매핑 누락이 곧 무료 개방이다** 【경계 판정】

`route-entitlement.test.ts:1-5` 가 이 위험을 이름으로 적는다:

> 매핑이 틀리면 게이팅이 **조용히 무력해진다** — 오타 하나면 그 화면은 영원히 granted 이고(**fail-open 이라 아무도 눈치채지 못한다**), 두 모듈이 같은 화면을 주장하면 판정이 목록 순서에 좌우된다. **둘 다 화면에는 아무 증상이 없어서 테스트 말고는 잡을 방법이 없다.**

**이것이 §7.2 의 뒷면이다.** fail-closed 축에서 매핑을 빠뜨리면 **화면이 막혀서** 즉시 발각된다. fail-open 축에서는 **화면이 열려서** 아무도 모른다 — 새 화면을 추가하고 `MODULE_RESOURCES` 에 넣는 것을 잊으면 **상위 플랜 기능이 무료 플랜에서 조용히 열린다.** 발각되는 시점은 대개 청구 대사(帳合)이고, 그때는 이미 여러 달이 지나 있다.

**지금 이 앱이 가진 방어는 단위 테스트 3건뿐이다**(`route-entitlement.test.ts:21-59`):

1. **모든 매핑 대상이 실재하는 권한 리소스인가**(`:26-34`) — 오타를 잡는다.
2. **한 화면을 두 모듈이 주장하지 않는가**(`:36-51`) — 판정이 목록 순서에 좌우되지 않게 한다.
3. **역인덱스가 매핑된 화면만 답하는가**(`:53-58`).

**이 셋이 잡지 못하는 것이 정확히 문제다** — 셋 다 '적힌 매핑이 옳은가'만 묻는다. **'적혀야 하는데 안 적힌 화면'은 아무도 묻지 않는다.** 실제로 `MODULE_RESOURCES` 를 통째로 비워도 위 테스트 중 둘은 통과한다(그래서 `:50` 이 `seen.size > 0` 앵커를 덧붙였다 — 그러나 그것도 '하나라도 있으면' 통과한다).

**판정 — 서버가 이 구멍을 막는다.**

1. **매핑의 정본을 서버가 갖는다.** 프론트의 `MODULE_RESOURCES` 는 **사본이며 UX 용**이다(§7.3). 서버가 모듈↔엔드포인트 매핑을 독립적으로 갖고 거절한다면, 프론트 매핑이 빠져도 **화면은 열리지만 데이터가 오지 않는다** — 조용한 무료 개방이 조용한 실패로 바뀐다(그것도 나쁘지만 발각된다).
2. **배포 검증으로 부재를 잡는다.** '새로 추가된 nav 잎 중 `MODULE_RESOURCES` 에도, 명시적 제외 목록에도 없는 것'을 세는 검사를 CI 에 둔다. **지금은 그 검사가 없고, 없다는 사실이 코드 주석 어디에도 적혀 있지 않다.** 명시적 제외 목록이 필요한 이유는 `module-resources.ts:15-17` 이 말한 대로 **대부분의 화면이 정당하게 매핑 밖**이기 때문이다 — 그 정당함을 **암묵이 아니라 목록으로** 만들어야 검사가 성립한다.
3. **서버가 자기 매핑의 커버리지를 청구 대사와 대조한다** — 어느 테넌트가 어느 모듈의 API 를 실제로 호출했는가와 그 테넌트가 산 것을 주기적으로 맞춘다.

**한 가지를 덧붙인다 — 이 판정이 §7.2 를 뒤집지는 않는다.** 매핑 누락의 대가가 크다고 해서 fail-closed 로 바꾸면, **오타 하나가 산 기능을 정지시킨다.** 두 실패의 비용이 다르다는 것이 ADR-0013 의 결정이고, 이 문서는 그 결정을 **서버 강제로 보완**할 뿐 방향을 되돌리지 않는다.

### 7.5 사용량은 파생값이다 — 그러나 그 숫자의 정본은 서버다 【정합 판정】

**코드로 확인한 것.**

- 화면이 그리는 `'상품 200/200'` 은 **매 렌더 계산되는 파생 문자열**이다 — `moduleDetail`(`PlanPage.tsx:100-112`) → `planQuotaStatus`(`entitlement-store.ts:138-140`) → `quotaStatusOf`(`plan.ts:431-456`)가 `text` 를 조립한다. **어디에도 저장하지 않는다.**
- `exhausted`(`plan.ts:450`)도 파생이다. `quotaCreateBlock`(`:465-473`)의 사유 문구도 파생이다.
- **저장되는 것은 `Entitlement` 의 두 숫자뿐**이다 — `{ kind:'quota', limit, usage }`(`plan.ts:92`). `limit` 은 계약값, `usage` 는 **서버가 아는 사용량**이다.
- **실제 건수는 주입된 조회기가 안다**(`entitlement-store.ts:51-77`). 배선 지점은 `src/wiring.ts:178-180` 하나이고 거기서 `key === 'commerce.products' ? listProducts().length : null` 을 꽂는다. 소비는 `quotaStatusOf` 의 `const used = usage ?? value.usage`(`plan.ts:442`) — **주입된 값이 저장값을 이긴다.**
- **모르는 키에는 `0` 이 아니라 `null` 을 준다**(`entitlement-store.ts:60-63`) — `0` 은 '아무것도 안 썼다'로 읽혀 **한도가 찬 계정의 등록을 열어 버린다.**

**판정**:

1. **서버는 `usage` 를 응답에 싣되, 그것이 표시의 유일한 근거가 아님을 안다.** 프론트에 조회기가 배선돼 있으면 로컬 실제 건수가 이긴다 — **픽스처 시절의 이음매**이며(`entitlement-store.ts:51-63`) 서버 연동 시 **둘 중 하나를 정본으로 정해야 한다.** 지금 두 값이 갈리면 화면이 서버와 다른 숫자를 말한다(§7.9 #4).
2. **한도 판정(`exhausted`)의 정본은 서버다.** 프론트의 `>=` 비교는 UX 이고, 실제 생성 차단은 **`POST` 시점에 서버가 자기 카운트로** 판정한다 — §7.3 과 같은 층 구조다. 그때의 거절도 **플랜 계열(402)** 이지 권한(403)이 아니다.
3. **한도 초과 상태(`201/200`)는 정상이다**(`plan.ts:424-427,458-463`). 다운그레이드 직후 실제로 존재하며, 서버는 ① **초과분을 삭제하지 않고** ② **읽기·내보내기를 열어 두며** ③ **어느 항목이 초과분인지 정하지 않는다.** 남는 조치는 '더 만들지 못하게'뿐이고 그것은 권한 액션 5종 중 `create` 하나로 이미 표현된다 — 새 개념을 만들지 않는다.
4. **청구 미납(`past_due`·`suspended`)도 같은 방향이다** — 기능을 `locked` 로 만들지 않고 **쓰기만 잠근다**(`plan.ts:396-404`). 서버의 거절 범위도 쓰기에 한정한다. **조회를 막으면 운영자는 무엇을 결제해야 하는지조차 확인할 수 없다.**

**미결 — 확인하지 못한 것**: 쿼터 종 모듈이 `commerce.products` 하나뿐이라 **실질 배선도 한 건**이다(FS-079-EL-016). 다른 도메인이 쿼터를 갖게 될 때 조회기가 어떻게 늘어나는지는 코드에 없다. **`level` 종(`stats.advanced`)의 수치 표현도 사용량이 아니라 등급이라 이 축이 아니다.**

### 7.6 화면 안 잠금이 배선되지 않았다 — 계약이 약속을 앞서 있다 【경계 판정】

**코드로 확인한 사실**: `useCreateBlock` · `usePlanWriteBlock` · `useEntitlement` · `useQuota`(`RequireEntitlement.tsx:61-92`) 네 훅의 **소비처가 리포 전체에 0건**이다(FS-079 §7 #6 의 전수 grep).

그 결과 두 가지가 **사실이 아니다**:

1. **쿼터 소진이 등록을 실제로 막지 않는다.** 판정은 완성돼 있고(`quotaCreateBlock` — `plan.ts:465-473`) 회귀도 있는데(`plan.test.ts:124-166`) **그것을 읽는 화면이 없다.**
2. **미납·정지가 쓰기를 실제로 잠그지 않는다.** 이 화면의 배너(FS-079-EL-001)가 '**지금은 조회만 가능합니다**'라고 **말하는데 앱은 그렇게 동작하지 않는다.**

**판정 — 이 부재가 서버 계약을 바꾸지는 않는다. 오히려 그 반대다.** 프론트가 집행하지 않으므로 **지금 이 두 규칙을 집행하는 유일한 층은 서버다.** §7.3·§7.5 의 요구가 여기서 더 무거워진다:

- **한도가 찬 테넌트의 `POST` 를 서버가 막지 않으면 아무도 막지 않는다.**
- **미납 테넌트의 쓰기를 서버가 막지 않으면 아무도 막지 않는다.** 그리고 이 화면은 **막힌다고 말하고 있다** — 화면이 사실이 아닌 것을 말하는 상태다.

**따라서 서버는 프론트 배선을 기다리지 않는다.** 프론트가 그 훅을 소비하게 되면 **같은 판정을 두 층이 하는 정상 상태**가 되고, 그때 프론트 쪽은 UX(미리 알려 주기)이며 서버 쪽이 강제다.

### 7.7 저장값 방어가 **넓히는 쪽**으로 수렴한다 — 서버 응답도 같은 문을 통과한다 【정합 판정】

`normalizePlanState`(`plan.ts:575-603`)는 **권한 축의 `normalizeMatrix` 와 정확히 반대 방향**이다(`:571-573`). §3 의 표가 그 전수이며, 요점은 **어느 갈래로 실패해도 기능이 늘어나는 쪽으로 수렴한다**는 것이다.

**그중 하나는 특히 계약 문장이 필요하다** — **응답에 없던 키를 티어 정의로 채우지 않는다**(`plan.ts:592-598`):

> 채우면 **앱이 서버 카탈로그보다 앞선 순간** 새 모듈이 전 고객에게서 잠긴다(minTier 가 그 계정의 티어보다 높게 잡히므로). 없는 채로 두면 `resolveEntitlement` 가 `undefined` 를 주고 그것은 granted 로 읽힌다.

**판정**:

1. **서버 응답도 `normalizePlanState` 를 통과시킨다**(§6.1 #1). 신뢰하는 출처라도 형태 방어를 건너뛰지 않는다 — 그래야 서버 스키마가 흔들려도 앱이 잠기지 않는다.
2. **서버는 자기가 모르는 키를 응답에서 빼도 된다.** 그것이 정상이며 프론트는 그 화면을 연다. **BE-079 가 이것을 명시하지 않으면 새 모듈이 출시일에 전 고객에게서 사라진다** — FS-079 §5 각주가 이 문서에 남긴 요구다.
3. **`version` 은 형태 버전이지 플랜 버전이 아니다**(`plan.ts:269-270`). 서버가 형태를 바꾸면 올리고, 낮은 값은 **기본 상태(전 기능 가용)로 되돌린다** — 마이그레이션 실패가 정지가 되지 않게.

### 7.8 크로스탭 전파는 브라우저의 일이고 웹훅은 서버의 일이다

`subscribeToOtherTabs`(`entitlement-store.ts:154-163`)가 `storage` 이벤트에서 키 `'tds-admin.plan'` 만 골라 스토어를 다시 읽는다. AppShell 이 마운트 시 한 번 건다(`AppShell.tsx:226` · `RequireEntitlement.tsx:47-49`). **`storage` 이벤트는 그 이벤트를 낸 탭 이외에서만 발화하므로 저장→재로드 루프가 생기지 않는다**(`:148-149`).

**판정**: 이 메커니즘은 **연동 후에도 남는다** — 다만 트리거가 바뀐다. 지금은 DEV 패널의 저장이 트리거이고, 서버가 붙으면 **조회 응답의 착지(`receivePlan`)가 `saveState` 를 거쳐 같은 이벤트를 낸다.** 근거(`:151-152`): '사내 어드민이 플랜을 내리면 **이 앱의 모든 탭이 같은 순간에 같은 화면을 보여야 한다.**'

**서버 쪽 요구는 하나다** — **다운그레이드가 즉시 반영되도록 응답에 강한 캐시를 걸지 않는다**(`no-store` — §4). 그리고 **토큰 클레임에 싣지 않는다**(§7.3 판정 5). 웹훅 수신·서명 검증·재전송 방어는 전부 서버의 몫이며 **프론트가 흉내 내지 않는다**(심 `entitlement-store.ts:15`).
